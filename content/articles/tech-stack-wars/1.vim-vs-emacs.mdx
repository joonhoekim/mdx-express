# Vim vs Emacs — 에디터 전쟁의 시작

*"Vim 유저가 Emacs 유저를 만나면 뭐가 되냐? 둘 다 VS Code 쓰게 됨"*

---

인류 역사상 가장 오래된 개발자 전쟁.
1970년대에 시작되어 반세기가 넘도록 이어진 이 전쟁은,
결국 양쪽 모두가 예상하지 못한 제3자의 승리로 (일단은) 막을 내림.

하지만 아직도 터미널에서 Vim을 쓰는 사람이 "나 Vim 씀"이라고 말할 때의 그 은근한 우월감,
그리고 Emacs 유저가 "Emacs는 에디터가 아니라 운영체제"라고 말할 때의 그 자부심.
이건 영원히 사라지지 않을 것임.

<Callout type="info" title="사전 지식">
이 글을 읽기 전에 알아야 할 것: Vim에서 나가는 방법.
모른다고? 걱정 마. Stack Overflow에서 가장 많이 본 질문 중 하나가 "How to exit Vim"임.
2017년 기준으로 100만 뷰를 넘겼음. ㄹㅇ 실화.
</Callout>

## 전쟁의 기원: 1970년대

### ed → ex → vi → Vim

모든 건 Ken Thompson이 1969년에 만든 `ed`에서 시작됨.
`ed`는 라인 에디터였음. 화면에 텍스트를 보여주지도 않았음.
지금 기준으로 보면 "이걸 어떻게 쓰냐"인데, 당시엔 터미널이 텔레타이프라서 화면 자체가 없었음.

```
$ ed
a
Hello, World!
.
w hello.txt
13
q
```

이게 진짜 에디터였음. 화면 출력 없이 명령어만으로 텍스트를 편집했음.
"어디를 편집하고 있는지 모르는 에디터" — 이게 시작이었음.

<Callout type="note" title="ed 일화">
ed의 에러 메시지는 딱 하나였음: `?`
뭘 잘못하든 그냥 물음표 하나. 친절함의 극치.
"파일이 없어도 ?, 명령어가 틀려도 ?, 인생이 잘못돼도 ?"
</Callout>

Bill Joy가 1976년에 `ex`를 만들고, 여기에 비주얼 모드를 추가해서 `vi`가 탄생함.
드디어 화면에서 텍스트를 볼 수 있게 된 것임. 혁명이었음.

그리고 1991년, Bram Moolenaar가 vi를 개선해서 **Vim** (Vi IMproved)을 만듦.
구문 강조, 다중 실행 취소, 플러그인 시스템... vi의 모든 불만사항을 해결한 버전이었음.

### TECO → Emacs

한편 MIT에서는 다른 길을 걷고 있었음.
1960년대 TECO (Text Editor and Corrector)라는 에디터가 있었는데,
Richard Stallman이 여기에 매크로 시스템을 넣어서 1976년에 **Emacs** (Editor MACroS)를 만듦.

Emacs의 철학은 처음부터 달랐음:
**"에디터 안에서 모든 걸 할 수 있어야 한다."**

이메일도 읽고, 파일 관리도 하고, 게임도 하고, 심리 상담도 받을 수 있는 에디터.
(진짜로 `M-x doctor`라는 명령어가 있음. Eliza 심리 상담 프로그램임.)

```
;; Emacs에서 심리 상담 받기
M-x doctor

I am the psychotherapist. Please, describe your problems.
Each time you are finished talking, type RET twice.

> My code doesn't compile
What makes you believe your code doesn't compile?
```

## 철학의 충돌

### Vim: 모달 편집의 미학

Vim의 핵심 철학은 **"편집은 언어다"**임.

Vim에는 모드가 있음:
- **Normal 모드**: 이동과 명령 (기본 상태)
- **Insert 모드**: 텍스트 입력
- **Visual 모드**: 선택
- **Command 모드**: ex 명령어

이 모드 시스템이 Vim을 Vim답게 만듦. 키보드 조합이 아니라 **동사 + 명사** 구조로 편집함:

```
d   → delete (동사)
w   → word (명사)
dw  → delete word (단어 삭제)

c   → change (동사)
i"  → inside quotes (명사)
ci" → change inside quotes (따옴표 안의 내용 변경)

y   → yank/copy (동사)
ap  → a paragraph (명사)
yap → yank a paragraph (문단 복사)
```

이게 왜 강력하냐면, 동사와 명사를 조합해서 무한한 편집 명령을 만들 수 있기 때문임:

```
d3w  → 3단어 삭제
c$   → 줄 끝까지 변경
yi{  → 중괄호 안의 내용 복사
da(  → 괄호 포함해서 삭제
gUaw → 현재 단어 대문자로
```

<Callout type="success" title="Vim의 진짜 장점">
Vim을 제대로 배우면, 마우스 없이 생각의 속도로 코드를 편집할 수 있음.
타이핑이 아니라 "편집"에 최적화된 도구임.
코드를 새로 작성하는 시간보다 기존 코드를 수정하는 시간이 훨씬 많다는 걸 이해하면,
Vim의 설계가 얼마나 천재적인지 알 수 있음.
</Callout>

### Emacs: 확장성의 끝판왕

Emacs의 핵심 철학은 **"에디터는 플랫폼이다"**임.

Emacs는 Emacs Lisp (elisp)라는 완전한 프로그래밍 언어를 내장하고 있음.
에디터의 모든 동작이 elisp 함수이고, 사용자가 마음대로 수정하고 확장할 수 있음.

```elisp
;; 현재 줄 복제하는 함수 만들기
(defun duplicate-line ()
  "Duplicate current line"
  (interactive)
  (let ((line (thing-at-point 'line t)))
    (move-end-of-line 1)
    (newline)
    (insert line)))

;; 키 바인딩
(global-set-key (kbd "C-c d") 'duplicate-line)
```

```elisp
;; Emacs에서 할 수 있는 것들
M-x tetris          ;; 테트리스
M-x snake           ;; 스네이크 게임
M-x calendar        ;; 달력
M-x calc            ;; 계산기 (과학용 계산기 수준)
M-x gnus            ;; 뉴스/이메일 클라이언트
M-x eww             ;; 웹 브라우저 (진짜로)
M-x org-mode        ;; 문서 작성 시스템 (이것만으로 박사 논문 쓰는 사람 있음)
M-x eshell          ;; 셸
M-x doctor          ;; 심리 상담
M-x dunnet          ;; 텍스트 어드벤처 게임
```

<Callout type="warning" title="Emacs의 유명한 별명">
**"Emacs Makes A Computer Slow"**
**"Eight Megabytes And Constantly Swapping"**
**"Escape Meta Alt Control Shift"**

이건 1980년대 농담인데, 당시 8MB 메모리가 엄청난 양이었을 때 나온 것임.
지금 기준으로 보면 Electron 앱 하나가 500MB 먹는 시대에 Emacs는 오히려 가벼운 편임.
</Callout>

## 키 바인딩 전쟁

이 전쟁의 가장 치열한 전선은 키 바인딩이었음.

### Vim 키 바인딩

```
h j k l     → 좌 하 상 우 (화살표 키 안 씀)
w b e       → 단어 단위 이동
0 $ ^       → 줄 시작/끝/첫 글자
gg G        → 파일 시작/끝
/pattern    → 검색
n N         → 다음/이전 검색 결과
.           → 마지막 명령 반복 (이거 진짜 사기임)
u Ctrl-r    → undo/redo
dd yy pp    → 줄 삭제/복사/붙여넣기
ciw         → 현재 단어 변경
:%s/old/new/g → 전체 치환
```

### Emacs 키 바인딩

```
C-f C-b     → 앞/뒤 (Character)
M-f M-b     → 단어 앞/뒤
C-a C-e     → 줄 시작/끝
C-n C-p     → 다음/이전 줄
C-s         → 검색
C-x C-s     → 저장
C-x C-f     → 파일 열기
C-x C-c     → 종료
C-w         → 잘라내기
M-w         → 복사
C-y         → 붙여넣기
C-/         → undo
C-x u       → undo (대체)
```

<Callout type="error" title="손가락 건강 경고">
Emacs의 Ctrl 키 조합을 장시간 사용하면 "Emacs Pinky"라는 증상이 올 수 있음.
새끼손가락으로 Ctrl을 계속 누르다가 손가락이 아파오는 건데,
이거 때문에 Caps Lock을 Ctrl로 바꾸는 게 Emacs 유저의 통과의례임.

Vim 유저들은 이걸 보며 "우리는 hjkl만 쓰면 되는데ㅋ"라고 도발함.
</Callout>

## 설정 파일 비교: 지옥의 시작



<Callout type="info" title="설정 파일의 늪">
Vim과 Emacs 유저의 공통점: 둘 다 에디터 설정하는 데 실제 코딩하는 것보다 시간을 더 많이 씀.
이걸 "yak shaving"이라고 함. 코드를 짜려고 했는데 에디터 설정을 하고 있고,
에디터 설정을 하려고 했는데 플러그인을 만들고 있고,
플러그인을 만들려고 했는데 Lisp를 공부하고 있는 자기 자신을 발견하게 됨.
</Callout>

## Neovim의 등장: Vim의 부활

2014년, Vim의 코드베이스가 너무 레거시라서 현대화가 필요하다는 의견이 나옴.
Bram Moolenaar가 보수적으로 운영하던 Vim에 불만을 가진 개발자들이 포크해서 **Neovim**을 만듦.

Neovim이 바꾼 것들:

```lua
-- Neovim은 Lua를 쓸 수 있음 (Vimscript 대신)
-- 이게 게임 체인저였음

-- init.lua 예시
vim.opt.number = true
vim.opt.relativenumber = true
vim.opt.tabstop = 4
vim.opt.shiftwidth = 4
vim.opt.expandtab = true
vim.opt.smartindent = true
vim.opt.wrap = false
vim.opt.termguicolors = true

-- 키 매핑
vim.g.mapleader = " "
vim.keymap.set("n", "<leader>pv", vim.cmd.Ex)
vim.keymap.set("n", "<leader>w", ":w<CR>")

-- LSP 설정 (Language Server Protocol)
local lspconfig = require('lspconfig')

lspconfig.ts_ls.setup({
  on_attach = function(client, bufnr)
    vim.keymap.set("n", "gd", vim.lsp.buf.definition, { buffer = bufnr })
    vim.keymap.set("n", "K", vim.lsp.buf.hover, { buffer = bufnr })
    vim.keymap.set("n", "<leader>ca", vim.lsp.buf.code_action, { buffer = bufnr })
    vim.keymap.set("n", "<leader>rn", vim.lsp.buf.rename, { buffer = bufnr })
  end,
})
```

```lua
-- Telescope (퍼지 파인더) — Neovim의 킬러 플러그인
require('telescope').setup({
  defaults = {
    file_ignore_patterns = { "node_modules", ".git" },
    layout_strategy = "horizontal",
    layout_config = {
      horizontal = { preview_width = 0.6 },
    },
  },
})

vim.keymap.set("n", "<leader>ff", ":Telescope find_files<CR>")
vim.keymap.set("n", "<leader>fg", ":Telescope live_grep<CR>")
vim.keymap.set("n", "<leader>fb", ":Telescope buffers<CR>")
```

```lua
-- Treesitter (구문 분석) — 정규식 기반 하이라이팅은 이제 그만
require('nvim-treesitter.configs').setup({
  ensure_installed = {
    "typescript", "javascript", "tsx", "lua",
    "rust", "go", "python", "json", "yaml",
    "html", "css", "markdown",
  },
  highlight = { enable = true },
  indent = { enable = true },
  incremental_selection = { enable = true },
})
```

<Callout type="success" title="Neovim의 현재 위상">
2024-2025년 기준으로 Neovim은 "터미널 에디터" 카테고리에서 사실상 1위임.
ThePrimeagen, TJ DeVries 같은 유튜버들이 Neovim을 쓰면서 인기가 폭발함.
"Neovim 설정 영상" 조회수가 수백만을 찍는 시대가 됨.

하지만 여전히 전체 시장 점유율은 VS Code에 비하면 한 자릿수 %임.
</Callout>

## VS Code: 어부지리의 승자

2015년, Microsoft가 VS Code를 출시함.
처음엔 "MS가 에디터를? ㅋㅋ"라는 반응이었는데...

VS Code가 승리한 이유:

### 1. 진입 장벽이 거의 없음

```
Vim 학습 곡선:
|        _____
|       /
|      /
|     /
|    /
|___/
+------------------→ 시간

VS Code 학습 곡선:
|  _______________
| /
|/
+------------------→ 시간

Emacs 학습 곡선:
|            _____
|           /
|    ___   /
|   /   \_/
|  /
|_/
+------------------→ 시간 (중간에 한번 포기했다가 다시 시작)
```

### 2. 확장성 + 사용성의 균형

```json
// VS Code settings.json — 설정이 JSON이라 이해하기 쉬움
{
  "editor.fontSize": 14,
  "editor.fontFamily": "'JetBrains Mono', monospace",
  "editor.tabSize": 2,
  "editor.formatOnSave": true,
  "editor.minimap.enabled": false,
  "editor.bracketPairColorization.enabled": true,
  "editor.guides.bracketPairs": true,
  "editor.cursorSmoothCaretAnimation": "on",
  "workbench.colorTheme": "One Dark Pro",
  "terminal.integrated.fontFamily": "'JetBrains Mono', monospace",
  "files.autoSave": "afterDelay",
  "files.autoSaveDelay": 1000
}
```

### 3. Vim 키 바인딩도 지원함

```json
// VS Code에서 Vim 쓰기
// 확장: vscodevim.vim
{
  "vim.leader": "<space>",
  "vim.hlsearch": true,
  "vim.useSystemClipboard": true,
  "vim.easymotion": true,
  "vim.normalModeKeyBindings": [
    { "before": ["<leader>", "w"], "commands": [":w"] },
    { "before": ["<leader>", "q"], "commands": [":q"] }
  ]
}
// Vim의 80%를 VS Code 안에서 쓸 수 있음
// 나머지 20%는... 그냥 마우스 쓰면 됨
```

<Callout type="warning" title="VS Code의 그림자">
VS Code가 Electron 기반이라 메모리를 많이 먹는다는 건 팩트임.
하지만 "메모리 좀 먹으면 어때, 잘 되기만 하면 되지"가 현대 개발자의 마인드셋.
RAM 32GB 시대에 에디터가 500MB 먹는 게 뭐가 문제냐는 거임.

Vim 유저: "내 에디터는 5MB도 안 먹는데?"
VS Code 유저: "그래서? 나는 마우스 클릭 한 번으로 Git blame 보는데?"
</Callout>

### 각 진영의 현재 상태

**Vim/Neovim 진영:**
- Neovim이 커뮤니티를 부활시킴
- LSP 지원으로 IDE급 기능 확보
- 유튜브/Twitch에서 "Neovim으로 개발하기" 콘텐츠 인기
- 하지만 학습 곡선이 여전히 높음
- "dotfiles" 공유 문화가 활발

**Emacs 진영:**
- Doom Emacs, Spacemacs 같은 "배터리 포함" 배포판이 진입 장벽을 낮춤
- Org-mode는 여전히 대체 불가 (이것 때문에 Emacs 쓰는 사람 많음)
- Magit은 최고의 Git 인터페이스라는 데 Vim 유저도 인정함
- 하지만 새로운 유저 유입이 거의 없음

**VS Code 진영:**
- 압도적 시장 점유율
- GitHub Copilot 통합
- Remote SSH, WSL 지원
- 확장 마켓플레이스의 방대한 생태계
- 하지만 "개발자 정체성"을 부여하지는 못함

<Callout type="note" title="2025년의 새로운 도전자들">
Zed, Helix, Lapce 같은 새로운 에디터들이 나오고 있음.
특히 Zed는 Rust로 만들어서 성능이 미쳤고, AI 기능을 네이티브로 탑재함.
Helix는 "Vim을 현대적으로 다시 설계한" 에디터로 관심을 받고 있음.

하지만 VS Code의 아성을 넘기엔... 아직 갈 길이 멂.
</Callout>

## 유명한 밈과 농담들

### "Vim을 종료하는 방법"

```
StackOverflow 질문 (2012):
"How do I exit Vim?"

조회수: 3,000,000+
투표수: 5,000+

가장 많이 추천된 답변:
:q!

세상에서 가장 많은 개발자를 구한 4글자.
```

### Vim 유저의 일상

```
상사: "이 파일 좀 고쳐줘"
Vim 유저: "네" (5초 만에 수정 완료)
상사: "...방금 뭐 한 거야?"
Vim 유저: "ciw 하고 새 단어 치고 :wq 했는데요"
상사: "한국어로 다시 설명해줘"
```

### Emacs 유저의 일상

```
동료: "점심 뭐 먹을래?"
Emacs 유저: "잠깐, Emacs에서 주변 맛집 검색하고 있음"
동료: "...그냥 네이버 지도 쓰면 안 됨?"
Emacs 유저: "M-x restaurant-finder 패키지가 있는데 왜 굳이"
동료: (걍 혼자 감)
```

### VS Code 유저의 일상

```
동료: "어떤 에디터 써?"
VS Code 유저: "VS Code"
동료: "아 나도"
(대화 끝)

이게 VS Code가 이긴 이유임.
아무도 VS Code에 대해 열정적으로 이야기하지 않음.
그냥 쓰는 거임. 도구는 그래야 하는 건데.
```

## 실무에서의 선택 가이드

솔직히 말하면 이 전쟁은 이미 끝났고, 정답은 없음.
하지만 실무에서 뭘 선택할지 고민된다면:

### VS Code를 쓸 때

- 빨리 생산성을 내야 할 때
- 팀원 모두가 같은 에디터를 쓰면 좋을 때
- 확장 기능 생태계가 중요할 때
- 디버거, Git 통합 등 GUI 도구를 자주 쓸 때
- 99%의 경우에 해당함

### Neovim을 쓸 때

- SSH로 원격 서버에서 자주 작업할 때
- 키보드에서 손을 떼기 싫을 때
- 터미널을 좋아할 때
- 커스터마이징이 취미일 때
- "도구를 mastery하겠다"는 의지가 있을 때

### Emacs를 쓸 때

- Org-mode가 필요할 때 (이게 유일한 합리적 이유)
- Lisp를 사랑할 때
- 에디터를 운영체제처럼 쓰고 싶을 때
- 독특한 워크플로우를 구축하고 싶을 때

<Callout type="success" title="결론">
최고의 에디터는 **니가 가장 생산적인 에디터**임.
Vim이든 Emacs든 VS Code든, 도구는 목적을 위한 수단이지 정체성이 아님.

...이라고 쓰지만 Vim 유저는 지금 "ㅋㅋ 그래도 Vim이 최고지"라고 생각하고 있을 것이고,
Emacs 유저는 "이 글을 Emacs에서 읽고 있는데"라고 생각하고 있을 것임.

전쟁은 끝나지 않았다. 영원히.
</Callout>
