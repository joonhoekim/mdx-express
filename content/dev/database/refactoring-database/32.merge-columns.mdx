## 이게 뭔데

Merge Columns. 단어 그대로다. **한 테이블 안에 흩어져 있던 여러 컬럼을, 사실 같은 한 가지를 가리키고 있었길래 하나로 합치는 것**이다.

비유하자면 이렇다. 이사 갈 때 짐을 싸는데, 옷장을 열어보니 똑같은 검정 양말이 서랍 세 개에 나뉘어 있다. 처음엔 "발목용", "정장용", "운동용"으로 나눠 넣었던 것 같은데, 막상 신을 때 보면 그냥 다 검정 양말이다. 구분해서 넣은 사람도 이제 어느 게 뭔지 모른다. 그럼 답은 하나다. 서랍 하나로 합쳐서 "양말"이라고 라벨 붙이는 거다. 컬럼 병합도 똑같다. 한때는 의미가 있었던(혹은 있다고 *믿었던*) 구분이 실무에서 사라졌을 때, 그 구분을 들고 있는 컬럼들을 하나로 접는다.

<Callout type="info" title="한 줄 정의">
단일 테이블 내 둘 이상의 컬럼이 사실상 동일한 데이터를 표현하고 있을 때, 그것들을 하나의 컬럼으로 병합하는 구조 리팩토링.
</Callout>

은행 도메인으로 가보자. `Customer` 테이블에 전화번호를 이렇게 쪼개 넣은 설계가 있다.

```sql
CREATE TABLE Customer (
  CustomerID       BIGINT PRIMARY KEY,
  PhoneCountryCode SMALLINT,   -- 82
  PhoneAreaCode    SMALLINT,   -- 2
  PhoneLocal       INTEGER     -- 12345678
);
```

설계 당시엔 그럴듯했다. "국가코드, 지역번호, 가입자번호를 따로 보관하면 나중에 지역별 통계도 내고, 국제전화 라우팅도 하고…" 그런데 5년이 지난 지금, 아무도 `PhoneAreaCode`로 통계를 안 낸다. 화면에서도, API에서도, 청구서에서도 셋을 매번 다시 붙여서 `02-1234-5678`을 만들어 쓴다. 즉 **세 컬럼이 사실 한 컬럼이었던 거다.** 이게 Merge Columns가 답인 전형적 상황이다.

## 언제 쓰나

책은 컬럼이 쪼개진 채로 남는 동기를 세 가지로 짚는다. 셋 다 "처음엔 이유가 있었거나, 사고였거나"로 요약된다.

**1) 중복 생성 — 서로 다른 팀이 같은 걸 따로 만들었을 때.** 코어뱅킹 팀이 `Customer.MobilePhone`을 만들었는데, 마케팅 팀이 같은 테이블에 `Customer.ContactNumber`를 또 추가했다. 둘 다 고객 휴대폰 번호다. 한동안 둘 다 채워지다가, 어느 순간 한쪽만 신뢰할 수 있게 되고 다른 쪽은 썩은 데이터가 된다. 이건 조직 구조가 스키마에 새어 들어온 흔한 흉터다.

**2) 과잉 설계 — 처음부터 너무 잘게 쪼갰을 때.** 위의 `PhoneCountryCode` / `PhoneAreaCode` / `PhoneLocal`이 정확히 이 케이스다. "혹시 나중에 쓸지도 몰라서" 미리 정규화해 둔 구분이, 실제로는 한 번도 그 단위로 안 쓰인다.

**3) 시간이 흐르며 용도가 같아졌을 때.** 처음엔 `PreferredContact`(선호 연락처)와 `BackupContact`(보조 연락처)가 정말 다른 의미였는데, 비즈니스가 바뀌어 이제 그냥 "연락 가능한 번호 아무거나"로 동일하게 취급된다.

<Callout type="note" title="냄새로 알아보기">
코드베이스에서 **"이 컬럼들을 항상 같이 읽고, 항상 같이 붙여서 쓴다"**는 패턴이 보이면 병합 후보다. `concat(a, b, c)`, `a + b + c`, 혹은 "둘 중 NULL 아닌 거 쓰기"(`COALESCE(x, y)`)가 코드 곳곳에서 반복된다면, 그 컬럼들은 이미 사용 양태상 하나다.
</Callout>

## 주의할 점

병합은 정보를 *합치는* 게 아니라 종종 정보를 *버리는* 작업이다. 여기서 사고가 난다.

<Callout type="warning" title="정밀도 손실 — 합치면 못 돌아온다">
`PhoneAreaCode`(지역번호)와 `PhoneLocal`(가입자번호)을 하나의 `PhoneNumber` 문자열로 합치는 순간, "지역번호만 따로 뽑아내기"가 다시 어려워진다. 만약 둘이 *진짜로* 다른 용도였다면(예: 지역번호로 콜센터 라우팅을 한다거나), 병합 후엔 그 분리를 복원해야 하는 Split Column 작업으로 되돌아가야 한다. **병합 여부는 "데이터를 실제로 어떻게 쓰는가"로 판단해야지, "구조가 예뻐 보이는가"로 결정하면 안 된다.**
</Callout>

특히 조심할 케이스가 두 가지 더 있다.

- **타입을 좁히는 병합.** 세 개의 정수 컬럼을 하나의 `INTEGER`로 합치는데, 합친 값이 `INT` 범위를 넘어가면 오버플로로 데이터가 깨진다. 전화번호 같은 건 산술 합성하지 말고 그냥 문자열(`VARCHAR`)로 두는 게 안전하다. 앞자리 0(`02`)도 정수로 만들면 날아간다.
- **두 컬럼 값이 서로 다를 때.** 중복 생성 케이스에서 `MobilePhone`과 `ContactNumber`가 *불일치*하는 행이 있을 수 있다. 어느 쪽이 진실인가? 이건 코드로 못 정한다. 마이그레이션 전에 불일치 건수부터 세고, 비즈니스에 어느 쪽이 정답인지 물어봐야 한다.

```sql
-- 병합 전 반드시: 두 컬럼이 얼마나 어긋나 있는지부터 센다
SELECT COUNT(*) AS conflicting_rows
FROM Customer
WHERE MobilePhone IS NOT NULL
  AND ContactNumber IS NOT NULL
  AND MobilePhone <> ContactNumber;
```

이 숫자가 0이 아니면, 마이그레이션 스크립트를 짜기 전에 멈추고 데이터부터 정리하는 게 맞다.

## 이렇게 한다

2006년 원본은 새 컬럼을 추가하고, 손으로 짠 트리거로 원본 컬럼과 신규 컬럼을 양방향 동기화하고, 전환 기간이 끝나면 원본을 드롭하는 3단계를 제시한다. 이 골격은 지금도 유효하다. 다만 트리거를 손코딩하는 부분은 현대 도구로 많이 가벼워졌다. 핵심 아이디어는 하나다. **읽기와 쓰기를 한 번에 바꾸지 말고, 신·구를 잠시 공존시키며 갈아탄다(expand-contract, 일명 parallel change).**

`PhoneAreaCode + PhoneLocal → PhoneNumber` 병합을 예로 전 과정을 본다. (`PhoneCountryCode`는 진짜로 따로 쓰니까 그대로 둔다 — 모든 걸 합치는 게 아니라는 점이 중요하다.)

<Steps>
<Step title="Expand — 새 컬럼을 더한다 (원본은 그대로)">
새 `PhoneNumber` 컬럼을 추가한다. 이 단계에선 아무것도 깨지지 않는다. 기존 코드는 여전히 옛 컬럼을 읽고 쓰고, 새 컬럼은 비어 있을 뿐이다.

```sql
ALTER TABLE Customer ADD COLUMN PhoneNumber VARCHAR(20);
```

대용량 테이블이라면 이 `ADD COLUMN`조차 락을 길게 잡을 수 있다. PostgreSQL 11+는 NULL 허용 컬럼 추가가 메타데이터만 바꾸므로 거의 즉시지만, 기본값을 걸거나 NOT NULL을 한 번에 주려 하면 테이블을 다시 쓴다. MySQL이라면 `gh-ost`나 `pt-online-schema-change`로 온라인 변경하는 걸 고려한다.
</Step>

<Step title="동기화 — 신·구를 잠시 같이 채운다">
전환 기간 동안 옛 컬럼과 새 컬럼이 어긋나지 않게 묶어야 한다. 여기엔 두 갈래의 현대적 선택지가 있다.

**선택 A — Generated Column (DB가 알아서 파생).** 새 컬럼이 옛 컬럼들로부터 *순수하게 계산*되는 거라면, 트리거 손코딩 자체가 필요 없다. DB에게 "이건 저 두 컬럼으로 만든 값"이라고 선언하면 끝이다.

```sql
-- PostgreSQL: 항상 옛 컬럼에서 파생되는 저장 컬럼
ALTER TABLE Customer
  ADD COLUMN PhoneNumber VARCHAR(20)
  GENERATED ALWAYS AS (
    LPAD(PhoneAreaCode::text, 2, '0') || LPAD(PhoneLocal::text, 8, '0')
  ) STORED;
```

이러면 동기화가 깨질 일이 없다. 옛 컬럼이 바뀌면 새 컬럼도 자동으로 따라온다. 트리거 순환(원본→신규→원본…)을 걱정할 필요도 없다. 단, generated column은 직접 쓸 수 없으므로, 다음 단계에서 코드가 새 컬럼에 직접 쓰기 시작하려면 이걸 일반 컬럼으로 바꿔야 한다. "읽기만 전환"하는 동안엔 이 방식이 제일 깔끔하다.

**선택 B — 양방향 트리거 (양쪽 다 쓰는 동안 필요).** 전환 기간에 *어떤 앱은 옛 컬럼에, 어떤 앱은 새 컬럼에* 쓰는 진짜 다중 애플리케이션 상황이라면, 양쪽을 맞춰주는 트리거가 필요하다. 이때 **트리거 순환을 반드시 막아야 한다.**

```sql
-- PostgreSQL: 값이 실제로 달라졌을 때만 반대편을 갱신 → 순환 차단
CREATE OR REPLACE FUNCTION sync_phone() RETURNS trigger AS $$
BEGIN
  -- 새 컬럼에 들어온 값으로 옛 컬럼을 역산
  IF NEW.PhoneNumber IS DISTINCT FROM OLD.PhoneNumber THEN
    NEW.PhoneAreaCode := LEFT(NEW.PhoneNumber, 2)::smallint;
    NEW.PhoneLocal    := RIGHT(NEW.PhoneNumber, 8)::integer;
  -- 옛 컬럼이 바뀌었으면 새 컬럼을 합성
  ELSIF NEW.PhoneAreaCode IS DISTINCT FROM OLD.PhoneAreaCode
     OR NEW.PhoneLocal    IS DISTINCT FROM OLD.PhoneLocal THEN
    NEW.PhoneNumber := LPAD(NEW.PhoneAreaCode::text, 2, '0')
                    || LPAD(NEW.PhoneLocal::text, 8, '0');
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sync_phone
  BEFORE INSERT OR UPDATE ON Customer
  FOR EACH ROW EXECUTE FUNCTION sync_phone();
```

핵심은 `IS DISTINCT FROM`으로 **"실제로 값이 달라진 쪽"만** 반대편에 반영한다는 것이다. 이게 책이 말하는 트리거 순환 회피의 본질이다. 무조건 양쪽을 덮어쓰면 INSERT/UPDATE가 서로를 호출하며 무한 루프 또는 불필요한 재계산에 빠진다.
</Step>

<Step title="백필 — 기존 데이터를 1회 변환한다">
트리거나 generated column은 *앞으로 들어올* 데이터만 챙긴다. 이미 쌓인 행은 한 번 직접 채워야 한다.

```sql
-- 기존 모든 행을 병합 컬럼으로 1회 변환
UPDATE Customer
SET PhoneNumber = LPAD(PhoneAreaCode::text, 2, '0')
              || LPAD(PhoneLocal::text, 8, '0')
WHERE PhoneNumber IS NULL;
```

원본 책은 `SET PhoneNumber = PhoneAreaCode * 10000000 + PhoneLocal` 같은 *산술* 합성을 보여준다. 이건 함정이다. 앞자리 0이 날아가고(`02` → `2`) 오버플로 위험도 있다. 전화번호는 산술이 아니라 **문자열 연결**로 합쳐야 한다. 책의 골격은 따르되, 이런 디테일은 현대 감각으로 고친다.

대용량이면 이 `UPDATE`를 한 방에 치지 말고 PK 범위로 잘라 배치로 돌린다. 한 트랜잭션에 600만 행을 잠그면 그 시간 동안 테이블이 사실상 멈춘다.
</Step>

<Step title="Contract — 읽기·쓰기를 새 컬럼으로 갈아타고 옛 컬럼을 버린다">
이제 애플리케이션 코드가 새 컬럼만 보도록 바꾼다. 충분한 전환 기간(외부 프로그램 수정 시간 포함)이 지나면, 옛 컬럼과 트리거를 드롭한다.

```sql
DROP TRIGGER trg_sync_phone ON Customer;
DROP FUNCTION sync_phone();
ALTER TABLE Customer DROP COLUMN PhoneAreaCode;
ALTER TABLE Customer DROP COLUMN PhoneLocal;
```

다중 앱 환경이라면 옛 컬럼·트리거에 **동일한 드롭 날짜(drop date)**를 박아두고 모든 팀에 공지한다. 한 팀이라도 아직 옛 컬럼을 읽고 있으면 드롭하는 순간 그쪽이 깨진다.
</Step>
</Steps>

### 접근 프로그램(코드) 수정

스키마만 바꾼다고 끝이 아니다. 진짜 일은 컬럼을 *쓰던 코드*를 정리하는 데 있다. 병합의 보상이 바로 여기서 나온다 — 컬럼을 조립하던 너저분한 코드가 통째로 사라진다.

<Tabs defaultValue="before">
<TabsList>
<TabsTrigger value="before">Before (쪼개진 컬럼)</TabsTrigger>
<TabsTrigger value="after">After (병합된 컬럼)</TabsTrigger>
</TabsList>

<TabsContent value="before">

```typescript
// 읽을 때마다 세 조각을 손으로 붙인다
function formatPhone(c: CustomerRow): string {
  const area = String(c.phoneAreaCode).padStart(2, "0");
  const local = String(c.phoneLocal).padStart(8, "0");
  return `${c.phoneCountryCode}-${area}-${local}`;
}

// 쓸 때마다 거꾸로 쪼갠다 — 그리고 이게 틀릴 여지가 곳곳에 있다
function savePhone(c: CustomerRow, raw: string) {
  const digits = raw.replace(/\D/g, "");
  c.phoneAreaCode = Number(digits.slice(0, 2));
  c.phoneLocal = Number(digits.slice(2));
}
```

</TabsContent>

<TabsContent value="after">

```typescript
// 읽기: 그냥 컬럼 하나
function formatPhone(c: CustomerRow): string {
  return c.phoneNumber;
}

// 쓰기: 정규화만 하고 그대로 저장. 쪼개는 로직 자체가 사라졌다
function savePhone(c: CustomerRow, raw: string) {
  c.phoneNumber = raw.replace(/\D/g, "");
}
```

</TabsContent>
</Tabs>

책이 "접근 프로그램 수정"에서 콕 집는 게 이거다. 병합 컬럼을 쓰도록 바꾸고, **컬럼을 조합하던 코드를 제거하고, 두 컬럼이 같은지 검증하던 코드를 들어낸다.** 중복 생성 케이스(`MobilePhone` vs `ContactNumber`)였다면 "둘 중 NULL 아닌 거 골라 쓰기"(`COALESCE`) 같은 분기도 통째로 없어진다.

<Callout type="success" title="병합의 진짜 이득">
컬럼 병합은 디스크를 아끼려는 게 아니다. **컬럼을 매번 붙였다 쪼갰다 하던 애플리케이션 코드를 없애는 것**이 핵심 이득이다. 그 조립/분해 코드가 곳곳에 흩어져 있을수록, 그 안에 버그가 숨어 있을 확률도 높았다. 병합은 그 표면적을 통째로 줄인다.
</Callout>

### 마이그레이션 도구에 얹기

위 DDL/DML을 손으로 운영 DB에 치지 말고, 버전 관리되는 마이그레이션으로 남긴다. Flyway라면 `V32__merge_phone_columns.sql` 한 파일에 Expand와 백필을, Contract는 충분한 전환 기간 뒤 별도 버전(`V40__drop_old_phone_columns.sql`)으로 분리한다. **이 둘을 같은 릴리스에 묶으면 expand-contract의 의미가 없어진다** — 옛 컬럼을 읽던 구버전 앱이 롤백 불가 상태가 되기 때문이다. Liquibase/Alembic도 같은 원칙으로, "추가·백필" 변경과 "드롭" 변경을 시간차를 두고 별도 changeset으로 나눈다.

전체 흐름을 그림으로 보면 이렇다.

```mermaid
flowchart LR
  A["원본 스키마<br/>AreaCode + Local"] --> B["Expand<br/>PhoneNumber 추가"]
  B --> C["동기화<br/>generated col 또는 트리거"]
  C --> D["백필<br/>기존 행 1회 변환"]
  D --> E["코드 전환<br/>새 컬럼만 읽고 쓰기"]
  E --> F["Contract<br/>옛 컬럼·트리거 드롭"]
  F --> G["결과 스키마<br/>PhoneNumber"]
```

## 정리

Merge Columns는 작고 얌전한 리팩토링처럼 보이지만, 본질은 **"우리가 이 컬럼들을 사실은 하나로 쓰고 있었다"는 사실을 스키마에 솔직하게 반영하는 일**이다.

> 합치기 전에 물어야 할 단 하나의 질문: **"이 컬럼들을 따로 쓴 적이 진짜로 있었나?"**

따로 쓴 적이 없으면 합쳐서 코드를 단순하게 만든다. 따로 쓴 적이 있다면, 그건 병합 대상이 아니라 그냥 정규화된 좋은 설계다. 그리고 합칠 땐 한 번에 들이받지 말고 expand-contract로 — 새 컬럼을 더하고(expand), generated column이나 순환 안 도는 트리거로 동기화하고, 기존 데이터를 문자열로(산술 말고) 백필하고, 코드를 갈아탄 뒤, 충분히 기다렸다가 옛 컬럼을 버린다(contract). 그러면 가동 중인 은행 DB에서도 아무도 모르게 양말 서랍을 하나로 합칠 수 있다.
