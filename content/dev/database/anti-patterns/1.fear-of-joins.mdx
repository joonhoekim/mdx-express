## 이게 뭔데

JOIN 공포증. 이름은 거창한데, 증상은 단순하다. **두 테이블을 한 쿼리로 묶기 싫어서, 따로따로 조회한 다음 애플리케이션 코드에서 손으로 합치는 습관**이다.

비유를 하나 들어보자. 마트에서 비빔밥을 만들어 먹는다고 치자. 정상적인 사람은 비빔밥 재료 세트를 한 번에 산다. 근데 JOIN 공포증 환자는 이렇게 한다. 밥 사러 한 번 갔다 오고, 나물 사러 또 갔다 오고, 고추장 사러 또 갔다 오고, 계란 사러 또 갔다 온다. 재료가 다섯 가지면 마트를 다섯 번 왕복한다. 그러고는 집에서 직접 비빈다. "어차피 비비는 건 내가 하니까 따로 사 와도 똑같잖아?" 똑같지 않다. 왕복 비용이 다섯 배다.

DB에서 이 "마트 왕복"이 바로 쿼리 라운드트립이다. JOIN 하나면 끝날 일을, 테이블마다 따로 긁어서 코드에서 비빈다. 그리고 이게 사람을 잡는 가장 흔한 형태가 바로 **N+1 쿼리**다.

<Callout type="warning" title="한 줄 요약">
JOIN을 피해 코드에서 데이터를 합치면, 쿼리 한 방이면 될 일이 N+1방이 된다. 관계형 엔진이 가장 잘하는 일을 일부러 안 쓰는 셈이다.
</Callout>

## 시나리오: 이런 적 있을 거임

주문 목록 화면을 만든다. 주문 100건을 띄우는데, 각 줄에 주문한 사람 이름도 같이 보여줘야 한다. ORM을 켜고, 너무나 자연스럽게 이렇게 친다.

```typescript
const orders = await orderRepo.findAll({ take: 100 });

for (const order of orders) {
  // order.user는 lazy 관계. 여기서 접근하는 순간 쿼리가 나간다
  console.log(`${order.id} - ${order.user.name}`);
}
```

화면은 멀쩡히 잘 뜬다. 코드도 깔끔하다. PR에 "LGTM 👍"이 달린다. 아무 문제 없어 보인다.

그런데 DB 로그를 켜고 보면 풍경이 다르다.

```text
SELECT * FROM orders LIMIT 100;             -- 1번
SELECT * FROM users WHERE id = 1;           -- +1
SELECT * FROM users WHERE id = 2;           -- +1
SELECT * FROM users WHERE id = 3;           -- +1
...
SELECT * FROM users WHERE id = 100;         -- +1
```

쿼리 **1번이 101번**이 됐다. 주문 100건을 한 번에 긁고, 그다음 `order.user.name`을 찍을 때마다 ORM이 유저를 **하나씩** 다시 조회한 거다. 이게 그 악명 높은 N+1이다. 1번의 목록 쿼리 + N번의 추가 쿼리.

여기서 진짜 무서운 건 **로컬에선 절대 안 보인다는 것**이다. 개발 DB엔 주문이 10건이라 101번이 아니라 11번이고, 데이터도 메모리에 다 올라와 있어서 0.02초에 끝난다. 그래서 아무도 눈치 못 챈다. 리뷰어도 못 잡는다. 줄 단위로 보면 죄다 멀쩡한 `findAll`과 프로퍼티 접근이니까.

문제는 **운영 DB에 주문이 600만 건 쌓이고, 그 화면에 동시 접속 300명이 붙는 날** 터진다. 한 명이 페이지를 열 때마다 101번의 쿼리가 나가고, 300명이면 30,000번이 된다. APM 그래프에 빨간 스파이크가 찍히고, 슬랙에 "주문 페이지 왜 이렇게 느려요?"가 올라온다. 원인을 추적하다 `git blame`을 찍으면, 3개월 전의 내가 나온다.

정작 이걸 부른 건 코드가 아니라, 그 순간의 **행동 패턴**이다.

- "JOIN 쿼리 직접 짜기 귀찮은데, 그냥 관계 따라가면 ORM이 알아서 가져와 주잖아?" — **편해서**.
- "JOIN 잘못 걸면 카테시안 곱 터진다던데... 무서우니까 가급적 안 쓸래." — 막연한 **두려움**.
- "어차피 ORM이 최적화해서 한 방에 가져오겠지." — 위임했다는 **착각**.

이 셋 중 어느 하나라도 마음 한구석에 있었다면, 당신은 이미 N+1을 한 번쯤 심어봤을 가능성이 높다. 그리고 그건 신입만의 일이 아니다. lazy loading이 기본인 ORM 환경에선, 시니어 코드에도 이 패턴이 조용히 박혀 있다.

## 무슨 일이 벌어지나 (비용)

N+1이 왜 그렇게 비싼지, 비용을 세 갈래로 뜯어보자.

**1. 라운드트립 폭증.** 쿼리 한 번은 단순히 "DB가 일하는 시간"만 드는 게 아니다. 애플리케이션 → DB로 요청을 보내고, 결과를 받아오는 **왕복(round-trip)** 비용이 매번 붙는다. 같은 데이터센터 안이라도 한 번에 보통 0.2~1ms. 별것 아닌 것 같지만 100번이면 20~100ms가 통째로 왕복에만 날아간다. DB가 다른 가용영역에 있거나, 커넥션 풀이 빡빡하면 이 숫자는 더 커진다.

**2. 커넥션 점유와 메모리.** 쿼리 100번을 순차로 날리는 동안 커넥션 하나가 그만큼 오래 잡혀 있다. 동시 사용자가 늘면 커넥션 풀이 마르고, 풀이 마르면 다른 요청까지 줄을 선다. 게다가 따로 가져온 결과를 앱 메모리에 쌓아두고 코드로 병합하니, DB가 인덱스로 효율적으로 할 일을 앱이 풀스캔하듯 떠안는다.

**3. N+1로 직결.** 위 둘이 합쳐지면 부하는 사용자 수에 그대로 곱해진다. 1명이 101쿼리면 300명은 30,300쿼리. 선형이 아니라 **사용자 수 × 목록 길이**로 늘어난다.

<Callout type="error" title="뭐가 문제냐면">
- **느려지는데 원인을 못 찾는다**: 쿼리 하나하나는 0.1ms로 빠르다. 느린 건 "느린 쿼리"가 아니라 "쿼리가 너무 많다"는 거라, 슬로우 쿼리 로그에도 안 걸린다.
- **로컬에서 재현이 안 된다**: 데이터가 적으면 멀쩡하다. 운영 규모에서만 터진다.
- **부하에 비례해 악화된다**: 평소엔 괜찮다가 트래픽 몰리는 순간 커넥션 풀이 고갈되며 연쇄적으로 무너진다.
</Callout>

## 옵티마이저는 JOIN 하라고 만든 물건임

JOIN 공포증의 밑바닥엔 "JOIN은 무겁고 위험하다"는 오해가 깔려 있다. 실상은 정반대다. **관계형 데이터베이스가 가장 잘하는 일이 바로 JOIN이다.** 수십 년간 그것 하나 잘하려고 갈고닦인 물건이다.

쿼리를 던지면 옵티마이저가 통계와 인덱스를 보고 조인 전략을 고른다. 대표적으로 셋:

- **Nested Loop Join** — 한쪽 테이블을 돌면서 다른 쪽을 인덱스로 콕콕 찍는다. 한쪽이 작고 조인 키에 인덱스가 있을 때 최고.
- **Hash Join** — 한쪽으로 해시 테이블을 만들고 다른 쪽을 흘려보낸다. 큰 테이블끼리 등치 조인에 강하다.
- **Merge Join** — 양쪽을 정렬해 지퍼처럼 맞물린다. 이미 정렬돼 있거나 범위 조인에 유리.

핵심은, **우리가 이걸 고민할 필요가 없다는 것**이다. 옵티마이저가 데이터 분포를 보고 알아서 고른다. 우리가 코드에서 `for` 루프로 합치면, 옵티마이저는 그 의도를 볼 수조차 없으니 이 똑똑한 전략들을 하나도 못 쓴다. **JOIN을 피하는 순간, 우리는 엔진의 가장 비싼 기능을 일부러 끄는 셈이다.**

단, 전제가 하나 있다. **조인 키에 인덱스가 있어야 한다.** `orders.user_id`에 인덱스가 없으면 옵티마이저도 Nested Loop에서 매번 풀스캔을 돌아야 해서 제 실력이 안 나온다. "JOIN이 느리다"는 경험담의 상당수는 사실 "인덱스 없는 JOIN이 느렸다"는 얘기다.

<Callout type="info" title="카테시안 곱이 무서웠다면">
JOIN을 피하게 만드는 대표적 트라우마가 "결과가 폭발하는" 경험이다. 그건 보통 조인 조건(`ON`)을 빠뜨렸거나 잘못 걸어서 모든 행이 모든 행과 곱해진(Cartesian product) 경우다. JOIN 자체가 위험한 게 아니라, **조인 조건을 안 건 게** 위험한 거다. 조건만 제대로 걸면 카테시안 곱은 안 난다.
</Callout>

## 이렇게 하면 됨

해법은 단순하다. **데이터를 한 번에 모아 가져온다.** 방법은 두 갈래다.

**방법 1 — JOIN으로 한 방에.** 가장 직관적이다. 주문과 유저를 한 쿼리로 묶는다.

```sql
-- Before: 주문 100건 + 유저 100번 = 101 쿼리
SELECT * FROM orders LIMIT 100;
-- ...그리고 코드에서 유저를 하나씩 조회

-- After: 1 쿼리
SELECT o.id, o.total, u.name
FROM orders o
JOIN users u ON u.id = o.user_id
ORDER BY o.created_at DESC
LIMIT 100;
```

ORM에서도 관계를 미리 같이 가져오라고 명시하면 된다. lazy가 아니라 eager로.

```typescript
// Before: lazy 관계를 루프에서 건드려 N+1
const orders = await orderRepo.findAll({ take: 100 });

// After: 조인해서 한 번에 (TypeORM 예시)
const orders = await orderRepo.find({
  relations: { user: true },   // JOIN으로 같이 로드
  take: 100,
});
```

**방법 2 — `IN`으로 배치 로딩.** 관계가 복잡하거나 컬럼 폭이 부담되면, 굳이 한 쿼리로 안 묶어도 된다. 쿼리 **2번**으로 줄이는 것만으로 N+1은 사라진다.

```sql
-- 1) 주문을 가져온다
SELECT * FROM orders LIMIT 100;

-- 2) 등장한 user_id를 모아 단 한 번에
SELECT * FROM users WHERE id IN (1, 2, 3, ..., 100);
```

GraphQL 쪽이라면 DataLoader가 정확히 이 일을 한다. 한 틱 동안 들어온 `user_id`들을 모아 `IN` 한 방으로 묶어주는 배치 로더다. 핵심은 어느 방법이든 **"항목마다 한 번"을 "전부 모아 한 번"으로 바꾸는 것**이다.

<Callout type="note" title="그럼 분리 조회는 항상 나쁜가?">
아니다. 항상 JOIN이 정답은 아니다. (1) 가져온 관계를 화면에서 안 쓰는데 무겁게 다 조인하면 오히려 낭비고, (2) 한쪽을 캐시에서 거의 항상 맞힐 수 있다면 따로 조회가 쌀 수 있다. 안티패턴은 "분리 조회" 자체가 아니라 **"루프 안에서 항목마다 조회"**다. 둘을 구분하자.
</Callout>

## 정리

JOIN 공포증은 자연스럽게 생긴다. 누가 악의로 N+1을 심지 않는다. 그냥 lazy 관계를 무심코 루프에서 건드렸을 뿐이고, 로컬에선 멀쩡했을 뿐이다. 하지만 원리는 단순하다.

> **JOIN은 피할 대상이 아니라, 엔진한테 떠넘기는 도구다.**

데이터를 합치는 일은 관계형 엔진이 우리보다 압도적으로 잘한다. 통계도 보고, 인덱스도 타고, 조인 전략도 고른다. 우리가 코드에서 `for` 루프로 합치는 순간 그 모든 걸 포기하는 거다. 그러니 "이 데이터들을 코드에서 합치고 있다"는 생각이 들면, 그건 대개 JOIN이나 `IN`으로 내려보내야 한다는 신호다.

다음 편에서는 결이 살짝 다른 문제를 본다. "이 조건들을 *전부* 만족하는 행"을 찾는 일 — 태그 세 개를 다 가진 사용자를 찾으랬더니 `WHERE`에 `AND`를 줄줄이 박는, 관계형 나눗셈 얘기다.
