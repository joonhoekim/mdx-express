## 이게 뭔데

데드락 공포증. 증상은 이렇다. 운영 로그에서 `deadlock detected`를 **단 한 번** 본 순간, "이건 절대 일어나면 안 되는 사고"라고 단정하고, 그날부터 데드락을 **0으로 만드는 데** 모든 에너지를 쏟는 습관이다.

마음은 이해한다. "데드락"이라는 단어 자체가 무섭게 생겼다. 죽을 사(死)에 자물쇠(lock)까지 붙었으니, 무슨 시스템이 통째로 얼어붙는 치명적 버그처럼 들린다. 그래서 본능적으로 이걸 "절대 나면 안 되는 것" 칸에 넣는다. NPE나 무한 루프 옆자리쯤에.

근데 여기에 함정이 있다. 데드락은 버그가 아니다. **여러 트랜잭션이 동시에 자원을 잡으려고 경쟁하는 환경에서 자연스럽게 발생하는 정상적인 동시성 현상**이다. 두 사람이 좁은 복도에서 마주쳐 서로 비키려다 같은 방향으로 두세 번 우왕좌왕하는 그거랑 똑같다. 사람이 못된 게 아니다. 그냥 좁은 복도에서 동시에 움직이면 가끔 일어나는 일이다.

그래서 "데드락 0"이라는 목표는, 마치 "복도에서 절대 안 부딪히게 하겠다"는 목표와 같다. 방법이 없는 건 아니다. 한 번에 한 명씩만 복도를 지나가게 하면 된다. 근데 그러면 복도는 멀쩡한데 사람들이 줄을 서다 하루가 다 간다. 데드락을 잡으려고 동시성을 죽이는 게 정확히 이 짓이다.

<Callout type="warning" title="목표를 다시 잡자">
데드락은 박멸 대상이 아니라 **완화 대상**이다. 동시성 환경에서 데드락 발생 확률을 수학적으로 0으로 만드는 유일한 길은 동시성 자체를 없애는 것(직렬화)뿐이다. 현실적 목표는 "안 나게 막기"가 아니라 "**자주 안 나게 줄이고, 나면 우아하게 회복하기**"다.
</Callout>

## 시나리오: 이런 적 있을 거임

운영 로그에 `deadlock detected`가 한 번 뜬다. 충격받는다. "데드락? 절대 일어나면 안 되는 거 아냐?" 그래서 결심한다. "다시는 데드락이 안 나게 막겠다." 비장하다. 마치 보안 사고 회고 끝에 "다시는 뚫리지 않겠다"고 선언하는 보안팀처럼.

가장 먼저 떠오르는 해법은 단순하다. **잠금을 더 세게 걸면 충돌이 줄겠지.** 그래서 손이 닿는 모든 트랜잭션에 가장 강한 잠금을 박기 시작한다. `SELECT`에도 일단 `FOR UPDATE`를 붙이고, 격리 수준도 `SERIALIZABLE`로 올리고, 한술 더 떠 테이블 단위 락까지 손이 간다. "이러면 동시에 같은 데이터를 못 건드리니까 데드락도 안 나겠지."

```sql
-- "안전하게" 만든다고 모든 조회에 잠금을 박기 시작
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;   -- 읽기만 하는데도 잠금
SELECT * FROM accounts WHERE id = 2 FOR UPDATE;
-- ... 길고 무거운 로직 ...
COMMIT;
```

처음 며칠은 행복하다. 데드락 로그가 잠잠하다. "거봐, 세게 걸면 되잖아." 그런데 다른 그래프가 빨개지기 시작한다. **처리량이 바닥**난다. 트랜잭션들이 서로의 잠금이 풀리길 기다리며 줄을 선다. 동시에 100명이 들어와도 사실상 한 명씩 처리되는, 직렬화에 가까운 상태가 된다. 데드락은 없는데 시스템이 그냥 느리다. 데드락 에러가 "응답 지연" 에러로 바뀌었을 뿐이다.

그리고 결정타. **그렇게 세게 걸었는데도, 부하가 몰리면 또 데드락이 뜬다.** 두 트랜잭션이 같은 두 행을 **반대 순서로** 잠그는 순간, 잠금을 아무리 세게 걸어도 데드락은 발생한다. 오히려 잠금 보유 시간이 길어진 탓에 충돌 창이 넓어져서 더 잘 난다.

```text
-- 트랜잭션 A
LOCK accounts(1)   ✅ 잡음
LOCK accounts(2)   ⏳ 대기 (B가 쥐고 있음)

-- 트랜잭션 B  (같은 시각)
LOCK accounts(2)   ✅ 잡음
LOCK accounts(1)   ⏳ 대기 (A가 쥐고 있음)

→ A는 B를, B는 A를 기다린다. 순환 대기. DEADLOCK.
```

A는 2번을 기다리고, B는 1번을 기다린다. 서로가 서로를 영원히 기다리는 **순환 대기**가 만들어졌다. DB는 이걸 감지하고 둘 중 하나를 강제로 죽여(`victim`) 풀어준다. 즉 데드락 감지 자체는 DB가 알아서 해주는, 이미 잘 도는 기능이다. 막으려고 한 게 아니라 회복하려고 만들어진 메커니즘이다.

여기서 마지막 비극이 터진다. 정작 애플리케이션 코드엔 **데드락이 떴을 때 재시도하는 로직이 없다.** DB가 트랜잭션 하나를 죽이고 "다시 시도하세요"라고 친절하게 에러를 던졌는데, 코드는 그걸 그냥 500 에러로 사용자에게 토스해버린다. 막는 데 집착하느라, 정작 떴을 때 우아하게 회복하는 길은 안 만들어둔 거다. 데드락 1건이 그대로 사용자 에러 화면 1건이 된다.

처리량을 죽이고, 그러고도 에러를 못 막은 건 데드락이 아니라 그걸 대하는 태도다.

- "데드락은 버그니까 0으로 만들어야 해" — 정상적 동시성 현상을 절대악으로 오해.
- "잠금을 더 세게 걸면 안전하겠지" — 안전과 처리량을 맞바꾸는 줄 모름.
- "재시도? 일단 안 나게 하는 게 먼저지" — 회복 설계를 후순위로 미룸.

이 세 마음 중 하나라도 있었다면, 이미 "잠금 더 세게" 루트를 한 번쯤 타봤을 가능성이 높다. 그리고 그 끝은 대개 느리고, 그러면서도 가끔 데드락이 나고, 날 때마다 사용자가 에러를 보는 시스템이다.

## 무슨 일이 벌어지나 (비용)

"데드락 0"에 집착하면 비용이 세 갈래로 청구된다.

**1. 과한 잠금 → 동시성과 처리량 붕괴.** 잠금을 강하게, 넓게, 오래 걸수록 동시에 진행될 수 있는 트랜잭션 수가 줄어든다. 극단으로 가면 사실상 한 번에 하나씩 처리하는 직렬 시스템이 된다. 멀티코어 서버를 사놓고 싱글 스레드로 도는 셈이다. 데드락은 가끔 나는 사고지만, 이건 **상시로 내는 세금**이다. 가끔 나는 사고를 막겠다고 매 순간 비용을 무는 거라 대개 손해다.

**2. 그럼에도 데드락은 남는다 — 박멸 불가.** 가장 허무한 비용이다. 처리량을 그렇게나 갈아 넣었는데도, 반대 순서 잠금이라는 근본 원인이 살아 있는 한 데드락은 발생한다. 돈을 다 쓰고도 목표를 못 이룬 거다. "원천 봉쇄"는 원리적으로 불가능하다는 사실을, 보통 처리량을 다 태운 뒤에야 깨닫는다.

**3. 재시도 없는 코드 = 데드락 1건이 사용자 에러 1건.** DB는 데드락을 감지하면 한 트랜잭션을 죽이고 "이건 일시적 충돌이니 다시 해보라"는 신호(예: PostgreSQL `40P01`, MySQL `1213`)를 던진다. 이건 "다시 하면 십중팔구 성공할 일"이라는 뜻이다. 그런데 재시도 래퍼가 없으면 이 회복 가능한 에러가 그대로 위로 전파돼 사용자에게 500이 뜬다. **회복 가능한 사고를 회복 불가능한 사고로 만드는** 거다.

<Callout type="error" title="뭐가 문제냐면">
- **상시 세금을 낸다**: 가끔 나는 데드락을 막겠다고 모든 트랜잭션에 강한 잠금을 걸면, 평소 처리량을 항상 까먹는다. 사고 방지 비용이 사고 자체보다 비싸진다.
- **돈 쓰고도 못 막는다**: 반대 순서 잠금이 남아 있는 한 데드락은 난다. 잠금 강화는 발생 빈도를 줄이지도 못하고 오히려 충돌 창을 넓히기도 한다.
- **회복 기회를 버린다**: DB가 "다시 시도하라"고 알려주는데, 재시도 로직이 없으면 그 친절이 그냥 에러 화면이 된다.
</Callout>

## 정석: 박멸이 아니라 완화

목표를 "0건"에서 "드물게, 그리고 나도 회복"으로 바꾸면 길이 보인다. 세 가지를 함께 쓴다.

**1. 일관된 잠금 순서로 순환 대기를 끊는다.** 데드락의 4대 조건 중 우리가 가장 쉽게 깰 수 있는 게 **순환 대기**다. 모든 트랜잭션이 자원을 **항상 같은 순서로** 잠그도록 규칙을 정하면, 두 트랜잭션이 서로를 기다리는 사이클 자체가 만들어질 수 없다. 위 시나리오에서 A도 B도 무조건 "작은 id 먼저"로 잠그게 하면, 둘 다 1번을 먼저 노리니 한쪽은 그냥 기다렸다가 순서대로 끝난다. 충돌은 있어도 사이클은 없다. 이게 데드락 완화의 절반 이상을 차지한다.

**2. 트랜잭션을 짧게 유지한다.** 잠금을 쥐고 있는 시간이 짧을수록 다른 트랜잭션과 겹칠 확률이 낮아진다. 트랜잭션 안에서 외부 API를 호출하거나, 무거운 계산을 하거나, 사용자 입력을 기다리는 건 잠금을 쥔 채 낮잠 자는 거랑 같다. 잠금이 필요한 최소 구간만 트랜잭션으로 감싸고, 준비 작업은 트랜잭션 밖에서 끝내자.

**3. 데드락은 잡는 게 아니라 받아내는 거다 — 재시도 + 백오프.** 일관된 순서와 짧은 트랜잭션으로 빈도를 낮춰도 데드락은 가끔 난다. 그게 정상이다. 그러니 날 걸 전제로 설계한다. 데드락 에러를 감지하면 잠깐(가급적 살짝 랜덤한) 백오프 후 다시 시도한다. 데드락은 본질적으로 **타이밍 충돌**이라, 다시 하면 대개 성공한다. 재시도 몇 번이면 사용자는 데드락이 났는지조차 모른다.

<Callout type="info" title="데드락 vs 락 경합, 헷갈리지 말 것">
**데드락**은 서로가 서로를 기다리는 순환 대기라, DB가 감지해서 한쪽을 죽인다(빠르게 에러로 끝남 → 재시도 대상). **락 경합/대기**는 사이클이 아니라 그냥 줄을 서서 기다리는 것(타임아웃 전까지 느려질 뿐). 둘 다 "느림"으로 보이지만 원인과 처방이 다르다. 잠금을 세게 거는 건 데드락을 줄이기는커녕 락 경합을 늘려 전체를 더 느리게 만들곤 한다.
</Callout>

### 미니: 스레드 간 커넥션 공유

데드락 얘기를 하다 보면 자주 묶여 나오는, 결이 비슷한 동시성 함정이 하나 있다. **하나의 DB 커넥션을 여러 스레드가 동시에 공유**하는 짓이다.

"커넥션 만드는 거 비싸다며? 그럼 하나 만들어서 전역으로 같이 쓰면 효율적이잖아?" 직관은 그럴듯한데, 결과는 참사다. **DB 커넥션(세션)은 스레드 안전하지 않다.** 커넥션은 단순한 소켓 핸들이 아니라, 진행 중인 트랜잭션 상태·미전송 쿼리 버퍼·프로토콜 시퀀스 같은 **상태 머신**을 품고 있다. 한 커넥션 위에서 두 스레드가 동시에 쿼리를 밀어 넣으면 이 상태가 엉킨다.

```typescript
// ❌ 절대 금지: 커넥션 하나를 여러 스레드/요청이 공유
const sharedConn = await createConnection();

async function handleRequest() {
  // 여러 요청이 같은 sharedConn에 동시에 쿼리를 던진다
  await sharedConn.query("BEGIN");
  await sharedConn.query("UPDATE ...");  // 다른 요청의 트랜잭션과 뒤섞인다
  await sharedConn.query("COMMIT");
}
```

증상은 데드락보다 더 음험하다. 깨진 프로토콜 상태, "다른 쿼리 응답을 받았다"는 식의 정체불명 오류, 한 요청이 시작한 트랜잭션을 다른 요청이 커밋해버리는 데이터 정합성 사고. 게다가 부하에 따라 났다 안 났다 해서 재현이 지옥이다.

해법은 단순하다. **커넥션 풀에서 스레드(또는 요청)별로 하나씩 빌려 쓰고, 끝나면 반납한다.** 풀이 커넥션 생성 비용 문제도 해결해주고, 동시에 "한 커넥션은 한 시점에 한 사용자"라는 안전도 보장한다.

```typescript
// ✅ 풀에서 요청마다 빌리고, 끝나면 반납
async function handleRequest() {
  const conn = await pool.acquire();   // 내 요청 전용 커넥션
  try {
    await conn.query("BEGIN");
    await conn.query("UPDATE ...");
    await conn.query("COMMIT");
  } finally {
    pool.release(conn);                // 반드시 반납
  }
}
```

요약하면, 데드락이든 커넥션 공유든 본질은 같다. **동시성을 만만하게 보면 상태가 엉킨다.** 데드락은 DB가 감지라도 해주지만, 커넥션 공유는 조용히 데이터를 망가뜨려서 더 무섭다.

## 이렇게 하면 됨

핵심은 두 가지를 동시에 손보는 거다. **잠금 순서를 통일**해서 빈도를 낮추고, **재시도 래퍼**를 씌워서 그래도 난 걸 회복한다.

**Before — 반대 순서 잠금 + 재시도 없음.** 계좌 이체에서 흔히 나오는 형태다. "보내는 계좌 먼저, 받는 계좌 다음"으로 잠그면, A→B 이체와 B→A 이체가 동시에 들어올 때 정확히 반대 순서로 잠궈서 데드락이 난다.

```typescript
// ❌ Before: 잠그는 순서가 인자 순서에 끌려다님 + 재시도 없음
async function transfer(fromId: number, toId: number, amount: number) {
  const conn = await pool.acquire();
  try {
    await conn.query("BEGIN");
    // 보내는 쪽 먼저, 받는 쪽 다음 — 호출에 따라 순서가 뒤집힌다
    await conn.query("SELECT * FROM accounts WHERE id = $1 FOR UPDATE", [fromId]);
    await conn.query("SELECT * FROM accounts WHERE id = $1 FOR UPDATE", [toId]);
    await conn.query("UPDATE accounts SET balance = balance - $2 WHERE id = $1", [fromId, amount]);
    await conn.query("UPDATE accounts SET balance = balance + $2 WHERE id = $1", [toId, amount]);
    await conn.query("COMMIT");
  } catch (e) {
    await conn.query("ROLLBACK");
    throw e;   // 데드락이면 그대로 사용자에게 500
  } finally {
    pool.release(conn);
  }
}

// transfer(1, 2, ...) 와 transfer(2, 1, ...) 가 동시에 → 반대 순서 잠금 → DEADLOCK
```

**After — 잠금 순서 통일 + 재시도 래퍼.** 두 군데를 고친다. 첫째, 어떤 호출이 와도 **항상 작은 id를 먼저** 잠근다. 그러면 두 트랜잭션이 같은 두 행을 건드려도 순서가 같으니 순환 대기가 안 생긴다. 둘째, 그래도 날 수 있는 데드락을 위해 백오프 재시도로 감싼다.

```typescript
// ✅ After: 항상 id 오름차순으로 잠근다 (순환 대기 제거)
async function transfer(fromId: number, toId: number, amount: number) {
  const conn = await pool.acquire();
  try {
    await conn.query("BEGIN");
    // 인자 순서와 무관하게 잠금 순서를 고정
    const [firstId, secondId] = [fromId, toId].sort((a, b) => a - b);
    await conn.query("SELECT * FROM accounts WHERE id = $1 FOR UPDATE", [firstId]);
    await conn.query("SELECT * FROM accounts WHERE id = $1 FOR UPDATE", [secondId]);
    await conn.query("UPDATE accounts SET balance = balance - $2 WHERE id = $1", [fromId, amount]);
    await conn.query("UPDATE accounts SET balance = balance + $2 WHERE id = $1", [toId, amount]);
    await conn.query("COMMIT");
  } catch (e) {
    await conn.query("ROLLBACK");
    throw e;
  } finally {
    pool.release(conn);
  }
}

// ✅ 그래도 날 수 있으니, 데드락이면 백오프 후 재시도하는 래퍼로 감싼다
const DEADLOCK_CODES = new Set(["40P01" /* postgres */, "1213" /* mysql */]);

async function withDeadlockRetry<T>(fn: () => Promise<T>, maxRetries = 3): Promise<T> {
  for (let attempt = 0; ; attempt++) {
    try {
      return await fn();
    } catch (e: any) {
      const isDeadlock = DEADLOCK_CODES.has(e.code);
      if (!isDeadlock || attempt >= maxRetries) throw e;
      // 지수 백오프 + 지터 — 두 트랜잭션이 동시에 재시도해 또 부딪히는 걸 방지
      const delay = 2 ** attempt * 20 + Math.random() * 20;
      await new Promise((r) => setTimeout(r, delay));
    }
  }
}

// 사용
await withDeadlockRetry(() => transfer(2, 1, 10_000));
```

순서 통일로 빈도를 거의 0에 수렴시키고, 재시도로 잔여 데드락을 사용자 눈에 안 보이게 흡수한다. 잠금을 더 세게 거는 게 아니라, **순서를 맞추고 회복을 준비하는** 방향이라는 점이 핵심이다.

**데드락 로그를 읽고 순서를 추적하는 법.** 어떤 순서로 잠궈야 할지 모를 땐, DB가 로그에 다 적어준다. PostgreSQL은 `log_lock_waits`를 켜면, 그리고 데드락이 나면 어느 프로세스가 어떤 락을 쥐고 무엇을 기다렸는지 친절하게 남긴다. MySQL/InnoDB는 `SHOW ENGINE INNODB STATUS`의 `LATEST DETECTED DEADLOCK` 섹션에 두 트랜잭션이 무슨 인덱스의 어떤 락을 잡고 기다렸는지 찍힌다.

```text
*** (1) TRANSACTION:
  ... holds lock on accounts index PRIMARY (id=1)
  ... waiting for lock on accounts index PRIMARY (id=2)
*** (2) TRANSACTION:
  ... holds lock on accounts index PRIMARY (id=2)
  ... waiting for lock on accounts index PRIMARY (id=1)
```

이 로그에서 "TX1은 1을 쥐고 2를 기다림 / TX2는 2를 쥐고 1을 기다림"을 읽으면, 잠금 순서가 엇갈렸다는 게 그대로 보인다. 처방은 명확하다. **둘 다 같은 순서로 잠그게 코드를 고치는 것.** 추측으로 잠금을 더 거는 게 아니라, 로그가 가리키는 사이클을 끊는 게 정공법이다.

## 정리

데드락 공포증은 단어가 무섭게 생긴 죄가 크다. "절대 나면 안 되는 것"으로 분류하는 순간, 잠금을 세게 걸어 처리량을 죽이고, 그러고도 못 막고, 회복 설계는 빼먹는 삼중고로 직행한다. 하지만 데드락은 동시성의 정상적인 부산물이고, 다스리는 방법은 정해져 있다.

> **데드락은 막는 게 아니라 다스리는 것 — 순서·짧은 트랜잭션·재시도.**

일관된 잠금 순서로 순환 대기를 끊어 빈도를 낮추고, 트랜잭션을 짧게 유지해 충돌 창을 좁히고, 그래도 난 데드락은 백오프 재시도로 흡수한다. 그러면 데드락은 났는지조차 모르게 지나간다. "데드락을 0으로"가 아니라 "데드락을 안 보이게"가 현실적이고 올바른 목표다.

다음 편은 시리즈의 마지막이다. 지울 수 있는 데이터를 못 버리고, 모든 걸 영원히 끌어안다가 테이블이 호수가 되는 이야기 — 데이터 호더다.
