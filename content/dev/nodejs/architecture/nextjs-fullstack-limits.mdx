거의 모두가 한 번은 던지는 질문이다. **"Next.js 하나로 백엔드·프론트엔드 다 되는데, 왜 굳이 나눠?"** 나도 그랬고, 한동안 Next.js 하나로 전부 해결하려 했다.

정직하게 먼저 답하면 — **상당히 넓은 범위에서 그 직관은 옳다.** 그런데 어느 지점에서 갈린다. 이 글은 그 갈림목이 어디고, 무엇이 그걸 만드는지를 짚는다. 결론부터 말하면, 갈림을 만드는 건 트래픽도 팀 규모도 아니라 **횡단 관심사(cross-cutting concerns)** 다.

## 먼저 인정하자 — 분리는 대체로 이르다

작은 앱에서 백/프론트 분리는 보통 **이른 최적화**다. Next.js의 route handler와 server component만으로 인증, DB 접근, API까지 한 프로젝트에서 다 된다. 분리하는 순간 공짜로 따라오는 비용이 있다.

- 배포 단위가 둘로 늘고, 그 사이에 **네트워크 경계**가 생긴다.
- 양쪽 데이터 타입을 어긋나지 않게 맞추는 일([별도 글의 주제](/docs/dev/nodejs/architecture/trpc))이 새 숙제로 추가된다.
- 운영·로깅·로컬 개발 환경이 두 배가 된다.

그러니 "남들이 나누니까", "왠지 그게 정석 같아서" 나누는 거라면 — 나누지 마라. Next.js 단독으로 버틸 수 있는 만큼 버티는 게 맞다. 문제는 *언제 그 버팀이 한계에 닿는가*인데, 그 신호는 트래픽 그래프가 아니라 **코드의 모양**에서 먼저 온다.

## 갈림목 — 횡단 관심사가 쌓이기 시작할 때

앱이 자라면 어느 라우트 *하나*에 속하지 않고 **거의 모든 라우트를 가로지르는** 관심사들이 등장한다.

- 인증(이 요청은 누구인가) · 인가(이걸 할 권한이 있는가)
- 로깅 · 감사(audit) · 추적(trace)
- 입력 검증 · 에러 매핑(도메인 오류 → HTTP 응답)
- 트랜잭션 경계 · 레이트 리밋

이것들의 공통점은 — **도메인 로직이 아니다.** "주문을 생성한다"는 본질이 아니라, *모든* 핸들러가 똑같이 둘러야 하는 껍질이다. 이렇게 여러 모듈을 *가로질러(cross-cut)* 반복되는 관심사를 횡단 관심사라 부른다. 그리고 이걸 다루는 방식이 코드 품질을 가른다.

## AOP — 한 곳에 모으거나, 모든 곳에 흩뿌리거나

횡단 관심사를 핵심 로직에서 떼어내 **한 곳에 모아 선언적으로 적용**하는 접근을 AOP(Aspect-Oriented Programming, 관점 지향)라 한다. 거창한 이름이지만 핵심은 단순하다 — *복붙하지 말고 한 군데 꽂아라.*

이게 되느냐 안 되느냐는 **프레임워크가 그 "꽂을 자리"를 구조로 떠받쳐주는가**에 달렸다. 같은 핸들러를 두 세계에서 써 보면 차이가 한눈에 보인다.

프레임워크 지원이 없을 때 — 모든 핸들러가 같은 껍질을 복붙한다:

```ts
// app/api/orders/route.ts
export async function POST(req: Request) {
  const user = await authenticate(req);   // 인증     ┐
  if (!user) return unauthorized();        // 인가      │ 이 다섯 줄이
  log("POST /orders", user.id);            // 로깅      │ /products, /users …
  try {                                    //          │ 모든 핸들러에
    const body = orderSchema.parse(await req.json()); // 검증  │ 그대로 반복된다
    const result = await createOrder(body, user);     // ← 진짜 로직은 이 한 줄
    return Response.json(result);
  } catch (e) {
    return mapError(e);                    // 에러 매핑 ┘
  }
}
```

핸들러 하나에 도메인 로직은 한 줄, 껍질은 다섯 줄이다. 라우트가 50개면 이 보일러플레이트가 50번 복제된다. 인증 로직을 한 줄 고치려면 50곳을 찾아 고쳐야 하고, 한 곳을 빠뜨리면 그게 보안 구멍이 된다. **"AOP가 나올 때 프레임워크가 지원 안 하면 코드 품질이 답이 없다"** 는 게 정확히 이 그림이다.

프레임워크가 떠받쳐줄 때 — 껍질이 핸들러 밖으로 빠진다:

```ts
@Controller("orders")
@UseGuards(AuthGuard)                  // 인증·인가 — 한 곳에 선언, 전 핸들러 적용
@UseInterceptors(LoggingInterceptor)   // 로깅 — 한 곳에 선언
export class OrderController {
  @Post()
  create(@Body() body: OrderDto) {     // 검증은 DTO + Pipe가 자동으로
    return this.service.create(body);  // 핸들러엔 진짜 로직만 남는다
  }
}
```

에러 매핑은 전역 `ExceptionFilter` 하나가 처리한다. 횡단 관심사가 전부 핸들러 밖의 *선언적 슬롯*(guard / interceptor / pipe / filter)으로 빠지고, 핸들러엔 도메인 로직만 남는다. 인증을 고치려면 `AuthGuard` 한 곳만 고치면 되고, 새 라우트는 자동으로 같은 껍질을 입는다.

<Callout type="info" title="🌱 처음이라면 — 'Next.js로는 못 한다'는 뜻이 아니다">
Next.js에서도 횡단 관심사를 뺄 수는 있다. `middleware.ts`(단, edge 런타임이라 할 수 있는 일이 제한적), 핸들러를 감싸는 고차 함수(`withAuth(withLogging(handler))`) 같은 패턴이 있다. 핵심 차이는 — 그걸 **네가 직접 만들고, 팀에 컨벤션으로 강제해야 한다**는 점이다. 프레임워크가 구조로 떠받치는 게 아니라서, 코드베이스와 팀이 커질수록 "누군가 wrapper를 안 씌운 핸들러"가 새어 나온다. 작을 땐 충분하지만, 규모가 임계를 넘으면 강제력의 부재가 품질로 드러난다.
</Callout>

## 두 번째 갈림목 — 요청 컨텍스트는 어디로 흐르는가

횡단 관심사가 *공간*을 가로지르는 문제라면, 이건 *시간*을 가로지르는 문제다. 그리고 내가 가장 결정적으로 등을 떠밀린 지점이기도 하다 — **결재(approval)**.

먼저 흔한 오해부터 풀자. "Next는 백그라운드 작업을 못 한다"는 말이 아니다. 오히려 배포가 잦은 현실에선 큐·스케줄을 **로컬(인메모리) 상태에 의존하도록 설계하지 않는** 게 당연하다 — 인스턴스는 배포마다 갈아엎히니까. 상태는 DB 같은 영속성 레이어에 두고, 스케줄은 `instrumentation.ts`에서 `node-cron`으로 띄우는 게 자연스러운 귀결이고, 이건 Next에서도 깔끔하게 된다. **스케줄링 자체는 벽이 아니었다.**

진짜 차이는 그다음에 왔다 — **요청 컨텍스트(request context)** 를 어떻게 다루느냐.

거의 모든 레이어가 공유해야 하는 주변 정보가 있다 — 현재 사용자, 테넌트/조직, 트랜잭션 핸들, 추적(trace) id, 권한, 로케일. 질문은 이것들이 *진입점에서 깊은 서비스 함수까지, 그리고 요청 경계 너머까지 어떻게 흐르느냐*다. 두 군데서 갈렸다.

**① 요청 경로 안에서의 전파.** Next route handler엔 구조적 컨텍스트가 없어서, 이 값들을 함수 인자로 일일이 꿰거나 `AsyncLocalStorage`로 컨텍스트 저장소를 직접 만들어야 한다. DI도 없으니 컨텍스트를 쓰는 서비스들도 손으로 엮는다. 레이어가 깊어질수록 시그니처가 배관(plumbing)으로 더러워진다.

**② 백그라운드로 건너갈 때의 재구성.** 그리고 결재가 정확히 여기서 아팠다. cron이 "기한 초과 자동 반려"를 돌릴 때는 *요청이 없다.* HTTP 요청이 없으니 "현재 사용자·트랜잭션·감사 주체"가 비어 있다. 그런데 이 잡은 요청 경로가 쓰는 *바로 그 도메인 서비스*를 호출해야 한다 — 반려 처리도, 알림도, 감사 로그도 요청에서든 cron에서든 똑같아야 하니까.

코드로 보면 차이가 또렷하다. 프레임워크 지원이 없을 때 — 컨텍스트가 시그니처를 타고 흐르고, cron에선 그걸 손으로 합성한다:

```ts
// 요청 경로 — 모든 레이어가 컨텍스트를 인자로 나른다
async function rejectApproval(id: string, actor: User, tx: Tx, traceId: string) { /* ... */ }

export async function POST(req: Request) {
  const actor = await authenticate(req);
  const tx = await db.begin();
  await rejectApproval(id, actor, tx, traceOf(req));        // 손으로 전달
}

// 백그라운드 경로 (instrumentation.ts + node-cron) — actor가 없다, 만들어내야 한다
cron.schedule("0 * * * *", async () => {
  for (const a of await findOverdue()) {
    const systemActor = makeSystemActor();                 // 가짜 주체를 합성
    const tx = await db.begin();
    await rejectApproval(a.id, systemActor, tx, newTraceId()); // 같은 함수, 컨텍스트는 수동
  }
});
```

NestJS는 요청 컨텍스트를 *주변(ambient)* 에 두고, DI/CLS(`AsyncLocalStorage` 래퍼)로 시그니처를 더럽히지 않고 꺼낸다. 백그라운드에선 같은 서비스를 *구성한 컨텍스트* 안에서 부르면 된다:

```ts
@Injectable()
class ApprovalService {
  constructor(private cls: ClsService) {}
  async reject(id: string) {
    const actor = this.cls.get("actor");   // 시그니처에 없어도 꺼낸다
    // 트랜잭션·감사 주체도 같은 컨텍스트에서 — 요청이든 잡이든 동일
  }
}

@Cron("0 * * * *")
async sweepOverdue() {
  for (const a of await this.repo.findOverdue())
    await this.cls.run({ actor: SYSTEM }, () => this.approval.reject(a.id));
}
```

같은 `ApprovalService.reject`가 요청에서든 cron에서든 *그대로* 돈다. 컨텍스트는 주변에 있고, 진입점(인터셉터든 cron이든)이 그걸 어떻게 채울지만 정한다. 프레임워크 지원이 없으면 — 요청 경로에서도 배관을 손으로 깔고, 백그라운드 경계에선 그 부재가 두 배로 드러난다. 요청용 로직과 배치용 로직이 미묘하게 갈라지고, 컨텍스트 누락으로 감사 주체가 비는 식의 버그가 샌다.

핵심 차이는 "Next로는 불가능하다"가 아니다. 다 된다. 차이는 — **요청 컨텍스트를 프레임워크가 소유하고 전파하느냐, 네가 레이어마다·경계마다 손으로 나르느냐**다. 결재처럼 *컨텍스트가 깊은 호출 그래프를 타고 백그라운드까지 이어져야 하는* 영역에선, 이 배관 비용이 누적돼 임계를 넘는다.

<Callout type="note" title="🔍 그럼 모든 게 다 분리를 정당화하나 — 아니다, 비용엔 결이 있다">
모든 관심사가 똑같이 분리를 부르진 않는다. 직접 겪어 보니 결이 둘로 갈렸다.

- **로깅** — 횡단 관심사이긴 한데, *요구하는 곳이 많지 않으면* 핸들러 몇 곳에 직접 넣어도 견딜 만했다. 퍼짐(spread)이 낮으면 프레임워크 지원이 없어도 그리 아프지 않다.
- **결재의 요청 컨텍스트** — 한 군데만 있어도 아팠다. 퍼짐의 문제가 아니라, *컨텍스트를 깊은 레이어와 백그라운드 경계 너머로 흘려보낼 구조 자체가 없는* 문제였기 때문이다.

그래서 비용은 대략 이렇게 갈린다 — 횡단 관심사는 **퍼진 정도 × 프레임워크 미지원**, 요청 컨텍스트 전파는 **퍼짐과 무관하게 구조 자체가 필요**. 로깅이 견딜 만했던 것과 결재가 결정타였던 건 같은 저울의 다른 눈금이다.
</Callout>

## 그래서 진짜 질문은

지금까지 두 개의 갈림목을 봤는데, 둘은 사실 한 뿌리다 — **관심사가 "요청 하나"의 경계를 넘어선다**는 것. 횡단 관심사는 그 경계를 *공간으로*(모든 라우트를 가로질러) 넘고, 요청 컨텍스트는 *시간으로*(요청이 끝난 뒤 백그라운드까지) 이어져야 한다. Next.js의 핸들러는 요청 하나에 묶인 단위라, 둘 중 어느 쪽도 구조로 떠받치지 못한다.

그래서 "Next.js냐 분리냐"가 진짜 질문이 아니다. 진짜 질문은 이거다 — **내 일이 요청 하나의 경계(공간이든 시간이든)를 얼마나 자주, 얼마나 본질적으로 넘는가?**

이건 사실 [tRPC 글](/docs/dev/nodejs/architecture/trpc)에서 다룬 "request lifecycle을 누가 소유하는가"와 *같은 축*이다. 거기선 NestJS가 lifecycle(guard·interceptor·pipe)을 통째로 소유하는 게 tRPC와 충돌하는 *비용*이었다. 그런데 여기선 바로 그 소유권이 **이득**이 된다 — guard·interceptor·pipe가 곧 AOP의 슬롯이기 때문이다. 같은 성질이 한 이야기에선 비용, 다른 이야기에선 자산이다. 그래서 무엇이 "더 좋은 도구"인지가 아니라, *내 문제가 그 성질을 필요로 하는 지점에 왔는가*가 결정의 전부다.

<Callout type="note" title="🔍 횡단 관심사 말고도, 분리를 부르는 신호들">
횡단 관심사와 요청 컨텍스트는 *내가 직접 겪은* 가장 또렷한 이유라 본문 중심에 뒀지만, 분리를 정당화하는 축은 더 있다. 함께 점검할 가치가 있다.

- **독립 배포·스케일** — 백엔드만 따로 무겁게 굴리거나 따로 배포해야 할 때
- **다중 소비자** — 웹 외에 모바일·서드파티·비-TS 클라이언트가 같은 API를 쓸 때(이건 [tRPC 글의 consumer 질문](/docs/dev/nodejs/architecture/trpc)과 직결)
- **팀 경계** — 프론트팀·백엔드팀이 나뉘어 배포 주기가 다를 때

이 중 어느 것도 없고 횡단 관심사도 가볍고 요청 밖의 일도 없다면 — 분리는 여전히 이르다.
</Callout>

## 점진적으로 가도 된다

좋은 소식은, 이게 한 번에 정하는 양자택일이 아니라는 것이다. 대부분은 이 경로를 밟는다.

1. **Next.js 단독** — 작을 땐 이게 정답. 분리 비용을 치르지 않는다.
2. **Next.js + 잘 짜인 wrapper 유틸** — 횡단 관심사가 보이기 시작하면, 핸들러를 감싸는 고차 함수와 컨벤션으로 버틴다. 직접 만든 얇은 AOP 층이다.
3. **백엔드 프레임워크 분리** — wrapper로는 컨벤션 강제가 새고 보일러플레이트가 다시 번지기 시작하면, 그때 프레임워크가 제 무게값을 한다.

핵심은 하나다. **분리는 목표가 아니라 증상에 대한 처방이다.** "Next.js 하나면 되지 않나"라는 질문의 정답은 "대체로 맞다, 횡단 관심사가 프레임워크의 지지를 요구하는 그 지점까지는" 이다. 그 지점이 오기 *전에* 나누면 이른 최적화고, 온 *뒤에도* 안 나누면 코드 품질로 비용을 치른다. 트래픽이 아니라 **코드의 모양과 도메인의 성격**을 보고 판단하면 된다.
