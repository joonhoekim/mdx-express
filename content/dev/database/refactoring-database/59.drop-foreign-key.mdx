## 이게 뭔데

외래 키 제약 제거(Drop Foreign Key Constraint). 이름 그대로다. **테이블에 걸려 있던 FK 제약을 떼서, DB가 더 이상 "이 참조가 진짜로 존재하는지"를 강제하지 않게 만드는 리팩토링**이다.

비유를 들어보자. FK는 클럽 입구의 신분증 검사원 같은 거다. `Account` 테이블에 행을 하나 넣으려면, 입구에서 "너 `StatusCode`가 `AccountStatus`에 진짜 있는 코드 맞아?" 하고 매번 확인한다. 없는 코드면 그 자리에서 입장 거부(예외)다. 이 검사 덕분에 클럽 안엔 항상 "유효한 상태값을 가진 계좌"만 존재한다는 게 보장된다.

FK를 제거한다는 건, **이 검사원을 입구에서 빼는 일**이다. 줄이 빨리 줄어들어서 입장은 훨씬 빨라진다. 대신 가짜 신분증을 든 손님이 들어올 수도 있다. 그러니 떼는 순간 "그럼 검사는 누가 해?"라는 질문이 따라온다. 이 리팩토링의 절반은 떼는 거고, 나머지 절반은 그 질문에 답하는 거다.

<Callout type="warning" title="한 줄 요약">
FK를 제거하면 쓰기 성능은 얻지만, 무결성을 보장하던 안전망이 사라진다. 그 책임을 애플리케이션이나 서비스 경계가 명시적으로 떠안지 않으면, DB엔 조용히 고아 데이터가 쌓인다.
</Callout>

## 언제 쓰나

FK는 공짜가 아니다. `Account` 행을 하나 넣거나 고칠 때마다 DB는 `AccountStatus`에 그 코드가 실제로 있는지 조회해서 확인한다. 평소엔 이 비용이 무시할 만하다. 근데 다음 같은 상황에선 이게 발목을 잡는다.

**1. RI 강제의 성능 비용을 DB가 감당 못 할 때.** 초당 수만 건씩 쓰기가 쏟아지는 테이블, 또는 대량 배치 적재(bulk load)를 돌리는데 행마다 부모 존재를 검증하느라 적재가 기어가는 경우다. 수천만 건을 밀어 넣는 ETL 잡에서 FK 검증은 종종 가장 큰 단일 병목이다.

**2. RI 규칙이 앱마다 다를 때.** 같은 DB를 여러 애플리케이션이 공유하는데, 어떤 앱은 "휴면 계좌도 일단 저장 허용", 다른 앱은 "활성 상태만 허용" 식으로 무결성 규칙이 제각각이라면, DB에 박힌 단일 FK가 오히려 걸림돌이 된다. 이때는 규칙을 각 앱(또는 각 서비스)으로 끌어올리는 게 맞다.

**3. 서비스 경계가 테이블 사이를 가를 때.** 이건 책이 쓰이던 2006년엔 없던 동기다. 모놀리식 DB를 마이크로서비스로 쪼개면서 `Account`는 계좌 서비스가, `AccountStatus`는 다른 서비스가 소유하게 됐다면, 두 테이블을 잇는 물리 FK는 애초에 성립할 수가 없다. 한쪽 서비스가 자기 테이블을 다른 DB로 옮기는 순간 FK는 끊어진다. 이런 경계를 넘는 무결성은 DB 제약이 아니라 애플리케이션 레벨에서 책임지게 된다.

<Callout type="info" title="여기서 짚고 갈 것">
FK를 떼는 게 "무결성을 포기하는" 게 아니다. **무결성 검증의 위치를 DB에서 애플리케이션·서비스 경계로 옮기는** 거다. 책임이 사라지는 게 아니라 이사하는 거다. 이걸 헷갈리면 그냥 데이터 품질을 버리는 결정이 돼버린다.
</Callout>

### 현실 시나리오

은행 시스템을 운영 중이라고 치자. `Account` 테이블엔 `StatusCode` 컬럼이 있고, `AccountStatus` 룩업 테이블을 참조하는 FK가 걸려 있다.

```text
Account (AccountID, CustomerID, Balance, StatusCode) ──FK──▶ AccountStatus (StatusCode, Description)
```

평소엔 아무 문제 없었다. 그런데 분기마다 한 번씩, 외부 코어뱅킹 시스템에서 받은 계좌 스냅샷 수천만 건을 통째로 `Account`에 적재하는 배치가 돈다. 새벽 2시에 시작해서 6시간이 걸린다. DBA가 실행 계획을 까보니, 적재 시간의 상당 부분이 **행마다 `AccountStatus`를 조회해서 `StatusCode`가 유효한지 확인하는 FK 검증**에 쓰이고 있었다.

게다가 데이터를 보낸 코어뱅킹 쪽엔 `'PENDING_REVIEW'` 같은 우리 룩업 테이블에 없는 상태값이 가끔 섞여 들어온다. 그러면 그 행에서 FK 위반 예외(예: 부모 미존재, Oracle `ORA-02291`)가 터지고, 배치 전체가 멈춘다. 새벽 4시에 알람이 울리고, 당직자가 일어나서 어느 행이 문제인지 찾아 수동으로 손본다.

여기서 팀의 결론은 이거였다. "이 적재 경로에선 FK 검증을 DB가 행마다 하는 게 비싸고 위험하다. 검증을 적재 파이프라인 쪽으로 옮기고, FK는 떼자." 성능과 운영 안정성을 위해 품질 검증의 위치를 바꾸기로 한 거다.

## 주의할 점

이 리팩토링의 본질은 한마디로 **성능 대 품질의 맞교환**이다. 공짜 점심은 없다.

<Callout type="warning" title="FK를 떼는 순간 잃는 것">
FK는 갱신마다 검증 비용을 치르는 대가로 **데이터 유효성을 DB가 보증**한다. 이걸 떼면:

- **무효 데이터 유입 위험.** 앱이 쓰기 전에 검증을 안 하면, `AccountStatus`에 없는 `StatusCode`를 가진 `Account` 행이 아무 저항 없이 들어온다. DB는 더 이상 막아주지 않는다.
- **고아 행(orphaned row)이 쌓인다.** 참조 대상이 사라져도 DB가 손대지 않으니, 어느 곳도 가리키지 않는 행이 조용히 누적된다.
- **버그가 늦게 드러난다.** FK가 있으면 잘못된 쓰기는 그 자리에서 예외로 터진다. FK가 없으면 한참 뒤 리포트가 안 맞거나 조인 결과가 비는 식으로, 원인에서 먼 곳에서 터진다. 디버깅이 훨씬 고약해진다.
</Callout>

그래서 "FK를 떼도 되나?"의 진짜 질문은 "**검증을 앱이 빠짐없이 할 수 있나?**"다. 답이 "그렇다, 모든 쓰기 경로가 한 곳을 지나고 거기서 검증한다"이면 떼도 된다. 답이 "글쎄, 여기저기서 막 쓰는데..."이면 떼면 안 된다. 그건 성능 개선이 아니라 데이터 품질을 운에 맡기는 거다.

## 이렇게 한다

크게 세 단계다. **스키마 변경(FK를 어떻게 떼느냐) → 데이터 마이그레이션(이 경우 거의 없음) → 접근 프로그램 수정(검증 책임을 누가 떠안느냐)**. 마지막 단계가 사실상 이 리팩토링의 전부라고 봐도 된다.

### 1단계: 스키마 변경 — DROP 말고 DISABLE

FK를 떼는 방법은 둘이다.

```sql
-- 방법 A: 완전 제거 (DROP)
ALTER TABLE Account DROP CONSTRAINT FK_Account_AccountStatus;

-- 방법 B: 비활성화 (DISABLE) — 권장
ALTER TABLE Account DISABLE CONSTRAINT FK_Account_AccountStatus;
```

둘 다 "더 이상 강제하지 않는다"는 결과는 같다. 하지만 **DISABLE을 권장한다.** 이유는 단순하다. DISABLE은 강제는 안 하되 **제약 정의 자체는 스키마에 남는다.** 즉 `Account.StatusCode`가 `AccountStatus`를 참조한다는 **관계가 문서로 보존**된다.

이게 왜 중요하냐면, 6개월 뒤에 이 스키마를 처음 보는 신입이 "어, `Account`랑 `AccountStatus`가 무슨 관계지?"라고 물었을 때, DROP해버렸으면 그 관계는 어디에도 안 남아 있다. ERD 자동 생성 도구도, DB 메타데이터를 읽는 ORM 인트로스펙션도 그 연결을 못 본다. DISABLE이면 비활성 상태라도 제약이 카탈로그에 남아 있어서 "원래는 이런 관계였고, 지금은 성능 때문에 강제만 꺼둔 상태"라는 의도가 읽힌다.

<Callout type="note" title="DB 제품마다 다른 점">
`DISABLE CONSTRAINT`는 Oracle, SQL Server 등에서 쓰는 문법이고, PostgreSQL은 FK를 "끄는" 직접 명령이 없다. PostgreSQL에선 (1) 그냥 `DROP CONSTRAINT`하거나, (2) 떼기 전에 `NOT VALID`로 만들어 신규 행만 검증을 건너뛰게 하는 식의 다른 결을 쓴다. 핵심은 문법이 아니라 **"관계의 의도를 어딘가에 명시적으로 남겨라"**다. PostgreSQL이라 DISABLE이 없으면, 마이그레이션 스크립트 주석이나 스키마 문서에 "FK_Account_AccountStatus를 성능 사유로 제거함, 검증은 AccountWriter에서 담당"이라고 남기는 게 그 역할을 대신한다.
</Callout>

이 변경은 손으로 콘솔에 치는 게 아니라 **마이그레이션 도구로** 버전 관리하는 게 현대 실무다. Flyway/Liquibase(JVM), Alembic(Python), 또는 ORM 마이그레이션(Prisma, TypeORM, Rails)에 올려서, "언제 누가 왜 이 FK를 뗐는지"를 코드 히스토리로 남긴다.

```sql
-- V59__drop_fk_account_status.sql (Flyway 마이그레이션 예시)
-- 사유: 분기 배치 적재 성능. RI 검증은 AccountWriter 서비스로 이관.
-- 관계는 보존하되 강제만 해제(DISABLE).
ALTER TABLE Account DISABLE CONSTRAINT FK_Account_AccountStatus;
```

### 2단계: 데이터 마이그레이션 — 없음

이 리팩토링엔 데이터를 옮기거나 고치는 단계가 없다. FK를 떼도 기존 행은 그대로다. 떼기 전까지 FK가 강제되고 있었으니 현재 데이터는 이미 다 유효한 상태라, 손볼 게 없다.

(반대로 FK를 *추가*하는 리팩토링이라면 기존 무효 데이터를 먼저 청소하는 마이그레이션이 한참 필요하다. 떼는 건 그게 없으니 그 점은 편하다.)

### 3단계: 접근 프로그램 수정 — 여기가 진짜다

FK가 사라졌으니, **이제 무결성을 누군가 챙겨야 한다.** 이게 이 리팩토링의 핵심 작업이다. 두 가지를 한다.

<Steps>
<Step title="앱이 쓰기 전에 RI를 검증하게 한다">
FK가 하던 "이 StatusCode가 AccountStatus에 있나?" 검사를, 이제 애플리케이션 쓰기 경로에서 한다. 핵심은 **모든 쓰기가 이 검증을 반드시 지나게** 만드는 것. 그래서 쓰기 진입점을 한 곳(리포지토리/서비스)으로 모으는 게 중요하다.
</Step>
<Step title="DB 예외 처리 코드를 재작업한다">
예전엔 "FK 위반 예외(부모 미존재 등)"를 잡아서 처리하던 코드가 있었을 거다. 이제 DB는 그 예외를 던지지 않는다. 그 catch 블록은 죽은 코드가 되거나, 앱이 직접 던지는 검증 실패 예외를 잡는 코드로 바뀌어야 한다.
</Step>
</Steps>

**Before — DB의 FK에 검증을 맡기던 코드.** 그냥 넣고, 터지면 DB 예외를 잡았다.

```typescript
async function createAccount(account: Account): Promise<void> {
  try {
    await db.insert("Account", account);
  } catch (e) {
    // FK 위반(부모 StatusCode 미존재)을 DB가 던져주면 그걸 잡았다
    if (e.code === "ORA-02291") {
      throw new InvalidStatusError(account.statusCode);
    }
    throw e;
  }
}
```

**After — 앱이 쓰기 전에 직접 검증한다.** DB는 더 이상 막아주지 않으니, 내가 먼저 확인한다.

```typescript
// AccountStatus는 행 수가 적은 룩업 테이블이라 메모리 캐싱이 잘 먹힌다.
// 부팅 시 한 번 로드하고, 변경이 드무니 TTL 캐시로 충분하다.
class StatusValidator {
  private validCodes = new Set<string>();

  async load(): Promise<void> {
    const rows = await db.query("SELECT StatusCode FROM AccountStatus");
    this.validCodes = new Set(rows.map((r) => r.statusCode));
  }

  isValid(code: string): boolean {
    return this.validCodes.has(code);
  }
}

async function createAccount(account: Account): Promise<void> {
  // DB가 안 막아주니 내가 먼저 막는다
  if (!statusValidator.isValid(account.statusCode)) {
    throw new InvalidStatusError(account.statusCode);
  }
  await db.insert("Account", account);
}
```

여기서 책이 짚는 실무 팁이 그대로 유효하다. **참조 테이블이 작으면 메모리에 캐싱하라.** `AccountStatus` 같은 룩업 테이블은 행이 수십 개뿐이고 거의 안 바뀐다. 매 쓰기마다 DB를 때려서 존재 확인하면 그건 FK를 뗀 의미가 없다(검증 비용을 앱으로 그냥 옮긴 것뿐). 작은 룩업은 앱 메모리에 올려두고 거기서 확인하면, FK 검증의 DB 왕복 비용 없이 같은 보증을 얻는다.

### 무결성 검증을 어디에 둘 것인가

같은 "앱 레벨 검증"이라도 두는 위치에 따라 결이 다르다. 상황에 맞게 고르면 된다.

<TabsList>
<TabsTrigger value="repo">단일 쓰기 경로</TabsTrigger>
<TabsTrigger value="service">서비스 경계</TabsTrigger>
<TabsTrigger value="audit">사후 감지</TabsTrigger>
</TabsList>

<TabsContent value="repo">
**모든 쓰기가 한 리포지토리/서비스를 지난다면**, 그 한 곳에 검증을 박는 게 가장 깔끔하다. FK가 하던 일을 그 진입점이 그대로 대신한다. 전제는 "정말 모든 쓰기가 여기를 지나는가"다. 누군가 우회해서 raw SQL로 직접 인서트하면 검증이 뚫린다. 그래서 이 패턴은 **쓰기 경로를 단일화하는 규율**과 한 세트다.
</TabsContent>

<TabsContent value="service">
**마이크로서비스로 쪼개서 `Account`와 `AccountStatus`가 다른 서비스 소유라면**, 검증은 서비스 경계에서 한다. 계좌 서비스가 상태 서비스에 "이 코드 유효해?"를 물어보거나(동기 호출), 상태 목록을 주기적으로 받아 로컬 캐시로 들고 있다가 검증한다(비동기). 경계를 넘는 무결성은 물리 FK로 못 거니까, 이건 선택이 아니라 필수다. 상태 코드가 바뀌면 CDC(Debezium)나 이벤트로 변경을 흘려보내 캐시를 갱신하는 식으로 일관성을 유지한다.
</TabsContent>

<TabsContent value="audit">
**FK를 뗐지만 100% 막을 자신은 없을 때**, 사전 검증과 별개로 사후 감지 그물을 깐다. 주기적으로 고아 행을 찾는 정합성 검사 쿼리를 돌리고, 걸리면 알람을 띄운다.

```sql
-- 야간 정합성 검사: AccountStatus에 없는 코드를 쓰는 Account 행 색출
SELECT a.AccountID, a.StatusCode
FROM Account a
LEFT JOIN AccountStatus s ON a.StatusCode = s.StatusCode
WHERE s.StatusCode IS NULL;
```

이건 검증을 대체하는 게 아니라 **보강**이다. 앱 검증은 사전에 막고, 이 쿼리는 그 그물을 빠져나간 게 있는지 사후에 잡는다.

## 정리

외래 키 제약 제거는 단순한 리팩토링처럼 보이지만, 실제로는 **데이터 무결성의 책임을 DB에서 다른 곳으로 이사시키는** 의사결정이다.

> **FK를 떼는 건 검증을 없애는 게 아니라, 검증을 옮기는 거다. 옮길 곳을 정하지 않고 떼면 그건 그냥 품질을 버리는 거다.**

요점만 다시 추리면 이렇다.

- 동기는 거의 항상 **성능**(검증 비용을 못 감당) 아니면 **규칙 분기**(앱·서비스마다 무결성 규칙이 다름)다. 현대엔 **서비스 경계**가 세 번째 동기로 추가됐다.
- 본질은 **성능 대 품질**의 맞교환이다. 얻는 게 있으면 잃는 게 있다.
- 뗄 땐 DROP보다 **DISABLE을 권장**한다. 강제는 끄되 관계는 문서로 남기기 위해서다. PostgreSQL처럼 DISABLE이 없으면 주석·문서로 의도를 남긴다.
- 데이터 마이그레이션은 없다. 대신 **접근 프로그램 수정이 작업의 전부**다. 앱이 쓰기 전 검증을 떠안고(작은 룩업은 캐싱), 죽은 DB 예외 처리 코드를 정리한다.
- 자신이 없으면 사후 정합성 검사로 그물을 한 겹 더 깐다.

FK를 떼야 할 진짜 이유가 있다면 떼는 게 맞다. 다만 떼기 전에 항상 같은 질문을 던지면 된다. **"그래서, 검증은 이제 누가 하지?"** 이 질문에 또렷이 답할 수 있으면, 그 제거는 안전하다.
