## 기술 FOMO: 새 기술이 무서운 이유

개발자 세계에는 독특한 공포증이 있음. FOMO — Fear Of Missing Out. 새 프레임워크가 나올 때마다 스멀스멀 올라오는 그 불안감. "나만 이거 안 쓰는 거 아니야?" "이거 안 배우면 도태되는 거 아니야?" "트위터에서 다들 좋다는데..."

<Callout type="warning" title="기술 FOMO 자가 진단">
다음 증상이 있으면 기술 FOMO임:
- Hacker News에 새 프레임워크 글이 올라오면 심장이 두근거림
- 새 도구의 "Getting Started"를 3개 이상 동시에 진행 중
- 현재 프로젝트의 기술 스택이 항상 "낡았다"고 느껴짐
- "다음 프로젝트에서는 꼭 이걸로 할 거야"를 분기마다 말함
- 사이드 프로젝트가 5개 있는데 전부 TODO 앱이고 전부 다른 기술 스택임
</Callout>

심리 메커니즘은 단순함. 새 기술을 도입한 첫 달엔 만족도가 99%임("와 이거 진짜 혁명이다!"). 6개월쯤 지나면 불편한 점이 보이고(75%), 1년쯤 되면 "보일러플레이트 너무 많다"(55%), 1년 반이면 "옆 동네 Svelte 좋아 보이는데...?"(40%). 그러다 Hacker News 글 하나 보면 만족도가 0%로 직행하면서 외침. "리라이트 합시다!" 그리고 새 프레임워크를 도입하면? 다시 첫 달 99%부터 사이클이 반복됨. 남의 떡은 언제나 커 보임.

## "이번엔 진짜 좋다" 연대기

프론트엔드 프레임워크의 리라이트 역사를 보면 데자뷰가 옴.

```
2006: "jQuery 쓰세요. DOM 조작이 너무 쉬워요!"     → 다 jQuery로 리라이트
2010: "Backbone.js가 MVC를 브라우저에!"           → jQuery 앱이 Backbone으로
2012: "AngularJS! 진짜 프레임워크! 양방향 바인딩!"  → Backbone 앱이 Angular로
2015: "React가 Virtual DOM으로 성능을 해결!"       → Angular 앱이 React로
2016: "Angular 2는 완전히 새로 만들었어요 (하위호환 없음)" → AngularJS 팀: "??????"
2021: "Next.js가 풀스택의 미래에요!"               → React 앱이 Next.js로
2023: "App Router가 패러다임을 바꿨어요!"          → Pages Router 앱이 App Router로
2025: "React Server Components가 진짜 미래에요!"   → 아직 이전 리라이트도 안 끝났는데?
```

빌드 도구도 똑같음. Grunt(2012) → Gulp(2014) → Webpack(2015) → Parcel(2018) → Snowpack(2020) → Vite(2021) → Turbopack(2022) → Rspack(2023). 매번 "이번엔 설정이 간단하다"고 했지만, 개발자는 여전히 `vite.config.ts` 100줄을 작성하고 있음.

<Callout type="error" title="리라이트 지옥의 현실">
위 타임라인에서 실제로 리라이트를 **완료한** 비율: 약 30%. 나머지는 — 40%는 리라이트 중간에 새 프레임워크가 나와서 또 갈아탔고, 20%는 예산/시간 부족으로 중단해 두 기술이 혼재하는 괴물을 낳았고, 10%는 리라이트했더니 원래보다 느려져서 롤백함.
</Callout>

## 아무도 안 하는 진짜 비용 계산

리라이트를 주장하는 사람은 항상 이점만 말하고 비용은 안 말함. 비용은 두 종류임.

**직접 비용**은 눈에 보임 — 개발 시간, 못 만든 기능의 기회비용, 전체 QA 재수행, 데이터 마이그레이션. 그리고 여기엔 철칙이 하나 있음. 리라이트 주장자는 "2명이 3개월이면 돼요"라고 하지만, 실제로 걸리는 시간은 그 추정의 **3~5배**임. "테스트도 같이 옮기니까 금방"이라던 게 QA 사이클 두 바퀴가 되고, "아키텍처가 좋으니 버그도 적을 것"이라던 게 대량의 새 버그가 됨.

**간접 비용**은 눈에 안 보여서 더 무서움. 그중 최악은 **비즈니스 로직의 유실**임. 이건 코드로 봐야 와닿음.

```typescript
// 리라이트 전: "왜 이렇게 지저분하지?" 싶은 레거시
function calculateDiscount(order: Order): number {
  let discount = 0;
  // 2019년 마케팅 이벤트: VIP + 10만원 이상 추가 할인
  if (order.customer.isVIP && order.total > 100000) discount += 0.05;
  // 2020년 세법 변경 대응 (국세청 가이드 2020-15호)
  if (order.region === 'domestic' && order.includesTax) discount = Math.min(discount, 0.10);
  // 2021년 버그 수정: 음수 할인 방지 (JIRA-4521, 고객이 돈을 더 내던 버그)
  discount = Math.max(0, discount);
  return discount;
}

// 리라이트 후: "깔끔하게 다시 짰습니다!"
function calculateDiscount(order: Order): number {
  return order.customer.isVIP ? 0.05 : 0;
}
```

깔끔해 보이는 한 줄짜리 버전은 세금 로직, 음수 할인 방지, 지역별 할인 제한을 전부 날려버렸음. 3년간 쌓인 엣지 케이스 처리 경험이 통째로 증발한 거임. **"깔끔한 코드" ≠ "올바른 코드".** 레거시 코드가 지저분한 데는 보통 이유가 있음.

<Callout type="warning" title="체스터턴의 울타리">
"울타리가 왜 세워져 있는지 이해하기 전까지는 울타리를 제거하지 마라." — G.K. 체스터턴

레거시 코드의 이상한 조건문도 마찬가지임. 왜 있는지 이해하기 전까지는 "정리"하지 마셈. 그게 길을 막는 울타리가 아니라, 누군가를 지키던 방어벽일 수도 있음.
</Callout>

## 리라이트가 거의 항상 실패하는 이유

조엘 스폴스키가 2000년에 쓴 글의 제목이 「You Should Never Do(절대로 하지 말아야 할 것)」임. 그 "절대로 하지 말아야 할 것"이 바로 **코드를 처음부터 다시 짜는 것**임. 그는 이걸 "기업이 저지를 수 있는 단 하나의 최악의 전략적 실수"라고까지 불렀음. 이유는 위 `calculateDiscount`가 다 설명함 — 지저분한 코드 줄줄이 안에는 누군가 실제 버그를 만나 고친 흔적이 박혀 있고, 새로 짜면 그 흔적이 사라지면서 같은 버그를 다시 만나게 됨.

프레드 브룩스의 **두 번째 시스템 효과**도 기억할 만함. 첫 시스템을 만든 사람이 두 번째를 만들 때, "이번엔 제대로"라는 욕심에 온갖 기능을 욱여넣어 결국 비대하고 망가진 시스템을 만든다는 거임. 리라이트는 정확히 이 함정에 빠지기 쉬움. "이번엔 처음부터 완벽하게"라는 자신감이 두 번째 시스템 효과의 연료임.

## 현명한 기술 선택의 기준

"새 기술이 더 좋으니까"는 기술 선택의 근거가 아님. 새 기술을 도입하기 전에 스스로 물어볼 질문들이 있음.

1. **현재 기술의 문제가 명확한가?** "그냥 좀 불편함"은 근거가 안 됨. "메모리 릭으로 매주 서버 재시작"은 근거가 됨.
2. **새 기술이 그 문제를 푸는가?** "DX가 좋아요"는 약함. "빌드가 10분에서 30초로 줄어듦"은 강함.
3. **새 기술이 가져올 새 문제는?** 모든 기술은 트레이드오프. 새로 생기는 문제가 더 크면 의미 없음.
4. **팀이 학습할 여유가 있는가?** 마감 3개월 전 새 프레임워크 도입 = 자살 행위.
5. **생태계가 충분히 성숙했는가?** bleeding-edge에서 프로덕션을 굴리면 스택오버플로우에 질문해도 답이 없음.
6. **점진적 마이그레이션이 가능한가?** 빅뱅 리라이트는 실패 확률 70%, 점진적은 30%.
7. **3년 후에도 살아있을 기술인가?** npm 패키지 중 1년 후까지 유지보수되는 비율은 30% 미만.

여기서 떠올릴 격언이 **"지루한 기술을 선택하라(Choose Boring Technology)"**임. 댄 매킨리의 유명한 주장인데, 회사가 쓸 수 있는 "혁신 토큰"은 한정돼 있으니 핵심이 아닌 곳엔 검증된 지루한 기술을 쓰고, 정말 차별화가 필요한 한두 곳에만 토큰을 쓰라는 거임. 트위터에서 핫한 기술은 대개 토큰을 낭비하는 쪽임.

기술 성숙도 곡선(가트너 하이프 사이클)도 같은 얘길 함. 리라이트러는 항상 **과대광고기**의 정점에서 뛰어듦. 정작 도입하기 좋은 시점은 환멸기와 계몽기를 지나 **안정기**임 — 스택오버플로우에 답변이 충분하고 "그냥 쓰면 됨" 소리가 나오는 때.

```
흥분도
  ▲      ★ ← 리라이터가 뛰어드는 지점 (과대광고기)
  │     ╱ ╲
  │    ╱   ╲___
  │   ╱        ╲___        ★ ← 도입하기 현명한 지점 (안정기)
  │  ╱             ╲______╱──────
  └────────────────────────────────▶ 시간
     과대광고  환멸기  계몽기   안정기
```

<Callout type="success" title="현명한 기술 선택 원칙">
1. **유행 대신 안정성**: 트위터에서 핫한 것보다 프로덕션에서 검증된 것
2. **전체 리라이트 대신 점진적 전환**: 스트랭글러 패턴(Strangler Fig) 활용
3. **기술이 아니라 문제에 집중**: "이 기술이 좋다"가 아니라 "이 문제를 풀어야 한다"
4. **1년 규칙**: 새 기술은 최소 1년 관망. 얼리어답터들의 피드백을 학습
5. **탈출구 확보**: 도입 시 롤백 플랜 필수
</Callout>

## 실전: "문제 형태로 고른다"는 게 뭔지 보고 싶다면

"기술이 아니라 문제에 집중하라"는 말은 멋있지만 추상적임. 실제로는 어떻게 하느냐? 핵심은 프레임워크 *이름*으로 비교하길 멈추고, 질문을 **내 문제의 형태**로 바꾸는 거임. 같은 사고법을 백엔드 아키텍처에서 끝까지 밀어붙인 글 세 편을 붙여둠. FOMO에 백신 한 방 맞는다 생각하고 읽어보면 좋음.

- [**tRPC는 어디서 1급 시민인가**](/docs/dev/nodejs/architecture/trpc) — "tRPC 쓸까 REST 쓸까?"는 거의 항상 잘못 던진 질문이라는 데서 출발함. 진짜 질문은 "누가 이 API를 소비하는가"와 "백엔드를 어떤 형태로 둘 것인가"임. 프레임워크 이름이 아니라 **request lifecycle을 누가 소유하는가**라는 *형태*가 답을 정함. 게다가 이 글의 결론은 우리 얘기랑 직결됨 — "지금 어느 쪽 DX가 좋은가(가장 빨리 바뀌는 축) 말고, consumer가 누구이고 앞으로 누가 더 붙을까(가장 오래 가는 축)로 정하라." 이게 곧 "유행 말고 본질로 고르라"의 다른 표현임.

- [**Next.js 하나면 되지 않나요?**](/docs/dev/nodejs/architecture/nextjs-fullstack-limits) — "백엔드를 왜 분리해?"라는 질문에 "대체로 분리는 이른 최적화"라고 먼저 인정하고 들어감. 분리(리라이트의 사촌임)는 트래픽이나 유행이 아니라 **횡단 관심사가 프레임워크의 지지를 요구하는 지점**에서만 정당해짐. 결정적 문장이 하나 있음 — *"분리는 목표가 아니라 증상에 대한 처방이다."* 이거 그대로 리라이트에 복붙해도 됨. 리라이트도 목표가 아니라 증상에 대한 처방이어야 함.

- [**Hono는 무엇에 베팅했나**](/docs/dev/nodejs/architecture/hono) — "요즘 핫한 Hono"를 하이프로 소비하지 않고 구조적으로 읽는 모범 사례임. "빠르고 가볍고 어디서나 돈다"는 표면 말고, 그게 *어떤 설계 결정의 배당금이고 무엇을 청구서로 치르는지*를 봄. 새 프레임워크를 평가하는 올바른 자세가 정확히 이거임 — "트위터에서 핫하다"가 아니라 "무슨 트레이드오프에 베팅한 거지?"

세 글의 공통 교훈을 한 줄로 줄이면 이거임. **프레임워크는 켜고 끄는 토글이 아니라 분기(branch)다.** "이거 도입할까 말까"가 아니라 "내 문제가 어느 형태인가"를 묻는 순간, FOMO가 비집고 들어올 틈이 사라짐. 앞에서 말한 "최고의 트레이드오프만 존재한다"가 바로 이 세 글이 처음부터 끝까지 증명하는 명제임.

## 그럼 리라이트는 언제 정당한가

물론 리라이트가 정답인 경우도 있음. 모든 리라이트가 나쁜 건 아님.

**리라이트가 합리적인 신호**: 프레임워크가 EOL이라 보안 패치가 안 나옴, 현재 아키텍처로는 10만 동시접속을 못 버팀, 의존성 보안 패치가 불가능, 근본 요구사항이 바뀜(SPA였는데 SSR이 필요해짐), 규모가 변함(5명용 코드를 50명이 개발), 해당 기술 개발자를 더 이상 채용 못 함.

**리라이트가 합리적이지 않은 신호**: "DX가 별로임"(불편해도 동작은 함), "코드가 못생김"(못생겨도 올바르면 됨), "기술이 오래됨"(오래됐지만 안정적이면 장점), "남들 다 씀"(남들이 뭘 쓰든 내 프로젝트와 무관), "이력서에 쓰고 싶어서"(사이드 프로젝트에서 하셈).

리라이트가 정당해도 방법은 **점진적**이어야 함. 빅뱅 리라이트는 새 코드베이스를 통째로 만들어 특정 날짜에 한 번에 전환하는 건데, 그 전환일에 폭발할 확률이 높음. 대신 **스트랭글러 무화과 패턴**을 쓰면, 새 기능만 새 기술로 받고 기존 기능을 하나씩 이전하다가 레거시가 0이 되면 제거함. 언제든 멈출 수 있고(예산 부족 시), 문제가 나도 영향 범위가 제한되고, 기존 서비스는 계속 굴러가고, 팀이 점진적으로 학습할 수 있음. 같은 "계단식으로 가라"를 백엔드 분리 결정에 적용한 구체적 사례가 [Next.js 글의 단계적 경로](/docs/dev/nodejs/architecture/nextjs-fullstack-limits)임 — *Next 단독 → 직접 짠 wrapper 층 → 프레임워크 분리*로, 임계를 넘었을 때만 다음 칸으로 올라감.

## 프레임워크 리라이터 자가 치료법

<Callout type="note" title="리라이트 충동 억제 가이드">
새 프레임워크가 핫해서 리라이트하고 싶을 때:

1. **72시간 규칙**: 충동이 들면 일단 72시간 기다리기. 대부분 사라짐
2. **비용 계산 먼저**: "3명 × 4개월 = 12인월 ≈ N천만원"으로 정량화
3. **사이드 프로젝트에서 먼저**: 정말 좋은지 2주 써보기
4. **프로덕션 레퍼런스 확인**: 비슷한 규모에서 쓰는 사례가 있는지
5. **탈출 비용 계산**: 안 되면 돌아올 수 있는지. 돌아오는 비용이 크면 위험한 베팅
</Callout>

근본 치료는 **현재 스택을 깊이 파는 것**임. "React 안다"에서 멈추지 말고 상태 관리, 성능 최적화, 서버 컴포넌트, 컴파일러까지 — 넓이보다 깊이가 커리어에 훨씬 도움됨. 새 기술은 "알고 있되 쓰지 않는" 관망 모드가 가장 이상적임. 충동적 도입을 막고 싶으면 ThoughtWorks Tech Radar처럼 기술을 적극사용(Adopt)·시험(Trial)·평가(Assess)·보류(Hold)로 분류하고 분기마다 업데이트하는 "기술 레이더"를 만들어보는 것도 방법임.

## 마무리

새 기술은 언제나 매력적임. 그건 당연함. 하지만 프로덕션 코드는 "매력"으로 운영하는 게 아님. "이번엔 진짜 좋다"는 말은 지난 10년간 매년 나왔고, 앞으로 10년도 매년 나올 것임. 5년 후에 살아남은 게 진짜 "진짜"고, 나머지는 Hacker News의 한때의 추억임.

가장 좋은 기술은 "지금 프로덕션에서 잘 돌아가는 기술"임. 두 번째는 "팀 전체가 잘 아는 기술", 세 번째는 "문서가 충분한 기술". 트위터에서 가장 핫한 기술은 이 목록 어디에도 없음.
