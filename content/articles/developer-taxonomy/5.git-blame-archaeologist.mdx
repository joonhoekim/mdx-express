## git blame: 코드 고고학의 시작

모든 버그에는 역사가 있음. 그리고 그 역사를 발굴하는 삽이 바로 `git blame`임. 어떤 개발자에게 blame은 범인을 찾는 수사 도구이고, 어떤 개발자에게는 "아 이거 내가 짠 거였구나..."를 깨닫는 자기 성찰의 시간임.

<Callout type="info" title="git blame의 철학">
git blame의 목적은 **범인 찾기**가 아님. 진짜 목적은 **맥락 복원**임. "이 코드를 왜 이렇게 짰는지"를 이해하려고 커밋 히스토리를 추적하는 것.

blame이라는 이름이 불행한 거지, `git context-history` 같은 이름이었으면 오해가 훨씬 덜했을 것임.
</Callout>

전형적인 고고학 발굴은 이런 식으로 흘러감. 코드에서 `if (user.age > 0 && user.age < 200)` 같은 이상한 조건문을 발견함("나이가 200 미만? 왜 하필 200?"). `git blame`을 찍어 범인 커밋을 찾고, `git show`로 커밋 메시지를 확인하는데 — 메시지가 그냥 **"fix"**임. `git log --follow -p`로 더 거슬러 올라가 보면 상한선이 120 → 150 → 200으로 계속 올라간 흔적이 보임. 알고 보니 QA에서 "200살인 사람도 있을 수 있다"는 엣지 케이스 리포트가 3번 들어왔고 매번 상한을 올린 거였음. 근데 커밋 메시지가 전부 "fix"라 이유를 알 수 없음. JIRA 티켓은 없고, 슬랙 대화는 삭제됐고, 작성한 김개발은 퇴사함. **고고학 실패. 미스터리로 남음.**

## 최악의 커밋 메시지 모음

커밋 메시지는 미래의 나에게 보내는 편지임. 근데 대부분의 개발자는 미래의 자기 자신을 별로 존중하지 않음.

**S티어 — 정보량 제로.** 실제로 야생에서 관측되는 메시지들:

```
a1b2c3d fix
b2c3d4e fix
c3d4e5f fix again
d4e5f6g fix for real this time
e5f6g7h ok now it actually works
f6g7h8i fuck
g7h8i9j undo fuck
h8i9j0k asdf
i9j0k1l .
j0k1l2m wip
```

<Callout type="error" title="커밋 메시지 범죄 기록">
가장 흔한 쓸모없는 커밋 메시지 순위: ① "fix" ② "update" ③ "wip" ④ "." ⑤ "asdf" ⑥ 빈 메시지 ⑦ "커밋"(한국 한정) ⑧ 이모지만 있는 경우 ⑨ "제발 되라" ⑩ 욕설.
</Callout>

**A티어 — 오해를 유발하는 메시지.** S티어보다 어떤 면에선 더 위험함. "사소한 변경"이라고 써놓고 실제론 47개 파일에 2,300줄을 바꾼 경우, "오타 수정"이라더니 "겸사겸사" 비즈니스 로직까지 손댄 경우(리뷰어는 오타인 줄 알고 LGTM), "성능 개선"이라며 캐시를 추가했는데 무효화 로직이 없어서 3일 후 "왜 데이터가 안 바뀌죠?" 장애가 난 경우, "코드 정리"라며 "안 쓰는 것 같은" 함수를 지웠는데 그게 월 1회 배치에서 쓰이던 거라 월말에 터진 경우. 이런 메시지는 정보가 없는 정도가 아니라 **틀린 정보를 줌.**

**B티어 — TMI 메시지.** 반대로 너무 많은 것도 문제임. "2024년 3월 15일 오후 3시 27분 회의실 B에서 진행된 스프린트 리뷰에서 박 PM이 요청한 프로필 사진 크기를 100px에서 120px로... 참고로 오늘 점심은 된장찌개였습니다." 유용한 정보는 "프로필 사진 100px → 120px" 하나고 나머지는 전부 노이즈임.

## 좋은 커밋 메시지 작성법

널리 쓰이는 형식은 **Conventional Commits**임. `<type>(<scope>): <subject>` 한 줄 요약 뒤에 본문과 푸터를 붙이는 구조. 타입은 `feat`(새 기능), `fix`(버그 수정), `refactor`(기능 변화 없는 리팩토링), `docs`, `test`, `chore`, `perf`, `style` 정도를 씀.

```
fix: prevent negative inventory count in concurrent orders

When multiple orders are placed simultaneously for the same product,
the inventory check and deduction were not atomic, leading to
negative inventory counts.

Added Redis distributed lock to ensure atomic check-and-deduct.

Closes #4521
```

이 메시지가 좋은 건 — 타입이 명확하고(fix), 요약이 구체적이고, 본문에서 **왜(WHY)** 바꿨는지 설명하고, 어떻게 해결했는지 간략히 적고, 관련 이슈를 참조하기 때문임.

여기서 핵심 원칙이 **WHAT보다 WHY**임. "change timeout from 30 to 60"은 WHAT만 있음 — 타임아웃이 60이라는 건 코드 diff를 보면 알 수 있는 정보임. 반면 "increase API timeout to 60s: payment gateway takes 40-50s during peak hours, 30s was causing false failures"는 WHY가 있음 — 결제 게이트웨이가 피크 시간에 느리다는 건 **코드 어디에도 없는 정보**임. 좋은 커밋 메시지는 코드를 봐도 모르는 걸 적음.

<Callout type="success" title="커밋 메시지 황금률">
**미래의 나에게 보내는 편지라고 생각하기.**

6개월 후 git blame으로 이 줄을 찾았을 때, 커밋 메시지만 읽고 "아, 그래서 이렇게 한 거구나"를 이해할 수 있으면 됨. 그때의 나는 지금의 맥락을 전혀 기억하지 못함. JIRA 번호도, 슬랙 대화도 다 까먹음. 커밋 메시지가 유일한 단서임.
</Callout>

## 고고학자의 도구 상자

git blame이 삽이라면, 다른 git 명령어들은 발굴에 필요한 나머지 장비임. 이 명령어들은 외워둘 가치가 있음.

**git log — 연대기 작성.**

```bash
git log --oneline --follow -- src/services/payment.ts  # 파일 변경 이력
git log -S "calculateDiscount" --oneline               # 특정 함수가 추가/삭제된 커밋
git log --author="김개발" --oneline                     # 특정 사람의 커밋
git shortlog -sn --no-merges                           # 누가 얼마나 기여했는지
```

**git bisect — 범인 찾기.** "이 버그가 언제부터 있었지?"를 이진 탐색으로 찾음.

```bash
git bisect start
git bisect bad                 # 현재: 버그 있음
git bisect good v2.1.0         # 이 버전: 정상이었음
# git이 중간 커밋을 자동 체크아웃 → 테스트 → bad/good 반복 → O(log n)으로 범인 발견

git bisect run npm test        # 테스트로 완전 자동화도 가능
```

**git reflog — 타임머신.** "실수로 브랜치 삭제했는데...", "hard reset 잘못했는데..." 할 때 구해줌.

```bash
git reflog
# def5678 HEAD@{1}: commit: feat: add payment  ← 이걸 살려야 함
git checkout def5678           # 삭제된 커밋으로 복귀 가능 (reflog는 기본 90일 보관)
```

**고급 blame 테크닉.** 옵션을 조합하면 진짜 원저자를 찾아낼 수 있음.

```bash
git blame -L 42,60 src/payment.ts   # 특정 줄 범위만
git blame -w src/payment.ts         # 공백 변경 무시 (포맷팅 커밋 건너뛰기)
git blame -M src/payment.ts         # 코드 이동 추적
git blame -w -M -L 42,60 src/payment.ts  # 셋 다 조합 → 포맷팅에 묻힌 원저자 발굴
```

<Callout type="note" title="git blame 에티켓">
blame으로 "범인"을 찾았을 때 —

**올바른 접근**: "이 코드의 배경이 궁금한데, 혹시 기억나세요?"
**나쁜 접근**: "이거 니가 짰는데 왜 이따위로 짠 거야?"

blame은 맥락 복원 도구이지 책임 추궁 도구가 아님. 3년 전의 코드로 지금의 사람을 판단하면 안 됨. 3년 전의 나와 지금의 나도 다른 사람인 것처럼.
</Callout>

## 좋은 히스토리를 만드는 습관

**논리적 단위로 커밋하기.** 하루 끝에 `git add .` 하고 "오늘 한 거"로 30개 파일을 한 커밋에 욱여넣으면 나중에 blame해도 도움이 안 됨. 검증 로직 변경은 검증 로직대로, 서비스 레이어 추출은 그것대로 따로 커밋하면 각 커밋이 하나의 논리적 변경 단위가 되어 추적이 쉬움.

**PR 전에 히스토리 정리하기.** 작업 중엔 "wip: 작업 중", "fix: 오타", "wip: 계속" 같은 커밋이 쌓이기 마련임. 이런 건 PR 올리기 전에 interactive rebase로 squash/reword해서 의미 있는 커밋 두어 개로 정리하면 됨.

**커밋 템플릿 쓰기.** `~/.gitmessage` 파일을 만들고 `git config --global commit.template ~/.gitmessage`로 등록해두면, 매번 빈 에디터 대신 `<type>: <subject>` / `Why:` / `Refs:` 같은 구조화된 양식이 떠서 좋은 메시지를 쓰도록 유도됨.

## git blame을 넘어서: 기록의 철학

코드는 컴퓨터에게 하는 명령이기도 하지만, 동시에 미래의 개발자(나 포함)에게 보내는 메시지임. `const x = y * 1.1`은 6개월 후의 나에게 "what?"을 안기지만, `TAX_RATE`라는 이름과 그걸 설명하는 커밋 메시지는 맥락을 전달함. **코드 + 커밋 메시지 + PR 설명**, 이 셋이 합쳐져야 미래의 고고학자가 맥락을 복원할 수 있음.

여기서 필 칼튼의 유명한 농담이 떠오름. **"컴퓨터 과학에 어려운 건 딱 두 가지뿐이다. 캐시 무효화, 그리고 이름 짓기."** 농담 같지만 핵심을 찌름 — 좋은 코드의 절반은 결국 *무언가에 올바른 이름을 붙이는 일*이고, 좋은 커밋 메시지도 결국 *변경에 올바른 설명을 붙이는 일*임. 이름과 설명, 둘 다 "남이 읽을 것"을 전제로 함. 그리고 코드는 쓰이는 횟수보다 **읽히는 횟수가 훨씬 많음.**

기록이 잘 돼 있으면 버그의 근본 원인을 찾고, 반복되는 실수를 피하고("5년 전에도 같은 시도를 했다가 실패한 기록이 있네"), 비즈니스 맥락을 이해하고("이 예외 처리는 2022년 세법 변경 때문이었구나"), 기술 부채의 이유를 파악할 수 있음. 반대로 기록이 없는 코드는 — 커밋이 전부 "fix"고, JIRA 티켓은 삭제됐고, 원작자는 퇴사한 코드는 — "건드리면 뭐가 터질지 모르는" 영원한 레거시가 됨.

<Callout type="warning" title="퇴사한 사람의 코드">
"이 코드 짠 사람 퇴사했는데요..."는 개발팀에서 가장 무서운 말 중 하나임. 코드는 남았는데 맥락이 사라진 상태.

방지하는 방법은 정해져 있음 — 좋은 커밋 메시지, 코드 리뷰(리뷰어도 맥락을 공유하게 됨), 핵심 결정의 문서화(ADR), 그리고 "왜 이렇게 했는지"를 적은 주석. 사람은 떠나지만 기록은 남음. 기록이 좋으면 사람이 떠나도 코드는 살아남음.
</Callout>

## 오늘부터 시작하기

"fix"만 쓰던 사람이 좋은 커밋 메시지를 쓰기 시작하는 건 코드 품질 향상의 첫 단계임. 거창하게 할 필요 없음. "fix" → "fix: resolve null pointer when user has no profile image (2024-01 이전 가입자엔 profile_image 없음)" 정도면 충분함. 참고로 "."은 커밋이 아니라 `git stash` 쓸 자리고, "asdf"는 키보드 테스트니까 메모장에서 하면 됨.

<Callout type="success" title="최소한의 규칙">
완벽할 필요 없음. "fix"보다 나은 한 줄이면 됨.

1. **무엇을** 고쳤는지 한 줄로 (50자 이내)
2. **왜** 고쳤는지 한 줄 추가 (선택이지만 강추)
3. 관련 **이슈 번호**가 있으면 추가

이것만 해도 상위 30%의 커밋 메시지임. "fix" 쓰는 사람보다 100배 나음.
</Callout>
