요즘 백엔드 얘기에 Hono가 부쩍 자주 등장한다. 보통 따라붙는 수식은 "초고속, 초경량, 어디서나 돈다"다. 다 맞는 말이지만, 그건 *표면*이다. 이 셋은 따로 노력해서 얻은 세 가지 장점이 아니라, **하나의 설계 결정에서 동시에 떨어진 결과**다.

그 결정은 한 문장으로 줄어든다 — **Hono는 플랫폼 위에 자기만의 추상화를 만들지 않는다.** 이 글은 그 한 줄이 어떻게 이식성·속도·thin함을 한꺼번에 설명하는지, 그리고 그 베팅이 무엇을 대가로 치르는지를 짚는다.

## 왜 지금 Hono인가 — 런타임이 갈라졌다

몇 년 전만 해도 "Node.js 서버"가 사실상 유일한 선택지였다. 지금은 아니다. Cloudflare Workers, Deno, Bun, Vercel·Netlify의 엣지 런타임, AWS Lambda… 자바스크립트가 도는 런타임이 갈라졌다. 그런데 **Express는 Node 전용**이다. Node의 `http` 모듈과 `req`/`res` 객체에 묶여 있어서, Workers나 Deno에 그대로 올라가지 않는다.

여기서 수요가 생긴다 — *같은 서버 코드를 어떤 런타임에든 올리고 싶다.* Hono가 뜬 진짜 이유는 "빠르다"가 아니라 이 파편화에 대한 답이라는 데 있다. 그리고 그 답이 곧 위에서 말한 한 가지 결정이다.

## 한 가지 결정 — Web 표준을 계약으로 삼는다

대부분의 프레임워크는 **자기만의 요청/응답 객체를 발명한다.** Express의 `req`/`res`가 그렇고, 무거운 프레임워크일수록 그 위에 더 두꺼운 추상화(컨트롤러, DTO, 데코레이터)를 얹는다. 편하지만, 그 추상화는 특정 런타임·특정 프레임워크에 코드를 묶는다.

Hono는 반대로 간다. 입력은 Web 표준 `Request`, 출력은 Web 표준 `Response` — 그게 외부 계약의 전부다. Hono 앱의 본질은 결국 이 시그니처 하나로 환원된다.

```ts
// Hono 앱의 정체 — Web 표준 fetch 핸들러 그 자체
app.fetch(request: Request): Response | Promise<Response>
```

`Request`를 받아 `Response`를 돌려주는 함수. 이건 Hono가 발명한 게 아니라 **플랫폼이 이미 정의한 표준**(Fetch API)이다. 그래서 같은 앱이 런타임을 가리지 않는다.

```ts
import { Hono } from 'hono'

const app = new Hono()
app.get('/', (c) => c.text('Hello'))

export default app   // Cloudflare Workers · Bun · Deno — app.fetch를 그대로 호출
```

```ts
import { serve } from '@hono/node-server'
serve(app)           // Node처럼 표준 fetch 핸들러를 모르는 런타임만 어댑터 한 줄
```

핵심은 이거다 — **이식성은 Hono가 따로 구현한 "기능"이 아니라, 추상화를 거절한 데서 나온 결과다.** 모든 모던 런타임이 이미 `Request`/`Response`를 말하니까, 그 위에 아무것도 덧씌우지 않으면 자동으로 어디서나 돈다. "빠르다"와 "가볍다"도 같은 뿌리다 — 두꺼운 추상화 레이어가 없으니 작고(초경량), 거칠 게 적으니 빠르다.

<Callout type="info" title="🌱 처음이라면 — 'Web 표준 Request/Response'가 뭔데">
브라우저에서 `fetch('/api')`를 부르면 `Request`가 나가고 `Response`가 돌아온다. 바로 그 `Request`/`Response` 객체가 이제 *서버 쪽 런타임에도* 표준으로 들어왔다(Fetch API). 즉 "요청 하나를 받아 응답 하나를 돌려주는 함수"를 플랫폼이 표준 규격으로 정해둔 것이다. Hono는 그 규격을 그대로 입출력으로 쓴다. 안에서는 `c`(Context)라는 얇은 도우미로 편의를 주지만, 경계(입력·출력)는 표준 그대로라 — 그 표준을 아는 런타임이면 어디든 얹힌다.
</Callout>

## thin의 자리 — lifecycle을 가져가지 않는다

이 결정은 Hono를 [tRPC 글](/docs/dev/nodejs/architecture/trpc)에서 그렸던 세 진영 중 **thin 진영**의 한복판에 앉힌다. Hono는 `Request → Response` 사이에 미들웨어와 핸들러만 둘 뿐, **request lifecycle을 소유하지 않는다.** DI 컨테이너도, guard/interceptor/pipe도, 데코레이터 생명주기도 없다. 그냥 "HTTP 서버"다.

그래서 [NestJS와는 정반대 선택](/docs/dev/nodejs/architecture/nextjs-fullstack-limits)이다. Nest는 lifecycle을 통째로 소유해서 구조를 주는 대신 무겁고 Node에 묶인다. Hono는 lifecycle을 비워두는 대신 가볍고 어디서나 돈다. 같은 축("lifecycle을 누가 소유하는가")의 양 끝이다.

## 그리고 thin이 자기 구조를 흡수한다 — hc RPC

trpc 글에서 "thin 프레임워크들이 tRPC의 niche를 점점 자기가 흡수한다"고 했는데, 그 대표 사례가 바로 Hono의 **`hc`(RPC 클라이언트)**다. 서버 라우트의 타입을 클라이언트가 그대로 추론해서, codegen 없이 end-to-end 타입 안전을 준다.

흥미로운 건 그 비밀이 *완전히 새로운 마법이 아니라는* 점이다. trpc 글에서 모든 걸 가른다고 했던 그 `typeof` 한 줄이, 여기서도 똑같이 등장한다.

```ts
// server.ts — 라우트를 체이닝해 하나의 정적 타입으로 모은다
const routes = app
  .get('/users/:id', (c) => c.json({ id: c.req.param('id'), name: 'Kim' }))
  .post('/users', (c) => c.json({ ok: true }))

export type AppType = typeof routes   // ← trpc 글에서 본 바로 그 한 줄
```

```ts
// client.ts — 서버 타입을 import해서 추론
import { hc } from 'hono/client'
import type { AppType } from './server'

const client = hc<AppType>('http://localhost:8787')
const res = await client.users[':id'].$get({ param: { id: '1' } })
// res의 타입이 서버 핸들러에서 그대로 흘러온다 — codegen 없이
```

[tRPC가 1급 시민이 되는 두 조건](/docs/dev/nodejs/architecture/trpc)(① lifecycle을 직접 소유 ② consumer가 타입을 import)을 떠올려 보면, Hono는 그 두 조건을 *자연스럽게* 충족한다 — thin이라 lifecycle을 자기가 갖고, 라우트가 `typeof`를 뜰 단일 객체로 모이기 때문이다. 그래서 "tRPC를 굳이 안 써도 비슷한 타입 안전을 얻는" 그림이 성립한다.

## 베팅의 대가

공짜는 아니다. Hono의 베팅에는 두 갈래의 청구서가 따라온다.

**thin이라는 건, 구조를 네가 떠안는다는 뜻이다.** lifecycle을 비워뒀다는 건 [횡단 관심사(AOP)와 요청 컨텍스트](/docs/dev/nodejs/architecture/nextjs-fullstack-limits)를 떠받칠 *선언적 슬롯이 없다*는 뜻이기도 하다. 작을 땐 미들웨어로 충분하지만, 도메인이 커지고 인증·트랜잭션·감사 컨텍스트가 깊은 호출 그래프를 타고 흘러야 하는 순간, Nest가 DI로 공짜로 주던 것을 직접 다시 짓게 된다. Hono가 나빠서가 아니라, 그게 thin을 고른 대가다.

**Web 표준에 베팅한 대가도 있다.** Node 생태계에만 있는 라이브러리·미들웨어와는 거리가 생기고(어댑터나 대체재가 필요하다), 런타임마다 미묘하게 다른 표준 구현의 틈(엣지 런타임의 API 제약 등)을 직접 만나기도 한다. "어디서나 돈다"는 "모든 곳에서 똑같이 다 된다"와는 다르다.

## 그래서 언제 Hono인가

이 클러스터의 결정 지도에 Hono를 얹으면 자리가 분명해진다.

- **엣지·멀티런타임이 타깃**이거나, 작고 빠른 API·BFF가 필요하고, 구조를 직접 통제하고 싶다 → Hono가 가장 자연스럽다. `hc`까지 쓰면 타입 안전도 공짜에 가깝다.
- **큰 도메인 + 무거운 횡단 관심사 + 백그라운드·요청 컨텍스트가 본질적**이다 → [NestJS 쪽](/docs/dev/nodejs/architecture/nextjs-fullstack-limits)이 제 무게값을 한다.
- 그 사이라면 — "Hono + 직접 짠 얇은 구조 층"이 [Next.js 글의 2단계](/docs/dev/nodejs/architecture/nextjs-fullstack-limits)와 같은 중간 지대로 존재한다. 임계를 넘으면 그때 무거운 프레임워크로 옮기면 된다.

핵심은 하나다. **Hono의 정체는 "플랫폼을 추상화하지 않겠다"는 베팅이다.** 빠름·가벼움·이식성은 그 베팅의 *배당금*이고, 비워둔 구조는 그 베팅의 *청구서*다. 둘은 같은 결정의 양면이라, Hono를 고른다는 건 이 베팅에 함께 올라탄다는 뜻이다 — 그게 맞는 판인지가 결정의 전부다.
