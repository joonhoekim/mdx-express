## 이게 뭔데

소프트 삭제(soft delete). 행을 진짜로 지우지 않고, **"지워진 셈 치자"는 표시만** 남기는 거다. `DELETE FROM Customer WHERE ...` 대신 `UPDATE Customer SET isDeleted = TRUE WHERE ...`로 바꾸는 게 전부다.

비유를 하나 들자. 회사에서 직원이 퇴사한다고 인사 기록을 파쇄기에 넣지는 않잖아. 그냥 "퇴사 처리"라는 도장을 쾅 찍고 캐비닛 뒤쪽으로 옮길 뿐이다. 나중에 경력 증명서를 떼러 오거나, 감사가 들어와서 "3년 전 이 사람 급여가 얼마였냐"고 물으면 꺼내 보여줄 수 있어야 하니까. 종이를 없애 버리면 그 순간 영원히 못 답한다. 소프트 삭제가 딱 이 "퇴사 도장"이다. 데이터는 캐비닛에 남고, 평소 업무에서는 안 보이게만 한다.

<Callout type="info" title="한 줄 요약">
삭제를 데이터의 소멸이 아니라 상태의 변화로 본다. 행은 살아 있고, `isDeleted` 플래그(또는 `deleted_at` 시각)가 TRUE/값으로 바뀔 뿐이다. 평소 화면에서만 안 보일 뿐, 복구도 감사도 가능하다.
</Callout>

## 언제 쓰나

동기는 책 표현 그대로 한 문장이다. **"모든 애플리케이션 데이터를 보존한다."** 주로 이력 목적이고, 현실에서는 다음 냄새가 날 때 답이 된다.

- **"실수로 지웠는데 되돌릴 수 있나요?"** 가 슬랙에 올라온다. 고객이 계좌(Account)를 잘못 해지했고, CS가 복구를 요청한다. 하드 삭제였다면 백업 복원밖에 답이 없다. 소프트 삭제였다면 `UPDATE ... SET isDeleted = FALSE` 한 줄이다.
- **감사·규제가 "지운 것도 기록으로 남겨라"** 라고 요구한다. 금융·보험 도메인이 특히 그렇다. 보험 증권(InsurancePolicy)을 해지했어도, 그 증권이 존재했었다는 사실 자체가 분쟁·재청구·감사의 증거다. 행을 날려 버리면 "원래 없었다"와 "있었는데 지웠다"를 구분할 수 없게 된다.
- **다른 행이 이 행을 참조**한다. 이미 삭제된 고객을 가리키는 거래 내역이 있는데, 고객 행을 하드 삭제하면 그 거래는 고아(orphan)가 되거나 FK 위반으로 삭제 자체가 막힌다. 행을 살려 두면 참조는 깨지지 않고, 화면에서만 숨기면 된다.

### 시나리오: 이런 적 있을 거임

은행 백오피스. 상담원이 고객 화면에서 "계좌 삭제" 버튼을 누른다. 코드는 충실하게 `DELETE FROM Account WHERE AccountID = ?`를 날린다. 깔끔하다. 행은 사라졌다.

다음 날, 그 고객이 전화해서 "어제 내 적금 계좌가 갑자기 없어졌는데요?" 한다. 알고 보니 상담원이 동명이인의 계좌를 잘못 지운 거다. 복구하려고 보니, 그 계좌 행은 이미 우주의 먼지가 됐다. 남은 선택지는 어젯밤 백업을 복원해서 그 행 하나만 떠 오는 것 — DBA를 호출하고, 운영 DB 스냅샷을 띄우고, 반나절이 날아간다. 그 사이 그 계좌에 걸려 있던 자동이체, 이자 정산, 거래 로그가 줄줄이 꼬인다.

여기서 더 무서운 건 따로 있다. 한 달 뒤 감사가 들어와서 "지난 분기 해지된 계좌 목록 주세요" 한다. 그런데 우리 DB엔 **해지된 계좌가 아예 없다.** 지웠으니까. "해지된 적이 없다"가 아니라 "기록이 없다"인데, 감사 입장에선 둘이 같은 의미로 안 읽힌다. 식은땀이 난다.

이 두 사건 모두, 삭제를 `DELETE`가 아니라 `UPDATE isDeleted = TRUE`로만 바꿨어도 5분 만에 끝났을 일이다.

## 주의할 점

공짜는 아니다. 책이 짚는 트레이드오프는 결국 **성능과 복잡도**다.

<Callout type="warning" title="삭제해도 안 줄어드는 테이블">
- **디스크가 안 줄어든다.** 소프트 삭제는 본질적으로 "절대 안 지우는" 정책이다. 지운 행이 테이블에 그대로 쌓이니, 1년 운영하면 죽은 행이 산 행보다 많아질 수도 있다. 디스크는 늘고, 풀스캔은 느려진다.
- **모든 읽기가 조건을 짊어진다.** 이제 모든 조회가 "산 행만 보여줘"를 챙겨야 한다. `WHERE isDeleted = FALSE`를 빼먹는 순간, 죽었어야 할 데이터가 화면에 부활한다. 이 조건을 쿼리마다 손으로 흩뿌리면 언젠가 반드시 하나는 빠진다. 그게 가장 흔한 소프트 삭제 버그다.
- **유니크 제약이 꼬인다.** 이메일 같은 컬럼에 UNIQUE를 걸어 뒀는데, 소프트 삭제된 고객이 같은 이메일을 점유하고 있으면 새 가입이 막힌다. 죽은 행은 유니크 검사에서 빠져야 자연스러운데, 순진하게 컬럼에 UNIQUE만 걸면 그게 안 된다.
</Callout>

마지막 항목은 책에는 명시적으로 안 나오지만, 소프트 삭제를 실전에 깔면 거의 반드시 밟는 지뢰라 미리 적어 둔다. 해법은 뒤에서 partial unique index로 푼다.

## 이렇게 한다

큰 그림은 네 단계다. 컬럼 하나 추가하고, 갱신을 트리거에 맡기고, 읽기를 한곳에 모으고, 인덱스로 성능을 메운다.

<Steps>
<Step title="식별 컬럼을 도입한다 (NOT NULL)">
Boolean `isDeleted`(삭제=TRUE) 또는 삭제 시각 timestamp 둘 중 하나를 추가한다. 둘 다 쓸 수 있는데, **현대 실무는 `deleted_at` timestamp 쪽을 표준으로 친다.** "삭제됐다"는 사실(`isDeleted`)뿐 아니라 "언제 삭제됐다"까지 공짜로 따라오기 때문이다. NULL이면 살아 있는 행, 값이 있으면 그 시각에 죽은 행 — 이 규칙 하나로 플래그와 삭제 이력을 동시에 표현한다.

책은 NULL 비허용(Make Column Non-Nullable)을 권하는데, 이건 Boolean `isDeleted`를 쓸 때 얘기다(`isDeleted`가 NULL이면 산 건지 죽은 건지 모호하니까). `deleted_at` 방식은 정반대로 **NULL을 "살아 있음"의 의미로 적극 활용**한다. 도메인에 맞게 고르면 된다.
</Step>

<Step title="갱신 방식을 정한다 — 트리거 선호">
삭제 시 플래그를 TRUE(또는 현재 시각)로 바꾸는 일을 누가 할 거냐. 앱이 할 수도, 트리거가 할 수도 있다. 책은 **트리거 방식을 선호**한다. 단순하고, 어느 앱이 `UPDATE`를 깜빡할 위험을 원천 차단하기 때문이다. DELETE 문을 트리거가 가로채서 "지우지 말고 플래그만 갱신해" 로 바꿔 버리면, 앱은 평소처럼 `DELETE`를 날려도 DB가 알아서 소프트 삭제로 처리한다.
</Step>

<Step title="읽기를 한곳에 캡슐화한다">
`WHERE isDeleted = FALSE`를 쿼리마다 복붙하지 마라. 비삭제 행만 돌려주는 **뷰로 감싸거나(Encapsulate Table With View)**, 공통 Read 메서드 한곳에 조건을 박아라. 조건은 한 군데서만 관리돼야 빠뜨릴 일이 없다.
</Step>

<Step title="인덱스로 성능을 메운다">
죽은 행이 쌓여도 산 행 조회가 안 느려지게, 살아 있는 행만 타는 partial index를 깐다.
</Step>
</Steps>

### 스키마 변경 (DDL)

먼저 컬럼을 추가한다. 운영 중인 큰 테이블이라면 추가 자체가 잠금을 유발할 수 있으니, 기본값 채우기와 NOT NULL 승격은 나눠서 한다(expand-contract).

<TabsContent value="pg">

```sql
-- 현대 실무 표준: deleted_at timestamp.
-- NULL = 살아 있음, 값 있음 = 그 시각에 삭제됨.
-- 기본값이 NULL이라 기존 행은 전부 "살아 있음"으로 해석되고,
-- 백필(batch update)이 필요 없다.
ALTER TABLE Account ADD COLUMN deleted_at timestamptz;

-- 산 행만 타는 partial index.
-- 죽은 행이 아무리 쌓여도 이 인덱스에는 안 들어가서
-- 평소 조회가 느려지지 않는다.
CREATE INDEX idx_account_alive
  ON Account (customer_id)
  WHERE deleted_at IS NULL;
```

</TabsContent>

<TabsContent value="book">

```sql
-- 책 카탈로그 스타일: Boolean isDeleted.
-- NULL 비허용 + 기본값 FALSE.
ALTER TABLE Account ADD COLUMN isDeleted BOOLEAN;

-- 기존 모든 행을 산 행으로 1회 백필
UPDATE Account SET isDeleted = FALSE WHERE isDeleted IS NULL;

-- 백필이 끝났으니 NOT NULL + 기본값 승격
ALTER TABLE Account
  ALTER COLUMN isDeleted SET DEFAULT FALSE,
  ALTER COLUMN isDeleted SET NOT NULL;
```

</TabsContent>
</Tabs>

여기서 partial index가 소프트 삭제의 진짜 핵심 무기다. 일반 인덱스는 죽은 행까지 다 색인하니 테이블이 부풀수록 같이 부푼다. 반면 `WHERE deleted_at IS NULL` 조건을 단 partial index는 **산 행만 색인**한다. 죽은 행이 90%여도 인덱스 크기는 산 행 10% 기준이다. 소프트 삭제의 가장 큰 약점(테이블이 안 줄어든다)을, 산 행 전용 인덱스로 정확히 우회하는 셈이다.

앞에서 예고한 유니크 제약 문제도 같은 도구로 푼다.

```sql
-- 이메일은 "살아 있는 고객" 사이에서만 유일해야 한다.
-- 죽은 고객이 점유한 이메일은 새 가입을 막으면 안 된다.
CREATE UNIQUE INDEX uq_customer_email_alive
  ON Customer (email)
  WHERE deleted_at IS NULL;
```

이러면 같은 이메일이 산 행에 둘 있으면 막히지만, 죽은 행은 검사에서 빠진다. 고객이 탈퇴 후 같은 이메일로 재가입해도 충돌이 안 난다.

### 데이터 마이그레이션 (DML)

`deleted_at` 방식은 기본값이 NULL이라 백필이 필요 없다(기존 행 = 다 살아 있음). Boolean 방식만 1회 배치로 기존 행을 채운다.

```sql
-- isDeleted 방식일 때만: 기존 행을 전부 "살아 있음"으로
UPDATE Customer SET isDeleted = FALSE WHERE isDeleted IS NULL;
```

이게 데이터 마이그레이션의 전부다. 소프트 삭제 도입은 기존 데이터를 옮기는 작업이 아니라, **삭제의 의미를 바꾸는** 작업이라 옮길 데이터가 거의 없다.

### 접근 프로그램 수정 (코드)

여기가 실제로 코드가 바뀌는 곳이다. 두 가지를 고친다. **삭제는 UPDATE로, 읽기는 캡슐화로.**

**1) 삭제를 물리 삭제에서 플래그 갱신으로.**

```sql
-- Before: 행이 우주에서 사라진다
DELETE FROM Account WHERE AccountID = ?;

-- After: 플래그만 갱신. 행은 살아 있고, 화면에서만 숨는다
UPDATE Account SET deleted_at = now() WHERE AccountID = ?;
```

앱마다 `DELETE`를 일일이 찾아 바꾸는 게 불안하면, 트리거로 DB 레벨에서 한 번에 가로챈다. 책이 말한 "DELETE를 가로채 플래그만 갱신하는 트리거"가 이거다.

```sql
-- 누가 어느 앱에서 DELETE를 날려도, DB가 소프트 삭제로 바꿔치기
CREATE OR REPLACE FUNCTION soft_delete_account()
RETURNS trigger AS $$
BEGIN
  UPDATE Account SET deleted_at = now() WHERE AccountID = OLD.AccountID;
  RETURN NULL;  -- 원래의 물리 DELETE는 취소
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_account_soft_delete
  BEFORE DELETE ON Account
  FOR EACH ROW EXECUTE FUNCTION soft_delete_account();
```

이게 트리거 선호의 핵심이다. 어떤 레거시 앱이 `DELETE Account`를 날려도, 그 앱을 고치지 않고도 소프트 삭제가 강제된다. 누락 위험이 0이 된다. 다만 트리거는 디버깅이 까다로우니, 팀이 "삭제는 항상 UPDATE로 짠다"는 규율을 지킬 수 있다면 앱 레벨로 가는 것도 정당한 선택이다.

**2) 읽기 조건을 흩뿌리지 말고 뷰/공통 Read로 모은다.**

가장 흔한 소프트 삭제 사고가 `WHERE deleted_at IS NULL`을 한 쿼리에서 빠뜨리는 거다. 조건을 쿼리마다 복붙하는 한 언젠가 반드시 하나는 샌다. 한곳에 모아라.

```sql
-- Encapsulate Table With View: 산 행만 보이는 뷰
CREATE VIEW Account_active AS
  SELECT * FROM Account WHERE deleted_at IS NULL;

-- 앱은 원본 테이블 대신 이 뷰를 읽는다.
-- 이제 조건을 깜빡할 곳이 없다 — 한 군데서만 관리된다.
SELECT * FROM Account_active WHERE customer_id = ?;
```

ORM을 쓴다면 프레임워크가 이 캡슐화를 대신 해 준다. 모델에 글로벌 스코프를 한 번 걸면 모든 조회에 조건이 자동으로 붙는다.

```typescript
// Prisma는 모델 수준 글로벌 필터가 (현재) 없어서
// 클라이언트 확장으로 모든 find에 조건을 끼워 넣는다
const prisma = basePrisma.$extends({
  query: {
    account: {
      async $allOperations({ args, query, operation }) {
        if (operation.startsWith('find')) {
          args.where = { ...args.where, deletedAt: null };
        }
        return query(args);
      },
    },
  },
});
```

</TabsContent>

<TabsContent value="hibernate">

```java
// 엔티티에 @SQLDelete로 삭제를 UPDATE로 바꾸고,
// @Where로 모든 조회에 조건을 자동으로 건다
@Entity
@SQLDelete(sql = "UPDATE Account SET deleted_at = now() WHERE AccountID = ?")
@Where(clause = "deleted_at IS NULL")
public class Account {
    // entityManager.remove(account) 가 DELETE가 아니라
    // 위 UPDATE로 나간다
}
```

</TabsContent>

<TabsContent value="rails">

```ruby
# default_scope로 모든 쿼리에 조건을 자동 부착.
# 죽은 행이 필요하면 .unscoped로 명시적으로 벗긴다.
class Account < ApplicationRecord
  default_scope { where(deleted_at: nil) }

  def soft_delete!
    update!(deleted_at: Time.current)
  end
end
```

<Callout type="note" title="RLS로 한 단계 더 — DB가 강제하는 캡슐화">
뷰나 ORM 스코프도 결국 "그 경로로 읽을 때만" 조건이 붙는다. 어드민이 psql로 직접 테이블을 긁으면 죽은 행이 그대로 보인다. PostgreSQL의 Row-Level Security(RLS)를 쓰면 캡슐화를 **DB 엔진 레벨**로 끌어내릴 수 있다.

```sql
ALTER TABLE Account ENABLE ROW LEVEL SECURITY;

CREATE POLICY account_alive ON Account
```

### 그 외 현대적 고려

- **죽은 행을 영원히 들고 있지는 마라.** 소프트 삭제는 "절대 안 지움"이 아니라 "당장 안 지움"으로 이해하는 게 건강하다. 감사 보존 기간(예: 7년)이 지난 죽은 행은 배치나 partition drop으로 진짜 비워 줘야 테이블이 무한정 부풀지 않는다. 시간 기준 파티셔닝을 깔아 두면 `deleted_at` 오래된 파티션을 통째로 떨어뜨릴 수 있어 깔끔하다.
- **CDC와 궁합이 좋다.** Debezium 같은 CDC는 물리 `DELETE`를 tombstone 이벤트로 흘리는데, 소프트 삭제는 그냥 `UPDATE` 이벤트라 다운스트림(검색 색인, 데이터 웨어하우스)이 변경을 일반 갱신처럼 다루기 쉽다. 삭제 이력이 이벤트 스트림에도 그대로 남는다.
- **마이크로서비스라면 소유권을 기억하라.** 소프트 삭제된 행을 다른 서비스가 여전히 참조할 수 있다. "내가 죽였으니 너도 죽은 줄 알겠지"는 위험한 가정이다. 삭제를 이벤트로 알리거나 outbox로 전파해, 죽음의 사실이 경계를 넘어 전달되게 하라.

## 정리

소프트 삭제는 삭제를 "데이터의 소멸"에서 "상태의 변화"로 바꾸는 리팩토링이다. `DELETE`를 `UPDATE deleted_at`으로 바꾸는 게 핵심이고, 그 대가로 복구 가능성과 감사 추적을 얻는다.

> **지우지 마라. 표시만 해라. 그리고 그 표시를 한곳에서만 관리해라.**

실패하는 소프트 삭제는 대개 두 가지에서 무너진다. 갱신을 깜빡하거나(트리거로 막는다), 읽기 조건을 흩뿌리다 하나를 빠뜨리거나(뷰·ORM 스코프·RLS로 캡슐화한다). 여기에 partial index로 "산 행만 타는" 길을 깔아 주면, 소프트 삭제의 유일한 약점인 테이블 비대마저 비껴간다. 감사와 복구가 중요한 도메인에서 이게 사실상 표준인 데는 이유가 있다.
