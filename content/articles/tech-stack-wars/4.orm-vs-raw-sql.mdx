# ORM vs Raw SQL — 추상화의 대가

*"모든 추상화는 결국 누수된다 — Joel Spolsky, The Law of Leaky Abstractions"*

---

ORM을 처음 쓰면 이런 생각이 듦:
"와 SQL 안 써도 되네? 이게 천국인가?"

6개월 후:
"이 쿼리가 왜 이렇게 느리지... 아 ORM이 서브쿼리를 7개 만들었네..."

이것이 추상화의 대가임. 편리함에는 항상 비용이 따르고,
그 비용은 보통 가장 안 좋은 타이밍에 청구됨.

<Callout type="warning" title="면책 조항">
이 글은 특정 ORM을 까기 위한 글이 아님.
모든 ORM에는 장단점이 있고, 상황에 따라 최선의 선택이 다름.
...이라고 쓰지만, 솔직히 ORM의 함정은 꽤 까야 할 게 많음.
</Callout>

## ORM이란 무엇인가

ORM (Object-Relational Mapping)은
데이터베이스 테이블을 프로그래밍 언어의 객체로 매핑하는 기술임.

```typescript
// SQL로 유저 조회
SELECT * FROM users WHERE id = 1;

// ORM으로 유저 조회 (Prisma)
const user = await prisma.user.findUnique({ where: { id: 1 } });

// ORM으로 유저 조회 (TypeORM)
const user = await userRepository.findOneBy({ id: 1 });

// ORM으로 유저 조회 (Drizzle)
const user = await db.select().from(users).where(eq(users.id, 1));

// 결과는 같음. 하지만 과정이 다름.
```

### ORM의 핵심 약속

```
1. SQL을 몰라도 DB를 다룰 수 있다
2. DB가 바뀌어도 코드를 안 바꿔도 된다 (이론상)
3. 타입 안전한 쿼리를 작성할 수 있다
4. 마이그레이션을 자동으로 관리해준다
5. 관계(Relations)를 객체처럼 다룰 수 있다
```

이 약속들이 지켜지는 경우도 있고, 안 지켜지는 경우도 있음.
하나씩 까보자.

## ORM의 편리함: 진짜 좋은 점들

### 1. 타입 안전성

```typescript
// Prisma의 타입 추론 — 이건 진짜 혁명적임
// schema.prisma
model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  name      String?
  posts     Post[]
  createdAt DateTime @default(now())
}

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  content   String?
  published Boolean  @default(false)
  author    User     @relation(fields: [authorId], references: [id])
  authorId  Int
}

// 자동 생성된 타입으로 쿼리
const user = await prisma.user.findUnique({
  where: { id: 1 },
  include: { posts: true },
});

// user의 타입이 자동으로 추론됨:
// { id: number; email: string; name: string | null; posts: Post[]; createdAt: Date }

// 없는 필드 접근하면 컴파일 에러
user.age; // ← TypeScript Error: Property 'age' does not exist

// select로 특정 필드만 가져오면 타입도 자동으로 좁혀짐
const user2 = await prisma.user.findUnique({
  where: { id: 1 },
  select: { name: true, email: true },
});
// user2 타입: { name: string | null; email: string }
```

<Callout type="success" title="Prisma의 타입 시스템">
Prisma의 타입 추론은 ORM 역사상 최고 수준임.
스키마에서 TypeScript 타입을 자동 생성하고,
select/include에 따라 반환 타입이 동적으로 결정됨.
이건 Raw SQL로는 절대 달성할 수 없는 영역임.
(물론 Drizzle이나 Kysely도 비슷한 수준으로 따라왔지만)
</Callout>

## ORM의 함정: 진짜 무서운 것들

### 1. N+1 문제 — ORM의 원죄

```typescript
// N+1 문제: ORM을 쓰면서 가장 많이 마주치는 성능 문제

// ❌ 이렇게 하면 N+1 발생
const posts = await prisma.post.findMany({ take: 100 });

for (const post of posts) {
  // 각 포스트마다 별도 쿼리 발생!
  const author = await prisma.user.findUnique({
    where: { id: post.authorId },
  });
  console.log(`${post.title} by ${author.name}`);
}

// 실제 실행되는 쿼리:
// 1. SELECT * FROM posts LIMIT 100
// 2. SELECT * FROM users WHERE id = 1
// 3. SELECT * FROM users WHERE id = 2
// ...
// 101. SELECT * FROM users WHERE id = 100
// 총 101번의 쿼리!

// ✅ include를 사용하면 해결
const posts = await prisma.post.findMany({
  take: 100,
  include: { author: true },
});
// 쿼리 2번으로 해결:
// 1. SELECT * FROM posts LIMIT 100
// 2. SELECT * FROM users WHERE id IN (1, 2, 3, ...)
```

<Callout type="error" title="N+1의 공포">
N+1 문제는 개발 환경에서는 잘 안 느껴짐.
데이터가 10개일 때 쿼리 11번은 체감이 안 되거든.

하지만 프로덕션에서 데이터가 10,000개가 되면?
쿼리 10,001번. DB 커넥션 풀 고갈. 서버 다운.

이게 ORM의 가장 큰 함정임. 동작하는 코드가 반드시 좋은 코드가 아니라는 것.
</Callout>

### 3. 복잡한 쿼리의 한계

```typescript
// ORM으로 하기 어려운 쿼리들

// 1. Window Functions
// SQL:
// SELECT *, RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) as rank
// FROM employees;
//
// Prisma: 네이티브 지원 안 됨. Raw SQL 써야 함.

// 2. CTE (Common Table Expression)
// SQL:
// WITH RECURSIVE subordinates AS (
//   SELECT id, name, manager_id FROM employees WHERE id = 1
//   UNION ALL
//   SELECT e.id, e.name, e.manager_id FROM employees e
//   JOIN subordinates s ON e.manager_id = s.id
// )
// SELECT * FROM subordinates;
//
// Prisma: 재귀 CTE? 그런 거 없음.

// 3. JSON 연산
// SQL (PostgreSQL):
// SELECT * FROM users
// WHERE metadata->>'preferences'->>'theme' = 'dark'
// AND metadata->'settings' @> '{"notifications": true}'::jsonb;
//
// Prisma: 일부 지원하지만 복잡한 JSON 쿼리는 Raw SQL 필요

// 4. 서브쿼리에서 외부 테이블 참조
// SQL:
// SELECT *, (
//   SELECT AVG(rating)
//   FROM reviews
//   WHERE reviews.product_id = products.id
//   AND reviews.created_at > NOW() - INTERVAL '30 days'
// ) as recent_avg_rating
// FROM products
// HAVING recent_avg_rating > 4.0;
//
// Prisma: 이건 진짜 Raw SQL 말고는 답이 없음
```

<Callout type="warning" title="Raw Query의 현실">
결국 복잡한 쿼리는 ORM의 Raw Query 기능을 쓰게 됨.
```typescript
const result = await prisma.$queryRaw`
  SELECT u.*, COUNT(p.id) as post_count
  FROM "User" u
  LEFT JOIN "Post" p ON p."authorId" = u.id
  WHERE u."createdAt" > ${thirtyDaysAgo}
  GROUP BY u.id
  HAVING COUNT(p.id) > 5
  ORDER BY post_count DESC
  LIMIT 20
`;
// 이 순간 ORM의 타입 안전성은 날아감
// result의 타입은 any...
```

"ORM을 쓰면 SQL을 안 배워도 된다"는 거짓말임.
ORM을 **잘** 쓰려면 SQL을 **더 잘** 알아야 함.
생성되는 SQL을 이해하고, 언제 Raw SQL로 전환해야 하는지 판단해야 하니까.
</Callout>

## Raw SQL의 세계

### Raw SQL의 장점

```typescript
// 1. 완전한 제어권
// 어떤 쿼리든 작성 가능, DB의 모든 기능 활용 가능

// 2. 성능 최적화
// EXPLAIN ANALYZE로 실행 계획을 보고 직접 최적화
const result = await pool.query(`
  EXPLAIN ANALYZE
  SELECT u.id, u.name, COUNT(p.id) as post_count
  FROM users u
  LEFT JOIN posts p ON p.author_id = u.id
  WHERE u.created_at > $1
  GROUP BY u.id
  ORDER BY post_count DESC
  LIMIT 20
`, [thirtyDaysAgo]);

// 3. DB별 특화 기능 활용
// PostgreSQL의 LATERAL JOIN, ARRAY_AGG, JSONB 연산 등
const result = await pool.query(`
  SELECT
    u.id,
    u.name,
    COALESCE(
      ARRAY_AGG(
        JSON_BUILD_OBJECT(
          'id', p.id,
          'title', p.title,
          'comment_count', (SELECT COUNT(*) FROM comments c WHERE c.post_id = p.id)
        )
      ) FILTER (WHERE p.id IS NOT NULL),
      '{}'::json[]
    ) as posts
  FROM users u
  LEFT JOIN LATERAL (
    SELECT * FROM posts
    WHERE posts.author_id = u.id
    ORDER BY created_at DESC
    LIMIT 5
  ) p ON true
  GROUP BY u.id
`);
```

### Raw SQL의 위험

```typescript
// 1. SQL Injection — 영원한 적

// ❌ 절대 이렇게 하면 안 됨
const query = `SELECT * FROM users WHERE name = '${userInput}'`;
// userInput이 "'; DROP TABLE users; --" 이면?
// SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
// 테이블이 날아감 ㅋㅋ

// ✅ 파라미터 바인딩 사용
const result = await pool.query(
  'SELECT * FROM users WHERE name = $1',
  [userInput]
);

// ✅ 또는 Tagged Template Literal (Prisma $queryRaw)
const result = await prisma.$queryRaw`
  SELECT * FROM users WHERE name = ${userInput}
`;
// Prisma가 자동으로 파라미터 바인딩 처리
```

```typescript
// 2. 타입 안전성의 부재

// Raw SQL의 결과는 any
const users = await pool.query('SELECT * FROM users');
// users.rows의 타입: any[]

// 수동으로 타입을 정의해야 함
interface User {
  id: number;
  name: string;
  email: string;
  created_at: Date;
}

const users = await pool.query<User>('SELECT * FROM users');
// 하지만 이 타입이 실제 DB 스키마와 일치하는지는 런타임에서만 알 수 있음
// 필드 이름 오타, 타입 불일치 등을 컴파일 타임에 잡을 수 없음
```

```typescript
// 3. 쿼리 관리의 어려움

// 프로젝트가 커지면 SQL 쿼리가 여기저기 흩어짐
// user-queries.ts
export const getUserById = 'SELECT * FROM users WHERE id = $1';
export const getUserPosts = `
  SELECT p.*, u.name as author_name
  FROM posts p
  JOIN users u ON u.id = p.author_id
  WHERE p.author_id = $1
  ORDER BY p.created_at DESC
`;

// 스키마가 바뀌면? 모든 쿼리를 하나하나 확인해야 함
// 컬럼 이름 변경? grep으로 찾아서 전부 수정
// 잊어버린 쿼리 하나가 프로덕션에서 에러...
```

<Callout type="error" title="Raw SQL의 유지보수 비용">
Raw SQL로 프로젝트를 시작하면 처음엔 빠르고 간단함.
하지만 규모가 커지면:

- 쿼리 중복이 발생
- 스키마 변경 시 영향 범위 파악이 어려움
- 새 팀원이 기존 쿼리를 이해하기 어려움
- 테스트가 어려움 (DB 의존적)
- 마이그레이션 관리가 수동

결국 "나만의 ORM"을 만들기 시작하는 순간이 옴.
그리고 그 ORM은 보통 기존 ORM보다 못함.
</Callout>

## TypeScript ORM 비교: 2025년 전장

### Prisma

```typescript
// Prisma — 가장 인기 있는 Node.js ORM
// 장점: 타입 안전성, DX, 마이그레이션, 스키마 파일
// 단점: 자체 쿼리 엔진 (Rust 바이너리), Cold Start 느림, 복잡한 쿼리 한계

// schema.prisma — 자체 스키마 언어 (DSL)
model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  name      String?
  role      Role     @default(USER)
  posts     Post[]
  profile   Profile?
}

model Post {
  id        Int       @id @default(autoincrement())
  title     String
  content   String?
  published Boolean   @default(false)
  author    User      @relation(fields: [authorId], references: [id])
  authorId  Int
  tags      Tag[]
}

model Tag {
  id    Int    @id @default(autoincrement())
  name  String @unique
  posts Post[]
}

// 쿼리 예시
const usersWithPosts = await prisma.user.findMany({
  where: {
    role: 'ADMIN',
    posts: { some: { published: true } },
  },
  select: {
    name: true,
    email: true,
    posts: {
      where: { published: true },

    },

```

### Drizzle

```typescript
// Drizzle — "SQL을 아는 사람을 위한 ORM"
// 장점: SQL에 가까운 문법, 가볍다, 타입 안전, Edge Runtime 호환
// 단점: 마이그레이션 도구 아직 발전 중, Prisma보다 생태계 작음

// 스키마 정의 — TypeScript로!
import { pgTable, serial, text, boolean, integer, timestamp } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: serial('id').primaryKey(),
  email: text('email').notNull().unique(),
  name: text('name'),
  role: text('role', { enum: ['USER', 'ADMIN'] }).default('USER'),
  createdAt: timestamp('created_at').defaultNow(),
});

export const posts = pgTable('posts', {
  id: serial('id').primaryKey(),
  title: text('title').notNull(),
  content: text('content'),
  published: boolean('published').default(false),
  authorId: integer('author_id').references(() => users.id),
});

// 쿼리 — SQL처럼 생겼지만 타입 안전함
const result = await db
  .select({
    userName: users.name,
    postTitle: posts.title,
    postCount: sql<number>`count(${posts.id})`,
  })
  .from(users)
  .leftJoin(posts, eq(posts.authorId, users.id))
  .where(eq(users.role, 'ADMIN'))
  .groupBy(users.id)
  .having(sql`count(${posts.id}) > 5`)
  .orderBy(desc(sql`count(${posts.id})`))
  .limit(20);

// SQL을 아는 사람이면 이 코드를 바로 이해할 수 있음
// Prisma처럼 새로운 API를 배울 필요가 없음
```

### TypeORM

```typescript
// TypeORM — 데코레이터 기반의 전통적 ORM
// 장점: Active Record + Data Mapper 패턴 둘 다 지원, 성숙한 프로젝트
// 단점: 타입 안전성 부족, 유지보수 속도 느림, 버그 많다는 평

import { Entity, PrimaryGeneratedColumn, Column, ManyToOne, OneToMany } from 'typeorm';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ unique: true })
  email: string;

  @Column({ nullable: true })
  name: string;

  @OneToMany(() => Post, (post) => post.author)
  posts: Post[];
}

@Entity()
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  title: string;

  @ManyToOne(() => User, (user) => user.posts)
  author: User;
}

// 쿼리
const users = await userRepository
  .createQueryBuilder('user')
  .leftJoinAndSelect('user.posts', 'post')
  .where('user.role = :role', { role: 'ADMIN' })
  .andWhere('post.published = :published', { published: true })
  .orderBy('user.createdAt', 'DESC')
  .take(20)
  .getMany();

// QueryBuilder가 강력하긴 한데,
// 문자열 기반이라 타입 안전성이 부족함
// 'user.role' ← 이게 오타면 런타임에서야 알 수 있음
```

### Kysely: 쿼리 빌더의 정수

```typescript
// Kysely — "타입 안전한 SQL 쿼리 빌더"
// ORM이 아니라 쿼리 빌더임. 관계 매핑 같은 건 안 해줌.
// 대신 SQL에 가장 가까우면서도 타입이 안전함.

import { Kysely, PostgresDialect } from 'kysely';

interface Database {
  users: {
    id: number;
    email: string;
    name: string | null;
    role: 'USER' | 'ADMIN';
    created_at: Date;
  };
  posts: {
    id: number;
    title: string;
    content: string | null;
    published: boolean;
    author_id: number;
  };
}

const db = new Kysely<Database>({ dialect: new PostgresDialect({ pool }) });

// SQL을 거의 그대로 쓰되, 타입 안전
const result = await db
  .selectFrom('users')
  .innerJoin('posts', 'posts.author_id', 'users.id')
  .select(['users.name', 'posts.title'])
  .where('users.role', '=', 'ADMIN')
  .where('posts.published', '=', true)
  .orderBy('users.created_at', 'desc')
  .limit(20)
  .execute();

// result의 타입: { name: string | null; title: string }[]
// 자동 추론!

// 없는 컬럼 접근하면 컴파일 에러
// .where('users.age', '=', 30)  ← Error: 'age' does not exist in 'users'
```

<Callout type="info" title="ORM 선택 가이드 (2025년 기준)">

| ORM | 추천 상황 |
|-----|----------|
| **Prisma** | 빠른 프로토타이핑, 스타트업, DX 우선 |
| **Drizzle** | SQL 잘 아는 팀, Edge Runtime, 성능 중요 |
| **TypeORM** | 기존 프로젝트, NestJS 사용 시 |
| **Kysely** | SQL 쿼리 빌더만 필요, 최대한 SQL에 가깝게 |
| **Raw SQL** | 극한 성능, DB 특화 기능, DBA가 있는 팀 |

2025년 트렌드: Prisma에서 Drizzle로 이동하는 흐름이 보임.
"Prisma가 너무 무겁다" "Edge에서 안 돌아간다" "SQL을 몰라도 되는 게 오히려 문제"
이런 의견들이 많아지면서 SQL에 가까운 Drizzle이 부상하고 있음.
</Callout>

### 문제: "최근 30일간 가장 많은 댓글을 받은 게시글 Top 10"

#### Raw SQL

```sql
SELECT
  p.id,
  p.title,
  u.name AS author_name,
  COUNT(c.id) AS comment_count,
  p.created_at
FROM posts p
JOIN users u ON u.id = p.author_id
LEFT JOIN comments c ON c.post_id = p.id
  AND c.created_at > NOW() - INTERVAL '30 days'
WHERE p.published = true
GROUP BY p.id, u.name
ORDER BY comment_count DESC
LIMIT 10;

-- 깔끔하고, 읽기 쉽고, DB가 최적화하기 좋음
-- 한눈에 뭘 하는지 알 수 있음
```

#### Prisma

```typescript
// Prisma로는... 이걸 한 번에 하기 어려움
// aggregation + filtering + join이 복잡하게 섞여 있어서

// 방법 1: Raw Query (결국 SQL)
const result = await prisma.$queryRaw`
  SELECT p.id, p.title, u.name AS author_name,
    COUNT(c.id)::int AS comment_count
  FROM "Post" p
  JOIN "User" u ON u.id = p."authorId"
  LEFT JOIN "Comment" c ON c."postId" = p.id
    AND c."createdAt" > NOW() - INTERVAL '30 days'
  WHERE p.published = true
  GROUP BY p.id, u.name
  ORDER BY comment_count DESC
  LIMIT 10
`;

// 방법 2: Prisma API (N+1 위험 + 비효율적)
const posts = await prisma.post.findMany({
  where: { published: true },
  include: {
    author: { select: { name: true } },
    _count: {
      select: {
        comments: {
          where: {
            createdAt: { gte: thirtyDaysAgo },
          },
        },
      },
    },
  },
});
// 이러면 모든 게시글을 가져온 후 JS에서 정렬해야 함
// DB에서 처리할 걸 앱에서 처리하는 건 안티패턴임
const top10 = posts
  .sort((a, b) => b._count.comments - a._count.comments)
  .slice(0, 10);
```

#### Drizzle

```typescript
const result = await db
  .select({
    id: posts.id,
    title: posts.title,
    authorName: users.name,
    commentCount: sql<number>`count(${comments.id})`.as('comment_count'),
    createdAt: posts.createdAt,
  })
  .from(posts)
  .innerJoin(users, eq(users.id, posts.authorId))
  .leftJoin(comments, and(
    eq(comments.postId, posts.id),
    gt(comments.createdAt, thirtyDaysAgo),
  ))
  .where(eq(posts.published, true))
  .groupBy(posts.id, users.name)
  .orderBy(desc(sql`count(${comments.id})`))
  .limit(10);

// SQL을 아는 사람이면 바로 읽히는 코드
// + 타입 안전성
```

#### Kysely

```typescript
const result = await db
  .selectFrom('posts as p')
  .innerJoin('users as u', 'u.id', 'p.author_id')
  .leftJoin('comments as c', (join) =>
    join
      .onRef('c.post_id', '=', 'p.id')
      .on('c.created_at', '>', thirtyDaysAgo)
  )
  .select([
    'p.id',
    'p.title',
    'u.name as author_name',
    db.fn.count('c.id').as('comment_count'),
  ])
  .where('p.published', '=', true)
  .groupBy(['p.id', 'u.name'])
  .orderBy('comment_count', 'desc')
  .limit(10)
  .execute();

// SQL 그 자체. 하지만 타입 안전.
```

## 결론: 추상화의 올바른 사용법

<Callout type="success" title="핵심 교훈">
ORM vs Raw SQL 논쟁의 진짜 답:

**"추상화의 수준을 상황에 맞게 선택하라."**

- 단순 CRUD → ORM (생산성)
- 복잡한 보고서 쿼리 → Raw SQL 또는 쿼리 빌더 (성능)
- 대시보드 → 쿼리 빌더 (타입 안전성 + 유연성)
- 마이그레이션 → ORM 도구 사용 (안전성)

하나만 고집하지 마. 같은 프로젝트에서도
간단한 건 ORM으로, 복잡한 건 Raw SQL로 쓰는 게 현실적인 답임.
</Callout>

### 필수 규칙

```
1. ORM을 쓰더라도 SQL은 반드시 배워라
   → 생성되는 쿼리를 이해하지 못하면 ORM은 시한폭탄

2. 항상 쿼리 로깅을 켜라 (개발 환경에서)
   → N+1이 몇 번 발생하는지 모르면 성능 지옥

3. EXPLAIN ANALYZE를 습관화하라
   → 느린 쿼리의 원인은 항상 실행 계획에 있음

4. 마이그레이션은 반드시 도구를 써라
   → 수동 ALTER TABLE은 프로덕션 사고의 지름길

5. 추상화 수준을 혼합해서 써라
   → ORM + Raw SQL을 같이 쓰는 건 부끄러운 게 아님
```
