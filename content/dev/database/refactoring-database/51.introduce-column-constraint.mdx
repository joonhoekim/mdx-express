## 이게 뭔데

컬럼에 `CHECK` 제약을 하나 박는 거다. `CreditLimit < 50000` 같은, "이 컬럼엔 이런 값만 들어올 수 있다"는 규칙을 **DB 스키마에 직접 새겨 넣는** 리팩토링이다.

비유를 하나 들자. 회사 출입문 얘기다. "외부인 출입금지"라는 규칙이 있다고 치자. 이걸 강제하는 방법은 두 가지다. 하나는 모든 직원한테 "혹시 모르는 사람 보이면 막아주세요"라고 부탁하는 거다. 다른 하나는 출입구에 게이트를 달고 사원증을 찍게 하는 거다. 전자는 직원 한 명이 깜빡하면 뚫린다. 후자는 게이트가 알아서 막는다. 누가 출근하든, 신입이든, 외주 직원이든.

`CHECK` 제약이 바로 그 **게이트**다. 어떤 애플리케이션이 INSERT를 날리든 — 잘 짠 메인 서비스든, 누가 새벽에 급하게 만든 배치 스크립트든, DBeaver 켜놓고 손으로 친 UPDATE든 — DB가 길목에서 값을 검사한다. 규칙을 애플리케이션 코드에 흩뿌리는 대신, **단 한 곳, 데이터가 반드시 통과하는 지점**에 박아두는 거다.

<Callout type="info" title="한 줄 요약">
공통 검증 규칙을 애플리케이션 N개에 N번 구현하는 대신, DB의 CHECK 제약 한 줄로 모든 경로에서 강제한다. 단, "진짜 공통 규칙"일 때만.
</Callout>

## 언제 쓰나

핵심 동기는 단순하다. **모든 애플리케이션이 이 컬럼에 유효한 데이터만 넣게 보장하고 싶을 때.** 비교적 단순하고 모두에게 똑같이 적용되는 검증 규칙을 DB에 박는 거다.

어떤 냄새가 나면 이 리팩토링이 답인지 보자.

- **같은 검증 코드가 여기저기 복붙돼 있다.** 결제 서비스에도 `if (balance < 0) throw ...`, 정산 배치에도 똑같은 검사, 어드민 콘솔에도 또. 셋 다 같은 규칙인데 코드가 세 벌이다. 그중 하나라도 빠뜨리면 그 경로로 더러운 값이 샌다.
- **"코드가 다 막는데 왜 음수 잔액이 있지?"** 분명 앱에선 다 검증하는데 운영 DB를 까 보면 규칙을 어긴 행이 있다. 범인은 보통 검증을 안 거치는 우회 경로다. 일회성 마이그레이션 스크립트, 데이터 패치 쿼리, 옛날 버전 앱이 남긴 흔적.
- **방금 표준화를 끝냈다.** `Apply Standard Codes`나 `Apply Standard Type`로 값을 한 차례 정리했다면, 이제 그 표준이 앞으로도 유지되도록 제약으로 못을 박을 차례다. (이때는 보통 `Drop Column Constraint`로 옛 제약을 떼어낸 뒤 새 제약을 다시 거는 흐름이 된다.)

### 시나리오: 이런 적 있을 거임

은행 시스템이다. `Customer.CreditLimit`(신용 한도)은 비즈니스 정책상 5만 달러를 넘으면 안 된다. 5만 이상은 별도 승인 프로세스를 타야 하니까, 일반 INSERT로는 절대 그 위로 못 올라가야 한다.

당연히 앱 코드엔 검증이 있다.

```typescript
function createCustomer(input: CustomerInput) {
  if (input.creditLimit >= 50000) {
    throw new ValidationError('신용 한도는 5만 달러 미만이어야 합니다');
  }
  return db.insert('Customer', input);
}
```

깔끔하다. 잘 돈다. 그런데 6개월 뒤, 정산팀에서 슬랙이 온다. "한도 8만짜리 고객이 왜 있어요?"

`git blame`을 찍고 추적해 보면, 범인은 앱이 아니다. 작년에 레거시 시스템에서 고객 데이터를 한 번 이관했는데, **그 마이그레이션 스크립트는 `createCustomer`를 안 거치고 DB에 raw INSERT를 때렸다.** 검증 로직은 애플리케이션 안에만 있었으니까, 옆문으로 들어온 데이터는 그냥 통과한 거다.

```text
-- 메인 앱이 들어가는 정문: 검증 있음 ✅
[createCustomer] → if(creditLimit < 50000) → INSERT

-- 마이그레이션 스크립트가 들어간 옆문: 검증 없음 ❌
[migrate_legacy.sql] → INSERT INTO Customer ... (CreditLimit = 80000)
```

문제의 본질은 **규칙이 데이터의 길목이 아니라, 길목 중 하나에만 있었다는 것**이다. 데이터가 거쳐 가는 길은 여러 개인데, 게이트는 정문에만 달려 있었다. 이럴 때 게이트를 **모든 길이 합쳐지는 한 지점** — 즉 DB 컬럼 — 에 옮겨 다는 게 `Introduce Column Constraint`다.

## 주의할 점

박기 전에 반드시 멈춰서 물어야 하는 질문이 하나 있다. 이게 이 리팩토링의 전부라고 해도 과언이 아니다.

<Callout type="warning" title="이거 진짜 모두에게 공통인 규칙 맞아?">
DB 제약은 그 컬럼에 손대는 **모든** 애플리케이션에 예외 없이 적용된다. 그러니 박기 전에 "정말 모든 앱이 이 규칙을 지켜야 하나?"를 확인해야 한다. 앱마다 규칙이 미묘하게 다르다면 — 예를 들어 한 앱은 `FavoriteColor`로 빨강·파랑만 받는데 다른 앱은 무지개색을 다 받는다면 — 그건 DB 제약이 아니라 **각 앱의 검증 로직**으로 가야 한다. 잘못 박으면 멀쩡히 돌던 다른 앱이 제약 위반으로 깨진다.
</Callout>

판단 기준을 정리하면 이렇다.

- **진짜 공통 + 단순** → DB 제약 ❍. "잔액은 음수 불가", "한도는 5만 미만", "상태는 A/I/C 셋 중 하나" 같은 건 모두에게 똑같고 표현도 단순하다. 게이트로 딱이다.
- **앱마다 다름 또는 복잡** → 애플리케이션 로직. "VIP 등급이면 한도 예외", "특정 지점은 다른 규칙" 같은 분기는 CHECK 한 줄로 안 끝난다. 억지로 박으면 제약이 누더기가 되거나, 정작 필요한 예외를 못 받는다.

그리고 거의 항상 따라오는 현실이 하나 더 있다. **기존 데이터엔 이미 규칙을 어긴 값이 있다.** 위 시나리오의 8만짜리 고객처럼. 그동안 게이트가 없었으니 더러운 값이 쌓여 있는 게 당연하다. 제약을 거는 순간 DB는 "기존 데이터부터 검사"하므로, 위반 행이 하나라도 있으면 제약 추가 자체가 실패한다. 그래서 **위반 데이터를 먼저 찾아 정제하는 것**이 사실상 이 리팩토링의 본체다.

## 이렇게 한다

순서가 중요하다. 제약을 먼저 박으면 위반 데이터 때문에 실패하니까, **(1) 위반 데이터 파악 → (2) 정제 → (3) 제약 추가 → (4) 앱의 예외 처리** 순으로 간다.

<Steps>
<Step title="위반 데이터부터 본다 (정제 아님, 보고서다)">

제약을 걸기 전에, 이 규칙을 지금 어기고 있는 행이 몇 개고 어떤 값인지부터 센다. 바로 UPDATE를 치지 말고 일단 SELECT로 현황 파악. 이해관계자(여기선 정산팀·리스크팀)와 "이 8만짜리들을 어떻게 처리할 거냐"를 합의해야 하기 때문이다. 5만으로 깎을지, 별도 승인 테이블로 옮길지, 예외 플래그를 줄지는 비즈니스 결정이다.

```sql
-- 규칙을 어기는 행이 몇 개? 어떤 값들?
SELECT CustomerOID, CreditLimit
FROM   Customer
WHERE  CreditLimit >= 50000.00
ORDER  BY CreditLimit DESC;
```

</Step>
<Step title="위반 데이터를 정제한다 (Update Data)">

합의된 처리 방안대로 기존 데이터를 손본다. 소량이면 SQL 한 방, 대량이면 배치로 나눠 친다. 여기선 "5만 이상은 일단 4만 9999로 캡하고 승인 대기 플래그"라고 합의했다 치자.

```sql
-- 위반 값을 규칙 안으로 끌어들인다
UPDATE Customer
SET    CreditLimit = 49999.00,
       NeedsApproval = TRUE
WHERE  CreditLimit >= 50000.00;
```

대량 테이블이라 전체 락이 부담되면 한 번에 다 갱신하지 말고 PK 범위로 쪼개 돌린다. 일관성(전체 락)과 가용성(부분 갱신) 사이의 트레이드오프는 데이터 양과 서비스 특성을 보고 고른다.

</Step>
<Step title="제약을 추가한다 (스키마 변경)">

이제 데이터가 깨끗하니 게이트를 단다. 책의 정석은 이 한 줄이다.

```sql
ALTER TABLE Customer
  ADD CONSTRAINT Check_Credit_Limit
  CHECK (CreditLimit < 50000.00);
```

`CHECK` 제약에 이름(`Check_Credit_Limit`)을 꼭 붙여라. 이름이 없으면 DB가 `customer_creditlimit_check1` 같은 자동 이름을 붙이는데, 나중에 떼거나 바꿀 때(`Drop Column Constraint`) 이름으로 못 찾아 고생한다.

</Step>
<Step title="앱이 제약 위반 예외를 처리하게 한다">

이제 규칙을 어긴 INSERT/UPDATE는 DB가 예외를 던진다. 앱은 그 예외를 잡아 사용자에게 의미 있는 메시지로 바꿔줘야 한다. 기존 앱의 `if` 검증을 지우라는 게 아니라 — 그건 빠른 사용자 피드백용으로 남겨두는 게 낫다 — DB가 던지는 최후의 방어선 예외도 처리하라는 거다.

```typescript
try {
  await db.insert('Customer', input);
} catch (err) {
  // PostgreSQL: CHECK 위반은 SQLSTATE 23514
  if (err.code === '23514' && err.constraint === 'check_credit_limit') {
    throw new ValidationError('신용 한도는 5만 달러 미만이어야 합니다');
  }
  throw err;
}
```

`errorCode`로 분기하는 건 이식성이 떨어지니, 이왕이면 제약 **이름**으로 분기하자. 그래서 3단계에서 이름을 붙인 거다.

</Step>
</Steps>

### 무중단으로 박기: PostgreSQL NOT VALID → VALIDATE

여기서 2006년 책엔 없던 현대 실무 한 스푼. 위의 정석에는 함정이 하나 있다. `ADD CONSTRAINT ... CHECK (...)`를 그냥 실행하면, PostgreSQL은 제약을 추가하면서 **테이블의 모든 기존 행을 즉시 검사**한다. 이 검사 동안 테이블에 강한 락(`ACCESS EXCLUSIVE`에 준하는 락)이 잡혀서, 행이 수억 개인 테이블이면 그 시간 내내 INSERT/UPDATE/SELECT가 줄줄이 막힌다. 운영 중에 이걸 그냥 때리면 사실상 장애다.

PostgreSQL은 이걸 **두 단계로 쪼개는** 길을 준다. 검증을 나중으로 미루는 `NOT VALID`다.

<TabsTrigger value="naive">그냥 박기 (위험)</TabsTrigger>
<TabsTrigger value="online">무중단 박기 (NOT VALID)</TabsTrigger>
</TabsList>
<TabsContent value="naive">

```sql
-- 기존 행 전부를 즉시 검사하며 테이블을 오래 잠근다.
-- 작은 테이블이면 OK, 대형 테이블이면 운영 장애.
ALTER TABLE Customer
  ADD CONSTRAINT Check_Credit_Limit
  CHECK (CreditLimit < 50000.00);
```

</TabsContent>
<TabsContent value="online">

```sql
-- 1단계: 제약을 "걸되 기존 행은 검사하지 않음"으로 추가.
-- 메타데이터만 바꾸므로 락이 짧다. 이 순간부터 '새로 들어오는'
-- INSERT/UPDATE는 제약을 적용받는다 (= 더 이상 더러운 값 못 들어옴).
ALTER TABLE Customer
  ADD CONSTRAINT Check_Credit_Limit
  CHECK (CreditLimit < 50000.00) NOT VALID;

-- 2단계: 기존 행을 검사. 이건 ACCESS EXCLUSIVE가 아니라
-- 약한 락(SHARE UPDATE EXCLUSIVE)으로 진행돼 읽기/쓰기를 막지 않는다.
-- (전제: 1·2단계 사이에 위반 데이터 정제가 끝나 있어야 함)
ALTER TABLE Customer
  VALIDATE CONSTRAINT Check_Credit_Limit;
```

</TabsContent>
</Tabs>

핵심은 `NOT VALID`가 **"앞으로 들어올 데이터는 막되, 과거 데이터 전수 검사는 미룬다"**는 점이다. 그래서 운영 중에도 짧은 락으로 게이트를 먼저 닫아 새 오염을 멈추고, 무거운 전수 검사(`VALIDATE`)는 약한 락으로 따로 돌릴 수 있다. 이 순서라면 정제(2단계)와 게이트 설치(NOT VALID) 사이의 틈에 새 위반 데이터가 끼어드는 것도 막힌다.

<Callout type="note" title="다른 엔진은?">
이 두 단계 트릭은 PostgreSQL의 NOT VALID/VALIDATE 조합이 제일 깔끔하다. MySQL은 8.0.16부터 CHECK 제약을 실제로 강제하고, 큰 테이블의 무중단 스키마 변경은 보통 gh-ost나 pt-online-schema-change 같은 온라인 DDL 도구로 처리한다. Flyway·Liquibase·Alembic 같은 마이그레이션 도구를 쓴다면, 정제(UPDATE)와 제약 추가를 같은 버전 스크립트에 순서대로 넣어 배포 파이프라인이 "정제 → 제약"을 원자적으로 보장하게 하자. 위반 데이터 정제 없이 제약만 추가하는 스크립트는 운영 배포에서 그냥 실패한다.
</Callout>

## 정리

`Introduce Column Constraint`는 검증 규칙을 애플리케이션이라는 **여러 길목**에서 DB 컬럼이라는 **단 하나의 길목**으로 옮기는 일이다. 어떤 앱이 어느 옆문으로 데이터를 밀어 넣든, 게이트는 거기 있다.

> **규칙을 N개의 앱에 N번 적는 대신, 데이터가 반드시 지나가는 한 곳에 한 번 새긴다.**

대신 두 가지를 잊지 마라. 첫째, **"진짜 공통인가"**를 먼저 따져라. 앱마다 다른 규칙을 DB에 박으면 멀쩡한 앱이 깨진다. 둘째, **기존 위반 데이터를 먼저 정제하라.** 게이트는 더러운 마당을 청소하고 나서 다는 거다. 운영 규모라면 PostgreSQL의 `NOT VALID → VALIDATE`로 게이트를 먼저 닫고 청소 검증을 뒤로 미뤄, 락 없이 부드럽게 박을 수 있다.
