# 왜 SAP 필드명은 이렇게 생겼나

*`LIFNR`, `MATNR`, `EBELN` — 이것들은 암호가 아니라 독일어 축약이다*

---

SAP WSDL을 처음 열었을 때 가장 먼저 부딪히는 벽은 필드명이다.

```
LIFNR, MATNR, EBELN, BANFN, EKGRP, WERKS, MWSKZ, LOEKZ, FRGKZ, BSTYP...
```

한 줄 읽는 데 5분이 걸린다. 알파벳 조합이 무작위 같지만, 사실은 **독일어 단어의 축약**이고, 규칙을 알고 나면 처음 보는 필드도 대강 추측할 수 있다. 이 글의 목표는 그 규칙을 머릿속에 심는 것이다.

---

## 왜 이렇게 짧게 줄였나

SAP은 1972년 독일에서 창업했다. 당시 메인프레임 환경의 DB 컬럼명 제한이 짧았고, 초기 SAP 제품군에서 ABAP Dictionary의 필드명 길이는 **최대 5자**로 굳어졌다. 이 5자 제한은 후대 제품에서 완화됐지만, 이미 굳어진 **SAP 표준 필드(standard fields)** 수만 개는 오늘날까지 그대로 남아 있다. 새로 만드는 커스텀 필드도 관행상 짧게 짓는다.

게다가 SAP 본사는 독일이고 초기 명명 규칙은 **독일어 기반**이다. 영어 ERP 제품에 익숙한 개발자에게 낯설게 보이는 이유가 여기 있다.

<Callout type="info" title="왜 아직도 안 바꾸나">
필드명 하나가 바뀌면 전 세계 수만 개 고객사의 ABAP 코드·인터페이스 정의서·문서가 전부 깨진다. SAP은 하위 호환을 신성시하는 제품이라 한 번 공개된 표준 필드명은 사실상 영구 보존된다. 우리가 1970년대식 독일어 약어를 2026년에도 읽어야 하는 이유다.
</Callout>

## 독일어 어원 해부

자주 등장하는 필드를 독일어 어원과 함께 뜯어보자. 한 번 보면 그 다음부터는 눈에 들어온다.

| 필드 | 독일어 어원 | 의미 |
|------|-------------|------|
| `LIFNR` | **Lief**erant + **Nr**. | 공급업체(Vendor) 번호 |
| `MATNR` | **Mat**erial + **Nr**. | 자재 번호 |
| `EBELN` | **E**in**bel**eg**n**ummer | 구매 문서(Einkaufsbeleg) 번호 — 발주·계약 공용 |
| `EBELP` | **E**in**bel**eg**p**osition | 구매 문서 항목 |
| `BANFN` | **Ban**f**n**ummer | 구매요청(Bestellanforderung) 번호 |
| `BANPO` | **Ban**f**po**sition | 구매요청 항목 |
| `ANFNR` | **Anf**rage**nr**. | 견적요청(RFQ, Anfrage) 번호 |
| `ANFPS` | **Anf**rage**p**o**s**ition | RFQ 항목 |
| `EKORG` | **E**in**k**aufs**org**anisation | 구매조직 |
| `EKGRP` | **E**in**k**aufs**gr**u**p**pe | 구매그룹 |
| `WERKS` | **Werk**s | 플랜트(공장, Werk) |
| `LGORT` | **L**a**g**er**ort** | 저장위치 |
| `BUKRS` | **Bu**chungs**kr**ei**s** | 회사코드(Buchungskreis) |
| `MENGE` | Menge | 수량 (그 자체가 독일어 단어) |
| `MEINS` | **Me**ng**ein**heit**s** | 기본 단위(Mengeneinheit) |
| `WAERS` | **Waer**s | 통화 키(Währung) |
| `NETWR` | **Net**to**w**e**r**t | 순 금액 |
| `NETPR` | **Net**to**pr**eis | 순 가격 |
| `LFDAT` | **Lie**ferdatum | 납기일(Lieferdatum) |
| `BEDAT` | **Be**leg**dat**um | 문서일자 |
| `AEDAT` | **Ae**nderungs**dat**um | 변경일자(Änderungsdatum) |
| `ERNAM` | **Er**fasser**nam**e | 생성자 이름 |

몇 개만 외우면 나머지는 패턴으로 풀린다. "Lieferant가 공급업체니까 납기일은 Lieferdatum이구나" 식의 연결이 된다.

<Callout type="note" title="왜 E/EK가 Einkauf(구매)인가">
독일어로 Einkauf = 구매, Verkauf = 판매. SAP 구매 모듈(MM)의 필드가 `EK-`로 시작하는 이유다. 영업(SD) 모듈에는 `VK-`로 시작하는 필드들이 있다. 한 번 이 매핑을 알면 모듈 소속을 접두사만 보고 추측할 수 있다.
</Callout>

## 접두사 패턴

SAP 필드는 같은 문서나 같은 도메인에 속한 필드끼리 **공통 접두사**를 공유한다. 접두사 사전이 있으면 처음 보는 필드도 어느 문서 소속인지 알 수 있다.

### 문서(Document) 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `BAN-` | Purchase Requisition (구매요청) | `BANFN`(번호), `BANPO`(항목) |
| `EBE-` | Purchase Order / Contract (발주·계약) | `EBELN`(번호), `EBELP`(항목) |
| `ANF-` | RFQ / Quotation (견적요청) | `ANFNR`(번호), `ANFPS`(항목) |

### 조직 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `EK-` | Einkauf(구매) | `EKORG`, `EKGRP` |
| `BU-` | Buchungskreis(회사코드) | `BUKRS` |
| `WE-` | Werk(공장) | `WERKS` |
| `LG-` | Lager(창고) | `LGORT` |

### 자재·수량 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `MAT-` | Material(자재) | `MATNR`(번호), `MATKL`(그룹) |
| `ME-` | Menge(수량) | `MENGE`, `MEINS` |

### 금액·가격 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `WAE-` | Währung(통화) | `WAERS` |
| `NET-` | Netto(순) | `NETWR`(금액), `NETPR`(가격) |
| `BRT-` | Brutto(총) | `BRTWR` |
| `PR-` | Preis(가격) | `PREIS`, `NETPR` |

### 날짜·시간 접미사

날짜는 접미사로 끝나는 경우가 많다.

| 접미사 | 의미 | 예시 |
|--------|------|------|
| `-DAT` / `-DT` | Datum(날짜) | `LFDAT`, `BEDAT`, `AEDAT` |
| `-TM` | Time (시간) | 커스텀 시간 필드에서 자주 사용 |

<Callout type="success" title="접두사로 모듈 추측하기">
처음 보는 필드를 마주쳤을 때 접두사만 봐도 대략 분류된다. `EK-`/`BAN-`/`EBE-`/`ANF-`로 시작하면 MM(구매) 필드고, `SAKTO`·`GSBER` 같은 계정 관련은 FI, `KOSTL`은 CO, `POSID`·`PSPID`는 PS(프로젝트) 쪽이다.
</Callout>

## Z-, Y- 접두사 — 커스텀 필드의 표식

SAP 표준 필드는 **Z**나 **Y**로 시작하지 않는다. 반대로 **Z나 Y로 시작하는 필드는 100% 고객사 커스텀**이다.

```
EBELN  ← 표준 발주번호 (SAP이 정의)
ZPO_CNFM_STAT  ← 고객사 커스텀 (해당 회사 ABAP 개발자가 추가)
ZRFQ_TRS_DT   ← 고객사 커스텀
```

이 규칙은 SAP이 예약한 **커스터마이징 네임스페이스**에서 온다. SAP 표준 업그레이드가 고객사 커스텀 필드를 덮어쓰지 않도록 Y·Z 접두사를 예약해뒀다. 실무에서 Y는 드물고 Z가 압도적으로 많다.

<Callout type="warning" title="Z-필드는 표준 문서에 없다">
SAP Help Portal이나 표준 테이블 가이드에 Z-필드는 절대 나오지 않는다. 그 회사 내부에서만 쓰는 필드이기 때문이다. **Z로 시작하는 필드의 의미를 모를 때는 SAP 담당자에게 묻는 것이 유일한 방법**이다. 구글 검색은 시간 낭비다.

그 회사의 ABAP 개발자가 의미를 정의했고, 보통은 인터페이스 정의서 문서에 설명이 남아 있다. WSDL 주석에 Description이 붙어 있을 수도 있다(5편에서 예제).
</Callout>

## 상태·플래그 필드 패턴

SAP 필드에는 "yes/no"나 "삭제됐는지" 같은 상태값이 많이 들어 있다. 자주 등장하는 패턴.

| 필드 | 독일어 | 의미 | 값 예시 |
|------|--------|------|---------|
| `LOEKZ` | **Lö**schkenn**z**eichen | 삭제 지시자 | 빈값=정상, `L`=삭제됨 |
| `STATU` | Status | 상태 | 문서 종류별로 의미 다름 |
| `FRGKZ` | **Fr**ei**g**abekenn**z**eichen | 승인 지시자 | `X`=승인됨, 빈값=미승인 |
| `BSTYP` | **Bel**eg**typ** | 문서 유형 | `F`=PO, `K`=Contract, `A`=RFQ |
| `ELIKZ` | **E**ndge**li**efert**kz** | 최종 납품 지시자 | `X`=완료 |

공통 패턴:
- `-KZ` = Kennzeichen(지시자) — "이 값이 뭔가를 표시한다"는 뜻의 접미사
- `X` 또는 특정 문자 한 글자로 플래그 표현. 불린이 `true/false`가 아니라 `'X'`/`''`(빈 문자열)다.
- 빈 값이 "의미 없음"이 아니라 **"정상 상태"** 또는 **"false"**인 경우가 많다.

<Callout type="error" title="빈 문자열 처리의 함정">
`LOEKZ` 필드가 빈 문자열이면 "정상"이고 `'L'`이면 "삭제됨"이다. 외부 시스템이 이걸 boolean으로 저장할 때 `!!loekz` 같은 truthy 체크를 쓰면 빈 문자열이 false로 떨어져서 "정상"이 된다. 맞는 결과긴 하지만 **명시적으로 `loekz === 'L'`로 비교**해야 다른 플래그 값이 들어왔을 때 안전하다.
</Callout>

## 처음 보는 필드를 마주쳤을 때 — 의사결정 트리

새 프로젝트에서 인터페이스 정의서를 받으면 모르는 필드가 수십 개 나온다. 다음 순서로 해석하면 혼자 힘으로 대부분 풀린다.

```
필드 이름 확인
    │
    ├── Z 또는 Y로 시작?
    │       ├── YES → 커스텀 필드. SAP Help Portal에 없음.
    │       │          인터페이스 정의서나 SAP 담당자에게 문의.
    │       └── NO  → 표준 필드. 다음 단계로.
    │
    ├── 접두사 3~4자가 위의 패턴표에 있나?
    │       ├── YES → 소속 도메인 추측 가능. 구글 검색 힌트.
    │       └── NO  → 직접 검색.
    │
    └── SAP Help Portal 검색 or "SAP field <이름>" 구글 검색
        → 대부분의 표준 필드는 공식 문서에 설명이 있음
```

실전에서 가장 많이 쓰게 되는 건 **구글 검색 `"SAP field <필드이름>"`**이다. 공식 문서, 커뮤니티 답변, 필드 딕셔너리 사이트 등이 검색 결과로 뜬다.

<Callout type="note" title="SAP Data Dictionary의 기원, DDIC">
SAP ABAP에는 **Data Dictionary(DDIC)**라는 메타데이터 저장소가 있다. 모든 표준 필드는 여기 정의돼 있고 설명·길이·타입·관계 정보를 담고 있다. T-Code `SE11`로 열 수 있다. 외부 개발자가 직접 접근할 일은 없지만, "이 필드는 DDIC에 정의된 표준이다"라는 말의 의미가 여기서 나온다.
</Callout>

## 정리

- SAP 필드명은 **1970년대 독일어 기반 5자 제약**의 유산이다. 무작위처럼 보여도 규칙이 있다.
- 자주 보는 필드는 어원을 한 번 뜯어보면 외워진다. `LIFNR=Lieferant(공급업체)+Nr(번호)`.
- **접두사로 도메인 추측**이 가능하다. `EK-`는 구매, `MAT-`는 자재, `NET-`는 순 금액, `-DAT`는 날짜.
- **Z·Y 접두사는 100% 고객사 커스텀**이다. 표준 문서에 없으니 구글에 묻지 말고 SAP 담당자에게 물어야 한다.
- 상태 플래그는 `X`/빈값의 한 글자 코드다. 빈 값이 "정상"인 경우가 많아 명시적 비교를 쓰는 것이 안전하다.

다음 편은 이 필드들이 실제로 **어떻게 생긴 값**으로 전달되는지를 다룬다. 숫자가 문자열로, 날짜가 `YYYYMMDD`로, 음수가 뒤에 마이너스가 붙어서 오는 이야기.
