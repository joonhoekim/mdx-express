## 이게 뭔데

뷰 도입(Introduce View)은 한 문장으로 이렇다. **기존 테이블 여러 개를 묶어서 만든 "저장된 SELECT문"에 이름을 붙여, 마치 테이블인 것처럼 쓰는 것**이다.

비유하자면 즐겨찾기다. 매번 고객 테이블, 계좌 테이블, 보험 테이블을 손으로 조인하고 합계를 내는 똑같은 쿼리를, 여기저기서 복붙해 쓰고 있다고 치자. 이걸 한 번 잘 짜서 `CustomerPortfolio`라는 이름으로 저장해두면, 그다음부터는 `SELECT * FROM CustomerPortfolio`만 치면 된다.

뷰는 데이터를 복사해서 따로 들고 있는 게 아니다. 그냥 **쿼리를 저장한 별명**일 뿐이다. 호출하는 순간 그 안의 SELECT가 실행되고, 항상 원본 테이블의 최신 데이터를 비춘다. 그래서 마이그레이션할 데이터가 없다. `CREATE VIEW` 한 줄이면 끝이고, 원본 행은 단 한 개도 안 건드린다.

여기까지는 책 그대로다. 그런데 이 변환은 **2006년의 전제 위에서 쓰였다는 점**을 짚고 가야 한다. 그때는 ORM이 성숙하지 않았고, DB 배포를 손으로 했고, 무엇보다 **하나의 DB를 여러 애플리케이션이 공유하는 게 기본값**이었다. 그 전제가 바뀌면 이 변환의 손익도 바뀐다.

## 먼저 물어야 할 것: 누가 이 DB를 보는가

"뷰를 도입할까?"의 진짜 질문은 ORM이 성숙했냐가 아니다. **이 변환이 풀려는 문제(흩어진 쿼리를 한 곳에 모으기)에서, 그 '한 곳'이 DB여야 하느냐**다. 그리고 그 답은 딱 하나로 갈린다.

<Callout type="info" title="결정 규칙 — 데이터 경계가 어디 있나">
- **이 DB를 보는 게 단 하나의 서비스뿐이라면** → 정의를 모을 곳은 그 서비스의 코드다. 뷰를 만들지 마라. service layer가 거의 모든 축에서 우위다.
- **DB를 보는 쪽이 여럿이라면**(리포팅 툴·타 팀 서비스·애드혹 SQL·ETL·polyglot) → 정의를 **모든 SQL 클라이언트에게** 강제할 유일한 지점은 뷰다. 이때 뷰가 밥값을 한다.

요지: **뷰는 코드 경계가 아니라 데이터 경계에 놓는 계약이다.** 경계가 코드 안에 있으면 함수로 모으고, 프로세스 밖으로 새면 뷰로 모은다.
</Callout>

책이 드는 다섯 가지 동기(보고용 요약, 중복 읽기 대체, 데이터 보안, 테이블 접근 캡슐화, SQL 중복 감소)는 전부 "흩어진 것을 한 곳에 모은다"는 한 가지 문제의 다른 얼굴이다. 그래서 위 규칙 하나로 거의 다 갈린다.

## 소비자가 하나면 — service layer가 낫다

요즘 흔한 형태부터 보자. **drizzle-orm을 쓰는 Node 서비스 하나가 자기 DB를 소유**한다. 이 DB를 직접 때리는 건 그 서비스뿐이다. 리포팅도 그 서비스의 API를 거친다.

이 상황에서 "활성 고객 정의가 일곱 군데에 복붙됐다"는 문제를 뷰로 푼다고 해보자. 가능은 하다. 그런데 더 나은 답은 그 정의를 **타입이 끝까지 흐르는 재사용 가능한 조건절**로 빼는 것이다.

```ts
// 활성 고객의 정의를 한 곳에 — 그런데 그 '한 곳'은 코드다
export const activeCustomer = () =>
  and(eq(customer.status, 'A'), isNull(customer.closedDate));

// 호출부: 뷰의 FROM을 갈아끼우는 것과 똑같은 DRY 효과
db.select({ id: customer.customerId, name: customer.name, phone: customer.phoneNumber })
  .from(customer)
  .innerJoin(insurance, eq(insurance.customerId, customer.customerId))
  .where(and(activeCustomer(), gte(insurance.startDate, '2026-04-01')));
```

"휴면 고객도 활성에서 빼주세요"가 와도 고칠 곳은 `activeCustomer()` **한 곳**이다. 뷰로 모으는 것과 동일한 단일 진실. 그런데 뷰 대비 이점이 분명하다.

- **단일 진실·단일 배포.** 뷰는 마이그레이션 세계에 살고, 소비자는 TS 세계에 산다. "ActiveCustomer가 어떤 컬럼을 갖나"가 두 곳에 존재하고 lockstep을 강요당한다. 함수는 한 곳, 한 PR, 한 배포다.
- **타입 안전성.** drizzle은 스키마에서 결과 타입을 추론한다. 뷰는 그 추론 바깥이라 타입을 다시 선언하거나 introspect해야 한다 — 경계가 하나 깨진다. 함수는 스키마 타입을 그대로 물고 흐른다.
- **테스트·리뷰.** 조건절은 로직을 쓰는 코드 바로 옆에서 리뷰되고, 쿼리 빌드 결과를 DB 왕복 없이 검증할 수 있다.
- **rename 캡슐화가 짐이 안 된다.** 컬럼을 rename하면 drizzle 스키마와 `activeCustomer()`를 **같은 PR에서 원자적으로** 고친다. 뒤에서 보겠지만, 뷰의 "rename 방패"는 소비자가 내 배포 주기와 다를 때만 가치가 있다 — 한 레포·한 배포면 방패가 오히려 유지 비용이다.

집계 뷰(`CustomerPortfolio`)도 마찬가지다. 그 3-way 조인과 `SUM`은 drizzle 쿼리로 그대로 표현할 수 있고, 재사용이 필요하면 함수로 감싸면 된다. 뷰가 주는 추가 가치가 사실상 없다.

<Callout type="note" title="'ORM의 추상화를 쓰자'와는 다른 얘기">
이건 "drizzle이 알아서 다 해주니 SQL을 몰라도 된다"는 게 아니다. 정반대로, **무슨 SQL이 나가는지 알면서** 그 정의를 코드 한 곳에 약속으로 못 박자는 거다. 조건절을 함수로 빼는 건 추상화 레이어를 늘리는 게 아니라, 같은 SQL을 타입과 함께 공유하는 것뿐이다. 책임을 뷰라는 별도 아티팩트로 분산시키지 않는다.
</Callout>

## 소비자가 여럿이면 — 뷰가 밥값을 한다

이제 경계를 넘겨보자. `activeCustomer()` 함수는 **TS 호출자만** 보호한다. DB를 직접 보는 다른 클라이언트에게는 그 함수가 존재하지 않는 것과 같다. 여기서 뷰만 할 수 있는 일이 두 가지 있다.

**1. out-of-process / polyglot 소비자에게 정의를 강제한다.** Metabase·Tableau·Looker 대시보드, 데이터팀의 파이썬 노트북, 다른 팀 서비스, 분석가의 애드혹 SQL, 야간 ETL. 이것들이 raw 테이블을 직접 때리면 네 TS 조건절은 아무것도 못 한다. "활성 고객"의 정의를 **모든 SQL 클라이언트가 공유**하게 만드는 유일한 지점이 뷰다. 책의 동기 2·5(중복 읽기 대체, SQL 중복 감소)와 동기 1(보고용 요약)이 진짜로 빛나는 건 바로 이 다중 소비자 환경이다.

**2. DB가 강제하는 보안 경계를 만든다.** 원본 테이블에 민감 컬럼이 섞여 있고 외부 사용자에게 일부만 노출하고 싶을 때, 뷰에 필요한 컬럼만 담고 사용자에게는 원본 테이블이 아니라 **뷰에만 `SELECT` 권한**을 준다(책의 동기 3). 앱 레이어는 DB 직결 클라이언트의 접근을 revoke할 수 없지만 `GRANT`는 할 수 있다. 이건 코드로 대체 불가능하다.

**3. 테이블 접근 캡슐화 — 단, 경계가 프로세스 밖일 때만.** 책이 가장 강조하는 동기 4다. 외부 프로그램이 원본 테이블 대신 뷰를 거치면, 나중에 `Rename Column`/`Rename Table`을 할 때 뷰 정의만 고치면 그 프로그램은 그대로 돌아간다. 핵심은 **"외부"가 진짜 외부일 때**(다른 팀, 다른 배포 주기) 가치가 있다는 점이다. 같은 레포 안에서는 앞 절처럼 원자적 변경이 더 단순하다.

<Callout type="success" title="반(反)책 입장이 아니다">
이 시리즈 안에는 `Replace View with Method`(74편)가 있다. 책 스스로 "뷰를 메서드(코드)로 되돌리는" 역방향 변환을 정식 리팩토링으로 인정한다. 즉 "뷰냐 코드냐"는 책도 양방향으로 열어둔 선택이고, 경계가 어디 있느냐에 따라 답이 달라진다는 게 핵심이다.
</Callout>

## 주의할 점 (뷰를 택했을 때)

다중 소비자라서 뷰로 가기로 했다면, 두 가지를 머리에 새겨두자.

<Callout type="warning" title="뷰는 공짜 추상화가 아니다">
- **성능은 뷰 안의 쿼리 그대로다.** 뷰는 데이터를 미리 계산해두지 않는다(일반 뷰 기준). `SELECT * FROM CustomerPortfolio` 한 줄이 깔끔해 보여도, 호출할 때마다 안에 든 3-way 조인과 `SUM` 집계가 통째로 다시 돈다. "테이블처럼 보여서" 사람들이 부담 없이 막 호출하다가 성능 사고를 친다. 조인이 응답 시간에 안 맞으면 뷰가 아니라 materialized view 쪽을 봐야 한다.
- **결합도가 올라간다.** 뷰 정의는 기반 테이블의 컬럼·구조에 묶여 있다. 원본 구조를 바꾸면 뷰도 같이 고쳐야 한다. (다만 다중 소비자 환경에선, 결합 지점을 소비자 수십 곳에서 뷰 한 곳으로 **모으는** 거니 총량으론 관리가 쉬워진다. 단일 소비자라면 이 이점이 없으니 앞 절로 돌아가라.)
</Callout>

뷰를 너무 겹쳐 쌓는 것도 조심하자. 뷰 위에 뷰, 그 위에 또 뷰를 올리면, 호출 한 번에 옵티마이저가 5단 중첩 쿼리를 펼쳐야 한다. EXPLAIN이 괴물 실행 계획을 토해낸다면 뷰 중첩이 범인인 경우가 많다.

## 이렇게 한다

은행 도메인으로 두 경로를 다 보여주자. 고객(`Customer`)·계좌(`Account`)·보험(`Insurance`) 세 테이블이 있고, "활성 고객" 정의가 쿼리마다 복붙된 상황이다.

<Steps>

<Step title="경로 A — 소비자가 하나면: service layer로 모은다">

DB를 보는 게 이 서비스뿐이라면, 흩어진 정의를 drizzle 조건절로 추출하고 끝낸다. 스키마 변경도, 마이그레이션도, 새 배포 아티팩트도 없다.

```ts
// 정의는 코드 한 곳에 — 타입은 스키마에서 그대로 흐른다
export const activeCustomer = () =>
  and(eq(customer.status, 'A'), isNull(customer.closedDate));

// Before: 활성 고객 조건이 쿼리마다 복붙됨 → After: activeCustomer() 한 번
const newInsuranceCustomers = await db
  .select({ id: customer.customerId, name: customer.name, phone: customer.phoneNumber })
  .from(customer)
  .innerJoin(insurance, eq(insurance.customerId, customer.customerId))
  .where(and(activeCustomer(), gte(insurance.startDate, '2026-04-01')));
```

"휴면 고객도 빼주세요"가 오면 `activeCustomer()`에 `gte(customer.lastTxnDate, ...)` 한 줄만 더한다. 그 함수를 쓰는 모든 쿼리가 동시에 갱신되고, 타입체커가 빠뜨린 호출부를 잡아준다.

</Step>

<Step title="경로 B — 소비자가 여럿이면: CREATE VIEW로 모은다">

리포팅 툴·타 팀·직결 SQL이 같은 테이블을 본다면, 그들 모두에게 정의를 강제할 곳은 뷰뿐이다. 마이그레이션할 데이터가 없으니 DDL 한 방이다.

```sql
-- 활성 고객의 정의를 모든 SQL 클라이언트에게 강제
CREATE VIEW ActiveCustomer AS
SELECT *
FROM Customer
WHERE Status = 'A'
  AND ClosedDate IS NULL;

-- 보고용 요약 뷰
CREATE VIEW CustomerPortfolio (
  CustomerID, Name, PhoneNumber,
  AccountsTotalBalance, InsuranceTotalPayment, InsuranceTotalValue
) AS
SELECT
  c.CustomerID, c.Name, c.PhoneNumber,
  COALESCE(SUM(a.Balance), 0),
  COALESCE(SUM(i.Payment), 0),
  COALESCE(SUM(i.Value), 0)
FROM Customer c
LEFT JOIN Account   a ON a.CustomerID = c.CustomerID
LEFT JOIN Insurance i ON i.CustomerID = c.CustomerID
GROUP BY c.CustomerID, c.Name, c.PhoneNumber;
```

<Callout type="note" title="2006년 책과 살짝 다르게 쓴 이유">
원서 예제는 `FROM Customer, Account, Insurance WHERE ...` 식의 옛날 콤마 조인에 `SUM`만 걸려 있다. 그대로 두면 두 함정이 있다. (1) 계좌나 보험이 없는 고객은 INNER JOIN이라 결과에서 통째로 사라진다. (2) 계좌 2개·보험 3개인 고객은 카테시안 곱으로 행이 6배 부풀어 `SUM`이 뻥튀기된다. 현대식으론 `LEFT JOIN` + 명시적 `GROUP BY`로 쓰고, 곱 부풀림이 걱정되면 각 집계를 서브쿼리로 따로 묶는다. **뷰 안의 SQL이 곧 모두가 믿게 될 진실**이니 "뷰니까 대충"이 제일 위험하다.
</Callout>

뷰를 쓰는 쪽은 그냥 **읽기 전용 테이블 하나**로 본다. 정의가 바뀌면 `ALTER VIEW`(또는 `CREATE OR REPLACE VIEW`) 한 곳만 고치면 모든 소비자가 동시에 갱신된다. drizzle 서비스도 이 뷰를 읽기 전용 엔티티로 매핑해 일반 테이블처럼 쓰면 된다.

</Step>

</Steps>

## 현대 실무로 강화하기

뷰 경로를 택했다면, 2006년의 `CREATE VIEW` 한 줄에 얹을 도구가 지금은 훨씬 많다.

<TabsContent value="migration">

뷰도 스키마 변경이다. 운영 DB에 손으로 `CREATE VIEW`를 치는 대신, **Flyway / Liquibase / Alembic**(또는 drizzle-kit) 같은 버전 관리 마이그레이션으로 넣는다. 데이터가 없어서 특히 다루기 좋다.

```sql
-- V96__introduce_active_customer_view.sql (Flyway)
CREATE OR REPLACE VIEW ActiveCustomer AS
SELECT *
FROM Customer
WHERE Status = 'A'
  AND ClosedDate IS NULL;
```

`CREATE OR REPLACE VIEW`는 멱등이라 정의가 바뀔 때마다 새 버전 파일로 안전하게 교체된다. 데이터 백필이 없으니 롤백도 `DROP VIEW ActiveCustomer;` 한 줄이다.

</TabsContent>

<TabsContent value="matview">

"뷰는 호출할 때마다 안의 쿼리가 다시 돈다" 문제. 무거운 집계 뷰를 리포트가 수시로 때린다면, 일반 뷰 대신 **materialized view**로 결과를 미리 계산해 저장한다. 책의 Introduce Read-Only Table이 현대 DB에선 이거다.

```sql
-- PostgreSQL: 집계 결과를 물리적으로 저장
CREATE MATERIALIZED VIEW CustomerPortfolio_MV AS
SELECT
  c.CustomerID, c.Name,
  COALESCE(SUM(a.Balance), 0) AS AccountsTotalBalance
FROM Customer c
LEFT JOIN Account a ON a.CustomerID = c.CustomerID
GROUP BY c.CustomerID, c.Name;

CREATE UNIQUE INDEX ON CustomerPortfolio_MV (CustomerID);
REFRESH MATERIALIZED VIEW CONCURRENTLY CustomerPortfolio_MV;
```

대가는 **데이터가 실시간이 아니라는 것**. `REFRESH` 전까지는 마지막 갱신 시점의 스냅샷이다. "약간 오래된 데이터여도 괜찮은" 리포트·대시보드용이지, 1원 단위로 최신이어야 하는 잔고 조회엔 부적합하다. 실시간성과 속도가 둘 다 필요하면 CDC(Debezium 등)로 변경분을 흘려 집계 테이블을 점진 갱신하는 쪽으로 더 나간다.

</TabsContent>

<TabsContent value="ownership">

캡슐화 동기는 마이크로서비스에서 그대로 빛난다. 어떤 서비스가 다른 팀 소유 DB 테이블을 직접 읽으면 그 구조에 발이 묶인다. 대신 소유 팀이 **읽기 전용 뷰를 공개 계약(published interface)으로 노출**하면, 내부 테이블은 자유롭게 리팩토링하면서 뷰의 컬럼 시그니처만 유지하면 된다.

이건 정확히 이 글의 결정 규칙이 "팀 경계"로 올라간 버전이다. 캡슐화 계층이 "앱과 테이블 사이"가 아니라 "팀과 팀 사이"에 놓인다 — 즉 데이터 경계가 프로세스 밖이라서 뷰가 정당해진다. 단, 뷰 공유도 강한 결합이니, 진짜 독립이 필요하면 뷰 공유보다 이벤트(outbox/CDC) 기반 복제가 정석이다.

## 정리

뷰 도입은 변환 중 가장 싸다. 데이터를 안 옮기니 적용도 롤백도 한 줄이다. 하지만 "싸다"가 "항상 옳다"는 아니다. 이 변환이 푸는 문제는 **흩어진 정의를 한 곳에 모으는 것**이고, 2026년의 진짜 질문은 그 "한 곳"이 DB여야 하느냐다.

> **데이터 경계가 코드 안에 있으면(소비자가 하나면) service layer의 타입 공유 함수로 모아라. 경계가 프로세스 밖으로 샐 때만(여러 소비자·polyglot·DB 보안) 뷰가 밥값을 한다.**

drizzle 단일 서비스가 DB를 소유한다면, "활성 고객" 같은 정의는 뷰가 아니라 재사용 가능한 조건절로 빼는 게 낫다 — 단일 배포, 끝까지 흐르는 타입, DB 왕복 없는 테스트를 얻는다. 뷰는 그 정의를 코드 밖 클라이언트에게까지 강제해야 할 때, 혹은 DB가 강제하는 보안 경계가 필요할 때 꺼내라. 같은 SQL 조각을 두 군데 이상에서 복붙하고 있다면 모으라는 신호인 건 맞다 — 다만 어디에 모을지는 누가 그걸 보느냐가 정한다.
