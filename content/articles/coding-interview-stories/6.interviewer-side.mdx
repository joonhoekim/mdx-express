시니어 되면 피할 수 없는 업무가 하나 있음.

**면접관.**

누구도 가르쳐주지 않음. "다음 주부터 면접 들어가세요" 라는 슬랙 메시지 하나 받고
갑자기 남의 인생을 판단해야 하는 자리에 앉게 됨.

면접 보는 것보다 면접 보게 하는 게 더 힘든 이유?
**면접은 틀리면 내가 떨어지고, 면접관은 틀리면 팀이 망하기 때문.**

---

## 면접관이 되면 알게 되는 것들

### 1. 이력서는 대체로 "가장 좋게 쓴 버전"이다

이력서 스크리닝을 시작하면 바로 깨달음. 새빨간 거짓말은 드물지만, *가장 유리한 각도로 쓴* 건 거의 전부임.

```
이력서에 적힌 것:
"MSA 기반 대규모 서비스 설계 및 운영"

면접에서 드러나는 현실:
"기존 모놀리스에서 API 하나 분리한 거"

이력서에 적힌 것:
"팀 리드로서 5명의 개발자 관리"

현실:
"스크럼 마스터 대리(?)로 데일리 스탠드업 진행한 거"

이력서에 적힌 것:
"서비스 성능 300% 개선"

현실:
"인덱스 하나 추가한 거"
(근데 솔직히 인덱스 하나로 300% 개선은 가능함 ㅋㅋ)
```

오해는 말 것 — 이게 *지원자의 90%가 거짓말쟁이* 라는 뜻은 아님. 대부분은 "한 일을 가장 좋게 표현한 것" 에 가깝고, 그건 이력서의 정상적인 기능임. 면접관의 일은 *거짓말을 잡는 것* 이 아니라 **"부각된 표현" 을 걷어내고 실제 기여의 크기를 가늠하는 것**. "MSA 운영" 이라고 쓰여 있으면 거짓이라 의심하는 게 아니라 "구체적으로 어느 부분을 직접?" 을 물어 *해상도를 높이는* 게 면접임.

<Callout type="warning" title="이력서 과장의 스펙트럼">
**허용 가능한 범위:**
- 역할을 약간 더 부각시키는 것
- 팀 성과를 본인 기여 관점에서 쓰는 것
- 기술 스택을 넓게 나열하는 것

**위험한 범위:**
- 하지 않은 일을 했다고 쓰는 것
- 팀 성과를 혼자 한 것처럼 쓰는 것
- 사용 경험 없는 기술을 "능숙"이라고 쓰는 것

면접에서 이력서를 기반으로 질문하니까,
**이력서에 쓴 모든 것은 30분 동안 설명할 수 있어야 함.**
"그냥 넣어본 건데..." 하면 신뢰가 무너짐.
</Callout>

### 2. 기술력만으로 판단하면 안 됨

처음 면접관 할 때 가장 많이 하는 실수:
"기술적으로 뛰어난 사람 = 좋은 채용"이라고 생각하는 것.

```
실제 채용 후 일어나는 일:

Case A: 기술 10점, 협업 3점
→ 코드는 잘 짜는데 코드 리뷰에서 항상 싸움
→ 팀 분위기 악화
→ 다른 팀원들 이직 고민 시작
→ 결국 순손실

Case B: 기술 7점, 협업 9점
→ 코드는 괜찮은 수준인데 팀원들과 시너지
→ 다른 팀원들 수준도 올라감
→ 팀 전체 생산성 향상

이걸 겪어봐야 "왜 행동 면접이 중요한지" 진짜 깨달음.
```

> 단, 현실의 결정은 이렇게 *10점 vs 3점* 처럼 극명하지 않음. 거의 항상 (기술 8, 협업 7) vs (기술 7, 협업 8) 같은 *미세한 차이* 에서 골라야 함. 그래서 진짜 어려운 질문은 "기술 vs 협업, 뭐가 더 중요해?" 가 아니라 **"이 포지션에서, 이 팀 구성에서, 지금 더 부족한 게 뭔가?"** 임. 시니어가 부족한 팀이면 기술 쪽에, 사일로가 심한 팀이면 협업 쪽에 가중치가 감. 정답은 *팀의 현재 상태* 가 정하지, 절대적 우선순위가 정하지 않음.

### 3. "합격/불합격" 결정의 무게

```
합격시키면:
- 이 사람의 연봉 (수천만원)이 회사 비용
- 온보딩 기간 (1-3개월) 동안 팀 생산성 하락
- 안 맞으면 수습 종료 = 그 사람 인생에 영향
- 팀 문화에 미치는 영향

불합격시키면:
- 혹시 좋은 인재를 놓친 건 아닌가?
- 면접 컨디션이 안 좋았던 건 아닌가?
- 내 질문이 공정했나?
- 이 사람의 다음 면접은 언제일까...
```

<Callout type="note" title="면접관의 책임감">
좋은 면접관은 **"이 사람을 떨어뜨릴 이유"**가 아니라
**"이 사람을 뽑아야 할 이유"**를 찾으려고 노력함.

불합격 사유가 "그냥 느낌이 안 좋았다"면 그건 면접관의 실패임.
구체적이고 설명 가능한 이유가 있어야 함.

"코딩 문제를 풀었지만 시간복잡도 분석을 못 했다"
"시스템 설계에서 캐싱 전략을 전혀 고려하지 않았다"
"팀 갈등 경험에서 자신의 역할을 설명하지 못했다"

이런 구체적 피드백을 쓸 수 있어야 함.
</Callout>

## 좋은 면접 질문 만들기의 어려움

면접관이 되면 "뭘 물어보지?"부터 고민됨.

### 나쁜 면접 질문의 특징

```
1. 정답만 있는 질문
   "JavaScript에서 var와 let의 차이점은?"
   → 구글 검색 3초면 나오는 걸 왜 물어봄?
   → 암기력을 테스트하는 건 의미 없음

2. 너무 모호한 질문
   "좋은 코드란 뭔가요?"
   → 답이 너무 열림. 지원자도 뭘 말해야 할지 모름
   → 면접관도 뭘 평가해야 할지 모름

3. 트릭 질문
   "typeof null이 뭔가요?" (정답: 'object')
   → 이거 맞힌다고 좋은 개발자? 아님
   → 자존심만 상하게 함

4. 면접관 자기 자랑 질문
   "저희 팀에서 사용하는 [매우 특수한 기술]에 대해 알고 있나요?"
   → 지원자가 모르는 게 당연한 걸 물어보고
   → 모르면 면접관이 설명해주면서 자기 자랑
```

### 좋은 면접 질문의 특징

```
1. 단계적으로 깊어지는 질문
   "배열에서 중복을 제거하는 방법을 설명해주세요"
   → Set 말하면: "Set의 내부 구현은?"
   → 해시 말하면: "해시 충돌은 어떻게 처리하나요?"
   → 체이닝 말하면: "시간복잡도가 최악인 경우는?"
   → 지원자 수준에 맞춰 깊이 조절 가능

2. 정답이 여러 개인 질문
   "이 코드의 성능을 개선하려면?"
   → 여러 접근 방식이 가능하고, 각각의 트레이드오프를 논의할 수 있음

3. 실무와 연결된 질문
   "이런 상황에서 어떻게 하실 건가요?"
   → 실제 업무에서 만날 수 있는 시나리오
   → 문제 해결 과정을 볼 수 있음

4. 열린 토론이 가능한 질문
   "마이크로서비스와 모놀리스, 어떤 상황에서 뭘 선택하실 건가요?"
   → 정답이 없고 맥락에 따라 달라짐
   → 지원자의 경험과 판단력을 볼 수 있음
```

<Callout type="success" title="면접 질문 설계 프레임워크">
좋은 면접 질문을 만들려면 이 체크리스트를 확인:

- **관련성**: 실제 업무와 연관이 있는가?
- **변별력**: 수준에 따라 다른 답이 나오는가?
- **공정성**: 특정 배경을 가진 사람에게 유리하지 않은가?
- **평가 가능성**: 답변을 객관적으로 평가할 수 있는가?
- **시간 적합성**: 주어진 시간 내에 답할 수 있는가?

이 중 하나라도 안 되면 질문을 수정해야 함.
</Callout>

> 그런데 이 체크리스트를 *다 만족하는 질문을 만드는 것 자체가 어렵다* 는 게 면접관이 잘 말 안 하는 부분임. **공정성과 변별력이 자주 충돌함** — 변별력을 높이려고 어려운 질문을 내면 특정 배경(CS 전공, 특정 회사 경험)에 유리해져 공정성이 깨짐. **일관성 유지도 어려움** — 같은 질문이라도 그날 5번째 면접에선 면접관이 지쳐 설명을 대충 하거나 힌트를 더 주게 됨. 그래서 좋은 팀은 *질문 은행 + 채점 루브릭* 을 문서로 관리하고, 정기적으로 "캘리브레이션"(면접관끼리 같은 답변에 같은 점수를 주는지 맞춰보기)을 함. 좋은 질문은 *개인의 재능* 이 아니라 *팀 차원의 유지보수* 의 산물임.

## 채용 프로세스의 현실

### 이력서 스크리닝

```
현실:
- 한 포지션에 이력서 100-300개 옴
- 스크리닝에 쓸 수 있는 시간: 이력서 당 30초-2분
- HR이 1차 스크리닝 → 개발팀에 20-30개 전달
- 개발팀이 2차 스크리닝 → 10-15명 면접 대상 선정

30초 안에 이력서에서 보는 것:
1. 최근 경력 (현 직장/최근 프로젝트)
2. 기술 스택 매칭 (우리가 쓰는 기술을 아는가)
3. 경력 연차 (우리가 원하는 레벨인가)
4. 특이사항 (오픈소스 기여, 블로그, 컨퍼런스 발표 등)
```

```
이력서 스크리닝 시 빨간불:
- 오타가 많음 (디테일 부족)
- 모든 기술을 "상" 또는 "Expert"로 표시 (과장 의심)
- 프로젝트 설명이 너무 추상적 ("서비스 개발" = 뭘 한 건지?)
- 이직이 너무 잦음 (6개월마다) → 무조건 탈락은 아니지만 질문 대상
- GitHub 링크 눌렀는데 잔디가 0 (개발에 관심이 없나?)

초록불:
- 구체적 수치가 있음 ("트래픽 50% 증가에 대응", "빌드 시간 3분 → 40초")
- 기여도가 명확함 ("5명 팀에서 API 서버 담당")
- 기술 블로그나 오픈소스 활동
- 이력서 형식이 깔끔하고 읽기 쉬움
```

> 면접관을 오래 하면 *이 신호들이 자주 배신한다* 는 걸 알게 됨. **이력서는 화려한데 면접은 비는 사람** — 글쓰기를 잘하거나 좋은 팀에 묻어간 경우. **이력서는 수수한데 면접이 빛나는 사람** — 자기 PR에 서툴 뿐 실력은 단단한 경우로, 특히 과묵한 엔지니어 유형에서 흔함. 그래서 노련한 스크리너는 이력서를 *합불 판정* 이 아니라 **면접에서 무엇을 검증할지의 가설** 로 씀. "수치가 구체적이네 → 진짜 본인이 한 건지 깊게 파보자", "설명이 추상적이네 → 한 단계만 더 물으면 드러나겠지". 빨간불이 *탈락 사유* 가 아니라 *질문 거리* 가 되는 것.

<Callout type="info" title="ATS(지원자 추적 시스템)의 현실">
대기업은 ATS를 사용함. 이력서를 시스템이 먼저 필터링함.

**ATS가 보는 것:**
- 키워드 매칭 (JD에 있는 기술 스택이 이력서에 있는가)
- 포맷 (PDF가 안전. 예쁜 디자인보다 텍스트 추출이 잘 되는 게 중요)
- 경력 연수

**ATS를 통과하려면:**
- JD에 있는 키워드를 이력서에 포함 (자연스럽게)
- 심플한 포맷 사용 (표, 그래프, 이미지는 파싱이 안 될 수 있음)
- PDF로 제출

**스타트업은?**
ATS 안 쓰는 곳도 많음. CTO가 직접 이력서 다 읽음.
이 경우 "사람 냄새"가 나는 이력서가 더 유리할 수 있음.
</Callout>

### 면접 프로세스 설계

```
일반적인 개발자 채용 파이프라인:

1단계: 이력서 스크리닝 (통과율 ~20-30%)
2단계: 코딩 테스트 or 과제 (통과율 ~40-50%)
3단계: 기술 면접 (1-2회) (통과율 ~30-40%)
4단계: 행동 면접 / 컬쳐핏 (통과율 ~50-60%)
5단계: 레퍼런스 체크 / 처우 협상

전체 파이프라인 통과율: ~3-10%
(100명 지원하면 3-10명 합격)
```

### 합불 결정의 어려움

```
면접 끝나면 면접관끼리 디브리핑을 함.

디브리핑에서 자주 벌어지는 상황:

면접관 A: "기술적으로는 좋았는데 소통이 좀..."
면접관 B: "소통은 괜찮았는데 코딩 실력이 좀..."
면접관 C: "저는 전체적으로 괜찮았다고 생각하는데요?"

이러면 어떻게 해야 함?

실제로는:
1. 각 면접관이 독립적으로 점수를 매김 (사전에 다른 면접관 의견 공유 X)
2. 구체적 근거를 기반으로 토론
3. "Strong Hire / Hire / No Hire / Strong No Hire" 같은 스케일 사용
4. 한 명이라도 "Strong No Hire"면 보통 불합격
5. 전원 합의가 어려우면 추가 면접 또는 매니저 결정
```

<Callout type="error" title="채용에서 가장 비싼 실수">
**False Positive (안 맞는 사람을 뽑는 것):**
- 온보딩 비용 (급여 3-6개월분)
- 팀 생산성 저하
- 수습 종료 시 법적/감정적 비용
- 다시 채용하는 비용과 시간
- **총 비용: 연봉의 1.5-3배**

**False Negative (맞는 사람을 놓치는 것):**
- 그 사람이 만들어낼 수 있었던 가치를 잃음
- 팀의 성장 기회를 잃음
- 채용 기간이 길어짐
- 근데 이건 **측정이 불가능함** (놓친 사람이 얼마나 좋았을지 알 수 없으니까)

대부분의 회사는 False Positive를 더 두려워해서,
**합격 기준이 불합격 기준보다 높음.**
"확실히 좋다"가 아니면 안 뽑는 경향.
이게 지원자 입장에서는 답답하지만, 회사 입장에서는 합리적인 선택임.

다만 *어느 쪽을 더 무서워하는지는 회사 규모에 따라 갈림*:
- **스타트업** — 한 명의 임팩트가 크고 자리가 비면 당장 제품이 안 나옴. 그래서 *False Negative(좋은 사람 놓치기)* 를 더 무서워하고 "일단 같이 해보자" 쪽으로 기욺. 대신 안 맞으면 빠르게 정리.
- **대기업** — 한 명 잘못 뽑아도 시스템으로 버티지만 해고가 어렵고 잘못된 채용의 *제도적 비용* 이 큼. 그래서 *False Positive* 를 더 무서워하고 면접 라운드가 많아짐.

같은 실력이어도 *스타트업에선 붙고 대기업에선 떨어지는* 일이 생기는 건 평가가 달라서가 아니라 **두 회사가 서로 다른 종류의 실수를 더 무서워하기 때문** 임.
</Callout>

## 면접관으로서 받는 스트레스와 책임감

### 스트레스 1: 시간 부족

```
1시간 면접으로 이 사람의 3-5년 경력을 판단해야 함.

1시간 동안 할 수 있는 것:
- 알고리즘 문제 1-2개
- 시스템 디자인 문제 1개
- 행동 질문 3-4개
- 지원자 질문 시간

이것만으로 이 사람이 팀에 적합한지 판단?
**불가능에 가까움.** 그래서 여러 라운드로 나눠서 봄.
```

### 스트레스 2: 무의식적 편향

```
면접관도 사람이라 편향이 있음:

- 후광 효과: 좋은 대학 출신 → 코딩도 잘할 거야
- 유사성 편향: 나와 비슷한 배경 → 호감
- 확증 편향: 첫인상이 좋으면 → 장점만 보임
- 앵커링: 첫 번째 지원자가 잘하면 → 기준이 높아짐

이걸 인식하고 있어도 완전히 없앨 수는 없음.
그래서 구조화된 면접(같은 질문, 같은 기준)이 중요함.
```

### 스트레스 3: 불합격 통보

```
합격 통보: "축하드립니다!" (보내기 쉬움, 기분 좋음)

불합격 통보: "검토 결과..." (보내기 어려움, 기분 안 좋음)

특히 면접에서 열심히 한 지원자에게 불합격 통보를 보낼 때,
"이 사람이 더 준비해서 다음에는 합격할 수 있을까?"
"내가 잘못 판단한 건 아닌가?"
이런 생각이 듦.

더 스트레스인 건:
좋은 피드백을 주고 싶어도 법적 이슈 때문에
구체적 불합격 사유를 알려줄 수 없는 경우가 많음.
"전체적인 역량을 종합적으로 판단한 결과..."
이 문장이 나오는 이유가 이거임.
```

### 스트레스 4: 우리 팀에 대한 세일즈

```
면접은 일방적 평가가 아님. 지원자도 우리를 평가하고 있음.

면접관이 해야 하는 것:
- 팀의 매력을 어필 (기술 스택, 문화, 성장 기회)
- 솔직하게 단점도 말하기 (진짜 열려있는 팀이라는 인상)
- 지원자의 질문에 성실하게 답하기

좋은 지원자일수록 다른 회사 옵션도 많으니까,
면접을 잘 봐야 합격시켜도 와줌.
면접 경험이 안 좋으면 합격해도 안 오는 경우도 있음.
```

<Callout type="note" title="면접관 교육의 현실">
많은 회사에서 면접관 교육을 제대로 안 함.

**이상적인 면접관 온보딩:**
1. 면접 가이드 문서 제공
2. 시니어 면접관 옆에서 3-5회 관찰
3. 함께 면접 진행하면서 피드백 받기
4. 단독 면접 + 사후 리뷰
5. 정기적 캘리브레이션 (면접관 간 기준 맞추기)

**현실:**
"다음 주부터 면접 들어가세요. 질문은 기존 거 참고하세요."
→ 끝.

그래서 면접 품질이 면접관에 따라 천차만별인 거임.
지원자가 "면접이 불공정했다"고 느끼는 경우의 상당수가
면접관 교육 부재에서 비롯됨.
</Callout>

## 면접관이 되어서 배운 것들

### 1. 질문하는 것보다 듣는 게 더 중요하다

```
초보 면접관: 질문 리스트를 순서대로 물어봄 (체크리스트 모드)
숙련 면접관: 지원자의 답변에서 다음 질문을 찾음

좋은 면접은 대화처럼 흘러감.
지원자가 "저는 성능 최적화를 많이 해봤어요"라고 하면
준비된 다음 질문 대신 "어떤 종류의 최적화를 해보셨나요?"로 이어감.

이렇게 해야 지원자의 진짜 역량을 볼 수 있음.
체크리스트 모드로 하면 "준비된 답변"만 듣게 됨.
```

### 2. 침묵을 두려워하지 마라

```
지원자가 생각하는 중일 때 면접관이 해야 할 것:
✅ 기다려준다 (30초-1분은 OK)
✅ "천천히 생각하셔도 됩니다"라고 말해준다
✅ 필요하면 힌트를 준다

면접관이 하면 안 되는 것:
❌ 답을 바로 말해버림
❌ 불편한 표정을 짓거나 시계를 봄
❌ "모르시면 다음 질문으로..."
```

### 3. 면접관의 태도가 지원자의 퍼포먼스에 영향을 줌

```
적대적 면접관:
- 팔짱 끼고 앉아있음
- 질문에 "아니요, 그게 아니라" 반복
- 표정이 없음
→ 지원자 위축 → 실력의 50%도 못 보여줌

우호적 면접관:
- 고개를 끄덕이며 들음
- "좋은 접근이네요" 같은 긍정적 반응
- 편안한 분위기 조성
→ 지원자가 편해짐 → 실력을 제대로 보여줌

같은 사람이어도 면접관에 따라 완전히 다른 결과가 나옴.
면접은 지원자의 실력을 보는 자리이지, 주눅 들게 하는 자리가 아님.
```

### 4. "문화 적합성(Culture Fit)"의 함정

```
"문화 적합성"이라는 이름 아래 벌어지는 것들:

좋은 의미:
- 팀의 일하는 방식과 맞는가 (빠르게 vs 신중하게)
- 소통 스타일이 맞는가 (직설적 vs 완곡한)
- 가치관이 맞는가 (품질 중시 vs 속도 중시)

위험한 의미:
- "나와 비슷한 사람" = 다양성 부족
- "우리랑 잘 어울릴 것 같은" = 동질성 편향
- "뭔가 안 맞는 느낌" = 설명할 수 없는 편견

최근에는 "Culture Fit" 대신 "Culture Add"라는 개념을 씀.
"우리와 얼마나 비슷한가"가 아니라
"우리에게 무엇을 더해줄 수 있는가"를 봄.
```

## 더 나은 면접관이 되기 위한 가이드

### 면접 전

```
1. 이력서를 꼼꼼히 읽고 질문 준비
   - "이 프로젝트에서 구체적으로 뭘 했는지 물어봐야겠다"
   - "이 기술 전환 경험이 흥미로우니 깊게 물어보자"

2. 평가 기준을 미리 정하기
   - 이 포지션에서 가장 중요한 역량 3가지는?
   - 합격/불합격의 경계는 어디인가?

3. 시간 배분 계획
   - 인트로: 5분
   - 기술 질문: 30분
   - 행동 질문: 15분
   - 지원자 질문: 10분
```

### 면접 중

```
1. 분위기 세팅
   "오늘 면접은 대화처럼 진행하려고 합니다.
   모르는 건 모른다고 하셔도 괜찮고,
   질문이 있으면 언제든 물어보세요."

2. 구조화된 질문
   - 같은 포지션 지원자에게 같은 핵심 질문 사용
   - 깊이는 지원자 수준에 맞게 조절

3. 노트 필기
   - 핵심 답변, 인상적인 점, 우려 사항 실시간 기록
   - 기억에 의존하면 편향이 생김

4. 공정한 힌트
   - 막히면 동일한 수준의 힌트 제공
   - "이 자료구조를 생각해보시면 어떨까요?" (방향 제시)
   - 답을 직접 말해주지는 않기
```

### 면접 후

```
1. 즉시 평가 기록
   - 면접 끝나고 30분 이내에 피드백 작성
   - 시간이 지나면 기억이 왜곡됨

2. 구체적 근거
   ❌ "전체적으로 좋았음"
   ✅ "이진탐색 구현에서 경계 조건을 정확히 처리했고,
       시간복잡도 분석도 정확했음. 다만 코드 가독성에서
       변수명이 약간 모호한 부분이 있었음."

3. 디브리핑 참여
   - 다른 면접관 의견 듣기 전에 자기 평가 먼저 정리
   - 의견이 다를 때 건설적으로 토론
```

<Callout type="success" title="좋은 면접관의 체크리스트">
- 지원자가 최선의 모습을 보여줄 수 있는 환경을 만들었는가?
- 질문이 공정하고 관련성이 있었는가?
- 지원자의 답변을 끝까지 들었는가?
- 구체적이고 설명 가능한 평가를 작성했는가?
- 지원자에게 회사에 대한 좋은 인상을 남겼는가?
- 무의식적 편향은 없었는가?

**모든 항목에 "예"라고 답할 수 있으면 좋은 면접을 진행한 것.**
아니라면 다음 면접에서 개선할 부분을 기록해두자.
</Callout>

## 결론: 면접은 양방향이다

면접관을 해보면 알게 됨.

**면접은 지원자를 평가하는 자리가 아니라, 서로를 알아가는 자리임.**

면접관도 긴장하고, 면접관도 실수하고, 면접관도 배움.
완벽한 면접은 없고, 완벽한 면접관도 없음.

근데 한 가지 확실한 건:

**면접 경험이 좋았던 회사는, 들어가서도 좋은 경우가 많음.**
면접에서 지원자를 존중하는 회사는, 직원도 존중할 가능성이 높음.

반대로 면접에서 지원자를 무시하거나, 압박하거나, 불쾌하게 하는 회사는...
들어가도 비슷할 가능성이 높음.

면접관으로서의 경험은 개발자로서의 성장에도 도움이 됨.
다른 사람의 코드를 읽고, 사고 과정을 듣고, 다양한 접근 방식을 보면서
자신의 시야도 넓어지니까.

면접 보는 것보다 면접 보게 하는 게 더 힘든 건 맞음.
근데 더 많이 배우는 것도 맞음.

그리고 만약 면접관이 되었는데 교육을 안 받았다면...
이 글이 조금이라도 도움이 됐으면 좋겠음.
최소한 "다음 주부터 면접 들어가세요" 보다는 나을 거임 ㅋㅋ
