## 이게 뭔데

**우리 서비스가 들고 있는 자체 `Customer` 테이블을 버리고, 회사가 공식적으로 인정한 고객 원본(보통 CRM)을 바라보게 바꾸는 리팩토링**이다. 한 문장 정의는 책 그대로다. "현재 쓰는 소스 대신, 그 엔티티의 공식 데이터 소스를 쓴다."

비유를 하나 들어보자. 가족 단톡방에 엄마 전화번호가 떠돈다고 치자. 나는 3년 전 번호를 저장해뒀고, 동생은 작년 번호를, 막내는 최신 번호를 갖고 있다. 평소엔 아무 문제 없다. 각자 자기 폰에서 잘 걸린다. 근데 막상 엄마 생일에 케이크 배달을 시키는 순간, 누구 번호가 "진짜"냐가 문제가 된다. 셋 다 "내 폰에 저장된 게 맞는데?" 한다. 정답은 하나뿐이다. **엄마 본인이 등록한 번호.** 그게 official data source다.

회사 데이터도 똑같다. 고객의 공식 소스는 보통 CRM이다. 그런데 결제팀도 자기 `Customer` 사본을 들고 있고, 마케팅팀도 따로 긁어둔 고객 리스트가 있고, 우리 서비스도 가입할 때 만든 `Customer`가 있다. 다들 "내 테이블이 맞는데?" 한다. 이 리팩토링은 그 사본들을 거두고 **다들 같은 원본을 보게** 정렬하는 작업이다.

<Callout type="info" title="한 줄 요약">
같은 엔티티의 사본이 여러 군데 있으면, 언젠가 반드시 불일치가 난다. 공식 소스 하나를 정하고 거기를 보게 만드는 게 이 리팩토링이다. 단, 키와 의미가 다르면 갈아타기 전에 변환부터 해야 한다.
</Callout>

## 언제 쓰나

핵심 동기는 단순하다. **데이터의 올바른 버전을 쓰기 위해서다.** 같은 데이터가 여러 곳에 흩어져 있으면 두 가지가 따라온다. 불일치(inconsistency)와 가용성(availability) 문제다.

불일치는 이런 거다. 고객이 CRM에서 주소를 바꿨다. CRM은 갱신됐다. 그런데 우리 서비스의 자체 `Customer` 테이블은 그대로다. 청구서가 옛날 주소로 날아간다. 반대 방향도 있다. CRM에 막 등록된 신규 고객이 우리 테이블엔 아직 없어서, 그 고객이 우리 화면에선 "존재하지 않는 사람"이 된다.

가용성은 좀 더 미묘하다. 자체 사본을 들고 있으면 "전사 고객 데이터의 일부만" 본다. 다른 팀이 등록한 고객, 다른 채널로 들어온 고객은 내 테이블엔 없으니 보이질 않는다. 회사 전체로 보면 고객은 한 명인데, 시스템마다 다른 조각만 쥐고 있는 거다.

그래서 이 리팩토링의 트리거는 대개 다음 냄새들이다.

- **"우리 DB 고객 정보가 CRM이랑 자꾸 안 맞아요"** 라는 CS 티켓이 주기적으로 올라온다.
- 같은 고객이 시스템마다 다른 ID, 다른 이름, 다른 상태로 존재한다.
- 신규 고객이 어떤 시스템엔 보이고 어떤 시스템엔 안 보인다.
- 누군가 "그래서 이 고객의 *진짜* 정보가 뭔데요?"라고 물었을 때, 아무도 자신 있게 답을 못 한다.

이 냄새들의 공통 원인은 하나다. **공식 소스가 합의돼 있지 않거나, 합의는 됐는데 다들 사본을 보고 있다는 것.**

### 시나리오: 이런 적 있을 거임

은행 시스템을 떠올려보자. 전사 고객 마스터는 CRM에 있다. 그런데 보험 상품팀이 몇 년 전 자기네 서비스를 띄우면서, 빠르게 가려고 CRM에서 고객을 한 번 덤프 떠서 자체 `Customer` 테이블을 만들었다. 그 위에 `Account`, `Policy`, `Insurance`를 붙였다. 당시엔 합리적이었다. CRM 붙이는 게 느렸고, 일단 출시가 급했으니까.

3년이 지났다. 이제 이런 일이 일상이 됐다.

```text
[CRM]       김민수, 010-1234-5678, 서울 강남구 (2026-05-01 갱신)
[보험팀 DB] 김민수, 010-9999-0000, 서울 서초구 (2023-08-12 덤프 당시 값)
```

고객이 콜센터에 전화해서 "주소 바꿨는데 보험 안내문이 옛날 집으로 와요"라고 항의한다. 콜센터 직원은 CRM을 본다. CRM엔 새 주소가 멀쩡히 있다. "고객님 정보 정상입니다"라고 답한다. 그런데 보험 안내문을 찍는 배치는 **보험팀 자체 `Customer` 테이블**을 본다. 거기엔 3년 전 주소가 박제돼 있다. 양쪽 다 자기 데이터는 맞다. 안 맞는 건 둘이 같은 사람을 다르게 알고 있다는 사실이다.

여기에 더 무서운 게 겹친다. 새로 가입한 고객은 CRM엔 있는데 보험팀 테이블엔 없다. 그래서 그 고객이 보험 화면에 로그인하면 "등록되지 않은 고객"이라고 튕긴다. 신규 고객한테 첫인상이 "당신은 존재하지 않습니다"인 거다.

원인을 추적하면, 3년 전의 그 "일단 덤프 떠서 쓰자"는 결정으로 돌아온다. 사본은 만든 순간이 가장 정확하고, 그 뒤로는 매일 조금씩 거짓말이 된다.

## 주의할 점

좋아, 그럼 자체 테이블 버리고 CRM 바라보면 끝이네? 안타깝지만 거기에 함정이 몇 개 있다.

<Callout type="warning" title="갈아타기 전에 반드시 확인할 것">
- **참조 비용**: 로컬 `Customer`를 참조하던 모든 쿼리·조인·코드를 공식 소스로 바꿔야 한다. `Account JOIN Customer` 같은 조인이 수십 군데 흩어져 있으면 그 전부가 작업 대상이다. 작은 변경이 아니다.
- **키 전략이 다르면 먼저 통합**: 우리 `customer_id`(예: auto-increment 정수)와 CRM `customer_no`(예: `CUST-00012345` 같은 문자열)가 다른 키 체계면, 곧바로 갈아탈 수 없다. **Consolidate Key Strategy를 선행**해서 두 세계를 잇는 키 매핑부터 만들어야 한다.
- **의미(semantics)·최신성이 다르면 변환 선행**: 같은 "상태" 컬럼이라도 우리는 `'A'/'I'`, CRM은 `'ACTIVE'/'INACTIVE'/'DORMANT'`를 쓸 수 있다. 코드 체계·타입·포맷이 다르면 Apply Standard Codes, Apply Standard Type, Introduce Common Format으로 데이터를 먼저 변환해야 한다. 의미를 안 맞추고 키만 이으면, 불일치를 다른 모양으로 옮겨놓은 것뿐이다.
- **공식 소스가 통제 밖일 수 있다**: CRM이 다른 팀, 심하면 외부 벤더 소유면 우리가 마음대로 스키마를 못 건드린다. 직접 접근하면 그쪽 다운타임·지연·릴리스 일정에 우리가 묶인다.
</Callout>

특히 마지막이 현대 마이크로서비스 환경에서 제일 자주 발목을 잡는다. "공식 소스"라는 건 보통 **다른 서비스가 소유한 데이터**라는 뜻이고, 남의 DB를 직접 들여다보는 건 데이터 소유권 경계를 침범하는 일이다. 그래서 직접 접근이냐 복제냐의 선택이 단순 성능 문제가 아니라 **아키텍처 결합도 문제**가 된다.

## 이렇게 한다

큰 그림은 두 단계다. **(1) 공식 소스를 식별하고 합의한다. (2) 어떻게 접근할지 전략을 고른다.** 그다음 디테일(키·의미 변환, 안 쓰는 테이블 정리)을 붙인다.

<Steps>
<Step title="공식 소스를 식별하고 합의한다">
"고객의 공식 소스는 CRM이다"가 자명해 보여도, 반드시 **이해관계자와 소스 소유자가 합의**해야 한다. 후보가 여러 개면(CRM, 회원 시스템, 빌링 등) 어느 것이 진짜 마스터인지 정하거나, 마스터 데이터 관리(MDM) 차원에서 통합한다. 이건 기술 결정 이전에 조직 합의다. 합의 없이 갈아타면 6개월 뒤 다른 팀이 "그건 우리가 마스터인데요?" 하고 나타난다.
</Step>

<Step title="키 전략을 맞춘다 (다르면)">
로컬 `Customer`와 공식 소스의 키가 다른 체계면 매핑 테이블부터 만든다. 이게 Consolidate Key Strategy의 역할이고, 이후 모든 작업의 기반이 된다.

```sql
-- 로컬 customer_id ↔ CRM customer_no 매핑
CREATE TABLE customer_id_map (
  local_customer_id  BIGINT       NOT NULL,
  crm_customer_no    VARCHAR(20)  NOT NULL,
  matched_by         VARCHAR(20)  NOT NULL,   -- 'ssn' | 'phone+name' | 'manual'
  confidence         SMALLINT     NOT NULL,   -- 매칭 신뢰도
  PRIMARY KEY (local_customer_id),
  UNIQUE (crm_customer_no)
);
```

여기서 진짜 일은 매핑을 *채우는* 거다. 주민번호 같은 강한 식별자가 양쪽에 있으면 깔끔하지만, 없으면 이름+전화번호+생년월일로 퍼지 매칭을 돌려야 하고, 애매한 건 사람이 손으로 확인한다(`matched_by = 'manual'`). 이 단계에서 "어? 우리 테이블엔 동일 인물이 중복 가입돼 있었네" 같은 데이터 품질 문제가 쏟아져 나오는 게 정상이다.
</Step>

<Step title="의미·포맷을 맞춘다 (다르면)">
키를 이었으면 값의 의미를 맞춘다. 코드·타입·포맷이 다르면 변환 규칙을 명시적으로 정의한다.

```sql
-- 로컬 상태 코드 → CRM 표준 코드로 정규화
-- 로컬: 'A'(활성) / 'I'(비활성)
-- CRM : 'ACTIVE' / 'INACTIVE' / 'DORMANT'
SELECT
  m.crm_customer_no,
  CASE c.status
    WHEN 'A' THEN 'ACTIVE'
    WHEN 'I' THEN 'INACTIVE'   -- 'DORMANT'는 로컬에 개념이 없음 → CRM 쪽이 더 풍부
  END AS crm_status
FROM customer c
JOIN customer_id_map m ON m.local_customer_id = c.customer_id;
```

여기서 주의. CRM에 있는 `'DORMANT'`(휴면)처럼 **로컬엔 없던 상태**가 드러나면, 그건 "우리가 그동안 휴면 고객을 활성으로 잘못 취급해왔다"는 뜻이다. 단순 매핑이 아니라 비즈니스 규칙을 다시 봐야 하는 신호다. 의미 정합을 건너뛰면 나중에 "휴면 고객한테 마케팅 메일 보냈다"는 사고로 돌아온다.
</Step>

<Step title="접근 전략을 고른다 — 직접 접근 vs 복제">
이게 핵심 갈림길이다. 책은 두 가지를 제시하고, 둘은 트레이드오프가 정반대다. 아래 탭에서 비교한다.
</Step>

<Step title="안 쓰는 테이블을 정리한다">
공식 소스로 완전히 전환됐고, 더는 로컬 `Customer`를 아무도 안 읽는 게 확인되면 Drop Table로 제거한다. 단, **곧바로 드롭하지 말 것.** 한동안 살려두고(읽기는 끊되 테이블은 유지) 모니터링해서 "정말 아무도 안 보는지" 확인한 뒤 떨어뜨린다. expand-contract의 contract 단계다.
</Step>
</Steps>

<TabsContent value="direct">

공식 소스에 **별도 연결이나 API/웹서비스로 그때그때 물어보는** 방식이다. 사본을 두지 않으니 항상 최신이다. 오래된 데이터(stale data) 문제가 원천적으로 없다는 게 가장 큰 장점이다.

```typescript
// Before: 로컬 Customer 테이블 직접 조회
async function getCustomer(localId: number) {
  return db.query(
    'SELECT * FROM customer WHERE customer_id = $1',
    [localId],
  );
}

// After: 공식 소스(CRM 서비스)에 직접 물어본다
async function getCustomer(localId: number) {
  const { crmCustomerNo } = await idMap.resolve(localId); // 키 매핑
  return crmClient.getCustomer(crmCustomerNo);            // 항상 최신
}
```

대가도 명확하다.

- **가용성 결합**: CRM이 죽으면 우리도 같이 죽는다. 우리 가용성이 남의 시스템에 묶인다.
- **지연·부하**: 매 요청마다 네트워크 왕복. CRM에 트래픽을 그대로 떠넘긴다.
- **분산 트랜잭션**: 우리 `Account` 갱신과 CRM `Customer` 갱신을 한 트랜잭션으로 묶어야 하면 분산 트랜잭션(2PC) 지옥이 열린다. 책이 명시적으로 경고하는 지점이다.

이 결합을 누그러뜨리려고 현대 실무에선 직접 호출을 **동기 API + 짧은 캐시**로 감싸거나, 회로 차단기(circuit breaker)로 CRM 장애가 우리까지 전파되는 걸 막는다. 하지만 본질적으로 "남의 가용성에 내가 묶인다"는 성질은 남는다.

</TabsContent>

<TabsContent value="replication">

공식 소스를 **우리 쪽으로 복제해두고 그 복제본을 읽는** 방식이다. 읽기는 단일 DB 안에서 끝나니 확장성·지연·가용성이 유리하다. 책도 "복제는 단일 DB 결합이라 확장성 면에서 유리하다"고 짚는다.

대신 핵심 문제는 다시 **오래된 데이터(stale data)** 다. 사본인 이상 동기화 지연만큼 거짓말을 한다. 그래서 복제를 고르면 **동기화 전략이 필수**다. 갱신까지 필요하면 다중마스터로 가야 하고, 이건 충돌 해소까지 떠안는 일이라 가급적 피한다(보통 공식 소스만 쓰기 가능, 우리는 읽기 전용 복제본).

2006년 책은 주기적 갱신·DB 복제·트리거 동기화를 든다. 골격은 같되, 요즘은 보통 **CDC(Change Data Capture)** 로 한다. CRM DB의 트랜잭션 로그를 Debezium 같은 도구가 읽어서, `Customer`가 바뀔 때마다 이벤트로 흘려보낸다. 우리는 그 이벤트를 받아 읽기 전용 복제본을 갱신한다.

```text
[CRM DB]  --(트랜잭션 로그)--> [Debezium] --> [Kafka: customer.changed]
                                                     |
                                                     v
                                        [우리 서비스: read-only Customer 복제본 갱신]
```

```sql
-- 우리 쪽 읽기 전용 복제본. 쓰기는 CDC 컨슈머만, 앱은 SELECT만.
CREATE TABLE customer_replica (
  crm_customer_no  VARCHAR(20) PRIMARY KEY,
  name             VARCHAR(100) NOT NULL,
  status           VARCHAR(10)  NOT NULL,
  address          VARCHAR(200),
  synced_at        TIMESTAMPTZ  NOT NULL   -- 동기화 시각: stale 여부 판단용
);
```

`synced_at`을 꼭 남겨라. "이 데이터가 얼마나 최신인지"를 사용자·운영자가 판단할 수 있어야 한다. 그리고 복제본은 **읽기 전용임을 강제**해야 한다. 누가 실수로 복제본을 직접 UPDATE 치면, 그 순간 다시 두 개의 진실이 생기고 우리는 처음 문제로 돌아간다. 읽기 전용 뷰로 감싸거나(Encapsulate Table With View), 권한을 SELECT만 주거나, 쓰기 시 에러를 던지는 트리거를 건다.

<Callout type="note" title="실무 기본값">
가용성·지연이 중요하고 약간의 지연(eventual consistency)을 비즈니스가 받아들일 수 있으면 **CDC 기반 읽기 전용 복제**가 현실적 기본값이다. "1초도 어긋나면 안 되는" 잔액·결제 같은 데이터만 직접 접근(또는 그 도메인 서비스에 위임)으로 간다. 고객 이름·주소처럼 변동이 드문 마스터 데이터는 복제가 잘 맞는다.
</Callout>

</TabsContent>

### 전환은 한 방에 하지 말 것 — expand-contract

위 전략을 고르고 나서도, 스위치를 한 번에 내리면 안 된다. 로컬 `Customer`를 보던 코드와 공식 소스를 보는 코드가 **잠시 공존**하게 두고 점진적으로 옮긴다. 이게 expand-contract(parallel change)다.

<Steps>
<Step title="Expand — 두 소스를 병행">
공식 소스 접근 경로(직접 또는 복제본)를 새로 깔되, 로컬 `Customer`는 그대로 둔다. 새 코드 경로를 만들고 피처 플래그로 일부 트래픽만 공식 소스로 보낸다. 양쪽 결과를 비교 로깅해서 "공식 소스로 바꿨을 때 값이 달라지는 케이스"를 미리 잡는다(shadow read).
</Step>
<Step title="Migrate — 읽기를 옮긴다">
검증이 쌓이면 읽기 트래픽을 점진적으로 공식 소스로 넘긴다. 키·의미 변환이 제대로 됐는지 운영에서 확인하는 단계다. 문제가 나면 플래그를 되돌려 즉시 롤백.
</Step>
<Step title="Contract — 로컬 테이블 제거">
100% 공식 소스로 넘어가고, 로컬 `Customer`를 아무도 안 읽는 게 확인되면 Drop Table. 의미·스키마가 같았으면 데이터 마이그레이션이나 앱 변경 없이 끝나기도 한다. 달랐으면 변환 스크립트를 돌리거나, Encapsulate Table With View로 "옛 스키마 모양"의 호환 뷰를 공식 소스 위에 씌워서 옛 코드가 당분간 그대로 돌게 해준다.
</Step>
</Steps>

<Callout type="success" title="호환 뷰라는 탈출구">
옛 코드를 한 번에 다 못 고치겠으면, 공식 소스(또는 복제본) 위에 **옛 `Customer`와 똑같은 컬럼 이름·타입을 가진 뷰**를 만들어 원래 테이블 이름으로 노출한다. 옛 쿼리는 자기가 여전히 로컬 테이블을 보는 줄 알고 돌아가지만, 실제 데이터는 공식 소스에서 나온다. 키·의미 변환을 이 뷰 정의 안에 흡수시키면, 호출하는 코드는 변환의 존재조차 모른다.
</Callout>

## 현대적으로 보면: 마스터 데이터와 데이터 소유권

이 리팩토링은 2006년 카탈로그 항목이지만, 사실 오늘날 가장 뜨거운 주제 두 개를 정통으로 관통한다.

**마스터 데이터 관리(MDM)와 single source of truth.** "고객의 공식 소스는 어디인가"를 정하는 일이 바로 MDM이다. 사본이 여러 개라 불일치가 난다는 책의 진단은, 현대 데이터 거버넌스 언어로 "single source of truth가 없다"는 말과 정확히 같다. 이 리팩토링은 그 SSoT를 *기술적으로 강제하는* 한 걸음이다.

**마이크로서비스의 데이터 소유권.** 마이크로서비스 원칙 중 하나가 "각 데이터는 하나의 서비스가 소유하고, 남은 그 서비스를 통해서만 접근한다"이다. 그러면 "고객 데이터는 고객 서비스(=공식 소스)가 소유한다"가 된다. 다른 서비스가 자기 `Customer` 사본을 들고 있는 건, 곧 소유권 경계를 어긴 상태다. 이 리팩토링은 그 경계를 다시 세우는 작업이다.

그래서 전략 선택도 마이크로서비스 어휘로 번역된다.

- **직접 접근** = 런타임에 고객 서비스 API를 동기 호출. 강한 일관성, 강한 결합.
- **복제(CDC)** = 고객 서비스가 발행하는 이벤트를 구독해 자기 로컬 읽기 모델을 만든다. 느슨한 결합, 결과적 일관성. (CQRS의 read model, outbox 패턴이 여기 붙는다.)

<Callout type="warning" title="흔한 함정: 가짜 SSoT">
복제본을 만들어 놓고, 어느 순간 그 복제본을 *직접 수정*하기 시작하면 SSoT가 깨진다. "우리 쪽에서 급해서 복제본 주소만 살짝 고쳤어요" 한 번이면, 공식 소스와 복제본이 갈라지고 처음 그 단톡방 전화번호 문제로 되돌아간다. **복제본은 끝까지 읽기 전용.** 쓰기는 반드시 공식 소스를 통해서만. 이 규율이 무너지면 이 리팩토링은 의미를 잃는다.
</Callout>

## 정리

데이터의 사본은 만든 순간이 가장 정확하고, 그 뒤로는 매일 조금씩 거짓말이 된다. 같은 엔티티를 여러 군데서 들고 있으면 언젠가 반드시 불일치와 가용성 문제가 터진다.

> **진실은 하나여야 한다. 공식 소스를 정하고, 나머지는 거기를 보게 만든다.**

다만 갈아타는 길은 한 번에 점프하는 게 아니다. 공식 소스를 합의로 정하고, 키가 다르면 매핑부터, 의미가 다르면 변환부터 깔고, 직접 접근이냐 복제냐를 트레이드오프로 고른 다음, expand-contract로 천천히 읽기를 옮기고 마지막에 사본을 떨군다. 현대 도구로 보면 CDC로 읽기 복제본을 만들고, 마이크로서비스의 데이터 소유권으로 "누가 마스터인지"를 못 박는 일이다. 핵심 규율은 끝까지 단순하다. **사본은 읽기만, 쓰기는 공식 소스로만.**
