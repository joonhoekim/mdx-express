## 이게 뭔데

Move Data는 한 문장이면 끝난다. **테이블에 담긴 데이터(전체든 일부 컬럼이든)를 다른 기존 테이블로 옮기는 것**이다. 스키마를 바꾸는 게 아니라 값을 이사시키는 일이라, 데이터 품질 리팩토링 칸에 들어가 있다.

비유를 하나 들자. 이삿짐 옮기기다. 집(테이블)을 새로 짓는 게 아니다. 짐(데이터)만 옆집으로 나르는 거다. 짐이 박스 다섯 개면 그냥 들고 가면 된다. 근데 짐이 5톤 트럭 열 대 분량이면? 그때부터 "어떻게 나르느냐"가 전부가 된다. 한 번에 다 들어내려고 현관을 막아버리면, 그 집에 사는 사람은 그동안 화장실도 못 간다. Move Data가 어려워지는 지점이 정확히 여기다. **옮기는 행위 자체가 아니라, 옮기는 동안 누가 그 테이블을 쓰고 있느냐**가 문제다.

<Callout type="info" title="한 줄 요약">
Move Data는 "데이터를 INSERT ... SELECT로 옮기고 원본을 지운다"가 본질이다. 행이 적으면 한 줄 SQL이고, 행이 수백만이면 그 한 줄이 테이블을 잠가버린다. 이 편의 진짜 주제는 후자다.
</Callout>

## 언제 쓰나

Move Data는 혼자 등장하는 일이 드물다. 대부분 **다른 리팩토링의 하위 단계**로 끌려 들어온다. 책이 나열하는 동기를 풀어보면 대충 이런 결들이다.

- **컬럼 이름 변경(Rename Column)** — 신규 컬럼을 만들고, 구 컬럼의 값을 신규 컬럼으로 옮기고, 구 컬럼을 지운다. 가운데의 "옮긴다"가 Move Data다.
- **테이블 수직 분할(Split Table)** — `Customer` 하나를 `Customer` + `CustomerProfile`로 쪼갤 때, 프로필성 컬럼들의 값을 새 테이블로 옮긴다.
- **테이블 수평 분할** — 한 테이블이 너무 무거워서 일부 행만 같은 구조의 다른 테이블로 뺀다. 예를 들어 유럽 고객만 `EuropeanCustomer`로 분리한다.
- **컬럼 분할/병합(Split/Merge Column)** — 컬럼을 쪼개거나 합칠 때 값이 새 위치로 이동한다.
- **테이블·컬럼 제거 전 데이터 보존** — 지우기 전에 살려둘 값을 다른 데 옮겨 백업한다.

그런데 Move Data가 **단독으로 의미를 갖는 시나리오**도 있다. 구조 변경 없이 데이터만 재배치하는 경우다. 조직 개편으로 BC주·Alberta주 고객을 다른 테이블로 옮긴다든가 — 그리고 이 편에서 주인공으로 삼을, **상태(status)를 이력으로 떼어내는 경우**.

### 현실 시나리오: 상태값에 역사가 필요해졌다

`Customer` 테이블에 `Status` 컬럼이 하나 박혀 있다고 하자. `ACTIVE`, `DORMANT`, `CLOSED` 같은 값이 들어간다. 처음엔 이걸로 충분했다. "지금 이 고객 상태가 뭐냐"만 알면 됐으니까.

그러다 어느 날 기획에서 이런 요구가 온다. "이 고객 언제 휴면 됐어요? 그리고 언제 다시 활성화됐죠? 작년 한 해 동안 휴면→활성 전환된 고객이 몇 명인지 뽑아주세요."

`Customer.Status`는 **현재 값 하나만** 들고 있다. 어제 `ACTIVE`였다가 오늘 `DORMANT`가 되면, 어제의 `ACTIVE`는 그냥 덮어쓰여 사라진다. 역사가 없다. 이력을 남기려면 별도 테이블이 필요하다.

```sql
CREATE TABLE CustomerStatusHistory (
  CustomerStatusHistoryOID BIGINT      NOT NULL,
  CustomerOID              BIGINT      NOT NULL,
  Status                   VARCHAR(20) NOT NULL,
  EffectiveDate            TIMESTAMP   NOT NULL,
  CONSTRAINT pk_csh        PRIMARY KEY (CustomerStatusHistoryOID),
  CONSTRAINT fk_csh_cust   FOREIGN KEY (CustomerOID) REFERENCES Customer (CustomerOID)
);
```

이제 `Customer`에 있던 **현재 상태값들을 이 이력 테이블로 옮겨서**, 첫 이력 레코드로 심어줘야 한다. 앞으로 상태가 바뀔 때마다 여기에 한 줄씩 쌓이고, "현재 상태"는 `EffectiveDate`가 가장 최근인 행을 보면 된다. 이게 바로 단독 Move Data다. `Customer`의 구조는 안 건드리고(`Status` 컬럼은 나중에 정리하든 비워두든), **값만 새 테이블로 이주**시킨다.

```sql
-- Move Data의 본질: 옮기고(INSERT...SELECT), 원본은 비우거나 지운다
INSERT INTO CustomerStatusHistory (CustomerStatusHistoryOID, CustomerOID, Status, EffectiveDate)
SELECT nextval('csh_seq'), CustomerOID, Status, CURRENT_TIMESTAMP
FROM Customer
WHERE Status IS NOT NULL;

UPDATE Customer SET Status = NULL;
```

행이 만 건이면 이 두 줄로 끝이다. 점심 먹고 오면 돼 있다. 문제는 `Customer`가 600만 건일 때다.

## 주의할 점

<Callout type="warning" title="수백만 행을 한 트랜잭션으로 옮기면 테이블이 잠긴다">
`UPDATE Customer SET Status = NULL` 한 방은 600만 행 전체를 한 트랜잭션으로 묶는다. 그동안:

- 그 행들에 **쓰기 락**이 잡혀 다른 트랜잭션이 줄을 선다 (DB·격리수준에 따라 락 범위는 다르지만, 대량 UPDATE는 락·롤백 세그먼트·WAL을 폭발시킨다).
- 트랜잭션 하나가 거대해져 **롤백/언두 영역**이 부풀고, 중간에 죽으면 그 전부를 되감느라 또 한참을 먹는다.
- 복제 지연(replication lag)이 튄다. 거대 트랜잭션 하나가 리플리카로 통째로 흘러가며 따라잡기가 밀린다.

즉 데이터 이동 자체는 성공해도, **그 시간 동안 서비스의 가용성과 성능이 같이 무너진다.** 책의 표현 그대로 "이동하는 동안 접근 불가"가 된다. 작은 테이블이면 무시해도 되는 트레이드오프지만, 큰 테이블에선 이게 전부다.
</Callout>

추가로 옮기기 전에 정해야 할 게 있다. 책이 "의존성 식별"이라 부르는 것들이다.

- **원본을 어떻게 할 것인가** — 옮긴 행을 삭제? `NULL`이나 `0`으로 비움? 아니면 일단 그대로 둠? 위 시나리오는 `Status`를 `NULL`로 비웠다. 이력 테이블이 진실의 원천(source of truth)이 됐으니까.
- **대상 테이블의 FK를 누가 참조하나** — 데이터가 이사 가면, 그 데이터를 FK로 참조하던 다른 테이블들이 **새 위치를 가리키도록** 같이 손봐야 한다. 이걸 빼먹으면 참조가 허공을 가리킨다.
- **이동 중 변환할 것인가** — 옮기면서 형식을 바꾸거나 합칠 거면 그 변환 로직도 마이그레이션에 녹인다.

## 이렇게 한다

큰 그림은 변하지 않는다. **옮기고(INSERT...SELECT) → 원본 정리(UPDATE/DELETE) → 접근 코드 수정.** 달라지는 건 "한 방에 하느냐, 잘게 나눠서 하느냐"뿐이고, 큰 테이블에선 무조건 후자다.

### 1. 작은 테이블: 그냥 한 방에

행이 수만 건 이하고 잠깐 락 걸려도 아무도 안 우는 테이블이면, 정석 SQL이 제일 깔끔하다.

```sql
BEGIN;

INSERT INTO CustomerStatusHistory (CustomerStatusHistoryOID, CustomerOID, Status, EffectiveDate)
SELECT nextval('csh_seq'), CustomerOID, Status, CURRENT_TIMESTAMP
FROM Customer
WHERE Status IS NOT NULL;

UPDATE Customer SET Status = NULL WHERE Status IS NOT NULL;

COMMIT;
```

대량인데 일회성 벌크라면, 책이 권하는 대로 export/import나 벌크 로더(Oracle SQL\*Loader, PostgreSQL `COPY`, `pg_bulkload` 등)로 내려서 통째로 밀어 넣는 것도 방법이다. 단 이건 보통 **다운타임 윈도우**를 전제로 한다. 새벽 점검 시간에 문 닫고 옮기는 그림이다.

### 2. 큰 테이블: 청크 단위 배치 백필

무중단으로 600만 행을 옮겨야 한다면, 거대 트랜잭션을 **수천 개의 작은 트랜잭션으로 쪼갠다.** PK 범위로 잘라서, 한 번에 몇천~몇만 행씩만 옮기고 짧게 커밋하고 다음 묶음으로 넘어간다. 각 묶음 사이엔 락이 풀려 있으니 운영 트래픽이 그 틈으로 흘러간다.

<Steps>
<Step title="배치 경계를 잡는다">
PK(`CustomerOID`)나 `created_at` 같은 인덱스 있는 단조 증가 컬럼을 커서로 삼는다. 인덱스 없는 컬럼으로 `WHERE`를 걸면 매 묶음이 풀스캔이라 의미가 없다.
</Step>
<Step title="한 묶음씩 옮기고 즉시 커밋한다">
한 트랜잭션 = 한 묶음. 묶음이 끝나면 바로 커밋해 락을 놓는다. 트랜잭션을 작게 유지하는 게 핵심이다.
</Step>
<Step title="묶음 사이에 숨을 쉰다">
각 묶음 뒤에 짧은 sleep을 둬서 복제 지연과 DB 부하를 식힌다. 리플리카 lag을 모니터링하며 보폭을 조절하면 더 좋다.
</Step>
<Step title="멱등하게 짠다">
중간에 죽어도 다시 돌리면 되도록, 이미 옮긴 행은 건너뛰게 한다. 마지막으로 처리한 PK를 기록하거나, 대상에 이미 있는지 확인하는 식으로.
</Step>
</Steps>

루프로 표현하면 이런 모양이다.

```sql
-- 한 묶음(5만 행)만 옮기고 커밋. 이 블록을 더 옮길 게 없을 때까지 반복.
WITH batch AS (
    SELECT CustomerOID, Status
    FROM Customer
    WHERE Status IS NOT NULL
      AND CustomerOID > :last_processed_oid   -- 커서
    ORDER BY CustomerOID
    LIMIT 50000
), moved AS (
    INSERT INTO CustomerStatusHistory
        (CustomerStatusHistoryOID, CustomerOID, Status, EffectiveDate)
    SELECT nextval('csh_seq'), CustomerOID, Status, CURRENT_TIMESTAMP
    FROM batch
    RETURNING CustomerOID
)
UPDATE Customer
SET Status = NULL
WHERE CustomerOID IN (SELECT CustomerOID FROM moved);
-- :last_processed_oid 를 이번 배치의 max(CustomerOID)로 갱신하고 다음 루프
```

이걸 잡(job) 코드로 감싸면 흐름이 더 명확해진다.

```typescript
const BATCH_SIZE = 50_000;
let lastOid = 0;

while (true) {
  const movedMaxOid = await db.tx(async (t) => {
    // 위 CTE를 그대로 실행. 옮긴 행이 없으면 null 반환
    return await moveOneBatch(t, lastOid, BATCH_SIZE);
  });

  if (movedMaxOid === null) break; // 더 옮길 게 없음 → 종료

  lastOid = movedMaxOid;
  await sleep(200);                // 복제 지연·부하 식히기
  await waitUntilReplicaLagOk();   // lag이 임계치 넘으면 더 쉰다
}
```

손으로 짜는 게 부담이면, 마이그레이션/배치 도구가 이 패턴을 거의 그대로 제공한다.

</TabsContent>

<TabsContent value="flyway">
Flyway/Liquibase에 데이터 이동을 SQL 마이그레이션으로 넣을 수 있지만, **대량 백필은 마이그레이션 한 단계에 묶지 않는 게** 정석이다. 마이그레이션이 길어지면 배포 파이프라인이 그만큼 잠긴다. 패턴: 마이그레이션은 대상 테이블 생성 같은 가벼운 DDL만 하고(빠르게 끝남), 데이터 이동은 멱등 백필 잡으로 분리해 배포와 비동기로 굴린다.
</TabsContent>

<TabsContent value="online">
"옮기는 김에 컬럼 타입까지 바꾼다"처럼 테이블 재구축을 동반하면, `gh-ost`나 `pt-online-schema-change`(MySQL)가 새 섀도 테이블에 데이터를 청크로 복사하고 트리거로 변경분을 따라잡은 뒤 원자적으로 스왑한다. PostgreSQL이라면 인덱스는 `CREATE INDEX CONCURRENTLY`, 제약은 `ADD CONSTRAINT ... NOT VALID` 후 `VALIDATE CONSTRAINT`로 풀 테이블 락을 피한다. Move Data의 락 문제와 정확히 같은 고민을 도구화한 것들이다.
</TabsContent>

### 3. 접근 프로그램 수정

데이터가 이사 갔으니 그걸 읽던 코드도 새 주소를 봐야 한다. 책 말대로, Move Data가 **다른 리팩토링의 하위 단계로 끌려온 거면 보통 그 상위 작업이 코드를 이미 고쳤을 것**이다. 하지만 우리 시나리오처럼 **단독 재구성**이면, "현재 상태"를 읽던 코드를 이력 테이블에서 최신 행을 뽑도록 바꿔야 한다.

```sql
-- Before: 현재 상태는 컬럼 하나에서
SELECT Status FROM Customer WHERE CustomerOID = :id;

-- After: 현재 상태는 이력의 최신 행에서
SELECT Status
FROM CustomerStatusHistory
WHERE CustomerOID = :id
ORDER BY EffectiveDate DESC
LIMIT 1;
```

```typescript
// Before: 컬럼 직독
async function getCustomerStatus(id: number): Promise<string | null> {
  const row = await db.one('SELECT Status FROM Customer WHERE CustomerOID = $1', [id]);
  return row.status;
}

// After: 이력 테이블의 최신 상태
async function getCustomerStatus(id: number): Promise<string | null> {
  const row = await db.oneOrNone(
    `SELECT Status FROM CustomerStatusHistory
     WHERE CustomerOID = $1
     ORDER BY EffectiveDate DESC
     LIMIT 1`,
    [id],
  );
  return row?.status ?? null;
}
```

<Callout type="note" title="이행 기간을 안전하게: expand-contract">
무중단이면 한 배포에 "스키마 + 데이터 + 코드"를 다 욱여넣지 말고 단계를 나누는 게 안전하다. (1) **Expand** — 이력 테이블을 만들고, 새로 쓰는 코드는 양쪽에 다 기록한다(`Customer.Status`도 갱신하고 이력에도 append). (2) **Backfill** — 위의 청크 잡으로 과거 데이터를 천천히 옮긴다. (3) **Contract** — 읽기를 이력 테이블로 전환하고, 다 안정되면 구 `Customer.Status` 쓰기·컬럼을 정리한다. 각 단계가 독립 배포라 언제든 롤백 가능하다. 읽기 부하가 크면 "최신 상태"를 빠르게 뽑도록 `(CustomerOID, EffectiveDate DESC)` 인덱스나 머티리얼라이즈드 뷰로 받쳐주면 된다.
</Callout>

<Callout type="success" title="현재값과 이력을 동시에 챙기고 싶다면">
이력화하면 "현재 상태" 조회가 매번 정렬+LIMIT이라 살짝 비싸진다. 타협안: `Customer`에 현재값 컬럼은 그대로 두되(빠른 읽기용), 변경 때마다 이력에 한 줄 append(감사·분석용)하는 이중 구조도 흔하다. 한 발 더 나가면 outbox/CDC(Debezium)로 상태 변경 이벤트를 잡아 이력·분석 저장소로 흘려보내, 운영 테이블의 읽기 경로는 가볍게 유지할 수도 있다. 정답은 "이력을 얼마나 자주, 어떤 모양으로 읽느냐"에 달렸다.
</Callout>

## 정리

Move Data는 정의만 보면 시시하다. 데이터를 옆 테이블로 옮기고 원본을 정리한다. 끝. `INSERT ... SELECT` 한 줄이면 되는 날도 많다.

> **어려운 건 옮기는 게 아니라, 옮기는 동안 서비스를 살려두는 거다.**

수백만 행을 한 트랜잭션으로 들어내는 순간 테이블이 잠기고 가용성이 무너진다. 그래서 큰 테이블의 Move Data는 사실상 **"거대 트랜잭션을 어떻게 잘게 썰 것인가"** 문제로 바뀐다. PK 커서로 청크를 잘라 짧게 커밋하고, 멱등하게 짜서 죽어도 이어 돌리고, 복제 지연을 보며 보폭을 조절한다. 2006년의 벌크 로더가 했던 일을, 지금은 청크 배치 잡과 gh-ost/pt-osc, expand-contract가 무중단으로 해낸다. 도구는 바뀌었지만 고민은 같다. **데이터는 가만히 있는 게 아니라, 사람이 쓰는 동안 옮겨야 한다는 것.**
