# 리버스 프록시, 뭘 골라야 할까 — NPM·Traefik·Caddy 실전 선택 가이드

## 들어가며 — 나도 삽질했다

홈랩을 운영하다 보면 한 번쯤 리버스 프록시 앞에서 멈추게 된다. 도메인 하나로 여러 서비스를 묶어야 할 때, SSL 인증서를 자동으로 발급받고 싶을 때, 혹은 그냥 "다들 뭐 쓰나"가 궁금할 때.

나의 여정은 이랬다.

**NPM(Nginx Proxy Manager)** 로 시작했다. GUI가 있고, 클릭 몇 번이면 프록시 규칙이 생겼다. 편했다. 그런데 어느 순간 "좀 더 코드답게 관리하고 싶다"는 생각이 들었다.

그래서 **Traefik** 을 설치했다. 서비스 디스커버리, 자동 설정, 대시보드까지. 기능 목록을 보면 완벽해 보였다. 세팅까지 다 마쳤다. 그런데 뭔가 이상했다. 내가 기대했던 것과 실제 동작이 달랐다.

결국 **Caddy** 로 갈아탔다. 그리고 지금은 만족하고 있다.

이 글은 그 여정에서 배운 것들을 정리한 선택 가이드다.

## 후보 솔루션 포지셔닝

본격적인 비교 전에, 각 솔루션이 어디에 서 있는지부터 짚고 넘어가자.

**Nginx Proxy Manager (NPM)**
Nginx를 기반으로 GUI를 입힌 솔루션이다. 비개발자도 클릭 몇 번으로 프록시 규칙과 SSL 인증서를 설정할 수 있다. 진입 장벽이 낮고, 팀원 중 서버에 익숙하지 않은 사람이 있을 때 특히 유용하다. 단, 설정을 코드로 관리하기 어렵고 복잡한 라우팅 규칙에는 한계가 있다.

**Traefik**
Docker/k8s 환경에 최적화된 동적 리버스 프록시다. 컨테이너 라벨만 붙이면 자동으로 라우팅 규칙이 생성된다. 대시보드도 있고, 미들웨어 체인 구성도 강력하다. 단, 이 자동화는 같은 Docker 데몬 안에서만 작동한다 — 이 부분이 핵심이다.

**Caddy**
자동 HTTPS가 기본이고, Caddyfile 문법이 직관적이다. 별도 설정 없이 Let's Encrypt 인증서를 발급·갱신한다. GUI는 없지만 설정 파일이 워낙 읽기 쉬워서 금방 익숙해진다.

**Nginx (직접)**
가장 오래되고 검증된 선택지다. 설정 파일 문법이 다소 복잡하지만, 세밀한 퍼포먼스 튜닝과 대규모 트래픽 처리에서 강점을 보인다. 프로덕션 대규모 서비스라면 여전히 강력한 후보다.

## 핵심 분기: 내 환경이 어디야?

리버스 프록시 선택에서 가장 먼저 따져야 할 건 기능이 아니라 **환경**이다.

### Proxmox / 온프레미스

Proxmox는 LXC 컨테이너와 VM을 함께 운영하는 구조다. 서비스들이 각자 독립된 네트워크 인터페이스와 IP를 가진다. 이 환경에서는 **서비스 디스커버리가 작동하지 않는다** — 적어도 Traefik이 기대하는 방식으로는.

Traefik의 서비스 디스커버리는 Docker 소켓을 마운트해서 컨테이너 라벨을 읽는 방식이다. LXC나 VM은 그 범위 밖이다. 결국 Proxmox 환경에서 Traefik을 쓰면, 자동화 기능을 전혀 못 쓰고 수동으로 설정을 작성해야 한다. 그 순간 Caddy나 Nginx 대비 장점이 사라진다.

**Proxmox 환경에서의 선택:**
- 전체 리버스 프록시 → **Caddy** (LXC/VM 불문하고 명시적으로 등록, 단순함)
- GUI가 필요하거나 비개발자가 관리한다면 → **NPM**
- 홈랩 안에 k8s 클러스터를 별도로 운영한다면 → 그 클러스터 내부에서만 **Traefik**

### 클라우드 (EC2 등)

클라우드 환경은 인프라 구성에 따라 선택지가 달라진다.

단일 서버에서 Docker Compose로 여러 서비스를 띄우는 구성이라면 Caddy가 가장 간단하다. 여러 서버에 걸쳐 서비스가 분산된다면 로드밸런서(ALB 등)와 조합하는 게 일반적이고, 리버스 프록시 레이어는 단순하게 유지하는 편이 낫다.

k8s를 쓴다면 ingress controller로 Traefik이나 ingress-nginx가 자연스러운 선택이다.

## 서비스 성격에 따른 선택

환경을 정했다면, 다음은 배포할 서비스의 성격이다.

| 상황 | 추천 | 이유 |
|---|---|---|
| 홈랩, LXC/VM 혼합 | **Caddy** | 명시적 설정, 단순함 |
| Docker Compose 단일 서버 | **Caddy** or NPM | 규모에 따라 |
| GUI 필요, 비개발자 포함 | **NPM** | 클릭 기반 관리 |
| k8s / Docker Swarm | **Traefik** | 서비스 디스커버리 진가 발휘 |
| 클라우드 + 간단한 서비스 | **Caddy** | 자동 HTTPS, 설정 단순 |
| 대규모 트래픽, 세밀한 튜닝 | **Nginx** | 성능과 유연성 |

## Traefik 오해 정정 — 서비스 디스커버리의 실제 범위

이게 내가 삽질한 포인트다.

Traefik의 "서비스 디스커버리"라는 말을 처음 들으면, 같은 내부 네트워크에서 동작하는 서비스들을 자동으로 찾아준다고 오해하기 쉽다. 실제로 나도 그렇게 이해했다. Proxmox 위에서 각 LXC에 서비스를 올려두면, Traefik이 알아서 감지해줄 것이라 기대했다.

현실은 달랐다.

<Callout type="warning" title="Traefik 서비스 디스커버리의 실제 범위">
Traefik의 서비스 디스커버리는 **같은 Docker 데몬 위에서 돌아가는 컨테이너**를 대상으로 한다. Docker 소켓(`/var/run/docker.sock`)을 마운트해서 컨테이너에 붙은 라벨을 읽는 구조다. **LXC나 VM은 이 범위 밖이다.**
</Callout>

Proxmox 환경에서 Traefik을 쓰려면 결국 `File Provider`를 통해 수동으로 서비스를 등록해야 한다. 이렇게 되면 Caddy의 Caddyfile과 비교해서 설정이 더 복잡하고 장점도 없다.

Traefik이 빛나는 환경은 **Docker Compose나 k8s처럼 오케스트레이터가 있는 환경**이다. 거기서는 라벨 하나로 라우팅 규칙, SSL, 미들웨어까지 한 번에 정의할 수 있어서 진짜 강력하다.

## CLI LLM과 함께 쓰기 좋은 Caddy

이건 Caddy를 선택한 이유 중에 요즘 들어 점점 비중이 커지는 부분이다.

Claude Code나 Codex 같은 CLI LLM 도구와 함께 작업할 때, Caddyfile은 유독 잘 맞는다.

이유는 단순하다. **사람이 읽기 쉬운 포맷은 LLM도 읽기 쉽다.**

Caddyfile은 선언적이고 문법이 간결하다. 현재 설정 파일을 통째로 붙여넣고 "여기에 `app.example.com` 서비스 추가해줘, 백엔드는 `localhost:3000`이야"라고 하면 바로 수정된 Caddyfile을 받을 수 있다.

```caddyfile
# 이 정도 맥락이면 LLM이 바로 파악한다
example.com {
    reverse_proxy localhost:3000
}

api.example.com {
    reverse_proxy localhost:4000
}
```

반면 Nginx 설정 파일은 문법이 장황하고, Traefik의 YAML 설정은 중첩 구조가 깊다. LLM이 못 읽는 건 아니지만, 컨텍스트 효율이 다르다.

AI 어시스턴트와 협업하는 개발 워크플로우가 일상이 된 지금, **"LLM이 읽고 수정하기 쉬운 설정 포맷"** 도 도구 선택 기준이 될 수 있다. 앞으로 이 관점은 더 중요해질 것 같다.

## 결론 — 레이어별로 생각해라

리버스 프록시를 고를 때 자주 하는 실수는 하나의 도구로 모든 걸 해결하려는 것이다.

실제로는 레이어가 다르다.

- **전체 인프라 진입점 (Proxmox, 클라우드 VM)** → Caddy, NPM, Nginx
- **클러스터 내부 ingress (k8s, Docker Swarm)** → Traefik, ingress-nginx

Proxmox 위에 Caddy를 전체 리버스 프록시로 쓰고, 만약 그 안에 k8s 클러스터를 올린다면 클러스터 내부에서만 Traefik을 쓰는 구성이 자연스럽다. 이 두 레이어를 하나의 도구로 통일하려다가 오히려 복잡해진다.

<Callout type="success" title="오버엔지니어링을 피하는 기준">
지금 당장 k8s가 필요 없다면, k8s에서 잘 작동하는 도구를 억지로 홈랩에 들여올 이유가 없다. 오버엔지니어링의 가장 흔한 패턴은 **"나중에 필요할 것 같아서"** 로 시작한다. 현재 환경에 맞는 가장 단순한 도구를 선택하고, 복잡도는 실제로 필요해졌을 때 올리면 된다.
</Callout>
