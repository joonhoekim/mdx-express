## 이게 뭔데

동적 피벗. 말은 어려운데 모양은 다들 안다. **행으로 쌓여 있는 데이터를 가로로 눕혀서, 값마다 열을 하나씩 만드는 것**이다. 흔히 크로스탭(crosstab)이라고도 부른다. "카테고리별, 월별 매출"을 떠올려 보자. DB에 들어 있는 원본은 보통 이렇게 세로로 길다.

```text
category | month   | amount
---------+---------+-------
가전     | 2024-01 |  120
가전     | 2024-02 |  150
의류     | 2024-01 |   80
의류     | 2024-03 |   95
...
```

그런데 기획이 원하는 표는 이렇게 가로로 펼쳐진 모양이다.

```text
category | 1월 | 2월 | 3월 | ... | 12월
---------+-----+-----+-----+-----+-----
가전     | 120 | 150 |  -  | ... |  ...
의류     |  80 |  -  |  95 | ... |  ...
```

세로로 쌓인 `month` 값(2024-01, 2024-02, ...)을 **열 이름으로 돌려세운** 거다. 행이 열이 됐다. 이게 피벗이다.

여기서 SQL의 본질적인 제약 하나가 발목을 잡는다. **SQL은 쿼리를 파싱하는 시점에 결과 컬럼이 확정돼야 한다.** `SELECT a, b, c`라고 쓰는 순간, 이 쿼리가 돌려줄 열은 a, b, c 세 개로 못 박힌다. 데이터를 한 줄도 읽기 전에, 문법만 보고 "이 쿼리의 출력 스키마는 이거다"가 정해진다.

근데 피벗에서 열의 개수와 이름은 **데이터에 따라 변한다.** 1월부터 12월까지 있으면 열 12개, 데이터가 3월까지만 있으면 열 3개. 2025년이 시작되면 열이 또 늘어난다. "출력 컬럼이 데이터에 의존한다" — 이건 SQL이 구조적으로 손쉽게 못 하는 일이다. 한 방에 멋있게 끝내려고 들이대는 순간, 이 벽에 머리를 박는다.

<Callout type="warning" title="한 줄 요약">
SQL은 파싱 시점에 컬럼이 고정되는 언어다. 그런데 피벗의 열은 데이터에 따라 변한다. 이 둘은 본질적으로 안 맞는다. 억지로 한 쿼리에 욱여넣으면 컬럼을 손으로 박게 되고, 그 순간부터 데이터가 바뀔 때마다 쿼리를 고쳐야 한다.
</Callout>

## 시나리오: 이런 적 있을 거임

기획에서 엑셀 캡처를 들고 온다. "이렇게 월이 가로로 쭉 펼쳐지고, 카테고리가 세로로 오는 표 만들어 주세요." 캡처 속 표는 이미 완성돼 있고, 그냥 저거랑 똑같이만 나오면 된다. 자연스럽게 생각한다. "SQL에 PIVOT 같은 거 있겠지, 한 방에 멋있게 뽑아주면 되겠네."

그래서 DB 문서를 뒤진다. 그리고 여기서부터 일이 꼬이기 시작한다.

- **표준 SQL엔 `PIVOT`이 없다.** ANSI SQL 표준에 피벗 연산자 같은 건 없다.
- SQL Server엔 `PIVOT` 키워드가 있긴 하다. 근데 이것도 **펼칠 컬럼 값을 `IN (...)` 안에 손으로 다 적어야 한다.** 동적이 아니다.
- Postgres는 `crosstab`이라는 게 있는데, 코어가 아니라 `tablefunc` **확장**을 깔아야 하고, 결과 컬럼 타입을 `AS (category text, jan int, feb int, ...)`로 또 손으로 선언해야 한다.
- MySQL/MariaDB는 아예 그런 거 없다. 직접 `CASE WHEN`으로 짜야 한다.

DB마다 문법이 다 다르고, 있다는 것들조차 결국 "펼칠 값을 미리 알아야" 쓸 수 있다. 한참 헤매다 결국 가장 원시적이고 확실한 방법으로 돌아온다. `CASE WHEN` 노가다다.

```sql
SELECT
  category,
  SUM(CASE WHEN month = '2024-01' THEN amount ELSE 0 END) AS jan,
  SUM(CASE WHEN month = '2024-02' THEN amount ELSE 0 END) AS feb,
  SUM(CASE WHEN month = '2024-03' THEN amount ELSE 0 END) AS mar,
  -- ... 4월, 5월, 6월, 7월, 8월, 9월, 10월, 11월 ...
  SUM(CASE WHEN month = '2024-12' THEN amount ELSE 0 END) AS dec
FROM sales
GROUP BY category;
```

열두 달을 손으로 하드코딩한다. 길긴 한데, 어쨌든 **동작은 한다.** 화면에 표가 뜨고, 캡처랑 똑같다. 기획도 만족, PR도 머지. 여기까지는 평화롭다.

그리고 해가 바뀐다. 2025년 1월 데이터가 들어온다. 그런데 쿼리의 컬럼은 2024년 열두 개로 박혀 있다. 2025-01 데이터는 어느 `CASE WHEN`에도 안 걸리니까 그냥 합계에서 빠진다. 표에 안 보인다. 누군가 "1월 매출 왜 0이에요?"라고 묻고, 그제야 쿼리를 또 연다. 컬럼을 더 추가한다. **"월"이 하나 늘어날 때마다 SQL을 손으로 고쳐야 하는 구조**가 돼버린 거다.

표를 망친 건 기획의 요구가 아니다. 엑셀 표를 가로로 펼쳐달라는 건 지극히 정상적인 요구다. 망친 건 그걸 받아낸 **개발자의 습관**이다.

- "SQL로 한 방에 끝내면 멋있지" — **한 방 집착**. 화면 레이아웃(가로로 펼친 모양)을 굳이 DB 쿼리로 떠넘기려 함.
- "컬럼 좀 하드코딩하면 어때, 어차피 12개잖아" — **정적으로 박는 습관**. 동적이어야 할 걸 상수로 못 박아 미래의 나에게 폭탄을 넘김.
- "그럼 동적으로... 문자열로 SQL을 조립하면 되나?" — 막히면 **동적 SQL 생성으로 도피**. 여기로 빠지는 순간 SQL 인젝션과 디버깅 지옥의 문이 같이 열린다.

이 셋 중 하나라도 마음에 있었다면, 당신도 한 번쯤 `CASE WHEN` 열두 줄을 박아본 적 있을 거다. 그리고 대개는 "급한데 일단 되니까" 그렇게 넘어간다.

## 무슨 일이 벌어지나 (비용)

`CASE WHEN` 하드코딩이 당장은 동작하니까 비용이 안 보인다. 근데 시간이 지나면 셋으로 청구된다.

**1. 컬럼이 바뀔 때마다 쿼리를 고쳐야 한다.** 피벗의 열은 본질적으로 데이터를 따라 움직인다. 월이 늘어나면, 카테고리가 추가되면, 새 지점이 생기면 — 그때마다 사람이 쿼리를 열어 컬럼을 추가하거나 빼야 한다. 즉 **유지보수 주기가 데이터의 변화 주기에 묶인다.** 데이터는 가만히 있어도 알아서 늘어나는데, 쿼리는 사람이 손대야만 따라간다. 이 간극에서 "1월 매출이 0으로 나오는" 사고가 조용히 발생한다. 게다가 빠뜨려도 에러가 안 난다. 그냥 값이 누락될 뿐이라 한참 뒤에야 발견된다.

**2. 동적 SQL로 도망가면 인젝션과 디버깅이 같이 따라온다.** "하드코딩이 싫으니 컬럼 목록을 데이터에서 뽑아 SQL 문자열을 동적으로 만들자"는 다음 단계로 넘어가는 사람이 많다. 그럴듯하지만, 여기엔 함정이 두 개 있다. 하나는 **SQL 인젝션**이다. 펼칠 컬럼 값(예: 카테고리 이름)을 문자열로 SQL에 끼워 넣는데, 그 값이 사용자 입력에서 왔다면 인젝션 표면이 된다. 컬럼 식별자는 파라미터 바인딩(`?`, `$1`)으로 보호할 수도 없어서 직접 화이트리스트 검증/이스케이프를 해야 한다. 다른 하나는 **디버깅 난이도**다. 코드가 만들어낸 SQL 문자열은 매번 모양이 달라서, 에러가 나도 "지금 실제로 실행된 쿼리"를 재구성해 봐야 원인이 보인다.

**3. DB에 종속돼 이식성을 잃는다.** SQL Server `PIVOT`으로 짰다면 Postgres로 못 옮긴다. Postgres `crosstab`으로 짰다면 그건 `tablefunc` 확장에 묶이고, 결과 컬럼 타입까지 손으로 선언해야 해서 호출부가 지저분해진다. "DB가 피벗을 해주니 편하다"고 느낀 그 편함의 대가로, 쿼리가 특정 벤더 문법에 박혀버린다. DB 마이그레이션이나 멀티 DB 지원이 필요해지는 날, 이 쿼리들이 발목을 잡는다.

<Callout type="error" title="뭐가 문제냐면">
- **에러 없이 조용히 틀린다**: 컬럼을 빠뜨려도 쿼리는 정상 실행된다. 값만 누락될 뿐이라 한참 뒤에야 들킨다.
- **유지보수가 데이터에 종속된다**: 데이터는 알아서 늘어나는데, 쿼리는 사람이 매번 손으로 따라가야 한다.
- **동적 SQL로 도망가면 더 깊은 늪이다**: 인젝션 표면이 생기고, 매번 모양이 달라지는 쿼리는 디버깅이 어렵다.
- **벤더에 묶인다**: PIVOT/crosstab은 DB마다 문법이 달라 이식성을 잃는다.
</Callout>

## 정석: 집계는 SQL, 배치는 애플리케이션

해법의 핵심은 일을 **두 단계로 쪼개는 것**이다. 그리고 둘 사이에 선을 긋는 기준이 명확하다.

**1단계 — 집계는 SQL이 한다.** SQL이 정말 잘하는 건 "묶어서 더하는 것"이다. `GROUP BY`로 카테고리·월 조합마다 합계를 낸다. 단, 결과는 **가로로 펼치지 않고 세로로 길게(long format)** 둔다.

```sql
SELECT category, month, SUM(amount) AS total
FROM sales
GROUP BY category, month
ORDER BY category, month;
```

이 쿼리의 출력 컬럼은 `category, month, total` **세 개로 고정**이다. 데이터가 몇 월까지 있든, 카테고리가 몇 개든, 컬럼 개수는 안 변한다. 2025년이 와서 행이 늘어나도 쿼리는 그대로다. 파싱 시점에 컬럼이 확정돼야 한다는 SQL의 제약과 정면으로 싸우지 않고, 그 제약 안에서 노는 거다.

**2단계 — 행→열 배치는 애플리케이션 코드가 한다.** "세로로 긴 결과를 가로 표로 돌려세우는" 일은 표현 계층의 몫이다. 코드는 결과를 한 번 순회하면서 `month` 값을 키로 묶어주면 끝이다. **월이 늘어나도, 카테고리가 추가돼도 이 코드는 한 줄도 안 바뀐다.** 동적인 부분(컬럼 목록)을 데이터로 다루지, 코드 구조로 박지 않기 때문이다.

이렇게 나누면 아까의 비용이 통째로 사라진다. 컬럼이 바뀌어도 쿼리를 안 고치고(1), 동적 SQL 문자열을 조립할 일이 없으니 인젝션·디버깅 문제도 없고(2), `GROUP BY`는 모든 DB에서 똑같이 도니 벤더 종속도 풀린다(3).

<Callout type="info" title="그럼 DB 피벗(PIVOT/crosstab)은 언제 쓰나?">
드물게 합당한 경우가 있긴 하다. (1) 펼칠 컬럼이 진짜로 고정이고 앞으로도 안 변할 때(예: "참/거짓" 두 값, "분기 4개"처럼 도메인상 절대 안 늘어나는 축), (2) 피벗된 결과를 DB 안에서 다른 쿼리/뷰에 바로 물려 써야 할 때. 이런 좁은 경우엔 `CASE WHEN`이나 벤더 PIVOT이 깔끔할 수 있다. 핵심 판단 기준은 하나다 — **펼칠 컬럼이 데이터에 따라 변하는가?** 변하면 코드로, 절대 안 변하면 그때만 SQL 안에서.
</Callout>

## 이렇게 하면 됨

Before와 After를 나란히 보자.

**Before — `CASE WHEN` 12개 하드코딩.** 동작은 하지만, 컬럼이 쿼리에 박혀 있어 데이터가 바뀌면 쿼리를 고쳐야 한다.

```sql
-- 월이 늘어날 때마다 이 쿼리를 손으로 수정해야 함
SELECT
  category,
  SUM(CASE WHEN month = '2024-01' THEN amount ELSE 0 END) AS jan,
  SUM(CASE WHEN month = '2024-02' THEN amount ELSE 0 END) AS feb,
  -- ... 열 줄 더 ...
  SUM(CASE WHEN month = '2024-12' THEN amount ELSE 0 END) AS dec
FROM sales
GROUP BY category;
```

**After — long 집계 + 코드 배치.** 쿼리는 컬럼 3개로 고정, 펼치기는 코드가 담당한다.

```sql
-- 출력 컬럼은 영원히 3개. 데이터가 변해도 안 고침
SELECT category, month, SUM(amount) AS total
FROM sales
GROUP BY category, month
ORDER BY category, month;
```

그리고 애플리케이션에서 long 결과를 wide로 돌려세운다. 한 번 순회하면 된다.

```typescript
type Row = { category: string; month: string; total: number };

function pivot(rows: Row[]) {
  const table = new Map<string, Record<string, number>>();
  const months = new Set<string>();

  for (const { category, month, total } of rows) {
    if (!table.has(category)) table.set(category, {});
    table.get(category)![month] = total;
    months.add(month);
  }

  // 등장한 월 전체가 자동으로 열이 됨. 12월이든 24개월이든 코드는 그대로
  const columns = [...months].sort();
  return { columns, table };
}
```

`months`는 데이터에 실제로 등장한 값에서 모으므로, 2025년이 와서 행이 늘어나든 카테고리가 추가되든 이 함수는 손댈 필요가 없다. **동적인 부분을 코드 구조가 아니라 데이터로 다룬 덕분**이다.

게다가 현실에선 이 펼치는 코드조차 직접 짤 일이 잘 없다. **프론트엔드 테이블 컴포넌트나 BI/리포트 라이브러리가 long → wide 변환을 이미 기본으로 지원**한다. 예를 들어 표 라이브러리에 "행 키는 category, 열 키는 month, 값은 total"이라고 설정만 던지면 알아서 크로스탭으로 그려준다. 엑셀 피벗 테이블이 하는 일이 정확히 이거다 — 긴 데이터를 받아서 사용자가 축만 지정하면 가로로 펼친다. **DB는 긴 집계까지만 내려주고, 펼치는 건 펼치는 걸 잘하는 도구한테 맡기면 된다.**

<Callout type="note" title="데이터 전송량도 덤으로 줄어든다">
long 포맷은 빈 칸(없는 월/카테고리)에 대해 아예 행을 안 만든다. 반면 wide로 펼치면 매트릭스가 sparse할 때 `0`이나 `NULL`로 칸을 다 채워야 한다. 카테고리 100개 × 월 24개인데 실제 데이터가 듬성듬성하다면, long으로 보내는 편이 전송량도 작다. 펼치는 비용은 받는 쪽에서 한 번 치르면 끝이다.
</Callout>

## 정리

동적 피벗이 사람을 잡는 이유는 단순하다. 화면에서 가로로 펼쳐 보이고 싶은 욕심을, SQL 쿼리 한 방에 욱여넣으려 하기 때문이다. 그런데 SQL은 파싱 시점에 컬럼이 고정되는 언어라, "데이터에 따라 열이 변하는" 일과는 본질적으로 안 맞는다. 억지로 밀어붙이면 `CASE WHEN`을 손으로 박게 되고, 그때부터 데이터가 변할 때마다 쿼리를 고치는 신세가 된다.

> **컬럼이 데이터에 따라 변하면, 그건 SQL이 아니라 표현 계층의 일이다.**

선 긋는 법은 명확하다. **집계(묶어서 더하기)는 SQL이, 배치(행을 열로 돌려세우기)는 애플리케이션이.** `GROUP BY`로 세로로 긴 결과까지만 내려보내고, 가로로 펼치는 건 코드나 리포트 라이브러리에 맡긴다. 그러면 컬럼이 늘어나도 쿼리는 그대로고, 동적 SQL의 인젝션·디버깅 늪도 피하고, 벤더 종속도 풀린다. "이 쿼리, 컬럼이 데이터 따라 늘었다 줄었다 하네?"라는 생각이 드는 순간, 그건 SQL에서 손을 떼고 코드로 내려야 한다는 신호다.

다음 편에서는 거의 모든 목록 화면이 한 번씩 겪는 문제를 본다. 페이지네이션 — `LIMIT`/`OFFSET`이 뒤 페이지로 갈수록 왜 느려지는지, 그리고 키셋(keyset) 방식이 그걸 어떻게 푸는지 얘기다.
