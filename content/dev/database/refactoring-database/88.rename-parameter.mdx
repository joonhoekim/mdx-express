## 이게 뭔데

Rename Parameter는 **저장 프로시저나 함수의 파라미터 이름을, 그 파라미터가 실제로 뜻하는 바에 맞게 바꾸는** 리팩토링이다. 끝. 이게 전부다. 카탈로그 통틀어 가장 작고, 가장 안 무서운 녀석.

비유하자면 서랍에 붙은 라벨을 바꾸는 거다. 서랍 안 물건은 그대로고, 서랍 위치도 그대로고, 그냥 "기타잡동사니"라고 적힌 라벨을 떼고 "충전케이블"이라고 다시 붙이는 거. 서랍을 통째로 옮기는 게 아니니까 옆 책상에 영향 갈 일이 없다. 라벨만 정확해지면 다음에 케이블 찾는 사람이 안 헤맨다.

DB 세계에서 이 "라벨"이 파라미터 이름이다. `@p1`, `@x`, `@flag`, `@dt` 같은 이름은 컴퓨터한텐 아무 상관 없지만, **6개월 뒤에 그 프로시저를 열어보는 사람한텐 라벨 떼고 던져준 서랍이랑 똑같다.** Rename Parameter는 그 라벨을 사람 말로 바꾸는 작업이다.

<Callout type="info" title="한 줄 요약">
파라미터 이름만 바꾸는, 시그니처는 그대로 두는 내부 리팩토링이다. 동작은 1도 안 변한다. 변하는 건 "읽는 사람의 이해도" 하나뿐.
</Callout>

## 이게 왜 "내부 리팩토링"이냐

여기서 한 가지는 짚고 가야 한다. 직관적으로는 "파라미터 이름을 바꾼다 = 인터페이스를 바꾼다"처럼 들린다. 근데 카탈로그상 Rename Parameter는 **내부 리팩토링(Internal Refactoring)** 으로 분류된다. 인터페이스 변경 리팩토링이 아니다.

이유는 **호출 규약(calling convention)** 에 있다. 대부분의 DB에서 프로시저는 **위치 인자(positional)** 로 호출된다.

```sql
-- 호출부는 이름이 아니라 '순서'로 값을 넘긴다
CALL GetAccountsForCustomer(12345, '2026-06-09');
```

호출부 입장에서 첫 번째 자리에 고객 ID, 두 번째 자리에 날짜를 넣은 거지, 그 자리의 파라미터가 `@CustomerID`로 불리든 `@cid`로 불리든 알 바 아니다. **이름은 프로시저 안에서만 쓰이는 지역 변수 같은 존재**라서, 이름을 바꿔도 위치로 부르는 호출부는 멀쩡하다. 그래서 외부 프로그램을 안 고쳐도 되고, 전환 기간(Transition Period)도 필요 없다. Add Parameter나 Rename Method 같은 인터페이스 변경 리팩토링이 호출하는 모든 앱·배치·리포트를 줄줄이 따라 고쳐야 하는 것과는 차원이 다르다.

<Callout type="warning" title="단, '명명 인자'로 부르고 있었다면 얘기가 달라진다">
일부 DB와 호출 방식은 **명명 인자(named argument)** 를 허용한다. 가령 PostgreSQL은 `CALL proc(customer_id => 12345)`처럼, 오라클은 `proc(p_customer_id => 12345)`처럼 부를 수 있다. 이렇게 **이름으로 값을 바인딩하던 호출부가 있다면, 파라미터 이름을 바꾸는 순간 그 호출은 깨진다.** 이 경우엔 더 이상 순수 내부 리팩토링이 아니라 인터페이스 변경에 가까워진다. 그러니 리네임 전에 호출부가 위치로 부르는지 이름으로 부르는지부터 확인할 것.
</Callout>

## 언제 쓰나

원리는 단순한데, 정작 "지금 이걸 해야 하나?" 판단이 더 중요하다. 다음 냄새가 나면 Rename Parameter 타이밍이다.

- **이름이 의미를 안 알려준다.** `@p1`, `@val`, `@x`, `@temp` 같은 류. 타입은 알겠는데 그래서 뭘 넣으라는 건지 모른다.
- **이름이 거짓말을 한다.** 더 위험한 케이스. `@CustomerName`이라고 적혀 있는데 실제론 고객 ID(숫자)가 들어온다든가, `@Date`인데 실은 기간(일수)이 들어온다든가. 틀린 라벨은 빈 라벨보다 나쁘다. 사람을 적극적으로 오해시킨다.
- **명명 규칙에 안 맞는다.** 팀은 `@CustomerID`로 통일했는데 옛날 프로시저만 `@cust_no`, `@CustID`, `@id_cust`로 제각각인 경우. 일관성이 깨지면 검색도 안 되고 자동완성도 헷갈린다.
- **약어가 너무 압축됐다.** `@accbal`, `@inscvrg` 같은 거. 본인은 알지만 남은 못 읽는다.

### 현실 시나리오 한 토막

은행 코어에 이런 프로시저가 있다고 치자. 2009년에 누가 짜놓고 떠난 거다.

```sql
-- 2009년의 누군가가 남긴 유산
CREATE PROCEDURE CalculateInterest
    @aid    INT,
    @r      DECIMAL(5,4),
    @d      INT
AS
BEGIN
    DECLARE @bal DECIMAL(15,2);
    SELECT @bal = Balance FROM Account WHERE AccountID = @aid;
    RETURN @bal * @r * @d / 365;
END;
```

신입이 이 프로시저를 처음 본다. `@aid`는 어찌어찌 Account ID겠거니 한다. `@r`은 rate겠지. 근데 `@d`는? days? date? 일수인가 날짜인가. 본문 `* @d / 365`를 보고 나서야 "아 일수구나" 한다. **코드를 다 읽어야 파라미터 의미를 역추적할 수 있는** 상태. 이게 매번 반복된다. 호출할 때마다, 디버깅할 때마다, 누군가 본문을 정독한다.

여기서 Rename Parameter가 답이다. 본문 한 줄 안 바꾸고, 이름만 사람 말로 바꿔서 이 고통을 영구히 제거한다.

```sql
-- After: 라벨만 정확해졌다
CREATE PROCEDURE CalculateInterest
    @AccountID       INT,
    @AnnualRate      DECIMAL(5,4),
    @PeriodInDays    INT
AS
BEGIN
    DECLARE @Balance DECIMAL(15,2);
    SELECT @Balance = Balance FROM Account WHERE AccountID = @AccountID;
    RETURN @Balance * @AnnualRate * @PeriodInDays / 365;
END;
```

이제 `@PeriodInDays`다. 더 이상 본문 정독 안 해도 된다. 호출부는 여전히 `CalculateInterest(12345, 0.0325, 30)`로 위치로 부르니까 한 군데도 안 깨진다.

## 주의할 점

너무 쉬워 보여서 방심하기 딱 좋은데, 함정이 몇 개 있다.

<Callout type="warning" title="리네임 전에 반드시 확인할 것">
- **명명 인자 호출부 확인.** 위에서 말했듯, 이름으로 바인딩하는 호출부(`p_customer_id => ...`)가 하나라도 있으면 그건 깨진다. 코드베이스 전체에서 해당 프로시저명을 grep해서 호출 방식을 먼저 확인하라.
- **동적 SQL 속 파라미터.** 본문 안에서 문자열로 SQL을 조립하면서 파라미터를 바인딩하는 경우, IDE 리팩토링이 문자열 안까지는 못 따라간다. `EXEC sp_executesql @sql, N'@aid INT', @aid = ...` 같은 데서 `@aid`를 누락하면 런타임에 터진다.
- **트리거 안의 의사 레코드.** 트리거의 `:NEW`/`:OLD`(오라클)나 `NEW`/`OLD`(PG)는 파라미터가 아니라 컬럼명을 따라간다. 이건 Rename Parameter가 아니라 Rename Column 영역이니 헷갈리지 말 것.
- **오버로드.** 같은 이름의 프로시저가 파라미터 시그니처만 다르게 여러 개 있으면(오라클 패키지 등), 어느 오버로드를 고치는 건지 명확히 해야 한다.
</Callout>

그리고 의외로 이게 가장 흔한 실수인데 — **"단순 리네임이니까 테스트 안 돌려도 되겠지"** 하는 마음. 동작이 안 변한다는 건 맞지만, 사람이 손으로 바꾸다 보면 본문 어딘가에서 옛 이름 하나를 안 바꾸고 빠뜨린다. 그러면 컴파일이 깨지거나(다행) 엉뚱한 변수를 참조한다(재앙). **단순할수록 자동화로 처리하고, 그래도 테스트는 돌리는 게** 안전하다.

## 이렇게 한다

### 스키마 변경 (DDL)

프로시저는 보통 "수정"이 아니라 `CREATE OR REPLACE`(혹은 `DROP` 후 `CREATE`)로 통째로 다시 정의한다. 그러니 DDL이라기보단 "전체 재정의"에 가깝다. 위 `CalculateInterest` 예시가 곧 before/after다. 핵심은 **파라미터 선언부와 본문의 모든 참조를 일관되게 같이 바꾸는 것** 하나뿐.

데이터 마이그레이션(DML)? **없다.** 테이블도, 데이터도, 컬럼도 안 건드린다. 이게 Rename Parameter가 카탈로그에서 가장 쌀 수밖에 없는 이유다. 다른 리팩토링들이 "스키마 바꾸고 → 기존 데이터 백필하고 → 듀얼라이트 걸고" 하는 동안 얘는 텍스트 한 줄 다시 정의하면 끝난다.

### 접근 프로그램(코드) 수정

위치 인자로만 부르고 있었다면 — **건드릴 게 없다.** 이게 베스트 케이스다.

명명 인자로 부르던 호출부가 있다면, 그것만 새 이름으로 같이 고친다.

```sql
-- 명명 인자 호출부가 있었다면 (PostgreSQL 예)
-- Before
CALL calculate_interest(aid => 12345, r => 0.0325, d => 30);
-- After
CALL calculate_interest(account_id => 12345, annual_rate => 0.0325, period_in_days => 30);
```

2006년 책은 이걸 손으로 찾아 바꿨다. 지금은 거의 다 자동이다. 도구별로 짚어보자.

DataGrip, SSMS + SQL Prompt(Redgate), Oracle SQL Developer 같은 툴은 파라미터 리네임을 **시맨틱하게** 처리한다. 단순 텍스트 치환이 아니라 "이건 파라미터, 저건 동명의 컬럼"을 구분해서 본문 참조까지 같이 바꿔준다. 우클릭 → Rename(보통 `Shift+F6`) 한 방.

단, 위에서 경고한 **동적 SQL 문자열 안쪽**은 시맨틱 분석 밖이라 못 따라간다. 리네임 후 해당 프로시저 본문에서 옛 이름을 한 번 더 grep해서 잔재가 없는지 확인하는 습관을 들이자.
  </TabsContent>

  <TabsContent value="migration">
운영 반영은 손으로 `CREATE OR REPLACE` 치지 말고 **버전 관리되는 마이그레이션**으로 내려보낸다. 프로시저를 멱등하게 재정의하면 되니, 보통 `CREATE OR REPLACE` 한 파일이면 충분하다.

```sql
-- Flyway: V0088__rename_param_calculate_interest.sql
CREATE OR REPLACE FUNCTION calculate_interest(
    account_id      INT,
    annual_rate     NUMERIC(5,4),
    period_in_days  INT
) RETURNS NUMERIC AS $$
    SELECT (SELECT balance FROM account WHERE account.account_id = calculate_interest.account_id)
           * annual_rate * period_in_days / 365;
$$ LANGUAGE sql;
```

Liquibase라면 `createProcedure`/`sqlFile` changeSet으로, Alembic이라면 `op.execute()`에 같은 DDL을 담는다. 핵심은 **"누가 언제 왜 이름을 바꿨는지"가 마이그레이션 히스토리에 남는 것**이다. 운영 DB에 직접 들어가 프로시저를 고치는 순간, 다음 사람은 또 똑같이 코드를 정독하는 신세가 된다.
  </TabsContent>

  <TabsContent value="orm">
앱이 프로시저를 부를 때 **위치 인자로 부르면** 이 리팩토링에 무관심해도 된다. 반대로 **명명 인자/딕셔너리로 바인딩**하던 코드는 같이 고쳐야 한다.

```typescript
// Before: 키 이름으로 바인딩하고 있었다면 깨진다
await db.query('CALL calculate_interest(:aid, :r, :d)', { aid, r, d });

// After: 키 이름을 새 파라미터명에 맞춘다
await db.query(
  'CALL calculate_interest(:account_id, :annual_rate, :period_in_days)',
  { account_id, annual_rate, period_in_days },
);
```

여기서 교훈 하나. **앱은 가급적 위치 인자보다 의미가 드러나는 바인딩을 쓰되, 프로시저 파라미터 이름에 결합하는 건 최소화**하는 게 좋다. 그래야 DB 쪽 내부 리팩토링이 앱까지 번지지 않는다.
  </TabsContent>
</Tabs>

### 전체 절차

<Steps>
  <Step title="호출 방식 정찰">
코드베이스 전체에서 프로시저명을 검색해, 호출부가 **위치 인자인지 명명 인자인지** 파악한다. 위치뿐이면 순수 내부 리팩토링, 명명이 섞였으면 영향 범위에 그 호출부들을 포함시킨다.
  </Step>
  <Step title="새 이름 합의">
팀 명명 규칙에 맞춰 새 이름을 정한다. 약어 풀고, 단위 드러내고(`@d` → `@PeriodInDays`), 거짓말하는 이름은 진실로 교정한다.
  </Step>
  <Step title="IDE로 시맨틱 리네임">
우클릭 Rename으로 선언부와 본문 참조를 한 번에 바꾼다. 동적 SQL 문자열 안쪽은 별도로 grep해 수동 확인.
  </Step>
  <Step title="마이그레이션 파일로 커밋">
손으로 운영에 반영하지 말고 Flyway/Liquibase/Alembic 마이그레이션에 `CREATE OR REPLACE`로 담아 버전 히스토리에 남긴다.
  </Step>
  <Step title="테스트 돌리기">
'단순 리네임'이라도 프로시저 회귀 테스트(혹은 최소한 스모크 호출)를 돌려 본문에서 옛 이름을 빠뜨린 데가 없는지, 명명 인자 호출부가 안 깨졌는지 확인한다.
  </Step>
</Steps>

<Callout type="success" title="왜 작은 리팩토링부터 손대기 좋은가">
프로시저용 자동화 테스트가 부실한 팀일수록, 인터페이스를 안 바꾸는 내부 리팩토링부터 시작하는 게 현실적이다. Rename Parameter는 그중에서도 가장 안전하다. 외부 영향 없고, 데이터 마이그레이션 없고, 롤백도 옛 정의를 다시 올리면 끝. "리팩토링 근육"을 붙이는 첫 운동으로 딱이다.
</Callout>

## 정리

Rename Parameter는 카탈로그에서 가장 작은 리팩토링이지만, 효과는 작지 않다. 코드는 한 번 쓰고 백 번 읽힌다. `@d`가 `@PeriodInDays`로 바뀌는 한 번의 수고가, 그 프로시저를 열어보는 백 명의 "이게 일수야 날짜야?" 정독을 없앤다.

> **동작은 안 변한다. 변하는 건 다음 사람이 코드를 이해하는 속도다.**

기억할 건 딱 하나. 위치 인자로 부르고 있었다면 이건 외부에 안 새는 순수 내부 리팩토링이고, 명명 인자로 부르던 호출부가 있었다면 거기까지가 영향 범위다. 그 경계만 정확히 긋고 IDE 리네임 → 마이그레이션 파일 → 테스트 순으로 내려보내면, 가장 쉽게 끝낼 수 있는 리팩토링이 맞다.
