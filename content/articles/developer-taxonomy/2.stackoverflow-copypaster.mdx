## 복붙의 황금기

2008년 스택오버플로우가 등장하면서 개발 생태계에 혁명이 일어났음. 그 전에는 뭐 하나 모르면 600쪽짜리 O'Reilly 책을 뒤져야 했는데, 이제는 구글에 에러 메시지만 복붙하면 답이 튀어나오는 세상이 된 거임.

개발자의 검색 진화사를 요약하면 대략 이렇다. 1990년대엔 man page를 읽다가 동료한테 물어보고 결국 포기했고, 2000년대엔 구글링하다 포럼을 뒤져도 여전히 몰랐다. 2010년대엔 스택오버플로우에서 복붙하면 일단 작동은 했고(왜 되는지는 몰랐고), 2020년대엔 ChatGPT한테 물어봐서 복붙하면 작동했다(역시 왜 되는지는 몰랐다). 그리고 지금은 AI가 코드를 다 짜주는데, 안 되면 AI한테 "왜 안 되냐"고 다시 물어보는 시대임.

<Callout type="info" title="스택오버플로우 재미있는 통계">
- 역대 가장 많이 조회된 질문: "How do I undo the most recent local commits in Git?" (1,400만+ 조회)
- 즉, 전 세계 개발자 1,400만 명이 git commit을 잘못 했다는 뜻
- 개발자는 실수하는 존재. 우리 모두 평등하게 무능함.
</Callout>

### 복붙에도 급이 있다

같은 복붙이라도 이해도에 따라 등급이 나뉨.

- **Lv.1 순수 복붙 (이해 0%)** — 좋아요 제일 많은 답변을 그대로 붙여넣음. "되긴 되는데 왜 되는지 모름"
- **Lv.2 변수명만 바꾸는 복붙 (이해 10%)** — `foo`를 `users`로 바꾸고 "이해했음"이라고 착각하는 단계
- **Lv.3 프랑켄슈타인 복붙 (이해 30%)** — 답변 A에서 함수 구조, B에서 에러 처리, C에서 타입을 가져와 합침. 셋이 서로 호환이 안 됨
- **Lv.4 복붙 후 커스터마이징 (이해 60%)** — 원리를 어느 정도 이해하고 프로젝트에 맞게 수정. 이 정도면 "참고"라고 부르는 게 맞음
- **Lv.5 읽고 영감받아 직접 작성 (이해 90%)** — 이건 복붙이 아니라 "학습"임. 여기 오면 졸업

복붙러의 전형적인 하루는 이렇다. 아침에 `Cannot read property 'map' of undefined`를 만나 2012년 답변을 복붙하는데, 그게 jQuery 기반이라 React 프로젝트에서 안 맞음. 그래서 "react equivalent of jquery each"를 또 복붙. 그러다 `this.setState is not a function`이 떠서 `bind(this)` 하라는 2016년 답변을 복붙하는데, 우리 프로젝트는 함수형 컴포넌트라 클래스 문법이 섞여 더 안 됨. 점심 무렵엔 원래 버그는 그대로고 새로운 버그만 3개 늘어나 있음.

## ChatGPT 의존러의 탄생

2022년 11월 ChatGPT가 등장하면서 판이 또 뒤집어졌음. 스택오버플로우 트래픽이 폭락하고 새로운 종족이 태어났는데, 바로 **ChatGPT 의존러**, 일명 "프롬프트 엔지니어"임.

<Callout type="warning" title="ChatGPT 의존러의 위험 신호">
- 간단한 for 루프도 ChatGPT한테 물어봄
- "ChatGPT가 이렇게 하라고 했어요"를 코드 리뷰 코멘트에 씀
- AI가 생성한 코드를 검증 없이 프로덕션에 배포함
- 에러가 나면 에러 메시지를 AI한테 던지고 나온 답을 또 복붙함
- AI가 만든 코드의 로직을 본인이 설명하지 못함
</Callout>

ChatGPT 의존러의 코딩은 보통 무한 루프로 끝남. "사용자 목록 컴포넌트 만들어줘" → 복붙, 작동함, 좋아. "여기 페이지네이션 추가해줘" → AI가 기존 코드를 무시하고 새 코드를 줌 → 복붙 → 충돌. "이 에러 고쳐줘" → 또 다른 코드 → 또 다른 에러. 4번째 프롬프트쯤이면 AI도 컨텍스트를 잃고, 결과물은 아무도(AI 포함) 이해 못 하는 프랑켄슈타인 코드가 됨.

스택오버플로우 복붙과 ChatGPT 복붙의 결정적 차이는 **검증 여부**임. `const unique = [...new Set(array)]` 같은 답변은 좋아요 수천 개, 댓글의 엣지 케이스 경고, 성능 벤치마크까지 딸려 있음 — 즉 검증된 집단지성임. 반면 ChatGPT는 보통은 맞지만 가끔 자신감 넘치게 **환각**함. 실제로 ChatGPT가 추천한 적 있는 존재하지 않는 것들: npm 패키지 `array-unique-deep`, `Array.prototype.unique()`, `lodash.deduplicate`. 전부 세상에 없는 것들인데 너무 당당하게 추천해서 더 무서움.

## 복붙의 함정들

### 1. 라이선스 지뢰밭

스택오버플로우의 코드는 CC BY-SA 4.0 라이선스임. 두 가지 의무가 따라옴. 하나는 **출처 표기**(이 코드는 스택오버플로우에서 가져왔다고 밝혀야 함 — 솔직히 아무도 안 함), 다른 하나는 **동일 라이선스 공유**임. 후자가 진짜 함정인데, MIT 라이선스 프로젝트에 CC BY-SA 코드를 넣으면 라이선스 충돌이 생김. "에이 설마 누가 고소하겠어"라고 넘기기엔, 프로덕션 코드의 약 30%가 스택오버플로우 출처라는 연구도 있음. 회사 법무팀이 이걸 알면 경악함.

<Callout type="error" title="실제 사례: 라이선스 문제">
2022년, 한 기업이 오픈소스 라이선스 감사를 받다가 스택오버플로우에서 가져온 코드 조각이 GPL 오염을 일으킨 걸 발견했음. 해당 모듈 전체를 다시 작성해야 했고 3개월이 걸렸음. "그냥 복붙한 건데..."가 3개월의 재작업으로 돌아온 사례.
</Callout>

### 2. 보안 취약점 복붙

스택오버플로우에서 가장 부지런히 복붙되는 게 사실 보안 취약점임. 단골 메뉴는 이런 것들임.

**SQL 인젝션** — 아래 같은 2011년 답변이 아직도 복붙됨. `userName`에 `'; DROP TABLE users; --`를 넣으면 테이블이 통째로 날아감. (정답: parameterized query)

```javascript
// 복붙되는 취약한 코드
const query = "SELECT * FROM users WHERE name = '" + userName + "'";
```

- **TLS 검증 비활성화** — "SSL 에러 나요" 질문의 최다 좋아요 답변이 인증서 검증을 끄라고 함. "야 이거 하니까 됨!!" ← 보안 구멍을 뚫어서 된 거임
- **CORS 전부 허용** — `origin: '*'`로 "오 됐다!" ← 전 세계 누구나 니 API를 호출할 수 있게 만든 거임

### 3. 버전 호환성 시한폭탄

2019년 답변(React 16 기준)을 2026년 프로젝트에 붙여넣으면 `componentWillMount`, `componentWillReceiveProps` 같은 게 튀어나옴. 이것들은 React 17에서 deprecated, 18에서 제거됐음. 복붙러는 "스택오버플로우에서 되는 코드인데 왜 안 되죠?"라고 하고, 팀원은 한숨 쉬며 말함. "그 답변 7년 전 거잖아요..."

## 검색도 실력이다

복붙 자체가 나쁜 건 아님. 문제는 **생각 없이** 복붙하는 것. 검색을 잘하는 것 역시 진짜 실력임.

0.1x식 검색은 "react not working"(결과 3억 개), "error in my code"(???), "javascript why"(이건 철학적 질문임) 같은 거임. 좋은 검색은 **구체적인 기술 + 구체적인 증상 + 구체적인 상황 + 버전**을 조합함. 예: "React useEffect cleanup function not called unmount", "Next.js 16 app router fetch cache revalidate". 버전 명시가 특히 중요함.

답변을 고를 때도 신호가 있음. 좋은 답변은 좋아요가 많고(100+), 최근에 수정됐고, 댓글에 엣지 케이스 토론이 있고, 왜 이렇게 하는지 설명이 있음. 나쁜 답변은 2015년 이전이고 수정 이력이 없고, "이거 해보세요" 한 줄짜리고, 댓글에 "이거 더 이상 안 됨"이 달려 있고, 보안 설정을 비활성화하라고 함.

<Callout type="success" title="효과적인 검색 프레임워크">
이 순서로 접근하면 됨:

1. **에러 메시지를 정확히 읽기** — 놀랍게도 대부분의 답이 에러 메시지 안에 있음
2. **공식 문서 먼저** — 검색어에 `site:nextjs.org` 추가
3. **버전 명시** — "React 19"를 검색어에 포함
4. **GitHub Issues 확인** — 라이브러리 버그일 수도 있음
5. **스택오버플로우/AI는 마지막** — 위 4단계로 안 될 때

이 순서를 지키면 복붙을 해도 양질의 코드를 복붙하게 됨.
</Callout>

AI도 마찬가지임. "사용자 관리 시스템 만들어줘"로 500줄 받아 통째로 붙여넣는 건 나쁜 활용이고, **구체적으로 질문하고 → 답변을 검증하고 → 학습 도구로 쓰는 것**이 좋은 활용임. "이 코드에서 exhaustive check가 왜 필요한지 설명해줘"라고 물어 **이해한 다음** 적용하는 것. 이해 없이 복붙하지 않는 것. 그 한 끗 차이임.

## 복붙러에서 개발자로

<Callout type="note" title="복붙에서 벗어나는 5단계">
**1단계 — 인정**: "나 복붙 좀 많이 하는 것 같음" 인정하기

**2단계 — 이해**: 복붙 전에 코드를 한 줄씩 읽기. 모르는 메서드는 MDN/공식 문서에서 확인

**3단계 — 변형**: 복붙한 코드를 프로젝트 스타일에 맞게 수정. 변수명·에러 처리·타입을 직접 고치기

**4단계 — 재작성**: 복붙한 코드를 참고만 하고 처음부터 직접 작성. 이 과정에서 진짜 이해가 됨

**5단계 — 기여**: 내가 겪은 문제와 해결법을 블로그/스택오버플로우에 공유. 복붙 당하는 입장이 되어보기
</Callout>

마지막으로 강조하고 싶은 게 있음. **시니어도 검색함.** 이건 부끄러운 게 아님. 기억력엔 한계가 있고 기술은 계속 변하니까. 차이는 검색의 *깊이*에 있음. 주니어가 "how to center a div"와 "왜 안됨"을 검색할 때, 시니어는 라이브러리 소스 코드(GitHub), RFC 문서, 벤치마크 결과, 릴리즈 노트, 보안 어드바이저리를 찾아봄.

우리는 모두 **거인의 어깨 위에 서 있음.** 남이 만들어둔 답을 활용하는 건 효율이지 부끄러움이 아님. 바퀴를 매번 다시 발명할 필요는 없으니까. 다만 그 바퀴가 어떻게 굴러가는지는 알아야 함. 중요한 건 **검색 결과를 어떻게 소화하느냐**임. 복붙은 시작점이지, 도착점이 아님.
