## 이게 뭔데

**임시 변수 분리(Split Temporary Variable)**는 한 마디로, **한 변수에 두 가지 다른 의미의 값을 번갈아 담아 쓰는 걸 그만두는 일**이다. 의미가 다르면 변수도 따로 두는 거다.

비유를 하나 들자. 주방에 컵이 하나밖에 없다고 치자. 아침엔 거기다 커피를 따라 마시고, 점심엔 같은 컵에 간장을 부어 양념을 재고, 저녁엔 또 거기다 물을 받아 마신다. 그때그때 헹구면 되긴 하는데, 어느 날 헹구는 걸 깜빡하면 커피에서 간장 맛이 난다. 컵을 용도별로 세 개 두면 이런 사고가 안 난다.

코드에서 `temp`, `v_result`, `n` 같은 만능 변수가 딱 그 하나뿐인 컵이다. 화씨를 섭씨로 바꾼 값을 잠깐 담았다가, 몇 줄 아래에서 같은 변수에 인치를 센티미터로 바꾼 값을 또 담는다. 동작은 한다. 근데 읽는 사람은 "이 시점의 `temp`가 온도냐 길이냐"를 매번 머릿속으로 추적해야 한다.

<Callout type="note" title="한 줄 정의">
변수 하나에 서로 다른 책임을 맡기지 마라. **루프 카운터·누산기를 제외하고, 변수는 살면서 딱 한 가지 의미만 가져야 한다.** 의미가 바뀌는 지점이 곧 새 변수를 만들어야 하는 지점이다.
</Callout>

이건 책에서 **10장 메서드 리팩토링**의 **내부 리팩토링(Internal Refactoring)**으로 분류된다. 저장 프로시저·함수·트리거 — 책 용어로 "메서드" — 의 **외부 인터페이스는 그대로 두고 내부 구현만 손보는** 부류다. 파라미터도, 반환 형태도, 호출하는 쪽 코드도 안 바뀐다. 그래서 외부 영향이 없고, 손대기 가장 안전한 축에 속한다.

## 언제 쓰나

냄새는 단순하다. **하나의 변수가 코드 흐름 중간에 의미를 갈아탄다.** 한 번 선언된 `temp`에 처음엔 환율 환산 결과가 들어갔다가, 스무 줄 아래에서 이자 계산 중간값이 들어가고, 또 아래에선 그냥 루프 임시값으로 재활용된다. 선언은 한 번, 대입은 서로 무관한 값으로 여러 번.

왜 이게 생기냐면, 대부분 **게을러서**다. 새 변수 선언하기 귀찮으니까 위에서 쓰던 거 그냥 재활용한다. "어차피 다 끝난 변수잖아, 재탕해도 결과는 같지" 싶은 거다. 결과는 같을 수 있다. 근데 **읽는 사람과 다음에 고칠 사람**한테 같지 않다.

이 리팩토링이 답이 되는 상황:

- 프로시저 한가운데 `v_temp`, `tmp`, `result1`, `x` 같은 **의미 없는 이름의 변수**가 여기저기서 재대입되고 있다.
- 변수 하나를 추적하려고 위아래로 스크롤하며 "지금 이 값이 뭐였더라"를 반복하게 된다.
- 버그를 잡는데, 원인이 **"이전 용도에서 남은 값이 다음 용도로 새어 들어간" 것**이더라.

<Callout type="info" title="Fowler 원조 예제 그대로">
이 리팩토링의 교과서 예제가 바로 단위 변환 두 개를 한 변수에 욱여넣는 거다. `temp`에 화씨→섭씨 결과를 담고, 같은 `temp`에 인치→센티미터 결과를 또 담는다. 이름이 `temp`니까 둘 다 "잠깐 쓰는 값"이라는 점에선 같지만, **온도와 길이는 다른 것**이다. 변수도 `celsius`, `centimeters`로 나누면 코드가 자기 자신을 설명한다.
</Callout>

### 현실 시나리오: 이런 적 있을 거임

은행 시스템의 `CalculateAccountFees`라는 저장 프로시저를 물려받았다고 하자. 작성자는 이미 퇴사했고, 주석은 "TODO: 나중에 정리"가 전부다. 계좌 수수료를 계산하는 200줄짜리 PL/SQL인데, 중간에 `v_temp`라는 변수가 등장한다.

처음엔 외화 잔액을 원화로 환산한 값이 `v_temp`에 담긴다. 그걸로 환전 수수료를 매긴다. 그러고 나서 한참 아래, 같은 `v_temp`에 이번엔 **이자 계산용 일할(日割) 중간값**이 대입된다. 변수 이름은 똑같이 `v_temp`다.

어느 날 환전 수수료 로직에 분기를 하나 추가한다. 특정 통화는 환산값에 0을 곱하도록. 무심코 분기를 잘못 배치해서, `v_temp`가 0인 채로 아래 이자 계산 구간까지 흘러간다. **이자가 전부 0으로 찍힌다.** 단위 테스트는 통과한다 — 환전 케이스만 봤으니까. 운영에서 "왜 이자가 안 붙었냐"는 민원이 들어오고, 추적하면 결국 그 만능 `v_temp` 하나가 범인이다.

문제의 본질은 분기 실수가 아니다. **서로 무관한 두 계산이 같은 변수를 공유하고 있었다**는 구조다. 변수가 둘로 나뉘어 있었으면, 환전 쪽 실수가 이자 쪽으로 새어 나갈 통로 자체가 없었다.

## 주의할 점

<Callout type="warning" title="루프 변수와 누산기는 예외다">
"변수는 한 번만 대입"이라는 규칙엔 정당한 예외가 둘 있다. **루프 카운터**(`i := i + 1`)와 **누산기/집계 변수**(`v_total := v_total + amount`)는 본질적으로 같은 의미를 누적하는 거라 여러 번 대입되는 게 맞다. 이건 "용도가 바뀐" 게 아니라 "같은 용도를 갱신"하는 거다. 분리 대상이 아니다. 분리해야 하는 건 **의미 자체가 달라지는** 재대입이다.
</Callout>

몇 가지 더 짚자.

- **인터페이스는 절대 건드리지 않는다.** 이건 내부 리팩토링이다. 변수를 나누느라 프로시저 시그니처나 반환값이 바뀐다면, 그건 더 이상 Split Temporary Variable이 아니라 인터페이스 변경 리팩토링이고, 호출하는 모든 앱·배치·리포트를 함께 고쳐야 하는 별개의 큰일이 된다. 선을 넘지 마라.
- **변수가 너무 많아지는 걸 두려워 말되, 남발도 말자.** 의미가 다르면 나눈다. 하지만 진짜 같은 의미를 단지 "scope를 좁히려고" 억지로 쪼개는 건 과하다. 기준은 항상 **의미**다.
- **이름이 핵심이다.** `v_temp1`, `v_temp2`로 나누면 절반만 한 거다. 분리의 진짜 가치는 `v_krw_balance`, `v_daily_interest`처럼 **이름이 용도를 말해주는** 데서 나온다.
- **전제는 테스트다.** 외부 영향이 없다지만, 프로시저 내부 로직을 만지는 건 맞다. 회귀를 잡아줄 자동화 테스트(또는 최소한 입력→출력 골든 케이스)가 있으면 마음 놓고 손댄다. 없으면 이 안전한 리팩토링부터 시작해서 테스트를 깔아가는 것도 방법이다.

## 이렇게 한다

핵심 절차는 짧다.

<Steps>
<Step title="재대입 지점을 찾는다">
변수가 선언된 뒤 서로 무관한 값으로 다시 대입되는 지점을 찾는다. 루프 카운터·누산기는 제외. "의미가 바뀌는 첫 대입"이 분리 경계다.
</Step>
<Step title="두 번째 용도부터 새 변수로 선언한다">
의미가 바뀌는 그 지점에 새 변수를 선언하고, 그 아래 모든 참조를 새 변수로 교체한다. 첫 용도는 원래 변수에 그대로 둔다.
</Step>
<Step title="이름을 용도에 맞게 고친다">
나뉜 변수 각각에 의미를 드러내는 이름을 준다. `v_temp` → `v_celsius` / `v_centimeters`처럼. 이게 리팩토링의 본체다.
</Step>
<Step title="테스트로 동작 동일성을 확인한다">
출력이 리팩토링 전과 한 비트도 다르지 않은지 회귀 테스트로 확인한다. 내부 리팩토링이므로 외부 결과는 바뀌면 안 된다.
</Step>
</Steps>

### 단위 변환: 교과서 예제

먼저 Fowler 원조 그대로. 화씨→섭씨, 인치→센티미터를 한 변수에 욱여넣은 코드다.

```sql
-- Before: temp 하나가 온도였다가 길이였다가
CREATE OR REPLACE FUNCTION report_measurements(
    p_fahrenheit  NUMBER,
    p_inches      NUMBER
) RETURN VARCHAR2 IS
    v_temp  NUMBER;          -- 만능 임시 변수
    v_out   VARCHAR2(200);
BEGIN
    v_temp := (p_fahrenheit - 32) * 5 / 9;       -- 지금은 섭씨
    v_out  := '온도: ' || ROUND(v_temp, 1) || 'C';

    v_temp := p_inches * 2.54;                   -- 이제는 센티미터 (의미가 바뀜!)
    v_out  := v_out || ', 길이: ' || ROUND(v_temp, 1) || 'cm';

    RETURN v_out;
END;
```

`v_temp`가 6번째 줄에선 온도, 9번째 줄에선 길이다. 같은 이름인데 의미가 통째로 갈아탔다. 분리하면 이렇게 된다.

```sql
-- After: 의미마다 변수가 따로, 이름이 곧 설명
CREATE OR REPLACE FUNCTION report_measurements(
    p_fahrenheit  NUMBER,
    p_inches      NUMBER
) RETURN VARCHAR2 IS
    v_celsius      NUMBER;   -- 온도 전용
    v_centimeters  NUMBER;   -- 길이 전용
    v_out          VARCHAR2(200);
BEGIN
    v_celsius     := (p_fahrenheit - 32) * 5 / 9;
    v_centimeters := p_inches * 2.54;

    v_out := '온도: ' || ROUND(v_celsius, 1) || 'C'
          || ', 길이: ' || ROUND(v_centimeters, 1) || 'cm';

    RETURN v_out;
END;
```

선언만 봐도 이 함수가 온도와 길이를 다룬다는 게 보인다. 각 변수는 딱 한 번 대입되고, 한 가지 의미만 산다. 누군가 온도 공식을 잘못 고쳐도 길이 쪽은 절대 오염되지 않는다.

```sql
-- v_temp가 환산값 → 이자 중간값으로 의미를 갈아탄다
PROCEDURE calculate_account_fees(p_account_id IN NUMBER) IS
    v_temp        NUMBER;       -- 만능
    v_fx_balance  NUMBER;
    v_rate        NUMBER;
BEGIN
    -- 1) 외화 잔액을 원화로 환산
    SELECT foreign_balance, fx_rate
      INTO v_fx_balance, v_rate
      FROM account WHERE account_id = p_account_id;

    v_temp := v_fx_balance * v_rate;            -- 지금은 '원화 환산 잔액'
    INSERT INTO fee_ledger(account_id, fee_type, amount)
    VALUES (p_account_id, 'FX', v_temp * 0.001);

    -- ... 수십 줄 뒤 ...

    -- 2) 같은 변수에 이자 계산 일할 중간값을 재대입
    v_temp := v_fx_balance * 0.03 / 365;        -- 이제는 '일할 이자' (의미 바뀜!)
    INSERT INTO fee_ledger(account_id, fee_type, amount)
    VALUES (p_account_id, 'INTEREST', v_temp);
END;
```

</TabsContent>
<TabsContent value="after">

```sql
-- 의미별로 변수를 가르고, 이름으로 용도를 박아둔다
PROCEDURE calculate_account_fees(p_account_id IN NUMBER) IS
    v_fx_balance       NUMBER;
    v_rate             NUMBER;
    v_krw_balance      NUMBER;   -- 원화 환산 잔액 전용
    v_daily_interest   NUMBER;   -- 일할 이자 전용
BEGIN
    SELECT foreign_balance, fx_rate
      INTO v_fx_balance, v_rate
      FROM account WHERE account_id = p_account_id;

    -- 1) 환전 수수료
    v_krw_balance := v_fx_balance * v_rate;
    INSERT INTO fee_ledger(account_id, fee_type, amount)
    VALUES (p_account_id, 'FX', v_krw_balance * 0.001);

    -- 2) 일할 이자 — 더 이상 1)과 변수를 공유하지 않는다
    v_daily_interest := v_fx_balance * 0.03 / 365;
    INSERT INTO fee_ledger(account_id, fee_type, amount)
    VALUES (p_account_id, 'INTEREST', v_daily_interest);
END;
```

</TabsContent>
</Tabs>

차이가 결과를 바꾸진 않는다. 두 코드가 토해내는 `fee_ledger` 행은 똑같다. 그게 **내부 리팩토링의 핵심**이다. 외부에서 본 동작은 한 톨도 안 변한다. 변한 건 **다음에 이 코드를 여는 사람의 인지 부하**고, **환전 실수가 이자로 새어 나갈 경로가 사라졌다**는 안전성이다.

아까 시나리오에서 환전 쪽에 0을 곱하는 분기를 잘못 넣었던 그 버그를, After 코드에선 애초에 만들 수 없다. `v_krw_balance`가 0이 되든 말든 `v_daily_interest`는 완전히 독립이니까.

<Callout type="success" title="공짜로 따라오는 것: 옵티마이저와 scope">
변수를 의미별로 나누고 가능하면 **쓰기 직전에 선언**하면, 각 변수의 생존 구간(scope)이 짧아진다. 사람이 추적하기 쉬워질 뿐 아니라, PL/SQL 컴파일러나 옵티마이저도 변수 생명주기를 더 정확히 파악한다. "이 변수는 여기서 죽는다"가 명확하면 불필요한 상태 유지가 줄어든다. 가독성으로 시작한 리팩토링이 덤으로 약간의 효율까지 챙기는 셈이다.
</Callout>

### 현대 실무에 얹기

2006년의 책은 이걸 PL/SQL 손코딩 관점에서 다뤘지만, 오늘날 이 작은 리팩토링을 안전하고 반복 가능하게 만드는 도구들이 있다.

- **마이그레이션 도구로 버전 관리.** 프로시저 본문을 손으로 고치고 운영 DB에 붙여넣지 마라. **Flyway**나 **Liquibase**의 repeatable migration(`R__calculate_account_fees.sql` 같은 파일)으로 프로시저 정의를 형상 관리하면, 리팩토링이 코드 리뷰·롤백 가능한 변경으로 남는다. 인터페이스가 안 바뀌는 내부 리팩토링이라 마이그레이션도 한 방에 안전하게 적용된다.

- **golden master 테스트로 동작 동일성 고정.** 내부 리팩토링의 유일한 합격 기준은 "출력이 전과 같다"는 거다. 리팩토링 **전에** 대표 입력들로 프로시저를 돌려 출력(여기선 `fee_ledger` 결과)을 스냅샷으로 박아두고, 리팩토링 후 같은 입력으로 돌려 diff가 0인지 본다. **utPLSQL**(PL/SQL용 단위 테스트 프레임워크)이 정확히 이 일을 해준다.

- **린터로 냄새를 자동 탐지.** `v_temp` 한 변수에 의미가 다른 두 값이 대입되는 패턴은 사람 눈으로 잡기 번거롭다. **SonarQube**의 PL/SQL/T-SQL 룰이나 정적 분석기는 "같은 변수에 무관한 재대입"이나 "의미 없는 변수명" 같은 코드 스멜을 PR 단계에서 짚어준다. 리팩토링을 사후가 아니라 사전에 막는 쪽으로.

- **로직을 애초에 DB 밖으로.** 더 근본적으로는, 단위 변환이나 수수료 계산 같은 순수 계산 로직이 꼭 저장 프로시저 안에 있어야 하는지 되물을 수 있다. 마이크로서비스로 데이터 소유권을 가져가는 추세에선 이런 계산을 애플리케이션 레이어로 끌어내 일반 코드 리팩토링·테스트의 풍부한 생태계 안에서 다루기도 한다. 다만 이건 큰 결정이고, **눈앞의 200줄짜리 프로시저를 당장 읽기 쉽게 만드는 데는 Split Temporary Variable 한 방이 가장 싸고 안전하다.**

## 정리

Split Temporary Variable은 책에 실린 리팩토링 중에서도 가장 소박한 축이다. 스키마를 안 바꾸고, 데이터 마이그레이션도 없고, 호출부도 안 건드린다. 하는 일이라곤 **"한 변수가 두 가지 의미를 겸직하던 걸 멈추는 것"**뿐이다.

> **변수 하나는 의미 하나. 의미가 바뀌는 자리가 곧 새 변수가 태어날 자리다.**

소박하다고 가치가 작은 건 아니다. 만능 `v_temp`는 가독성을 갉아먹는 데서 그치지 않고, 한 용도의 실수가 다른 용도로 새어 나가는 **숨은 결합**을 만든다. 변수를 가르면 그 통로 자체가 막힌다. 거기에 의미를 담은 이름까지 붙이면, 주석 없이도 코드가 자기를 설명한다.

그러니 저장 프로시저를 열었는데 `temp`나 `x`에 서로 다른 값이 번갈아 대입되고 있다면, 그건 컵 하나로 커피와 간장을 같이 쓰고 있다는 신호다. 컵을 나눠라. 비싸지도, 위험하지도 않다.
