## 이게 뭔데

남의 떡이 커 보인다. SQL을 쓰다 보면 어느 순간 NoSQL이 신세계처럼 보이는 시기가 온다. 스키마 마이그레이션은 매번 무섭고, JOIN은 짜기 귀찮고, 인덱스는 튜닝해도 또 느려진다. 그때 누군가 옆에서 속삭인다. "MongoDB 쓰면 그거 다 없어져요." 그 한마디가 사람을 흔든다.

근데 이 안티패턴의 진짜 정체는 "NoSQL이 나쁘다"가 아니다. **"도구를 바꾸면 내가 안 한 설계가 저절로 채워진다"는 기대**가 안티패턴이다. NoSQL은 잘못된 게 아니다. 오히려 그 반대다.

핵심 오해부터 깨고 가자. 많은 사람이 NoSQL을 "스키마를 안 짜도 되는 DB"로 이해한다. 절반만 맞다. **NoSQL은 스키마 정의를 안 받을 뿐, 모델링을 면제해 주는 게 아니다.** 오히려 NoSQL은 관계형보다 모델링을 **더 강하게** 강제한다. 관계형은 일단 정규화해서 넣어두면 어떤 쿼리든 JOIN으로 사후에 조립할 수 있다. NoSQL은 그게 잘 안 된다. 그래서 NoSQL은 "이 데이터를 **어떤 화면에서 어떻게 읽을 것인가**"를 먼저 정하고, 그 접근 패턴에 맞춰 document를 미리 짜 넣어야 한다. 이걸 access-pattern-first modeling이라고 부른다.

그러니까 SQL의 고통에 지쳐서 NoSQL로 도망치는 건, **설계 숙제를 안 한 채로 더 어려운 시험장에 들어가는 것**이다. 거기서 숙제를 안 하면 어떻게 되겠나. 똑같이 틀린다. 정확히는, 1편에서 봤던 그 N+1이 짐가방에 묻어서 같이 따라온다.

<Callout type="warning" title="한 줄 요약">
NoSQL은 "스키마를 면제"해 주는 게 아니라 "접근 패턴 중심 모델링을 강제"한다. 그 설계를 안 하고 도망치면, SQL에서 겪던 N+1과 정합성 문제가 도구만 바뀐 채 그대로 따라온다.
</Callout>

## 시나리오: 이런 적 있을 거임

스키마 마이그레이션에 데이고, JOIN에 데이고, 인덱스 튜닝에 지쳤다. 컨퍼런스 발표를 본다. "MongoDB로 갔더니 개발 속도가 10배!" 발표자 표정이 너무 평온하다. 결심한다. "스키마 고민 끝! 그냥 document로 막 넣자."

처음엔 진짜 편하다. 스키마 정의도, 마이그레이션 파일도 없다. 그냥 객체 하나 만들어서 `insert`. 필드 추가하고 싶으면 그냥 넣으면 된다. `ALTER TABLE`도, 다운타임도 없다. 첫 주는 마치 족쇄가 풀린 기분이다. "진작 갈아탈걸."

그러다 게시글 목록 화면을 만든다. 각 게시글 옆에 작성자 이름을 같이 보여줘야 한다. 그런데 게시글 document엔 `authorId`만 들어 있다. 관계? 그런 거 없다. NoSQL이니까. 그래서 이렇게 짠다.

```javascript
// 게시글 100개를 가져온다
const posts = await db.collection('posts').find().limit(100).toArray();

// 작성자 이름을 붙이려고... 하나씩 조회한다
for (const post of posts) {
  post.author = await db.collection('users').findOne({ _id: post.authorId });
}
```

화면은 잘 뜬다. 코드도 깔끔하다. 그런데 DB 프로파일러를 켜 보면 풍경이 1편과 소름 끼치게 똑같다.

```text
db.posts.find().limit(100)              -- 1번
db.users.findOne({_id: ...})            -- +1
db.users.findOne({_id: ...})            -- +1
...
db.users.findOne({_id: ...})            -- +1 (총 100번)
```

쿼리 **1번이 101번**이 됐다. 어... 이거 어디서 봤더라. **N+1이다.** SQL에서 도망쳐 왔는데, N+1이 먼저 와서 짐 풀고 기다리고 있었다. `JOIN`이라는 단어만 사라졌지, 코드에서 손으로 합치는 그 행동은 토씨 하나 안 바뀌었다.

"아 맞다, NoSQL은 임베드(embed)하는 거랬지." 그래서 작전을 바꾼다. 작성자 정보를 게시글 document 안에 통째로 박아 넣는다.

```javascript
// 게시글에 작성자 정보를 통째로 임베드
{
  _id: "post_1",
  title: "오늘 배운 것",
  authorId: "user_42",
  author: { name: "김개발", avatar: "/u42.png" }  // ← 박아둠
}
```

이제 목록 조회는 진짜 한 방에 끝난다. `find()` 하나면 작성자 이름까지 다 딸려 온다. 빨라졌다. 만족스럽다.

그런데 일주일 뒤, 김개발이 닉네임을 "김시니어"로 바꾼다. 그 순간 깨닫는다. 이 사람이 쓴 게시글이 3,400개고, 각 document에 박아둔 `author.name`을 **전부 갱신**해야 한다. 안 그러면 어떤 글엔 "김개발", 어떤 글엔 "김시니어"가 떠서 정합성이 깨진다. 이거... **6편에서 봤던 성급한 비정규화 정합성 문제**가 그대로 재현됐다. 임베드는 N+1을 없앤 게 아니라, N+1을 정합성 부채로 바꿔치기한 것뿐이었다.

결국 깨닫는다. **고통의 원인은 SQL이 아니라 "접근 패턴을 설계 안 한 나"였다.** "작성자 이름은 자주 읽히고 거의 안 바뀌니 임베드, 자주 바뀌는 정보는 참조" — 이런 결정을 처음에 안 하고 그냥 "막 넣은" 게 원인이었다. NoSQL은 그 결정을 SQL보다 **더 강하게** 요구할 뿐, 면제해 주지 않는다.

고통을 옮겨온 건 도구가 아니라, 우리가 그대로 들고 간 행동 패턴이다.

- **"도구를 바꾸면 문제가 사라지겠지"** — 은탄환(silver bullet) 기대. 문제의 원인이 도구가 아니라 설계였는데, 도구를 바꿔서 풀려고 했다.
- **"스키마 안 짜도 되니까 빠르겠지"** — 모델링을 **없앤** 게 아니라 **미룬** 것이다. 미룬 빚은 목록 화면 만들 때 이자까지 쳐서 돌아온다.
- **"관계는 코드에서 이으면 되지"** — 또 N+1. **1편 JOIN 공포증과 정확히 같은 행동**이다. 그땐 JOIN이 무서워서 코드에서 합쳤고, 이번엔 JOIN이 아예 없어서 코드에서 합쳤다. 동기는 다른데 결과는 같다.

## 무슨 일이 벌어지나 (비용)

도구만 바꾸고 설계를 안 가져갔을 때, 비용은 세 갈래로 청구된다.

**1. 접근 패턴 미설계 → 앱 레벨 조인 → N+1 재발.** NoSQL엔 옵티마이저가 알아서 골라주는 JOIN 전략이 없다(혹은 제한적이다). "관계를 코드에서 잇는다"는 건 곧 **앱 레벨 조인**이고, 앱 레벨 조인을 루프 안에서 하면 그대로 N+1이다. 1편에서 본 라운드트립 폭증이 그대로 재현된다. DB 종류만 바뀌었지, "항목마다 한 번씩 왕복"이라는 구조는 동일하다.

**2. 임베드 vs 참조 결정을 안 하면 정합성·중복 폭탄.** NoSQL 모델링의 본질은 "이 데이터를 document 안에 **품을(embed)** 것인가, 밖에서 **참조(reference)**할 것인가"의 결정이다. 이걸 명시적으로 안 하면 두 갈래로 터진다. 참조만 쓰면 위의 N+1, 임베드를 남발하면 같은 정보가 수천 군데 복제돼서 갱신 시마다 정합성이 흔들린다(6편). 정답은 데이터 성격마다 다르다. 자주 같이 읽히고 거의 안 바뀌는 값은 임베드, 자주 바뀌고 여러 곳에서 공유되는 값은 참조. 이 결정을 **안 하는 것**이 안티패턴이다.

**3. 트랜잭션·제약을 포기한 대가가 뒤늦게 청구된다.** 관계형이 공짜로 깔아주던 안전장치들 — 외래 키 제약, 다중 행 트랜잭션, `UNIQUE`, `NOT NULL`, `CHECK` — 상당수가 NoSQL에선 기본이 아니다. "유연함"이라는 이름으로 넘어가지만, 실은 **무결성 책임이 DB에서 앱으로 넘어온 것**이다. 부모 없는 자식 document, 절반만 반영된 갱신, 중복 키 같은 게 운영 중에 슬며시 쌓인다. 이건 로컬에선 안 보이다가, 데이터가 쌓이고 동시성이 붙는 운영에서 터진다. 1편의 N+1과 발현 패턴이 똑같다.

<Callout type="error" title="뭐가 문제냐면">
- **N+1이 도구만 바꿔 그대로 따라온다**: JOIN이라는 단어가 사라졌을 뿐, "루프 안에서 항목마다 조회"는 똑같이 N+1이다.
- **임베드는 만능이 아니다**: N+1을 없애는 대신 정합성 부채를 떠안는다. 어느 쪽이 싼지는 데이터 성격이 결정한다.
- **무결성 책임이 앱으로 넘어온다**: DB가 공짜로 막아주던 깨진 데이터를, 이제 우리 코드가 일일이 막아야 한다.
</Callout>

### 미니: N+1

이 시리즈에서 N+1은 1편의 주인공이었지만, 여기서 다시 등장한 이유는 분명하다. **N+1은 SQL의 병이 아니라 데이터 접근 설계의 병**이라서, DB를 바꿔도 안 낫는다.

- **증상**: 목록 1쿼리 + 항목마다 1쿼리 = **1+N**. 게시글 100개면 101번 왕복.
- **해결**: 본질은 하나, **"항목마다 한 번"을 "전부 모아 한 번"으로 바꾸는 것**이다. 도구별로는 이렇게 나타난다.
  - SQL: `JOIN`으로 한 방에, 또는 `WHERE id IN (...)`으로 배치 조회.
  - NoSQL(MongoDB): `$lookup` 단계로 서버 사이드 조인, 또는 등장한 `authorId`를 모아 `find({ _id: { $in: [...] } })` 한 번에 배치 조회.
  - 공통: GraphQL이라면 DataLoader류 배치 로더가 한 틱 동안 들어온 키를 모아 `IN`/`$in` 한 방으로 묶는다.

```javascript
// Before: 작성자를 하나씩 조회 → 1 + 100 쿼리 (N+1)
const posts = await db.collection('posts').find().limit(100).toArray();
for (const post of posts) {
  post.author = await db.collection('users').findOne({ _id: post.authorId });
}

// After: 등장한 authorId를 모아 단 한 번에 → 2 쿼리
const posts = await db.collection('posts').find().limit(100).toArray();
const ids = [...new Set(posts.map(p => p.authorId))];
const authors = await db.collection('users')
  .find({ _id: { $in: ids } })
  .toArray();

const byId = new Map(authors.map(u => [String(u._id), u]));
for (const post of posts) {
  post.author = byId.get(String(post.authorId));
}
```

쿼리 100번이 2번이 됐다. 코드 양은 거의 안 늘었는데 왕복은 50배 줄었다. **SQL이든 NoSQL이든 처방은 동일하다 — "한 번에 모아 가져와라."** 더 깊은 내용과 라운드트립 비용 분석은 [1편 JOIN 공포증](/docs/dev/database/anti-patterns/1.fear-of-joins)을 보자. 거기 적힌 처방이 DB 종류만 바꾸면 여기서도 그대로 통한다는 게, N+1이 도구의 병이 아니라는 가장 확실한 증거다.

## 정석: 도구가 아니라 접근 패턴을 먼저 설계

해법은 도구를 고르기 전 단계에 있다. **"이 데이터를 어떤 쿼리로, 어떤 화면에서, 얼마나 자주 읽을 것인가"를 먼저 정한다.** 이게 접근 패턴 설계고, 이게 SQL/NoSQL 선택보다 앞선다.

순서를 뒤집지 말자. 흔한 실수는 "MongoDB로 정했으니 이제 어떻게 모델링하지?"다. 올바른 순서는 그 반대다.

1. **접근 패턴을 먼저 나열한다.** "게시글 목록을 작성자 이름과 함께 20개씩 페이징", "작성자 프로필에서 그 사람 글 전체", "글 상세에서 댓글 함께" 등 실제 화면이 요구하는 읽기 패턴을 적는다.
2. **각 패턴의 빈도와 변경 특성을 본다.** 자주 같이 읽히는가? 그 값이 자주 바뀌는가? 여러 곳에서 공유되는가?
3. **그 다음에 모델과 도구를 고른다.** 같이 읽히고 잘 안 바뀌면 임베드(또는 비정규화), 자주 바뀌고 공유되면 참조(또는 정규화 + JOIN/배치).

이렇게 하면 SQL이냐 NoSQL이냐는 **결론**이지 출발점이 아니게 된다. 관계가 복잡하고 무결성·트랜잭션이 중요하면 관계형이 맞고, 접근 패턴이 단순·고정적이고 한 document로 깔끔히 떨어지면서 수평 확장이 절실하면 NoSQL이 빛난다. 어느 쪽이든 **접근 패턴을 먼저 설계한 사람만 제대로 쓴다.**

<Callout type="info" title="그럼 NoSQL은 쓰지 말라는 거임?">
전혀 아니다. NoSQL은 접근 패턴이 명확할 때 관계형이 못 하는 일을 해낸다 — 단일 document 읽기로 화면이 한 방에 그려지고, 샤딩으로 수평 확장이 자연스럽다. 안티패턴은 "NoSQL을 쓰는 것"이 아니라 **"SQL 설계가 싫어서 도망치는 용도로 NoSQL을 고르는 것"**이다. 도주처로 고른 도구는 어떤 도구든 배신한다.
</Callout>

<Callout type="note" title="N+1을 발견하면">
DB가 뭐든 "이 데이터들을 코드 루프에서 하나씩 합치고 있다"는 생각이 들면, 그건 거의 항상 배치로 내려보내라는 신호다. SQL이면 `JOIN`/`IN`, MongoDB면 `$lookup`/`$in`. 처방은 도구가 아니라 패턴을 따른다.
</Callout>

## 정리

NoSQL로 갈아탄 사람들이 종종 하는 말이 있다. "막상 와 보니 SQL 때랑 고민이 똑같던데요?" 당연하다. 고민이 똑같은 이유는, 그 고민의 정체가 도구가 아니라 **데이터를 어떻게 읽을지 설계하는 일**이었기 때문이다. 그 숙제는 DB를 바꾼다고 사라지지 않는다. 짐가방에 넣어서 같이 이사 온다.

> **고통의 원인은 보통 도구가 아니라 설계의 부재다. 갈아타도 따라온다.**

그러니 SQL이 힘들 때 "NoSQL로 도망갈까"가 떠오르면, 그 전에 한 번 물어보자. "내가 지금 힘든 게 정말 SQL 때문인가, 아니면 접근 패턴을 설계 안 해서인가?" 후자라면, 어떤 DB로 가도 똑같이 힘들다. 도구를 바꾸기 전에 설계를 먼저 채우자. 남의 떡은, 가서 베어 물어 보면 보통 내 떡이랑 맛이 똑같다.

다음 편에서는 한 도구 안에서 똑같은 회피가 벌어지는 경우를 본다. 관계형을 쓰면서도 모델링이 싫어서, 컬럼 하나에 JSON을 통째로 쑤셔 넣고 그 안에 또 JSON을 넣는 — JSON 마트료시카 얘기다.
