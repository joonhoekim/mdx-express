## 이게 뭔데

Substitute Algorithm, 알고리즘 치환. 한 문장으로 말하면 **저장 프로시저(나 함수, 트리거) 안에서 어떤 일을 하는 방식 자체를 더 명확한 방식으로 통째로 갈아끼우는 것**이다. 결과는 똑같이 나와야 한다. 바뀌는 건 "어떻게 그 결과에 도달하느냐"뿐이다.

비유를 하나 들자. 옛날에 이 동네 토박이가 알려준 길로 매일 출근한다고 치자. 골목을 일곱 번 꺾고, 공사장을 우회하고, 어느 집 담벼락 옆을 지나는 그 길. 30분 걸린다. 본인은 눈 감고도 가지만, 옆자리 신입한테 길을 설명하려면 칠판이 필요하다. 그런데 어느 날 새로 뚫린 대로를 보니, 직진 두 번이면 같은 회사 정문에 똑같이 30분에 도착한다. 거리도 시간도 같다. 근데 설명은 한 줄이면 된다. "쭉 가다가 사거리에서 우회전." 이게 알고리즘 치환이다. **도착지(결과)는 그대로, 가는 길(구현)을 누구나 읽을 수 있는 직선으로 바꾸는 것.**

<Callout type="info" title="한 줄 요약">
Substitute Algorithm은 인터페이스를 건드리지 않는 **내부 리팩토링**이다. 프로시저 시그니처도, 입력도, 출력도 안 바뀐다. 바깥에서 보면 아무 일도 없었던 것처럼 보여야 정상이다.
</Callout>

내부 리팩토링이라는 게 핵심이다. 호출하는 쪽 — 앱이든 배치든 리포트든 — 은 이 프로시저가 안에서 뭘 어떻게 바꿨는지 알 필요도, 알 방법도 없다. 입력 같으면 출력 같으니까. 그래서 인터페이스 변경 리팩토링처럼 전환 기간이니 drop date니 하는 게 필요 없다. 외부 의존성에 미치는 충격이 0이라, 메서드 리팩토링 중에서 제일 손대기 부담 없는 축에 든다.

## 언제 쓰나

이 리팩토링이 답인 상황은 냄새로 알아챌 수 있다.

- 프로시저 하나를 고치려고 열었는데, 로직을 **다 읽고 머릿속에 그림을 그리는 데만 10분**이 걸린다. 중첩 IF가 다섯 겹이고, 같은 계산을 세 군데서 미묘하게 다르게 한다.
- 분명 더 간단한 방법이 있는 걸 아는데, 기존 코드가 워낙 꼬여 있어서 **부분 수정으로는 도저히 못 펴겠다.** 실밥 하나 당기면 스웨터 전체가 풀릴 것 같은 그 느낌.
- 알고리즘 자체가 시대에 뒤처졌다. 옛날엔 윈도우 함수가 없어서 커서로 한 줄씩 돌며 누적합을 손으로 구했는데, 이제 DB가 `SUM() OVER ()`를 지원한다. 굳이 커서를 돌 이유가 없다.
- 성능 때문이 아니라 **그냥 읽기 힘들어서** 다음 사람(=3개월 뒤의 나)이 고생할 게 뻔하다.

여기서 한 가지 구분이 중요하다. **부분 수정으로 펼 수 있으면 그건 Substitute Algorithm이 아니다.** 중첩 IF를 보호 절로 펴는 건 `Replace Nested Conditional with Guard Clauses`고, 긴 블록을 떼어내는 건 `Extract Method`다. 알고리즘 치환은 그런 국소 수술로는 안 되고, **접근 방식 자체를 버리고 새로 써야 할 때** 쓰는 무거운 카드다.

책의 조언이 여기서 빛난다. **복잡한 알고리즘은 곧장 치환하려 들지 말고, Extract Method 등으로 먼저 단순화한 뒤 치환하라.** 800줄짜리 스파게티를 통째로 새로 쓰면서 "기능은 그대로 유지했음"을 보장하는 건 거의 도박이다. 먼저 의미 있는 조각으로 쪼개서 "아, 이 프로시저가 사실은 (1) 자격 검사, (2) 이율 계산, (3) 로깅 이 세 가지를 하는구나"가 눈에 보이게 만들면, 그제야 "(2) 이율 계산"만 안전하게 새 알고리즘으로 바꿀 수 있다.

### 현실 시나리오: 이런 적 있을 거임

은행 시스템에 `CalculateAccountInterest`라는 저장 프로시저가 있다. 계좌(Account)의 잔액(Balance)과 고객(Customer) 등급, 그리고 보험(Insurance) 가입 여부에 따라 적용 이율을 정하는 물건이다. 누가 2009년에 짰고, 그 뒤로 요구사항이 들어올 때마다 IF가 하나씩 붙었다. 지금은 이렇게 생겼다.

```sql
-- Before: 십 년치 IF가 쌓인 지층. 읽으면 두통이 온다
CREATE PROCEDURE CalculateAccountInterest(
  IN  p_account_id  BIGINT,
  OUT o_rate        DECIMAL(5,4)
)
BEGIN
  DECLARE v_balance  DECIMAL(15,2);
  DECLARE v_grade    CHAR(1);
  DECLARE v_has_ins  TINYINT;

  SELECT a.balance, c.grade,
         (SELECT COUNT(*) FROM Insurance i WHERE i.account_id = a.account_id)
    INTO v_balance, v_grade, v_has_ins
    FROM Account a JOIN Customer c ON c.customer_id = a.customer_id
   WHERE a.account_id = p_account_id;

  IF v_grade = 'A' THEN
    IF v_balance >= 100000 THEN
      IF v_has_ins > 0 THEN SET o_rate = 0.0350;
      ELSE SET o_rate = 0.0325; END IF;
    ELSE
      IF v_has_ins > 0 THEN SET o_rate = 0.0300;
      ELSE SET o_rate = 0.0275; END IF;
    END IF;
  ELSE
    IF v_grade = 'B' THEN
      IF v_balance >= 100000 THEN SET o_rate = 0.0250;
      ELSE SET o_rate = 0.0225; END IF;
    ELSE
      IF v_balance >= 100000 THEN SET o_rate = 0.0200;
      ELSE SET o_rate = 0.0175; END IF;
    END IF;
  END IF;
END;
```

읽었는가? 다시 읽어보라. "A등급에 잔액 10만 이상에 보험 있으면 3.5%"... 이걸 머릿속에서 8가지 경우로 펼치고 나면 이미 점심시간이다. 그리고 다음 분기에 "C등급 신설"이 떨어지면 이 IF 미로에 새 가지를 또 끼워 넣어야 한다. 끼우다가 한 칸 어긋나면, 누군가의 이자가 조용히 틀리게 나간다. 슬로우 쿼리에도 안 잡히고, 에러 로그에도 안 찍히고, 그냥 **숫자가 틀린다.** 제일 무서운 종류의 버그다.

이게 Substitute Algorithm의 정확한 적응증이다. 부분 수정으로는 이 IF 지층을 못 편다. 접근 방식 자체 — "조건 분기로 이율을 결정한다" — 를 "테이블에서 이율을 조회한다"로 갈아끼워야 한다.

## 주의할 점

신난다고 바로 새 알고리즘을 갈겨쓰기 전에, 이 리팩토링 특유의 함정 세 개를 짚고 가자.

<Callout type="warning" title="치환하기 전에 반드시 확인할 것">
- **인터페이스를 진짜로 안 건드렸는지** 확인하라. Substitute Algorithm은 정의상 시그니처·입력·출력이 동일해야 한다. 새로 짜다 보면 "이왕 고치는 김에 파라미터 하나 더 받자" 하는 유혹이 온다. 그 순간 이건 더 이상 내부 리팩토링이 아니라 인터페이스 변경(Add Parameter)이 되고, 전환 기간과 외부 호출부 수정이 줄줄이 딸려온다. 욕심내지 마라. 인터페이스는 다음 PR에서.
- **숨은 부수효과(side effect)를 빠뜨리지 마라.** 옛 알고리즘이 중간에 로그 테이블에 INSERT를 하거나, 세션 변수를 세팅하거나, NULL을 특정 방식으로 처리하고 있었을 수 있다. 결과 컬럼만 보고 "같네" 하면, 안 보이던 부수효과가 사라져서 다른 데가 터진다. 출력값뿐 아니라 **부수효과까지 동치**여야 진짜 치환이다.
- **엣지 케이스가 동치의 무덤이다.** 잔액이 정확히 100000일 때 `>=`냐 `>`냐, 등급이 NULL일 때 옛 코드는 어느 가지로 빠졌나, 보험이 2건 이상이면? 옛 알고리즘의 "버그까지 포함한 정확한 동작"을 새 알고리즘이 그대로 재현하는지를 따져야 한다. 여기서 함부로 "고치는 김에 버그도 고치자"를 끼우면 리팩토링이 아니라 동작 변경이다. 분리하라.
</Callout>

마지막 항목이 특히 중요하다. 리팩토링의 정의는 "동작 의미와 정보 의미를 **둘 다 보존**하면서" 설계를 개선하는 것이다. 옛 코드의 명백한 버그를 발견했더라도, 알고리즘 치환과 버그 수정은 **별개의 커밋**으로 가라. 섞으면 나중에 "이자가 바뀐 게 리팩토링 탓인가 버그 수정 탓인가"를 아무도 못 가린다.

## 이렇게 한다

절차는 셋이다. 먼저 단순화, 다음 치환, 그다음 동치 증명. 순서가 곧 안전벨트다.

<Steps>
<Step title="회귀 테스트로 현재 동작을 박제한다">
손대기 전에 옛 알고리즘의 동작을 **테스트로 먼저 고정**한다. 이게 전부의 전제다. 입력 조합을 망라해서 "지금 이 프로시저가 뱉는 값"을 캡처해두면, 새 알고리즘이 그걸 한 칸도 안 어기는지 기계가 검증해준다. 사람 눈으로 8가지 분기를 검수하는 것보다 정확하고 빠르다.
</Step>
<Step title="Extract Method로 먼저 단순화한다">
치환 대상을 곧장 새로 쓰지 말고, 의미 있는 조각으로 쪼갠다. "자격 판정", "이율 결정", "로깅" 같은 책임이 분리돼 눈에 보이면, 정말 갈아끼워야 할 부분(이율 결정)만 외과적으로 도려낼 수 있다.
</Step>
<Step title="알고리즘을 치환한다">
도려낸 조각을 더 명확한 방식으로 다시 쓴다. 여기선 "조건 분기"를 "테이블 룩업"으로 바꾼다. (이율표를 테이블에 두는 건 `Replace Literal with Table Lookup`과 짝을 이루는, 아주 흔한 조합이다.)
</Step>
<Step title="테스트 그린을 확인하고, 부수효과·엣지 케이스를 점검한다">
1단계에서 박제한 회귀 테스트가 전부 통과하는지 본다. 그린이면 출력 동치는 증명된 것이다. 그다음 부수효과와 NULL·경계값을 별도로 점검한다.
</Step>
</Steps>

### 1단계 — 동작을 박제하는 회귀 테스트

핵심은 **현재 프로시저가 뱉는 값을 신뢰의 기준선(golden master)으로 삼는 것**이다. "올바른 값"을 우리가 다시 계산하는 게 아니라, "지금 운영에서 나가는 값"을 그대로 떠서 고정한다. 그래야 동작 보존이 보장된다.

```sql
-- 입력 조합을 망라하는 케이스 테이블을 만든다
-- (등급 A/B/그외) × (잔액 경계 위/아래) × (보험 유무) = 12케이스
-- 각 케이스에 대해 옛 프로시저의 출력을 캡처해 기준선으로 저장
CREATE TEMPORARY TABLE interest_golden_master AS
SELECT a.account_id, NULL AS expected_rate
  FROM Account a
 WHERE a.account_id IN (/* 대표 케이스 계좌들 */);

-- 옛 프로시저를 돌려 기준선 채우기 (의사코드 수준)
-- for each account_id: CALL CalculateAccountInterest(id, @r); update expected_rate = @r;
```

테스트 러너 쪽에서 보면 이렇게 생긴다. 프로시저를 호출해 나온 값을, 미리 떠둔 기준선과 한 행씩 대조한다.

```typescript
// 동치 회귀 테스트: 옛 동작을 기준선으로 박제하고, 치환 후에도 같은지 검증
import { describe, it, expect } from "vitest";
import { callInterestProc, GOLDEN_MASTER } from "./fixtures";

describe("CalculateAccountInterest 동치 보장", () => {
  // GOLDEN_MASTER = 치환 직전에 옛 프로시저로 떠둔 [입력 -> 출력] 스냅샷
  it.each(GOLDEN_MASTER)(
    "account %s 는 이율 %s 를 유지해야 한다",
    async ({ accountId, expectedRate }) => {
      const rate = await callInterestProc(accountId);
      // 부동소수 비교가 아니라 DECIMAL 동치 — 문자열/정수 환산으로 정확히 본다
      expect(rate).toBe(expectedRate);
    },
  );
});
```

이 테스트가 그린인 상태에서 알고리즘만 바꾸고, 다시 그린이면 **출력은 동치임이 증명된다.** 2006년 책이 "실행하고 직접 결과를 비교하라"고 손으로 시키던 그 검증을, 지금은 `it.each`로 12케이스든 1200케이스든 자동으로 돌린다. 이게 현대화의 핵심이다 — 동치 보장을 사람의 눈썰미가 아니라 CI에 맡긴다.

<Callout type="success" title="golden master가 곧 안전벨트">
리팩토링이 무서운 진짜 이유는 "기능을 안 깼다는 걸 어떻게 확신하느냐"다. golden master 테스트는 그 확신을 기계가 대신 책임진다. 박제 → 치환 → 그린 확인. 이 사이클을 지키면, 800줄 스파게티를 새로 써도 손이 안 떨린다.
</Callout>

### 2단계 — Extract Method로 단순화

곧장 치환하지 않는다. 먼저 프로시저가 사실 무슨 일들을 하는지 드러낸다. 위 예시 프로시저는 결국 "(a) 계좌 상태 조회 → (b) 이율 결정"이다. (b)만 함수로 떼어내자.

```sql
-- Extract: 이율 결정 로직만 별도 함수로 도려낸다
-- (이 단계에선 알고리즘은 아직 그대로! 자리만 옮긴다)
CREATE FUNCTION DetermineRate(
  p_grade CHAR(1), p_balance DECIMAL(15,2), p_has_ins TINYINT
) RETURNS DECIMAL(5,4)
BEGIN
  -- ...옛 IF 지층이 여기로 그대로 이사옴...
  -- 핵심: 이사만 했지 로직은 한 글자도 안 바꿨다 -> 동작 보존
END;
```

이렇게 떼어내고 회귀 테스트를 한 번 돌린다. 그린이면 "도려내기"는 동작을 안 깬 것이다. 이제 메인 프로시저는 `SET o_rate = DetermineRate(v_grade, v_balance, v_has_ins);` 한 줄로 깔끔해지고, **갈아끼울 대상이 `DetermineRate` 하나로 격리**됐다. 스파게티 전체가 아니라 작은 함수 하나만 새로 쓰면 된다. 위험 표면이 확 줄었다.

### 3단계 — 알고리즘 치환 (조건 분기 → 테이블 룩업)

이제 진짜 치환이다. 8중 IF로 이율을 "계산"하던 걸, 이율표에서 "조회"하는 방식으로 갈아끼운다. 먼저 이율표 테이블을 둔다. (이게 `Replace Literal with Table Lookup`이다. 매직 넘버 0.0350 같은 게 코드에서 사라져 테이블로 이주한다.)

```sql
-- 이율 규칙을 데이터로: 운영 중 무중단 변경도 가능해진다
CREATE TABLE InterestRate (
  grade        CHAR(1)        NOT NULL,
  min_balance  DECIMAL(15,2)  NOT NULL,  -- 이 잔액 이상에 적용
  has_insurance TINYINT       NOT NULL,
  rate         DECIMAL(5,4)   NOT NULL,
  PRIMARY KEY (grade, min_balance, has_insurance)
);
```

이제 `DetermineRate`의 8중 IF를 단 한 번의 SELECT로 치환한다.

```sql
-- After: 알고리즘 치환 완료. IF 미로가 룩업 한 방으로 평탄해졌다
CREATE FUNCTION DetermineRate(
  p_grade CHAR(1), p_balance DECIMAL(15,2), p_has_ins TINYINT
) RETURNS DECIMAL(5,4)
BEGIN
  DECLARE v_rate DECIMAL(5,4);

  SELECT rate INTO v_rate
    FROM InterestRate
   WHERE grade = CASE WHEN p_grade IN ('A','B') THEN p_grade ELSE '*' END
     AND has_insurance = CASE WHEN p_has_ins > 0 THEN 1 ELSE 0 END
     AND min_balance <= p_balance
   ORDER BY min_balance DESC      -- 가장 높은 적용 구간 하나
   LIMIT 1;

  RETURN v_rate;
END;
```

같은 8가지 결과가 나온다. 그런데 이젠 "C등급 신설"이 떨어져도 코드는 한 줄도 안 건드린다. `InterestRate`에 행 두 개 INSERT면 끝이다. IF 미로에 가지를 끼워 넣다 한 칸 어긋나 이자가 틀리던 그 위험이, 통째로 사라졌다.

<Callout type="note" title="DB 코드 리팩토링도 결국 Fowler다">
눈치챘겠지만 여기서 한 일은 Martin Fowler의 평범한 코드 리팩토링 — Extract Method, Substitute Algorithm, Replace Magic Number — 을 PL/SQL·T-SQL 같은 DB 코드에 그대로 적용한 것뿐이다. 책의 메시지가 정확히 이거다. "검증된 코드 리팩토링을 저장 프로시저에도 똑같이 쓰라." 다만 DB 코드는 회귀 테스트 인프라가 부실한 현장이 많아, golden master를 먼저 깔고 시작하는 규율이 앱 코드보다 훨씬 더 중요하다.
</Callout>

### 접근 프로그램은 손댈 게 없다

여기가 내부 리팩토링의 즐거운 지점이다. `CalculateAccountInterest`를 호출하던 앱·배치·리포트는 **단 한 줄도 안 고친다.** 시그니처가 그대로고 출력이 그대로니까. 인터페이스 변경 리팩토링이었다면 전환 기간을 두고 호출부를 전수 조사해 고쳐야 했겠지만, 알고리즘 치환은 바깥에서 보면 아무 일도 없었다. 이게 내부 리팩토링을 "우선 손대기 좋은 안전한 카드"로 만드는 이유다.

배포 순서만 챙기면 된다. 마이그레이션 도구(Flyway, Liquibase, Alembic 등) 기준으로:

```text
V31__create_interest_rate_table.sql      -- 이율표 테이블 + 데이터
V32__replace_determine_rate_algorithm.sql -- 함수 본문 치환
-- 롤백 스크립트(U32, U31)도 짝으로 준비: 그린이 깨지면 즉시 되돌린다
```

치환된 함수는 옛 버전과 출력이 동치이므로, 배포 직후 golden master 테스트를 운영(혹은 스테이징)에서 한 번 더 돌려 그린을 확인하면 무중단으로 끝난다. 데이터 마이그레이션이랄 게 없다 — 알고리즘만 바뀌었지 저장된 데이터의 의미는 그대로다. 새로 생긴 `InterestRate`는 "코드에 박혀 있던 상수를 데이터로 옮겨 담은 것"일 뿐이다.

## 정리

Substitute Algorithm은 화려하지 않다. 새 컬럼이 생기는 것도, 테이블이 쪼개지는 것도 아니다. 그냥 **꼬인 길을 직선으로 펴는 일**이다. 그래서 가치도 분명하다 — 다음에 이 프로시저를 여는 사람이 10분이 아니라 30초 만에 이해하고, 가지를 끼우다 이자를 틀리게 만들 위험이 사라진다.

세 가지만 기억하면 된다.

> **하나, 곧장 새로 쓰지 말고 Extract Method로 먼저 단순화해서 치환 대상을 격리하라.**
> **둘, golden master 회귀 테스트로 동치를 기계에 증명시켜라 — 출력만이 아니라 부수효과·엣지 케이스까지.**
> **셋, 리팩토링과 버그 수정을 섞지 마라. 동작 보존이 깨지는 순간 그건 다른 작업이다.**

알고리즘 치환이 무섭게 느껴졌다면, 그건 "기능을 안 깼다는 확신"이 없어서다. 그 확신을 사람의 눈썰미가 아니라 CI의 그린 불빛이 책임지게 만드는 순간, 800줄 스파게티 앞에서도 손이 안 떨린다.
