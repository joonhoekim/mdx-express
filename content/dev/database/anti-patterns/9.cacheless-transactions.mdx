## 이게 뭔데

캐시 없는 트랜잭션. 정확히는 "생각 없는 캐시"에 가깝다. 증상은 이렇다. **어디가 느리다는 말만 들리면 반사적으로 캐시부터 붙이는 습관**이다. 무엇을 캐시할지, 그게 바뀌면 언제 버릴지는 안 정한다. 그냥 느린 곳 앞에 Redis 하나 세워두면 성능 문제가 알아서 증발할 거라고 믿는 거다.

비유를 하나 들어보자. 냉장고를 샀다고 치자. 냉장고가 있으면 음식이 안 상한다고 믿는 사람이 있다. 그래서 먹다 남은 거, 사 온 거, 어제 시킨 배달 음식까지 전부 냉장고에 욱여넣는다. 문제는 두 가지다. 첫째, **언제 버려야 하는지를 안 본다.** 한 달 전 반찬이 그대로 들어 있다가 어느 날 열었을 때 사람을 잡는다. 둘째, **다 같은 날 상한다.** 정전 한 번 나면 안에 있던 거 전부 동시에 못 먹게 된다. 냉장고가 있다고 음식이 안 상하는 게 아니다. 뭘 넣을지, 언제 버릴지를 사람이 정해야 한다.

캐시도 똑같다. 캐시를 붙인다는 건 "느린 곳에 마법 상자를 놓는 일"이 아니다. 본질은 두 개의 결정이다. **무엇을 캐시할까**, 그리고 **언제 무효화할까**. 이 둘을 안 정하면 캐시는 성능 도구가 아니라, 옛날 값을 뱉고 부하가 오면 같이 터지는 폭탄이 된다.

<Callout type="warning" title="한 줄 요약">
캐시의 핵심은 "저장"이 아니라 "무엇을 넣고 언제 버릴까"라는 두 결정이다. 이걸 안 정하고 Redis만 세우면, 평소엔 빨라 보이지만 부하가 오는 순간 성능이 신기루처럼 사라진다.
</Callout>

## 시나리오: 이런 적 있을 거임

API가 느리다는 리포트가 올라온다. 검색해보니 다들 그런다. "캐시 붙이세요." 그래서 붙인다. "느리네? Redis 앞에 깔면 되지." 너무나 자연스럽게 이렇게 친다.

```typescript
async function getProduct(id: string) {
  const cached = await cache.get(`product:${id}`);
  if (cached) return JSON.parse(cached);

  const product = await db.query("SELECT * FROM products WHERE id = $1", [id]);
  await cache.set(`product:${id}`, JSON.stringify(product));
  return product;
}
```

응답이 빨라진다. APM 그래프가 쭉 내려간다. 끝난 줄 안다. PR에 "성능 50% 개선 🚀"이 달리고, 회고에서 박수도 받는다. 아무 문제 없어 보인다.

그런데 며칠 뒤 CS에 문의가 들어온다. "상품 가격을 수정했는데 화면에 안 바뀌어요." DB를 보면 분명히 바뀌어 있다. 근데 캐시는 옛날 값을 계속 내주고 있다. **무효화(invalidation)를 안 걸었다.** `cache.set`만 있고, 데이터가 바뀔 때 캐시를 지우는 코드가 어디에도 없다. 한 번 들어간 값은 영원히 거기 있다.

급한 대로 막는다. TTL 60초를 박는다.

```typescript
await cache.set(`product:${id}`, JSON.stringify(product), { EX: 60 });
```

이제 "수정하고 최대 1분은 옛날 값이 보이는" 서비스가 됐다. 정합성 문제를 시간으로 덮은 거다. 운영팀은 "1분만 기다리세요"를 매크로로 만들어 둔다. 찝찝하지만 일단 굴러는 간다.

진짜 사고는 트래픽 몰리는 날 터진다. 세일 오픈 시각, 인기 상품 페이지에 동시 접속이 몰린다. 그런데 그 인기 키들은 비슷한 시각에 캐시에 들어갔으니 **TTL도 비슷한 시각에 한꺼번에 만료된다.** 만료된 순간, 그 키를 찾던 수천 개의 요청이 전부 캐시 미스가 나고, **동시에 DB로 쏟아진다.** 다 같은 쿼리를, 같은 순간에, 수천 번.

```text
12:00:00  product:hot 캐시 만료
12:00:00  요청 3,000개 동시 도착 → 전부 캐시 미스
12:00:00  SELECT * FROM products WHERE id = 'hot'  × 3,000  → DB로 직행
12:00:01  DB 커넥션 풀 고갈, CPU 100%, 응답 지연 폭발
```

이게 **캐시 스탬피드(cache stampede)**, 또는 썬더링 허드(thundering herd)다. 캐시를 붙이기 전이라면 DB가 적당히 분산된 부하를 받았을 텐데, 이젠 평소엔 캐시가 다 막아주다가 만료되는 그 한순간에 부하가 응축돼서 한꺼번에 터진다. **캐시를 붙이기 전보다 DB가 더 아프다.** 그렇게 빨라 보였던 성능이 부하가 오는 순간 신기루처럼 사라진다.

신기루를 부른 건 캐시 자체가 아니라, 그걸 다루는 방식이다.

- "느리면 캐시 붙이면 되지" — 캐시를 **설계**가 아니라 켜고 끄는 **스위치**로 취급. 무엇을 캐시할지 따져본 적이 없다.
- "무효화? TTL 짧게 주면 되겠지" — 정합성 문제를 **시간으로 얼버무림**. 캐시 무효화가 왜 어려운지를 통째로 과소평가한 것.
- "캐시 미스 나도 DB가 받아주겠지" — 미스가 **한꺼번에** 날 수 있다는 걸 고려 안 함. 스탬피드를 안 본 것.

이 셋 중 하나라도 마음에 있었다면, 당신은 이미 신기루 캐시를 한 번쯤 세워봤을 가능성이 높다. 그리고 이건 신입만의 일이 아니다. "일단 Redis 붙이자"는 시니어 회고에도 단골로 등장하는 멘트다.

## 무슨 일이 벌어지나 (비용)

캐시를 생각 없이 붙였을 때 치르는 비용을 세 갈래로 뜯어보자.

**1. Stale data — 옛날 값을 계속 서빙한다.** 무효화 전략이 없으면 캐시는 한 번 들어간 값을 고집한다. 가격이 바뀌어도, 재고가 빠져도, 게시글이 수정돼도 캐시는 모른다. TTL로 덮으면 "TTL만큼은 틀린 값이 보장된다"는 이상한 계약이 생긴다. 이게 상품 가격이면 컴플레인으로 끝나지만, 권한이나 잔액 같은 거라면 그냥 버그를 캐시에 박제하는 셈이다. **캐시 무효화가 컴퓨터 과학에서 어렵기로 손꼽히는 두 문제 중 하나**라는 농담이 괜히 나온 게 아니다.

**2. 캐시 스탬피드 — 동시 만료로 DB에 요청이 폭발한다.** 위 시나리오 그대로다. 인기 키일수록 비슷한 시점에 캐시되고 비슷한 시점에 만료된다. 만료 순간 모든 요청이 미스를 맞고 한꺼번에 원본으로 향한다. 평소 캐시가 막아주던 부하가 한 점으로 응축돼서 터지니, 오히려 캐시 없을 때보다 피크 부하가 커진다. 캐시가 부하 평탄화 장치가 아니라 부하 응축 장치가 되는 순간이다.

**3. 캐시 계층 자체가 새로운 장애 지점이다.** 공짜 성능은 없다. Redis를 하나 세웠다는 건 운영할 시스템이 하나 늘었다는 뜻이다. 메모리가 차면 eviction이 일어나 엉뚱한 키가 사라지고, 캐시와 DB 사이 일관성을 누가 보장하는지 애매해지고, Redis 자체가 죽으면 모든 미스가 DB로 직행하면서 전면 장애가 된다. 캐시를 잘못 다루면 "성능 문제를 해결한" 게 아니라 "성능 문제에 일관성 문제와 운영 부담을 추가한" 게 된다.

<Callout type="error" title="뭐가 문제냐면">
- **틀린 값을 빠르게 준다**: 캐시는 "빠른 정답"이 아니라 "빠른 옛날 답"이 되기 쉽다. 무효화가 없으면 속도를 정합성과 맞바꾼 것.
- **평소엔 멀쩡하다**: stale도 스탬피드도 트래픽이 적을 땐 잘 안 보인다. 인기 키에 부하가 몰리는 그 한순간에만 터진다. 로컬에서 절대 재현이 안 된다.
- **장애 지점이 하나 늘었다**: 캐시는 빼는 게 아니라 더하는 계층이다. DB가 아프던 시스템에 "캐시도 아플 수 있는" 시스템이 됐다.
</Callout>

### 미니: 쿼리 캐시의 함정

"애플리케이션에서 캐시 다루기 귀찮은데, DB가 알아서 쿼리 결과를 캐시해주면 안 되나?" 싶을 수 있다. 실제로 옛날 MySQL에는 **query cache**라는 기능이 있었다. 같은 SELECT가 또 들어오면 결과를 그대로 돌려주는, 말 그대로 DB 내장 캐시였다. 좋아 보인다. 그런데 MySQL 8.0에서 이 기능은 통째로 제거됐다. 이유가 정확히 이 글의 주제다.

문제는 무효화다. 쿼리 캐시는 **어떤 테이블이 한 번이라도 쓰이면, 그 테이블을 건드린 모든 캐시 엔트리를 싹 날린다.** 정합성을 지키려면 그럴 수밖에 없다. 그래서 읽기가 압도적으로 많고 쓰기가 거의 없는 워크로드에선 이득이지만, **쓰기가 조금이라도 섞이면 무효화 비용이 캐시 이득을 그대로 잡아먹는다.**

```sql
-- 이 SELECT 결과가 쿼리 캐시에 들어간다
SELECT COUNT(*) FROM orders WHERE status = 'paid';

-- 그런데 누가 주문 하나만 넣어도...
INSERT INTO orders (...) VALUES (...);
-- → orders 테이블이 걸린 캐시 엔트리 전부 무효화. 다음 SELECT는 다시 풀계산.
```

게다가 더 고약한 건, 캐시에 넣고 빼는 과정 자체가 **전역 락(global mutex)을 잡는다는 점이다.** 동시 접속이 많은 환경에선 이 락이 경합 지점이 돼서, "캐시가 있으니 빠르겠지" 했던 게 오히려 모두를 줄 세우는 병목이 됐다. 캐시가 성능을 깎아먹는, 이 안티패턴의 교과서적 사례다. 캐시는 무효화 비용을 안 따지면 어디에 두든 똑같이 배신한다.

## 정석: 무엇을·언제를 먼저 정한다

해법은 단순하다. 캐시를 켜는 스위치가 아니라 **두 개의 결정**으로 다루는 것이다. 캐시 붙이기 전에 이걸 먼저 적어라.

**1. 무엇을 캐시할까 — 대상 선정.** 아무거나 캐시하지 않는다. 좋은 캐시 대상은 세 조건을 만족한다.

- **읽기가 많다** — 같은 값을 자주 읽어야 캐시가 일을 한다. 한 번 읽고 마는 건 캐시할 이유가 없다.
- **변경이 적다** — 자주 바뀌면 무효화 비용이 이득을 먹는다. 쿼리 캐시가 죽은 그 이유.
- **stale이 허용된다** — 잠깐 옛날 값이어도 괜찮은 데이터부터. 환율 표시는 1분 늦어도 되지만, 계좌 잔액은 안 된다. **이 데이터가 몇 초 틀려도 되는가**를 먼저 답하라. 답이 "0초"면 그건 캐시 대상이 아니다.

**2. 언제 무효화할까 — 정합성 전략.** TTL "만"으로 때우지 않는다. 데이터 성격에 맞게 고른다.

- **write-through / write-on-update** — 데이터를 쓸 때 캐시도 같이 갱신하거나 지운다. 원본이 바뀌는 그 자리에서 캐시를 처리하니 stale 창이 거의 없다. 가장 기본.
- **이벤트 기반 무효화** — 쓰기 주체와 캐시가 멀리 떨어져 있으면, 변경 이벤트(메시지 큐, CDC 등)를 받아서 해당 키를 무효화한다. 여러 서비스가 같은 데이터를 캐시할 때 유용.
- **버전 키 / 키 네임스페이스** — 키에 버전을 박는다(`product:42:v7`). 데이터가 바뀌면 버전을 올려 새 키로 쓰니, 옛 키는 알아서 안 읽히고 TTL로 정리된다. 일괄 무효화가 쉽다.

**3. 스탬피드 대비 — 동시 만료를 흩는다.** 무효화를 정했어도 만료가 겹치면 터진다. 셋 중 하나는 깔아라.

- **TTL 지터(jitter)** — TTL에 랜덤을 더한다(`60s ± 10s`). 같은 시각에 들어간 키들이 같은 시각에 만료되지 않게 흩뿌리는 것. 한 줄로 스탬피드의 절반을 막는다.
- **single-flight / 락** — 같은 키에 미스가 동시에 나면 **딱 한 요청만** 원본을 조회하게 하고, 나머지는 그 결과를 기다린다. 수천 개가 DB로 직행하는 걸 한 개로 줄인다.
- **사전 워밍 / 백그라운드 갱신** — 만료되기 전에 미리 새 값으로 채워둔다. 사용자 요청이 만료 순간을 맞닥뜨리지 않게, 만료를 백그라운드 작업이 대신 맞는 것.

<Callout type="info" title="TTL은 안전장치지 전략이 아니다">
오해하지 말자. TTL이 나쁜 게 아니다. TTL은 "혹시 무효화를 빠뜨려도 영원히 stale하진 않게" 막아주는 **최후의 안전장치**로 훌륭하다. 문제는 TTL을 **유일한 무효화 전략**으로 쓰는 것이다. "60초 TTL"은 "60초간은 틀려도 된다"는 선언이지 정합성 전략이 아니다. write-through로 즉시 무효화하고, TTL은 그 뒤를 받치는 그물로 두는 게 정석이다.
</Callout>

## 이렇게 하면 됨

처음의 그 "맹목적 Redis" 코드를, 세 결정을 명시한 형태로 다시 써보자.

**Before — 대상도 무효화도 스탬피드도 없는 캐시.**

```typescript
// 무엇을 캐시하는지 따진 적 없고, 무효화 코드는 없고, 동시 만료는 무방비
async function getProduct(id: string) {
  const cached = await cache.get(`product:${id}`);
  if (cached) return JSON.parse(cached);

  const product = await db.query("SELECT * FROM products WHERE id = $1", [id]);
  await cache.set(`product:${id}`, JSON.stringify(product), { EX: 60 });
  return product;
}
```

**After — 대상·무효화·스탬피드 대비를 모두 명시.** 캐시 대상은 "읽기 많고, 변경 적고, 잠깐 stale 허용되는" 상품 정보로 한정한다(잔액·재고 같은 건 여기 안 넣는다). 만료는 지터로 흩고, 미스가 겹치면 single-flight로 한 번만 조회하고, 쓰기 시점에 즉시 무효화한다.

```typescript
// 1) 읽기: single-flight + TTL 지터로 스탬피드 방어
const inflight = new Map<string, Promise<Product>>();

async function getProduct(id: string): Promise<Product> {
  const key = `product:${id}`;

  const cached = await cache.get(key);
  if (cached) return JSON.parse(cached);

  // 같은 키 미스가 동시에 나도, 원본 조회는 딱 한 번만
  if (inflight.has(key)) return inflight.get(key)!;

  const promise = (async () => {
    const product = await db.query("SELECT * FROM products WHERE id = $1", [id]);
    const ttl = 60 + Math.floor(Math.random() * 20); // 60~80s 지터: 동시 만료 방지
    await cache.set(key, JSON.stringify(product), { EX: ttl });
    return product;
  })().finally(() => inflight.delete(key));

  inflight.set(key, promise);
  return promise;
}

// 2) 쓰기: 데이터가 바뀌는 그 자리에서 캐시를 즉시 무효화 (TTL에 의존 안 함)
async function updateProduct(id: string, patch: Partial<Product>) {
  await db.query("UPDATE products SET ... WHERE id = $1", [id]);
  await cache.del(`product:${id}`); // write-through invalidation
}
```

차이는 코드 줄 수가 아니라 **결정의 유무**다. Before는 "느리니까 캐시"라는 반사 신경이고, After는 "이 데이터는 이래서 캐시하고, 이렇게 무효화하고, 이렇게 동시 만료를 막는다"는 세 개의 의식적 결정이다. 같은 Redis인데 부하가 와도 신기루가 안 된다.

그리고 사실 그 전에 한 단계가 더 있다. **캐시는 마지막에, 측정 후에 붙인다.** [6편 성급한 비정규화](/docs/dev/database/anti-patterns/6.premature-denormalization)에서 말한 원칙과 정확히 같다. 진짜 병목이 어딘지 재보지도 않고 캐시부터 까는 건, 측정 없이 최적화하는 것과 똑같은 실수다. 인덱스 하나로 풀릴 쿼리를, 캐시 계층이라는 새 장애 지점으로 덮는 경우가 생각보다 많다. **캐시는 더 싼 해법이 없을 때 꺼내는 카드지, 느리면 일단 누르는 버튼이 아니다.**

<Callout type="note" title="그럼 캐시 자체가 나쁜가?">
아니다. 잘 설계한 캐시는 정말 강력하다. 읽기 많고 변경 적은 데이터를 원본 앞에서 막아주면, DB 부하가 한 자릿수 배로 줄기도 한다. 안티패턴은 "캐시" 자체가 아니라 **"무엇을·언제를 안 정한 캐시"**다. 무엇을 캐시할지, 바뀌면 언제 버릴지, 동시에 만료되면 어떻게 막을지 — 이 셋을 적고 시작하면, 캐시는 신기루가 아니라 진짜 성능이 된다.
</Callout>

## 정리

캐시는 매력적이다. 코드 몇 줄로 그래프가 쭉 내려가니까. 하지만 그 그래프가 내려간 건 부하가 사라져서가 아니라, **부하를 잠깐 미뤄둔 것**일 때가 많다. 미뤄둔 부하는 인기 키가 동시에 만료되는 그 한순간에, 이자까지 붙여서 한꺼번에 돌아온다.

> **캐시의 어려움은 저장이 아니라 무효화다. 그걸 설계 안 하면 캐시가 아니라 버그다.**

"느리네? 캐시 붙이자"는 생각이 들면, 그 자리에서 두 개만 먼저 답해라. **무엇을 캐시할 건가**(읽기 많고, 변경 적고, stale 허용되나?), 그리고 **언제 버릴 건가**(쓰기 시점에 어떻게 무효화하나, 동시 만료는 어떻게 막나?). 이 둘에 답이 없으면, 그건 캐시를 붙일 때가 아니라 측정부터 다시 할 때다.
