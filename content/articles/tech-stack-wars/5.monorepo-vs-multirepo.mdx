# 모노레포 vs 멀티레포 — 저장소 전쟁

*"Google은 하나의 저장소에 수십억 줄의 코드를 넣는다. 너네 회사가 Google은 아닌데?"*

---

회사에서 새 프로젝트를 시작할 때 가장 먼저 나오는 논쟁 중 하나:
"저장소를 하나로 합칠까, 따로 만들까?"

이 질문은 단순한 Git 구조 문제가 아님.
팀의 문화, 배포 전략, CI/CD 파이프라인, 코드 소유권, 의존성 관리...
모든 것에 영향을 미치는 아키텍처 결정임.

<Callout type="info" title="용어 정리">
**모노레포 (Monorepo)**: 여러 프로젝트/패키지를 하나의 Git 저장소에서 관리
**멀티레포 (Multi-repo/Polyrepo)**: 프로젝트/패키지마다 별도의 Git 저장소

주의: 모노레포 ≠ 모놀리스
모노레포는 저장소 구조이고, 모놀리스는 아키텍처 패턴임.
모노레포 안에 마이크로서비스를 넣을 수도 있음.
</Callout>

## 빅테크의 선택

### Google: 모노레포의 전설

```
Google의 모노레포 규모 (2015년 기준, 지금은 더 큼):
- 코드 라인 수: 20억 줄+
- 파일 수: 10억 개+
- 커밋 수: 하루 45,000건+
- 개발자 수: 25,000명+
- 저장소 크기: 86TB+

이걸 관리하기 위해 Google은 자체 도구를 만들었음:
- Piper: 자체 버전 관리 시스템 (Git 아님!)
- Blaze/Bazel: 빌드 시스템
- CitC: 클라우드 IDE
- Critique: 코드 리뷰 도구

Git으로는 이 규모를 감당할 수 없어서 아예 새로 만든 것임.
```

### Facebook/Meta: 모노레포 + 자체 도구

```
Meta의 선택:
- Mercurial (hg) 기반 모노레포
- Git이 대규모에 약하다고 판단해서 Mercurial을 선택
- 그마저도 성능이 안 돼서 자체 확장 (EdenSCM → Sapling)
- Buck: 자체 빌드 시스템 (Bazel과 유사)

2023년에 Sapling을 오픈소스로 공개함.
"Git보다 빠르고 사용하기 쉬운 VCS"를 목표로.
```

### Netflix, Amazon: 멀티레포의 대표

```
Netflix의 선택:
- 수천 개의 마이크로서비스 = 수천 개의 저장소
- 각 팀이 자기 서비스의 저장소를 완전히 소유
- 독립적 배포, 독립적 기술 스택 선택

Amazon의 유명한 원칙:
"Two-Pizza Team" — 피자 두 판으로 먹일 수 있는 크기의 팀이 하나의 서비스를 소유
→ 자연스럽게 서비스별 저장소 = 멀티레포
```

<Callout type="warning" title="빅테크 따라하기의 함정">
"Google이 모노레포 쓰니까 우리도 모노레포 쓰자" — 위험한 논리임.

Google은 모노레포를 운영하기 위해:
- 전용 버전 관리 시스템을 만들었음
- 전용 빌드 시스템을 만들었음
- 전용 IDE를 만들었음
- 전용 코드 리뷰 도구를 만들었음
- 전담 인프라 팀이 수백 명임

당신의 회사에 이런 리소스가 있는가?
없으면 Google의 사례는 참고만 하고, 자신의 상황에 맞게 판단해야 함.
</Callout>

### 2. 원자적 변경 (Atomic Changes)

```typescript
// 모노레포의 가장 강력한 장점:
// API + 프론트엔드 + 타입을 한 번의 커밋으로 변경 가능

// 커밋: "feat: add user avatar upload"
// 이 커밋에 포함된 변경:

// 1. shared-types: 타입 추가
// packages/shared-types/src/user.ts
interface User {
  id: string;
  name: string;
  avatar?: string;  // ← 추가
}

// 2. API: 엔드포인트 추가
// apps/api/src/routes/users.ts
router.post('/users/:id/avatar', uploadMiddleware, async (req, res) => {
  const url = await uploadToS3(req.file);
  await db.user.update({ where: { id: req.params.id }, data: { avatar: url } });
  res.json({ avatar: url });
});

// 3. Web: UI 추가
// apps/web/src/components/UserProfile.tsx
function UserProfile({ user }: { user: User }) {
  return (
    <div>
      <AvatarUpload userId={user.id} currentAvatar={user.avatar} />
      <h1>{user.name}</h1>
    </div>
  );
}

// 멀티레포라면?
// 1. shared-types 저장소에서 커밋 → npm publish
// 2. api 저장소에서 shared-types 업데이트 → 커밋
// 3. web 저장소에서 shared-types 업데이트 → 커밋
// 3번의 PR, 3번의 리뷰, 3번의 배포
// 순서가 잘못되면 프로덕션 에러
```

<Callout type="success" title="원자적 변경의 가치">
API 스키마를 변경했는데 프론트엔드를 안 고치면?
프론트엔드를 고쳤는데 API가 아직 이전 버전이면?
타입을 바꿨는데 npm publish를 깜빡하면?

모노레포에서는 이런 문제가 구조적으로 발생하지 않음.
하나의 커밋에 모든 변경이 포함되니까.
이건 진짜 큰 장점임.
</Callout>

## 모노레포의 단점

### 1. 느린 CI/CD

```yaml
# 모노레포의 CI가 느려지는 이유:
# 하나의 파일을 변경해도 전체 저장소를 체크아웃해야 함

# ❌ 순진한 CI 설정
name: CI
on: push
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4  # 전체 저장소 체크아웃
      - run: npm install            # 모든 패키지 의존성 설치
      - run: npm run build           # 모든 패키지 빌드
      - run: npm test                # 모든 패키지 테스트

# 10개 패키지 중 1개만 변경해도 10개 전부 빌드+테스트
# CI 시간: 30분+ (의존성 설치만 10분)
```

```yaml
# ✅ 영향 받는 패키지만 빌드 (Turborepo 사용)
name: CI
on: push
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 변경 감지를 위해 전체 히스토리 필요
      - uses: actions/cache@v4
        with:
          path: node_modules/.cache/turbo
          key: turbo-${{ hashFiles('**/turbo.json') }}
      - run: npm install
      - run: npx turbo run build test --filter='...[HEAD^1]'
      # 변경된 패키지와 그에 의존하는 패키지만 빌드+테스트
```

### 2. 저장소 크기 폭발

```bash
# 모노레포가 커지면 발생하는 문제들

# 1. git clone이 느려짐
$ time git clone https://github.com/company/monorepo.git
# Cloning into 'monorepo'...
# 시간: 5분... 10분... 15분...

# 해결: Shallow clone
$ git clone --depth 1 https://github.com/company/monorepo.git
# 최신 커밋만 가져옴. 빠름.

# 2. git status가 느려짐
$ time git status
# 파일 수만 개를 스캔하느라 느려짐

# 해결: fsmonitor 사용
$ git config core.fsmonitor true
# 파일 시스템 이벤트를 모니터링해서 변경된 파일만 체크

# 3. IDE가 느려짐
# VS Code에서 파일 10만 개를 인덱싱하면...
# "Extension Host is unresponsive" 메시지가 반겨줌
```

<Callout type="error" title="실화: 모노레포 크기 문제">
실제로 겪은 사례:
- 모노레포에 5년치 커밋이 쌓여서 .git 폴더가 15GB
- CI/CD에서 clone만 20분 걸림
- 새 팀원 온보딩 첫날: "git clone이 안 끝나요..."
- git gc --aggressive 돌리면 CPU 100%로 30분

결국 git-filter-repo로 대용량 파일 히스토리를 정리하고,
shallow clone + sparse checkout을 도입해야 했음.
이런 관리 비용을 미리 알았으면 좋았을 것.
</Callout>

## 멀티레포의 단점

### 1. 의존성 지옥

```
# "Diamond Dependency Problem"
# 다이아몬드 의존성 문제

service-a → shared-lib@1.0.0
service-b → shared-lib@2.0.0
service-c → service-a + service-b
            → shared-lib@1.0.0? @2.0.0? 둘 다?

# 버전 충돌!
# shared-lib의 breaking change가 있으면
# 모든 의존 서비스를 업데이트해야 함
# 하지만 누가 먼저? 순서가 중요함.
```

```bash
# 멀티레포에서 공통 라이브러리 업데이트하기
# (shared-types v1 → v2 breaking change)

# Step 1: shared-types 저장소에서 v2 릴리스
cd shared-types && npm version major && npm publish

# Step 2: api 저장소 업데이트
cd api && npm install @company/shared-types@2 && npm run test
# 테스트 실패! 타입이 바뀌었으니까. 코드 수정 필요.
# PR → 리뷰 → 머지 → 배포

# Step 3: web 저장소 업데이트
cd web && npm install @company/shared-types@2 && npm run test
# 여기도 테스트 실패. 코드 수정.
# PR → 리뷰 → 머지 → 배포

# Step 4: admin 저장소 업데이트
cd admin && npm install @company/shared-types@2 && npm run test
# PR → 리뷰 → 머지 → 배포

# Step 5: mobile 저장소 업데이트
# ...

# 이 과정이 며칠에서 몇 주 걸릴 수 있음
# 그 사이에 v1을 쓰는 서비스와 v2를 쓰는 서비스가 공존
# 호환성 문제 발생 가능
```

<Callout type="warning" title="멀티레포의 숨겨진 비용">
멀티레포에서 가장 고통스러운 시나리오:

1. 공통 라이브러리에 보안 취약점 발견
2. 패치 버전 릴리스
3. 이 라이브러리를 사용하는 저장소: 47개
4. 47개 저장소 전부 업데이트 + 테스트 + 배포
5. Dependabot이 47개의 PR을 자동 생성하긴 하는데...
6. 각 PR을 누가 리뷰하고 머지하지?
7. 테스트가 깨지는 저장소는?
8. 3일 후에도 15개 저장소가 업데이트 안 됨

모노레포였다면? 1번의 PR로 끝.
</Callout>

## 모노레포 도구 비교 (2025년)

### Turborepo

```json
// turbo.json — Turborepo 설정
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "test": {
      "dependsOn": ["build"]
    },
    "lint": {},
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

```bash
# Turborepo의 핵심 기능: 빌드 캐싱 + 병렬 실행

# 변경된 패키지만 빌드 (의존 패키지 포함)
npx turbo run build --filter='...[HEAD^1]'

# 원격 캐싱 — 팀원이 이미 빌드했으면 캐시에서 가져옴
npx turbo run build --team=my-team --token=xxxx
# ">>> FULL TURBO" — 캐시 히트! 0초 빌드!

# 의존 그래프 시각화
npx turbo run build --graph
```

<Callout type="success" title="Turborepo의 위상">
Vercel이 인수한 후 Next.js 생태계와 긴밀하게 통합됨.
- 설정이 간단 (turbo.json 하나)
- 원격 캐싱으로 CI 속도 획기적 개선
- Vercel 배포와 자연스러운 통합

JavaScript/TypeScript 모노레포의 사실상 표준.
</Callout>

### Nx

```json
// nx.json — Nx 설정
{
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "cache": true
    },
    "test": {
      "cache": true
    }
  },
  "namedInputs": {
    "default": ["{projectRoot}/**/*"],
    "production": ["default", "!{projectRoot}/**/*.spec.ts"]
  }
}
```

```bash
# Nx의 강점: 영향 분석 (Affected)
# 변경사항이 영향을 미치는 프로젝트만 빌드/테스트

npx nx affected --target=build
npx nx affected --target=test

# 프로젝트 그래프 시각화 (웹 UI)
npx nx graph
# 브라우저에서 프로젝트 간 의존 관계를 시각적으로 확인

# 코드 생성
npx nx generate @nx/react:library shared-ui
npx nx generate @nx/node:application api-server
```

### Turborepo vs Nx 비교

```
Turborepo:
✅ 설정 간단 (미니멀리즘)
✅ Vercel 생태계 통합
✅ package.json 기반 (기존 프로젝트 마이그레이션 쉬움)
❌ 기능이 Nx보다 적음
❌ 코드 생성 도구 없음

Nx:
✅ 기능 풍부 (코드 생성, 그래프 UI, 플러그인)
✅ 언어 무관 (JS 외에도 Go, Java, .NET 지원)
✅ 엔터프라이즈 기능
❌ 설정이 복잡 (학습 곡선)
❌ nx.json + project.json + workspace.json...
```

## 선택 가이드

<Callout type="success" title="모노레포를 선택해야 할 때">
- 같은 언어/프레임워크를 쓰는 프로젝트들
- 프론트엔드 + 백엔드가 같은 TypeScript
- 코드 공유가 빈번한 경우
- 원자적 변경이 중요한 경우
- 팀 규모가 작~중간 (50명 이하)
- 스타트업 / 빠르게 움직여야 할 때
</Callout>

<Callout type="info" title="멀티레포를 선택해야 할 때">
- 서비스별 기술 스택이 다른 경우 (Node.js + Python + Go)
- 팀 간 독립성이 중요한 대규모 조직
- 서비스별 배포 주기가 매우 다른 경우
- 보안/접근 권한을 서비스별로 분리해야 할 때
- 오픈소스 라이브러리를 관리할 때
</Callout>

```
결론:

10명 이하 팀, 같은 기술 스택 → 모노레포 (무조건)
10~50명, 혼합 기술 스택 → 상황에 따라
50명+, 마이크로서비스 → 멀티레포가 자연스러움
                       (하지만 관련 서비스끼리 미니 모노레포도 가능)

가장 안 좋은 선택:
"일단 멀티레포로 시작했다가 나중에 모노레포로 합치자" ← 마이그레이션 지옥
"Google처럼 모노레포 하자" ← Google이 아닌데

가장 좋은 선택:
현재 팀 규모와 상황에 맞는 걸 선택하고,
문제가 생기면 그때 전환을 고려하자.
```
