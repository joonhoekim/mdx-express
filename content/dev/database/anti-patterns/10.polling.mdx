## 이게 뭔데

폴링. 한 줄로 정의하면, **변경이 생겼는지 알고 싶어서 일정 주기로 같은 `SELECT`를 계속 던지는 것**이다. 데이터가 바뀌었나 궁금한데 DB가 먼저 알려주는 방법을 모르니까, 우리가 직접 5초마다 한 번씩 가서 "바뀌었어?" 하고 물어본다.

비유를 하나 들어보자. 택배를 시켜놓고 도착이 궁금한 사람이 있다. 정상적인 사람은 도착 알림을 켜놓고 자기 할 일을 한다. 알림이 오면 그때 나가서 받는다. 근데 폴링하는 사람은 이렇게 한다. 1분마다 현관문을 열어서 "왔나?" 확인한다. 안 왔다. 다시 닫는다. 1분 뒤 또 연다. 안 왔다. 또 닫는다. 택배가 3시간 뒤에 오면, 그 사이에 현관문을 180번 여닫는다. 그중 179번은 헛걸음이다.

DB한테 이 짓을 하는 게 폴링이다. "끝났나? 아직. 끝났나? 아직. 끝났나? 아직." 변경은 어쩌다 한 번 일어나는데, 우리는 변경 여부를 묻는 질문을 멈추지 않는다. 답의 대부분은 "아니, 아직"이다. 그 "아직"을 듣자고 DB는 매번 커넥션을 잡고, 쿼리를 파싱하고, 인덱스를 타고, 결과를 돌려준다.

<Callout type="warning" title="한 줄 요약">
폴링은 변경이 없어도 계속 묻는다. 답의 99%가 "아직"인 질문을, 사용자 수만큼 곱해서 5초마다 던지는 구조다. 묻는 비용은 변경 빈도가 아니라 폴링 주기와 사용자 수에 비례한다.
</Callout>

## 시나리오: 이런 적 있을 거임

결제 상태가 바뀌면 화면을 갱신해야 한다. 푸시는 복잡해 보인다. WebSocket 붙이고, 서버에서 이벤트 쏘고... 생각만 해도 머리 아프다. "그냥 5초마다 상태 조회하면 되지 뭐."

프론트에 이렇게 친다.

```javascript
// 결제 페이지: 5초마다 상태를 다시 물어본다
const timer = setInterval(async () => {
  const res = await fetch(`/api/payments/${paymentId}/status`);
  const { status } = await res.json();

  if (status !== "PENDING") {
    clearInterval(timer);
    renderResult(status);   // 완료/실패 화면으로 전환
  }
}, 5000);
```

그리고 서버는 이걸 받아 DB를 친다.

```sql
SELECT status FROM payments WHERE id = :id;
```

동작한다. 5분 만에 끝. 결제하고 기다리면 5초 안에 화면이 바뀐다. PR에 "LGTM 👍". 배포. 아무 문제 없어 보인다.

그런데 서비스가 큰다. 결제 페이지를 동시에 열고 있는 사용자가 1만 명이 됐다. 이제 무슨 일이 벌어지냐면, **5초마다 1만 번의 `SELECT`가 DB를 두드린다.** 초당 2천 쿼리. 그중 대부분은 결제 승인을 기다리는 중이라 답은 똑같다. **99%가 "아직 PENDING"**이다. 의미 있는 변경 하나를 잡자고, 의미 없는 "아직" 99개를 매번 같이 긁어온다.

게다가 응답 지연도 어정쩡하다. 결제 상태가 실제로 바뀐 직후 0.1초 만에 승인이 떨어졌어도, 다음 폴링 차례가 올 때까지 사용자는 그걸 모른다. 운 나쁘면 5초를 멀뚱히 기다린다. "결제됐는데 화면이 왜 안 바뀌어요?" 문의가 들어온다.

그래서 주기를 줄인다. 5초 → 1초. 그럼 지연은 1/5로 줄지만, **부하는 5배가 된다.** 초당 1만 쿼리. 0.1초로 줄이면 지연은 더 짧아지지만 부하는 또 10배. 줄이면 줄일수록 빨라지지만 그만큼 DB가 비명을 지른다. **부하와 지연을 맞바꾸는 시소**에 올라탄 거다. 어느 쪽으로 기울여도 반대쪽이 손해다. 이 시소에서는 절대로 둘 다 좋아질 수 없다.

DB를 괴롭힌 건 "결제 완료를 알려달라"는 요구 자체가 아니다. 그 요구를 가장 쉬운 길로 처리하려던 습관이다.

- "푸시는 복잡하니까 폴링이 쉽지" — 당장의 구현 **편의**로 영구적인 부하를 떠안았다. 5분 vs 반나절의 차이가, 운영 내내 따라붙는 비용으로 돌아온다.
- "5초마다 한 번인데 별거 있겠어" — **사용자 수를 곱하면** 폭증한다는 걸 간과했다. 1명이면 별거 아닌데, 1만 명이면 초당 2천이다. 폴링 비용은 항상 동시 접속자 수만큼 곱해진다.
- "주기 줄이면 빨라지겠지" — **부하-지연 트레이드오프**의 본질을 못 봤다. 주기를 만지는 건 시소 위에서 위치를 옮기는 것뿐, 시소 자체를 내려오는 게 아니다.
- "어차피 그 쿼리 빠르잖아, 0.1ms밖에 안 걸려" — 느린 건 쿼리 하나가 아니라 **쿼리가 너무 많다는 것**이다. 개별 쿼리가 빨라도 초당 2천 번이면 커넥션과 CPU가 갈린다.

이 중 하나라도 마음 한구석에 있었다면, 당신은 어딘가에 `setInterval` 폴링을 한 번쯤 심어봤을 가능성이 높다.

## 무슨 일이 벌어지나 (비용)

폴링이 왜 비싼지, 비용을 세 갈래로 뜯어보자.

**1. 변경 없는 조회가 대부분 낭비다.** 폴링의 목적은 "변경 감지"인데, 정작 던지는 쿼리의 절대다수는 변경이 없다는 걸 확인하는 데 쓰인다. 결제가 보통 수십 초 안에 끝난다면, 5초 폴링은 한 결제당 평균 대여섯 번 묻고 그중 마지막 한 번만 유의미하다. 나머지는 전부 "아직"을 받아오는 헛수고다. 그 헛수고가 커넥션을 잡고, 쿼리 플랜을 타고, 결과를 직렬화해서 네트워크로 돌려보낸다. 변경 빈도는 낮은데 조회 빈도는 고정이라, 둘 사이의 간극이 통째로 낭비가 된다.

**2. 주기와 지연이 묶여 있어서 벗어날 수가 없다.** 폴링에서는 "얼마나 빨리 아느냐(지연)"와 "얼마나 자주 묻느냐(부하)"가 한 손잡이로 묶여 있다. 손잡이를 한쪽으로 돌리면 다른 쪽이 반드시 따라온다. 지연을 줄이려면 부하를 키워야 하고, 부하를 줄이려면 지연을 감수해야 한다. 이건 튜닝으로 해결되는 문제가 아니다. **구조가 그렇게 생긴 것**이다. 그래서 폴링 시스템은 늘 "조금 느리거나, 조금 비싸거나" 둘 중 하나에 머문다.

**3. 사용자 수에 비례해 선형으로 폭증한다.** 가장 무서운 부분이다. 폴링 부하는 변경 이벤트 수와 무관하다. **동시 폴링 클라이언트 수 × (1 / 주기)**로 결정된다. 결제가 실제로 일어나든 말든, 페이지를 열어둔 사람이 1만 명이면 5초마다 1만 쿼리가 나간다. 화면을 열어두고 딴짓하는 사람, 탭만 켜둔 사람까지 전부 폴링한다. 사용자가 10배 늘면 폴링 부하도 정확히 10배다. 트래픽이 몰리는 바로 그 순간, 폴링이 DB 부하를 가장 크게 끌어올린다.

<Callout type="error" title="뭐가 문제냐면">
- **느려지는데 슬로우 쿼리 로그엔 안 걸린다**: 쿼리 하나는 0.1ms로 빠르다. 문제는 "느린 쿼리"가 아니라 "쿼리가 너무 많다"는 거라, 평균 응답 시간만 보면 멀쩡해 보인다.
- **DB 부하가 비즈니스 활동과 무관하다**: 결제가 한 건도 안 일어나도, 페이지만 열려 있으면 폴링은 계속 돈다. 부하 그래프가 실제 트래잭션 수가 아니라 "열려 있는 탭 수"를 따라간다.
- **튜닝의 끝이 없다**: 주기를 만질수록 부하-지연 시소만 오갈 뿐, 근본 비용은 그대로다. 사용자가 늘면 결국 다시 터진다.
</Callout>

## 정석: 묻지 말고 알림받기 (푸시)

해법의 핵심은 질문을 뒤집는 것이다. **"바뀌었어?"를 반복하지 말고, "바뀌면 알려줘"로 바꾼다.** 우리가 가는 게 아니라, 변경이 일어났을 때 그쪽에서 우리에게 온다. 그러면 묻는 비용 자체가 사라진다. 변경이 없으면 아무 일도 안 일어나고, 변경이 있을 때 딱 그만큼만 알림이 온다. 부하가 **폴링 주기**가 아니라 **실제 변경 빈도**에 비례하게 바뀌는 거다.

서버~DB 구간에서 쓸 수 있는 푸시 방식은 대표적으로 셋이다.

**1. Postgres `LISTEN/NOTIFY`** — DB 자체가 메시지 채널을 들고 있다. 한쪽은 채널을 구독(`LISTEN`)하고, 변경을 일으킨 쪽은 같은 채널로 신호를 쏜다(`NOTIFY`). 보통 트리거나 애플리케이션 코드에서 상태가 바뀌는 순간 `NOTIFY`를 호출한다.

```sql
-- 구독자 쪽: 이 채널을 듣겠다
LISTEN payment_status;

-- 변경을 일으킨 쪽: 상태가 바뀌면 알린다 (트리거 함수 안 등)
SELECT pg_notify('payment_status', json_build_object(
  'payment_id', NEW.id,
  'status',     NEW.status
)::text);
```

구독자는 그냥 커넥션 하나를 열어두고 대기하다가, 알림이 올 때만 깨어나 해당 결제를 처리한다. 5초마다 1만 쿼리를 던지던 게, "결제 상태가 실제로 바뀐 횟수"만큼의 알림으로 줄어든다.

**2. 메시지 큐 (이벤트 발행)** — 상태가 바뀐 쪽이 큐(Kafka, RabbitMQ, SQS 등)에 이벤트를 발행하고, 관심 있는 컨슈머가 구독한다. DB에 직접 묻는 대신 변경 스트림을 받는 구조라, 여러 소비자가 같은 이벤트를 나눠 받기에도 좋고, 일시적으로 컨슈머가 죽어도 큐가 메시지를 들고 기다려준다.

**3. CDC (Change Data Capture, WAL 구독)** — 애플리케이션이 별도로 이벤트를 쏘는 것조차 생략하고, DB의 트랜잭션 로그(Postgres라면 WAL)를 직접 구독한다. Debezium 같은 도구가 로그를 읽어 "어떤 행이 어떻게 바뀌었는지"를 스트림으로 뽑아준다. 코드에 알림 호출을 흩뿌리지 않아도, 커밋된 변경이 자동으로 흘러나온다는 게 강점이다.

그리고 마지막 한 단계, **클라이언트(브라우저)까지**는 WebSocket이나 SSE(Server-Sent Events)로 전달한다. 서버가 푸시로 변경을 받았으면, 그걸 폴링하던 브라우저에게도 폴링 없이 내려보낼 수 있다. 프론트의 `setInterval`도 같이 사라진다.

```javascript
// Before가 사라진 자리: 서버가 보내줄 때만 받는다
const es = new EventSource(`/api/payments/${paymentId}/stream`);
es.onmessage = (e) => {
  const { status } = JSON.parse(e.data);
  if (status !== "PENDING") {
    es.close();
    renderResult(status);
  }
};
```

### 미니: 커밋 전 큐 적재 레이스 → 아웃박스

푸시로 갈아타다 보면 거의 반드시 한 번은 밟는 함정이 있다. **트랜잭션 커밋 전에 큐에 작업을 넣는 레이스**다.

흔한 코드는 이렇게 생겼다. 주문을 저장하는 트랜잭션 안에서, 저장 직후 큐에 "이 주문 처리해" 메시지를 발행한다.

```typescript
// 위험한 코드: 커밋 전에 큐에 넣는다
await db.transaction(async (tx) => {
  const order = await tx.orders.insert({ ... });   // 아직 커밋 안 됨
  await queue.publish("order.created", { id: order.id });  // 큐에는 벌써 나감
  // ...트랜잭션이 여기서 끝나며 커밋
});
```

문제는 타이밍이다. `queue.publish`가 트랜잭션 커밋보다 **먼저** 큐에 메시지를 꽂는다. 큐는 빠르다. 워커가 거의 즉시 깨어나서 "id로 주문 좀 줘" 하고 DB를 조회한다. 그런데 이 주문은 아직 **커밋 전**이라, 워커가 보는 커넥션에서는 보이지 않는다. 워커는 빈손으로 "그런 주문 없는데?" 하고 실패한다. 정작 0.01초 뒤엔 커밋이 끝나서 멀쩡히 존재하는데도 말이다.

게다가 반대 방향 사고도 있다. 큐에 넣은 뒤 커밋이 **롤백**되면? 주문은 없는데 큐에는 "이 주문 처리해" 메시지가 남아 유령 작업이 돈다. DB에 없는 행을 처리하라는 명령만 떠도는 거다.

해결책이 **트랜잭셔널 아웃박스(transactional outbox)**다. 핵심 아이디어는 단순하다. **큐에 직접 쏘지 말고, 같은 트랜잭션 안에서 `outbox` 테이블에 "발행할 메시지"를 한 행으로 같이 INSERT한다.** 그러면 주문 행과 아웃박스 행이 한 커밋에 묶인다. 커밋되면 둘 다 있고, 롤백되면 둘 다 없다. 원자성이 보장된다.

```typescript
// 안전한 코드: 큐 대신 outbox 테이블에 같이 쓴다
await db.transaction(async (tx) => {
  const order = await tx.orders.insert({ ... });
  await tx.outbox.insert({
    topic:   "order.created",
    payload: { id: order.id },
    status:  "PENDING",
  });
  // 주문과 아웃박스가 한 커밋에 묶여 같이 살고 같이 죽는다
});
```

```sql
CREATE TABLE outbox (
  id         BIGSERIAL PRIMARY KEY,
  topic      TEXT        NOT NULL,
  payload    JSONB       NOT NULL,
  status     TEXT        NOT NULL DEFAULT 'PENDING',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

그리고 별도의 **릴레이(relay)**가 커밋된 아웃박스 행을 읽어 진짜 큐에 발행한다. 릴레이는 커밋이 끝난 뒤에만 행을 보게 되므로, 워커가 "아직 안 보이는 행"을 조회하는 레이스가 원천적으로 사라진다. 릴레이가 발행에 성공하면 해당 행을 `SENT`로 마킹한다.

여기서 헷갈리기 쉬운 지점 하나. "릴레이가 아웃박스를 읽는 건 결국 폴링 아니냐?" 맞다. 릴레이가 아웃박스를 주기적으로 긁는다면 그건 폴링이다. 하지만 차이가 있다. (1) 폴링 대상이 **자기 DB의 한 작은 테이블**로 한정돼 범위가 좁고, (2) 사용자 수가 아니라 **릴레이 인스턴스 수**(보통 한 줌)에 부하가 묶이며, (3) 릴레이의 폴링조차 CDC로 아웃박스 테이블의 WAL을 구독하면 푸시로 바꿀 수 있다. 즉 아웃박스는 "수많은 사용자의 광역 폴링"을 "소수 릴레이의 국소 폴링(또는 CDC 구독)"으로 좁혀주는 셈이다.

<Callout type="info" title="아웃박스가 보장하는 건 'at-least-once'다">
릴레이가 큐에 발행한 직후 죽으면, 재시작 후 같은 행을 다시 발행할 수 있다. 그래서 아웃박스는 보통 **최소 한 번(at-least-once)** 전달을 보장하고, 중복을 컨슈머 쪽에서 막는다. 메시지에 고유 ID를 실어 보내고 컨슈머를 **멱등(idempotent)**하게 짜는 게 한 세트다. "정확히 한 번"은 환상에 가깝고, "최소 한 번 + 멱등"이 현실적인 정석이다.
</Callout>

## 이렇게 하면 됨

흐름을 Before/After로 정리하자.

**Before — 사용자마다 `setInterval` 폴링.** 클라이언트 1만 개가 각자 5초마다 묻고, 서버는 그걸 그대로 DB 쿼리로 흘린다.

```javascript
// Before: 모든 클라이언트가 5초마다 DB로 흘러드는 질문을 던진다
setInterval(async () => {
  const { status } = await (await fetch(`/api/payments/${id}/status`)).json();
  if (status !== "PENDING") render(status);
}, 5000);
```

**After — 변경을 일으킨 쪽이 알리고, 서버가 받아 클라이언트로 푸시.** DB에서 상태가 바뀌면 `NOTIFY`, 서버는 그 알림을 `LISTEN`하다가 SSE로 해당 클라이언트에게만 내려준다.

```typescript
// After (서버): DB 알림을 듣다가, 바뀐 결제만 클라이언트로 푸시
await db.query("LISTEN payment_status");

db.on("notification", (msg) => {
  const { payment_id, status } = JSON.parse(msg.payload);
  const client = sseClients.get(payment_id);   // 그 결제를 보고 있는 사용자
  client?.send({ status });                     // 폴링 없이 한 번만 전송
});
```

이러면 "초당 2천 쿼리"가 "결제 상태가 실제로 바뀐 횟수만큼의 알림"으로 줄어든다. 변경이 없는 동안 DB는 아무 일도 안 한다. 부하가 사용자 수가 아니라 변경 빈도에 묶이는 게 핵심이다.

<Callout type="note" title="그럼 폴링은 항상 나쁜가?">
아니다. 폴링이 오히려 정답인 좁은 경우가 있다.

- **푸시 채널이 없는 외부 시스템**: 알림 훅을 제공하지 않는 서드파티 API라면, 상태를 알 방법이 폴링뿐이다. 다만 이때도 고정 주기 대신 **지수 백오프**(처음엔 자주, 점점 뜸하게)로 헛조회를 줄인다.
- **단순 배치/크론**: "매일 새벽 한 번 새 파일이 왔나 확인"처럼 빈도가 낮고 실시간성이 필요 없으면, 폴링이 가장 단순하고 충분하다.
- **방금 본 아웃박스 릴레이**: 자기 DB의 작은 테이블을 소수 인스턴스가 폴링하는 건, 범위가 좁아 감당 가능한 폴링이다.

안티패턴은 "폴링" 자체가 아니라, **수많은 사용자가 실시간성을 폴링 주기로 흉내 내며 DB를 광역으로 두드리는 것**이다. 둘을 구분하자.
</Callout>

## 정리

폴링은 자연스럽게 생긴다. 누가 작정하고 DB를 괴롭히지 않는다. 그냥 "5초마다 조회하면 되지 뭐"가 제일 빨라 보였을 뿐이고, 로컬에서 혼자 테스트할 땐 멀쩡했을 뿐이다. 하지만 원리는 단순하다.

> **"바뀌었어?"를 반복해 묻지 말고, "바뀌면 알려줘"로 뒤집는다.**

폴링의 비용은 변경 빈도가 아니라 폴링 주기와 사용자 수에 비례한다. 그래서 변경이 드물수록, 사용자가 많을수록 더 손해다. 묻는 구조를 알림받는 구조로 뒤집는 순간, 부하는 실제 변경량만큼으로 줄고 부하-지연 시소에서도 내려올 수 있다. 그리고 그 푸시로 갈아탈 때 거의 반드시 만나는 게 커밋 전 큐 적재 레이스인데, 트랜잭셔널 아웃박스로 "행과 메시지를 한 트랜잭션에 묶는" 한 수면 깔끔하게 풀린다.

다음 편에서는 결이 살짝 다른 문제를 본다. 비즈니스 로직을 트랜잭션 경계 안에 어떻게 가둘 것인가 — 트랜잭션 캡슐화 얘기다.
