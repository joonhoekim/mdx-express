## 화이트보드 코딩의 공포

화이트보드 코딩이 왜 무서운지 설명해줄게.

평소에 코딩할 때 우리가 얼마나 많은 도구에 의존하는지 생각해봐:

- **자동완성** — 함수명 절반만 치면 나머지 나옴
- **구문 강조** — 빨간 줄 뜨면 "아 여기 틀렸구나"
- **Google** — "javascript array sort" 검색 5초
- **TypeScript** — 타입 에러로 버그 미리 잡아줌
- **ChatGPT** — ~~이건 비밀~~

이걸 다 뺏기고 마커 하나 들고 화이트보드 앞에 서면?
**갑자기 신입 시절로 회귀함.**

<Callout type="error" title="실화: 화이트보드 공포의 순간들">
- 코드를 위에서 아래로 쭉 썼는데, 중간에 한 줄을 끼워넣어야 함. 끼울 공간이 없음. 화살표로 "여기 이거 들어감" 표시하다가 면접관도 나도 길을 잃음.

## 면접에서 자주 나오는 알고리즘 유형

실제로 코딩 면접에서 나오는 문제는 패턴이 있음. 크게 분류하면:

### 1. 배열/문자열 조작 — 가장 흔함

면접의 50% 이상이 이 카테고리에서 나옴. "쉬운" 문제라고 생각하지만 함정이 많음.

**대표 문제: Two Sum**

"배열에서 두 수의 합이 target이 되는 인덱스를 찾아라"

LeetCode의 첫 번째 문제이자, 면접에서 워밍업으로 자주 나오는 문제임.

```typescript
// 면접장에서 99%가 처음에 쓰는 코드 (브루트 포스)
function twoSum(nums: number[], target: number): number[] {
  for (let i = 0; i < nums.length; i++) {
    for (let j = i + 1; j < nums.length; j++) {
      if (nums[i] + nums[j] === target) {
        return [i, j];
      }
    }
  }
  return []; // 못 찾으면... 근데 면접에서 이 케이스 처리 안 하면 감점
}
// 시간복잡도: O(n²) — 면접관: "더 나은 방법은 없을까요?"
```

이 질문 들으면 뇌에서 "해시맵" 이라는 단어가 떠올라야 하는데,
긴장하면 안 떠오름. 진짜로.

```typescript
// "아 해시맵!" 하고 깨달은 후의 코드
function twoSum(nums: number[], target: number): number[] {
  const map = new Map<number, number>();

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];

    if (map.has(complement)) {
      return [map.get(complement)!, i];
    }

    map.set(nums[i], i);
  }

  return [];
}
// 시간복잡도: O(n) — 면접관: "좋습니다" (표정 변화 없음)
```

<Callout type="warning" title="면접장 리얼 상황">
면접관: "시간복잡도가 어떻게 되죠?"
나: "O(n)입니다"
면접관: "공간복잡도는요?"
나: "..." (해시맵 때문에 O(n)인데 이걸 단점처럼 말해야 하나? 장점처럼 말해야 하나?)
면접관: (무표정)
나: (심장 쿵쾅)
</Callout>

### 2. 이진탐색 — "쉽다"고 했다가 당하는 유형

이진탐색은 개념은 간단함. 정렬된 배열에서 반씩 나눠서 찾는 거.
근데 **직접 구현하면** 경계 조건에서 100% 버그남.

```typescript
// 이진탐색 — "간단한" 구현
function binarySearch(arr: number[], target: number): number {
  let left = 0;
  let right = arr.length - 1; // ← 여기서부터 고민 시작
                                // length? length - 1? 왜 -1이지?

  while (left <= right) {      // ← < 인가? <= 인가?
                                // 이거 틀리면 무한루프 or 못찾음
    const mid = Math.floor((left + right) / 2);
    // ↑ (left + right) 오버플로우? JS는 상관없지만 면접관이 물어볼 수 있음
    // 안전한 방법: left + Math.floor((right - left) / 2)

    if (arr[mid] === target) {
      return mid;
    } else if (arr[mid] < target) {
      left = mid + 1;          // ← mid + 1 안 하면 무한루프
    } else {
      right = mid - 1;         // ← 여기도 mid - 1
    }
  }

  return -1;
}
```

코드 자체는 20줄도 안 되는데, 주석에 적힌 모든 의문점을 면접관이 물어볼 수 있음.

**진짜 무서운 건 변형 문제:**

```typescript
// "정렬된 배열에서 target 이상인 첫 번째 인덱스를 찾아라" (lower bound)
function lowerBound(arr: number[], target: number): number {
  let left = 0;
  let right = arr.length; // ← 아까랑 다름! length - 1이 아님!

  while (left < right) {   // ← 아까랑 다름! <= 이 아님!
    const mid = Math.floor((left + right) / 2);

    if (arr[mid] < target) {
      left = mid + 1;
    } else {
      right = mid;          // ← mid - 1이 아님!
    }
  }

  return left;
}
```

이진탐색의 `left`, `right`, `<=` vs `<`, `mid + 1` vs `mid` 조합이 바뀌면
완전히 다른 결과가 나옴. 면접장에서 이걸 머릿속으로만 시뮬레이션하다가 멘탈 나가는 거임.

<Callout type="info" title="이진탐색의 진실 — 그래서 어떻게 대처하나">
Donald Knuth가 이런 말을 했음: "이진탐색의 아이디어는 1946년에 나왔지만,
버그 없는 구현이 나온 건 1962년이다." 학계가 16년 걸린 걸 면접에선 15분에 짜라는 거임.

그래서 결론은 *외워서 대응* 하는 게 정답에 가까움. 이진탐색은 창의력으로 푸는 문제가 아니라
**정확히 외워둔 템플릿을 꺼내는 문제** 임. 경계 조건(`<=` vs `<`, `mid+1` vs `mid`)을 매번
즉석에서 추론하면 거의 실수함. 한 가지 템플릿을 손에 익히고, 변형 문제(lower/upper bound)는
그 템플릿에서 *무엇이 바뀌는지* 만 기억하는 게 현실적 전략.

면접에서 "경계 조건이 까다로워서 제가 늘 쓰는 패턴으로 가겠습니다" 라고 *대놓고 말하는 것* 도
감점이 아님. 오히려 자기 약점을 알고 도구로 보완하는 사람으로 보임.
</Callout>

### 3. 동적 프로그래밍 (DP) — 면접의 최종 보스

DP가 나오면 대부분의 지원자가 두 부류로 나뉨:

1. "아 이거 패턴 외웠는데" → 풀음
2. "DP가 뭐였지..." → 멘탈 붕괴

**대표 문제: 계단 오르기**

"한 번에 1계단 또는 2계단 오를 수 있을 때, n번째 계단까지 가는 방법의 수는?"

```typescript
// 1단계: 재귀 (대부분이 처음 생각하는 방법)
function climbStairs(n: number): number {
  if (n <= 1) return 1;
  return climbStairs(n - 1) + climbStairs(n - 2);
}
// 시간복잡도: O(2^n) — 면접관: "n이 크면 어떻게 되죠?"
// 나: "...느려지겠죠?"
// 면접관: "얼마나?"
// 나: "...많이?"
```

```typescript
// 2단계: 메모이제이션 ("아 맞다 캐싱!")
function climbStairs(n: number): number {
  const memo = new Map<number, number>();

  function dp(step: number): number {
    if (step <= 1) return 1;
    if (memo.has(step)) return memo.get(step)!;

    const result = dp(step - 1) + dp(step - 2);
    memo.set(step, result);
    return result;
  }

  return dp(n);
}
// 시간복잡도: O(n) — 면접관: "공간복잡도도 O(n)인데, 줄일 수 있나요?"
// 나: (벌써 3번째 최적화 요구... 살려줘)
```

```typescript
// 3단계: 바텀업 + 공간 최적화 (면접관이 원하는 최종 답)
function climbStairs(n: number): number {
  if (n <= 1) return 1;

  let prev2 = 1; // dp[0]
  let prev1 = 1; // dp[1]

  for (let i = 2; i <= n; i++) {
    const curr = prev1 + prev2;
    prev2 = prev1;
    prev1 = curr;
  }

  return prev1;
}
// 시간복잡도: O(n), 공간복잡도: O(1)
// 면접관: (고개 끄덕) "피보나치랑 같은 구조네요"
// 나: (아 진작 말해주지)
```

<Callout type="error" title="DP 면접의 현실">
**면접관이 원하는 사고 과정:**
1. 문제를 부분 문제로 나눈다
2. 점화식을 세운다
3. 재귀 → 메모이제이션 → 바텀업으로 최적화한다

**실제 지원자 사고 과정:**
1. "이거 DP인가?"
2. "DP 맞는 것 같은데 점화식이 뭐지"
3. (5분 침묵)
4. "죄송한데 힌트 하나만..."
5. (면접관이 힌트를 줌)
6. "아!!!" (하지만 이미 시간의 60%가 지남)
</Callout>

### 4. 그래프/트리 — "재귀 아니면 못 풀죠?"

트리 문제는 재귀랑 친함. 근데 면접에서 재귀 짜면 스택 오버플로우 걱정하면서 짜야 함.

```typescript
// 이진트리 노드 정의
interface TreeNode {
  val: number;
  left: TreeNode | null;
  right: TreeNode | null;
}

// 면접 단골: 이진트리 최대 깊이
function maxDepth(root: TreeNode | null): number {
  if (root === null) return 0;
  return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
}
// 이건 쉬움. 근데 문제는 여기서 끝나지 않음.
```

```typescript
// 면접관: "반복문으로도 풀 수 있나요?"
// 나: (BFS? DFS? 스택? 큐?)

function maxDepthIterative(root: TreeNode | null): number {
  if (!root) return 0;

  const queue: [TreeNode, number][] = [[root, 1]];
  let maxD = 0;

  while (queue.length > 0) {
    const [node, depth] = queue.shift()!;  // ← shift()가 O(n)인 거 아는 면접관도 있음
    maxD = Math.max(maxD, depth);

    if (node.left) queue.push([node.left, depth + 1]);
    if (node.right) queue.push([node.right, depth + 1]);
  }

  return maxD;
}

// 면접관: "queue.shift()의 시간복잡도가 뭐죠?"
// 나: "O(1)... 아 O(n)이요"
// 면접관: "그럼 전체 시간복잡도에 영향을 주나요?"
// 나: (왜 이렇게까지 파고드는 거임...)
```

## LeetCode 그라인딩의 현실

면접 준비 = LeetCode 풀기라는 공식이 업계에 자리 잡은 지 오래됨.

### LeetCode 풀이 5단계

**1단계: 의욕 넘침 (1-3일차)**
- "하루에 3문제씩 풀면 한 달에 90문제다!"
- Easy 문제를 20분 만에 풀고 자신감 상승
- "이거 별거 아니네?"

**2단계: 현실 인식 (4-7일차)**
- Medium 문제에서 1시간 넘게 막힘
- "풀이 보면 이해되는데 왜 혼자 못 풀지?"
- 하루 3문제 → 하루 1문제로 목표 하향

**3단계: 패턴 학습 (2-3주차)**
- "아 슬라이딩 윈도우는 이렇게 푸는 거구나"
- 같은 패턴 문제를 연달아 풀면 좀 됨
- 근데 다른 패턴이 나오면 다시 원점

**4단계: 번아웃 (4주차~)**
- "이걸 매일 해야 한다고?"
- 실무에서 이런 코드 짤 일이 1도 없다는 자괴감
- LeetCode Premium 해지 고민 시작

**5단계: 타협 (면접 직전)**
- "Blind 75만 풀자"
- 75문제 중 40문제 풀고 면접 감
- "운 좋으면 아는 문제 나오겠지"

이 5단계 자체는 누구나 겪음. 중요한 건 *어느 단계에서 방식을 바꾸느냐* 임. 대부분 4단계(번아웃)에서 "양"으로 밀다가 지쳐 그만둠. 합격하는 사람은 3단계(패턴 학습)에서 전략을 트는데, 핵심은 **새 문제를 더 푸는 게 아니라 틀린 문제를 다시 푸는 것** 임. 새 문제 10개보다, 틀린 문제 1개를 *안 보고 다시 짤 수 있을 때까지* 반복하는 게 합격에 훨씬 가까움. "풀어본 문제 수" 는 허영 지표이고, **"안 보고 다시 짤 수 있는 문제 수"** 가 진짜 지표임.

<Callout type="note" title="LeetCode 풀이량 vs 면접 합격률">
**재미있는 통계 (비공식, 체감 기준):**
- 50문제 이하: 합격률 ~20%
- 50-150문제: 합격률 ~40%
- 150-300문제: 합격률 ~55%
- 300문제 이상: 합격률 ~60%

진짜 핵심은 *숫자* 가 아니라 **곡선의 모양** 임. 150문제까지는 가파르게 오르다가 그 뒤로는 거의 평평해짐. *수확 체감* 이 명확함 — 300 → 600문제로 늘려도 합격률은 거의 그대로.

이게 말해주는 것: 150문제쯤이면 *대부분의 패턴은 이미 다 만남*. 그 이후로 안 오르는 40%는 문제를 더 풀어서 메워지지 않음. **다른 축** — 라이브에서 사고를 말로 꺼내는 능력, 긴장 관리, 커뮤니케이션 — 에서 갈림. 300문제 풀고도 떨어진다면 *더 풀 게 아니라 모의 면접으로 축을 바꿔야 한다* 는 신호임.
</Callout>

## 시간복잡도 까먹는 순간

면접에서 코드를 짰으면 시간복잡도를 물어보는 건 당연한 수순임.
근데 이게 평소에는 O(n), O(n log n) 이런 거 자연스럽게 말하다가
면접장에서는 갑자기 헷갈리기 시작함.

### 자주 헷갈리는 시간복잡도

```typescript
// 이건 뭘까?
for (let i = 0; i < n; i++) {
  for (let j = i; j < n; j++) {  // j = i부터 시작
    // 뭔가 함
  }
}
// 정답: O(n²) — n(n+1)/2 이지만 빅오는 O(n²)
// 면접장에서 "O(n log n)?" 이라고 말하는 사람 의외로 많음
```

```typescript
// 이건?
function something(n: number): void {
  if (n <= 0) return;
  console.log(n);
  something(n - 1);
```

| 알고리즘 | 최선 | 평균 | 최악 | 공간 |
|---------|------|------|------|------|
| 버블 정렬 | O(n) | O(n²) | O(n²) | O(1) |
| 삽입 정렬 | O(n) | O(n²) | O(n²) | O(1) |
| 병합 정렬 | O(n log n) | O(n log n) | O(n log n) | O(n) |
| 퀵 정렬 | O(n log n) | O(n log n) | O(n²) | O(log n) |
| 힙 정렬 | O(n log n) | O(n log n) | O(n log n) | O(1) |
| Tim Sort | O(n) | O(n log n) | O(n log n) | O(n) |

참고로 V8(Chrome/Node.js)은 Tim Sort 사용함. 근데 이걸 면접에서 물어보면
"Tim Sort가 뭐였더라..." 하면서 침묵이 흐르는 경우가 대부분임.

<Callout type="success" title="꿀팁: 시간복잡도 외우는 법">
**외우지 말고 이해하기:**
- 한 번 순회 → O(n)
- 순회하면서 매번 또 순회 → O(n²)
- 반씩 나누기 → O(log n)
- 반씩 나누면서 각각 순회 → O(n log n)
- 모든 부분집합 → O(2^n)
- 모든 순열 → O(n!)

이것만 기억하면 대부분의 시간복잡도 질문에 대응 가능함.
</Callout>

## 면접관이 원하는 것 vs 지원자가 보여주는 것

### 면접관이 원하는 것

1. **문제 이해 능력** — 문제를 정확히 파악하고, 모호한 부분은 질문하는 것
2. **사고 과정** — 어떻게 접근하는지, 왜 그런 결정을 내리는지
3. **코딩 능력** — 깨끗하고 정확한 코드 작성
4. **최적화 능력** — 더 나은 방법을 찾을 수 있는지
5. **소통 능력** — 자신의 생각을 명확하게 전달하는 것

### 지원자가 실제로 보여주는 것

1. **문제 받자마자 코딩 시작** — 질문 안 하고 바로 풀기 시작
2. **침묵 속 코딩** — 아무 말 없이 10분간 코딩
3. **버그 투성이 코드** — 긴장해서 평소에 안 하던 실수
4. **최적화 포기** — "이게 최선입니다" (아닌데)
5. **패닉** — "죄송한데 잠시만요..." (하고 5분 동안 아무것도 안 함)

<Callout type="warning" title="면접관의 속마음">
면접관도 사람임. 지원자가 긴장하는 거 다 알고 있음.
근데 **사고 과정을 말로 설명하지 않으면** 뭘 생각하는지 알 수가 없음.

"음... 이 문제는 정렬된 배열이니까 이진탐색을 써볼 수 있을 것 같고요,
아니면 투 포인터로도 접근할 수 있을 것 같은데..." 이런 식으로
**생각을 소리 내어 말하는 것**이 가장 중요한 스킬임.

코드가 완벽하지 않아도 사고 과정이 논리적이면 합격하는 경우 많음.
반대로 코드를 완벽하게 짰는데 왜 그렇게 했는지 설명 못 하면 불합격하기도 함.
</Callout>

### 1. 엣지 케이스 무시

```typescript
// "배열이 비어있으면?" — 이 질문을 안 하면 감점
function findMax(arr: number[]): number {
  // 대부분이 이렇게 시작함
  let max = arr[0]; // ← arr이 빈 배열이면? undefined!

  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > max) max = arr[i];
  }
  return max;
}

// 이렇게 해야 함
function findMaxSafe(arr: number[]): number {
  if (arr.length === 0) throw new Error("빈 배열!");
  // 또는 return -Infinity; 등 요구사항에 따라 처리

  let max = arr[0];
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > max) max = arr[i];
  }
  return max;
}
```

> *왜 자꾸 빠뜨리는가*: 긴장 상태에선 작업 기억(working memory)이 *주 알고리즘* 을 붙드는 데 다 쓰임. "해시맵으로 O(n)" 하나를 머릿속에 띄워두는 것만으로 용량이 차서, 빈 배열·null·중복 같은 *주변 조건* 이 인지에서 밀려남. 평소엔 IDE가 `arr[0]` 에 빨간 줄로 대신 기억해주는데 화이트보드엔 그게 없음.
>
> 대처는 의지력이 아니라 *습관* 임. 코딩 *전에* "입력이 비면 / null이면 / 중복이면" 을 입으로 먼저 말하고 시작하면, 작업 기억이 아니라 *발화된 체크리스트* 로 처리됨. 엣지 케이스는 매번 기억해내는 게 아니라 *루틴으로 박아두는* 것.

## 알고리즘 면접 생존을 위한 필수 패턴 7가지

다 외울 필요 없고, 이 7가지 패턴만 확실히 알면 대부분의 면접을 커버할 수 있음:

```typescript
// 1. 투 포인터 — 정렬된 배열에서 쌍 찾기
function twoPointerExample(arr: number[], target: number): boolean {
  let left = 0, right = arr.length - 1;
  while (left < right) {
    const sum = arr[left] + arr[right];
    if (sum === target) return true;
    if (sum < target) left++;
    else right--;
  }
  return false;
}

// 2. 슬라이딩 윈도우 — 연속 부분 배열/문자열 문제
function maxSumSubarray(arr: number[], k: number): number {
  let windowSum = arr.slice(0, k).reduce((a, b) => a + b, 0);
  let maxSum = windowSum;
  for (let i = k; i < arr.length; i++) {
    windowSum += arr[i] - arr[i - k];
    maxSum = Math.max(maxSum, windowSum);
  }
  return maxSum;
}

// 3. 해시맵 — 조회 O(1) 활용
function hasDuplicate(arr: number[]): boolean {
  const seen = new Set<number>();
  for (const num of arr) {
    if (seen.has(num)) return true;
    seen.add(num);
  }
  return false;
}

// 4. BFS — 레벨별 순회, 최단 경로
function bfs(graph: Map<string, string[]>, start: string): string[] {
  const visited = new Set<string>();
  const queue = [start];
  const result: string[] = [];
  visited.add(start);

  while (queue.length > 0) {
    const node = queue.shift()!;
    result.push(node);
    for (const neighbor of graph.get(node) || []) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor);
        queue.push(neighbor);
      }
    }
  }
  return result;
}

// 5. DFS — 경로 탐색, 연결 컴포넌트
function dfs(graph: Map<string, string[]>, start: string): string[] {
  const visited = new Set<string>();
  const result: string[] = [];

  function explore(node: string) {
    visited.add(node);
    result.push(node);
    for (const neighbor of graph.get(node) || []) {
      if (!visited.has(neighbor)) explore(neighbor);
    }
  }

  explore(start);
  return result;
}

// 6. 이진탐색 — 정렬된 데이터에서 O(log n) 탐색
// (위에서 이미 다룸)

// 7. DP — 부분 문제의 답을 재활용
function longestIncreasingSubsequence(nums: number[]): number {
  const dp = new Array(nums.length).fill(1);

  for (let i = 1; i < nums.length; i++) {
    for (let j = 0; j < i; j++) {
      if (nums[j] < nums[i]) {
        dp[i] = Math.max(dp[i], dp[j] + 1);
      }
    }
  }

  return Math.max(...dp);
}
```

## 결론: 알고리즘 면접은 공정한가?

솔직히 *실무 능력을 정확히 재느냐* 로 보면 완벽하게 공정하진 않음.

- 실무에서 이진탐색을 직접 구현하는 경우는 거의 없음
- DP를 풀 줄 안다고 좋은 개발자인 건 아님
- LeetCode 많이 푼 사람이 프로덕션 코드를 잘 짜는 것도 아님

근데 알고리즘 면접이 *살아남는 진짜 이유* 는 "더 나은 대안이 없어서" 라는 체념이 아니라, 몇 가지 **비대칭** 때문임:

- **검증 비용이 싸다** — 채점이 거의 자동. 1시간에 여러 명을 같은 잣대로 거를 수 있음.
- **거짓 양성(잘못 뽑기)을 줄여준다** — 알고리즘을 *못 푸는* 사람 중에 좋은 개발자는 있어도, 알고리즘을 *깔끔하게 푸는* 사람 중에 기본기 없는 사람은 드묾. 회사는 놓치는 인재보다 *잘못 뽑는 비용* 을 더 무서워하므로, 이 필터는 회사 입장에선 합리적임.
- **준비로 메울 수 있다** — 운처럼 보이지만, 사실 *가장 노력으로 따라잡을 수 있는* 면접임. 시스템 디자인·행동 면접은 경험 없이는 단기에 못 메우지만 알고리즘은 됨.

즉, 지원자에겐 불공정해 *보이지만* 회사 입장에선 risk를 줄이는 합리적 도구임. 이 비대칭을 이해하면 "왜 이런 걸 시키지" 라는 분노가 "그럼 어떻게 통과하지" 로 바뀜. 후자가 훨씬 생산적인 질문임.

<Callout type="note" title="알고리즘 면접이 싫다면">
알고리즘 면접을 안 보는 회사도 있음:
- **과제 전형** — 집에서 과제를 풀어오는 방식
- **포트폴리오 리뷰** — 기존 프로젝트를 보여주는 방식
- **페어 프로그래밍** — 면접관과 같이 코딩하는 방식
- **실무 면접** — 실제 업무와 비슷한 문제를 푸는 방식

자기 스타일에 맞는 면접 방식을 채택하는 회사를 찾는 것도 전략임.
근데 FAANG 가고 싶으면... 어쩔 수 없음. LeetCode 켜.
</Callout>
