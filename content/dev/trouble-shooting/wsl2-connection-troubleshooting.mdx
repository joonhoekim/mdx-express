# WSL2 간헐적 연결 문제 해결 가이드

> VSCode Remote 연결 실패, WSL2 네트워크 불안정 등의 문제를 겪고 있다면 이 가이드를 참고하세요.

## 증상

- VSCode에서 WSL Remote 연결이 간헐적으로 실패하거나 끊김
- WSL2 내부에서 네트워크가 끊기거나 DNS 확인이 느려짐
- `wsl` 명령 실행 시 응답이 없거나 지연됨
- 절전/하이버네이션 복귀 후 문제가 두드러짐

## 진단 과정

<Steps>
<Step title="WSL 상태 확인">

PowerShell에서 기본 상태를 확인합니다:

```powershell
wsl --status
wsl -l -v
```

정상이라면 아래와 같이 출력됩니다:

```
Default Distribution: Ubuntu-24.04
Default Version: 2

  NAME                   STATE       VERSION
* Ubuntu-24.04           Running     2
```

</Step>
<Step title="WSL 내부 연결 상태 확인">

```bash
# 기본 연결 확인
echo 'WSL connectivity: OK'
uptime

# DNS 설정 확인
cat /etc/resolv.conf

# 네트워크 인터페이스 확인
ip addr show eth0
```

NAT 모드(기본값)에서는 아래와 같이 `172.x.x.x` 대역의 가상 IP가 할당됩니다:

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 qdisc mq state UP group default qlen 1000
    link/ether 00:15:5d:08:85:a8 brd ff:ff:ff:ff:ff:ff
    inet 172.29.142.123/20 brd 172.29.143.255 scope global eth0
```

이 NAT IP는 Windows 호스트와 직접 통신하는 것이 아니라 가상 스위치를 거치기 때문에 불안정의 원인이 됩니다.

</Step>
<Step title="커널 로그 확인 (dmesg)">

```bash
dmesg | tail -30
```

아래와 같은 **소켓 연결 실패 로그**가 보인다면, WSL2 VM과 Windows 호스트 간 통신에 문제가 있는 것입니다:

```
[    5.816600] WSL (270) ERROR: CheckConnection: connect() failed: 101
```

<Callout type="info" title="에러 코드 101">
에러 코드 101은 `ENETUNREACH` (Network is unreachable)로, VM이 호스트 네트워크에 도달하지 못하는 상태를 의미합니다.
</Callout>

추가로, 저널 파일 손상 로그도 함께 나타날 수 있습니다. 이는 WSL이 비정상 종료(절전 등)된 후 재시작될 때 발생합니다:

```
[    1.116339] systemd-journald[40]: File /var/log/journal/.../system.journal corrupted or uncleanly shut down, renaming and replacing.
```

</Step>
<Step title="systemd 로그 확인 (journalctl)">

```bash
journalctl --no-pager -n 30
```

**핵심 지표**: `Clock change detected` 메시지가 30~40초 간격으로 반복되는지 확인합니다.

실제 관측된 로그:

```
Mar 11 09:41:41 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:42:16 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:42:52 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:43:27 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:44:03 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:44:38 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:45:13 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:45:49 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:46:24 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
Mar 11 09:46:59 evot1 systemd-resolved[137]: Clock change detected. Flushing caches.
```

<Callout type="warning" title="DNS 캐시 flush 반복">
이 로그가 반복된다면 WSL2 VM의 시계가 Windows 호스트와 지속적으로 어긋나고 있는 것입니다. `systemd-resolved`가 시계 변경을 감지할 때마다 DNS 캐시를 전부 flush하므로, DNS 확인이 반복적으로 실패하거나 지연되고, 이것이 VSCode Remote Server 연결 실패의 직접적인 원인이 됩니다.
</Callout>

또한 아래 로그도 함께 나타날 수 있습니다:

```
Mar 11 09:41:29 evot1 wsl-pro-service[175]: WARNING Daemon: could not connect to Windows Agent: could not get address: could not read agent port file "/mnt/c/Users/jh/.ubuntupro/.address": open /mnt/c/Users/jh/.ubuntupro/.address: no such file or directory
Mar 11 09:41:29 evot1 wsl-pro-service[175]: WARNING Exiting after <nil>: check if the Windows agent is installed and running.
```

이 로그는 Ubuntu Pro 서비스 관련이며, 직접적인 원인은 아니지만 WSL↔Windows 간 통신 문제의 부수적인 증거입니다.

</Step>
<Step title="시계 동기화 상태 확인">

```bash
timedatectl status
```

```
               Local time: Wed 2026-03-11 09:49:35 KST
           Universal time: Wed 2026-03-11 00:49:35 UTC
                 RTC time: Wed 2026-03-11 00:49:35
                Time zone: Asia/Seoul (KST, +0900)
System clock synchronized: yes
              NTP service: active
          RTC in local TZ: no
```

`System clock synchronized: yes`로 표시되더라도, WSL2 VM 특성상 절전 복귀 시 일시적인 시계 드리프트가 발생하며, 이것이 위의 `Clock change detected` 반복 로그의 원인입니다.

</Step>
</Steps>

## 원인 요약

| 원인 | 로그 지표 | 영향 |
|------|-----------|------|
| 시계 동기화 드리프트 | `Clock change detected. Flushing caches.` 반복 | DNS 캐시가 주기적으로 소멸 → DNS 확인 실패/지연 → VSCode 연결 끊김 |
| NAT 네트워킹 소켓 통신 실패 | `CheckConnection: connect() failed: 101` | WSL↔Windows 간 직접 통신 불가 → VSCode Remote Server 접근 불가 |
| 비정상 종료로 인한 저널 손상 | `system.journal corrupted or uncleanly shut down` | systemd 서비스 초기화 지연 → 부팅 직후 불안정 |

이 세 가지 문제가 결합되면서 간헐적이고 재현이 어려운 연결 실패가 발생합니다.

## 해결 방법

<Steps>
<Step title=".wslconfig 생성 (Windows 측)">

`C:\Users\<사용자명>\.wslconfig` 파일을 생성합니다:

```ini
[wsl2]
networkingMode=mirrored
dnsTunneling=true
autoProxy=true
```

각 설정의 역할:

| 설정 | 기본값 | 변경 후 | 효과 |
|------|--------|---------|------|
| `networkingMode` | `nat` | `mirrored` | NAT 가상 네트워크 대신 Windows 호스트와 동일한 네트워크 스택을 공유. `172.x.x.x` 가상 IP 대신 호스트 네트워크 대역의 실제 IP를 사용하여 소켓 통신이 안정화됨 |
| `dnsTunneling` | `false` | `true` | DNS 요청을 WSL2 VM 내부의 `systemd-resolved` 대신 Windows 호스트의 DNS 스택을 통해 터널링. 시계 드리프트로 인한 DNS 캐시 flush 영향을 받지 않음 |
| `autoProxy` | `false` | `true` | Windows에 설정된 프록시를 WSL2에서 자동으로 상속. 회사 네트워크 등에서 유용 |

</Step>
<Step title="WSL 재시작">

설정 변경 후 반드시 WSL을 완전히 종료하고 다시 시작해야 합니다:

```powershell
wsl --shutdown
wsl
```

<Callout type="warning">
`wsl --shutdown`은 모든 WSL 인스턴스를 종료합니다. 실행 중인 작업이 있다면 먼저 저장하세요.
</Callout>

</Step>
<Step title="적용 확인">

WSL 내부에서 mirrored 모드가 정상 적용되었는지 확인합니다:

```bash
ip -4 addr
```

<TabsList>
<TabsTrigger value="before">변경 전 (NAT 모드)</TabsTrigger>
<TabsTrigger value="after">변경 후 (mirrored 모드)</TabsTrigger>
</TabsList>

<TabsContent value="before">
```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450
    inet 172.29.142.123/20 brd 172.29.143.255 scope global eth0
```
</TabsContent>

<TabsContent value="after">
```
5: eth2: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450
    inet 192.168.190.119/24 brd 192.168.190.255 scope global noprefixroute eth2
```
</TabsContent>
</Tabs>

호스트 네트워크 대역(`192.168.x.x` 등)의 IP가 보인다면 mirrored 모드가 적용된 것입니다.

추가로 외부 연결을 확인합니다:

```bash
ping -c 2 google.com
```


</Step>
</Steps>

## 부록: Docker Desktop + WSL Integration vs WSL2 내부 Docker Engine

WSL2 환경에서 Docker를 사용하는 방법은 크게 두 가지입니다. 특히 위에서 설명한 mirrored 네트워킹을 적용한 환경에서는 두 방식의 차이가 더 두드러집니다.

### 아키텍처 비교

<Tabs defaultValue="desktop">
<TabsList>
<TabsTrigger value="desktop">Docker Desktop + WSL Integration</TabsTrigger>
<TabsTrigger value="engine">WSL2 내부 Docker Engine</TabsTrigger>
</TabsList>

<TabsContent value="desktop">
```
Windows Host
├── Docker Desktop (Windows 프로세스)
│   └── docker-desktop VM (별도 WSL2 배포판)
│       └── dockerd (Docker 데몬)
└── Ubuntu-24.04 (사용자 WSL2 배포판)
    └── docker CLI → docker-desktop VM의 dockerd로 프록시
```

Docker Desktop은 `docker-desktop`과 `docker-desktop-data`라는 별도의 WSL2 배포판을 생성하고, 사용자의 WSL 배포판에서는 CLI만 제공하면서 실제 데몬은 별도 VM에서 실행합니다.
</TabsContent>

<TabsContent value="engine">
```
Windows Host
└── Ubuntu-24.04 (사용자 WSL2 배포판)
    ├── dockerd (Docker 데몬, 직접 실행)
    └── docker CLI → 로컬 Unix 소켓으로 직접 통신
```

사용자 배포판 내부에 Docker Engine을 설치하여, 데몬과 CLI가 같은 환경에서 동작합니다.
</TabsContent>
</Tabs>

### 상세 비교

| 항목 | Docker Desktop + WSL Integration | WSL2 내부 Docker Engine |
|------|----------------------------------|------------------------|
| **데몬 위치** | 별도 `docker-desktop` WSL2 VM | 사용자 WSL2 배포판 내부 |
| **CLI↔데몬 통신** | cross-distro 프록시 소켓 | 로컬 Unix 소켓 (`/var/run/docker.sock`) |
| **메모리 사용** | WSL2 VM 2개 (사용자 + docker-desktop) | WSL2 VM 1개 |
| **파일 I/O (볼륨 마운트)** | 배포판 간 파일 공유 시 9P 프로토콜 경유 → 느림 | 네이티브 ext4 직접 접근 → 빠름 |
| **네트워크** | docker-desktop VM의 네트워크 스택 별도 관리 | WSL2 배포판의 네트워크 스택 공유 |
| **mirrored 모드 호환** | 충돌 가능성 있음 (별도 VM이 네트워크 설정에 간섭) | 문제 없음 |
| **설치/업데이트** | GUI 설치 프로그램, 자동 업데이트 | `apt` 패키지 매니저로 관리 |
| **Kubernetes** | 내장 K8s 클러스터 지원 | 별도 설치 필요 (minikube, k3s 등) |
| **GUI 관리** | Docker Desktop 대시보드 | 없음 (CLI 또는 Portainer 등 별도 설치) |
| **라이선스** | 대규모 조직(250명+ 또는 매출 $10M+) 유료 | 무료 (Apache 2.0) |
| **리소스 제어** | Docker Desktop 설정에서 CPU/메모리 제한 | `.wslconfig` 또는 `cgroup`으로 직접 제어 |

### 성능 차이가 발생하는 이유

Docker Desktop의 WSL Integration은 사용자 배포판에 Docker CLI와 프록시 소켓만 주입하고, 실제 컨테이너 실행은 `docker-desktop` 배포판에서 처리합니다. 이 구조에서 발생하는 오버헤드:

1. **소켓 프록시**: `docker` 명령이 cross-distro 소켓을 통해 다른 VM의 데몬에 전달되므로 레이턴시가 추가됨
2. **파일 시스템 경유**: 볼륨 마운트 시 사용자 배포판의 파일을 docker-desktop VM에서 접근해야 하므로 9P 프로토콜을 거치게 됨 (특히 `node_modules` 같은 대량의 작은 파일에서 체감됨)
3. **이중 VM 메모리**: 두 개의 WSL2 배포판이 각각 메모리를 점유

### WSL2 내부 Docker Engine 설치

Docker Desktop을 사용하지 않기로 결정했다면, WSL2 내부에 Docker Engine을 직접 설치합니다. 배포판별 설치 방법은 [Docker 공식 문서](https://docs.docker.com/engine/install/)를 참고하세요.

<Callout type="info" title="GUI가 필요하다면?">
Docker Desktop 없이도 VSCode의 **Docker 확장**(ms-azuretools.vscode-docker)과 **Dev Containers 확장**을 설치하면 컨테이너/이미지 목록 조회, 로그 확인, 실행/중지 등 기본적인 GUI 관리가 가능합니다. Docker Desktop의 대시보드를 대부분 대체할 수 있으므로, GUI 때문에 Docker Desktop을 유지할 필요는 없습니다.
</Callout>

## 주의 사항

<Callout type="warning" title="Docker Desktop 충돌">
mirrored 모드와 Docker Desktop의 WSL integration이 충돌할 수 있습니다. 문제가 발생하면 Docker Desktop 설정 → Resources → WSL integration에서 해당 배포판을 비활성화 후 다시 활성화해보세요.
</Callout>

<Callout type="info" title="포트 포워딩">
NAT 모드에서는 `localhost`로 WSL 포트에 접근 가능했지만, mirrored 모드에서는 동작 방식이 달라질 수 있습니다. 대부분의 경우 동일하게 동작하지만, 특정 포트 바인딩 설정이 있다면 확인이 필요합니다.
</Callout>

<Callout type="warning" title="VPN 충돌">
일부 VPN 소프트웨어와 mirrored 모드가 충돌할 수 있습니다. VPN 연결 후 WSL 네트워크가 끊기면 `.wslconfig`에 `firewall=false`를 추가해보세요:

```ini
[wsl2]
networkingMode=mirrored
dnsTunneling=true
autoProxy=true
firewall=false
```
</Callout>

- Windows 11 Pro 10.0.26200
- WSL 2.6.3.0
- WSL Kernel 6.6.87.2-1
- WSLg 1.0.71
- Ubuntu 24.04 (systemd 활성화)

## 버전 요구 사항

- `networkingMode=mirrored`: WSL 2.0.0 이상
