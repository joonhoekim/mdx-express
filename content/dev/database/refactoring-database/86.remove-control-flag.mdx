## 이게 뭔데

제어 플래그(control flag) 제거. **루프 한가운데서 "이제 그만 돌자"는 의도를 boolean 변수에 담아 놓고, 루프 조건은 그 변수만 쳐다보게 만든 코드를 걷어내는 일**이다. 그 자리에 진짜 탈출 조건을 직접 박는다.

비유를 하나 들자. 운전 중에 "다음 휴게소에서 멈춰야지"라고 마음먹는다고 치자. 정상적인 사람은 휴게소가 보이면 그냥 깜빡이 켜고 들어간다. 근데 제어 플래그 환자는 이렇게 한다. 조수석에 "멈출까말까"라고 적힌 포스트잇을 붙여두고, 휴게소가 보이는 순간 포스트잇에 동그라미를 친 다음, 그 포스트잇을 5초마다 흘끔거리면서 "동그라미 쳐졌나? 아직? 쳐졌네, 그럼 멈춰야지" 한다. 멈추고 싶으면 그냥 멈추면 되는데, 굳이 중간에 메모지 한 장을 끼워 넣은 거다.

코드도 똑같다. `found = false` 같은 플래그 하나 선언해 놓고, 조건이 충족되면 `found = true`로 바꾸고, 루프는 `WHILE NOT found`로 도는 패턴. 이 플래그가 바로 그 포스트잇이다. 없어도 되는 중간 매개물이고, 읽는 사람 머릿속에 상태 하나를 더 얹는다.

이건 책에서 말하는 **내부 리팩토링(Internal Refactoring)** 에 속한다. 저장 프로시저의 **인터페이스(시그니처)는 1도 안 건드리고**, 내부 구현만 깔끔하게 바꾼다. 그래서 이 프로시저를 호출하는 앱·배치·리포트는 아무것도 몰라도 된다. 외부 영향 제로. 가장 마음 편하게 손댈 수 있는 부류다.

<Callout type="info" title="한 줄 요약">
"멈출지 말지"를 boolean 변수에 적어두고 그 변수를 검사하지 말고, 멈춰야 하는 조건을 루프 탈출 지점에 직접 적어라. 변수 하나가 사라지면 코드는 그만큼 읽기 쉬워진다.
</Callout>

## 언제 쓰나

플래그 변수가 나쁜 건 그게 **간접 표현(indirection)** 이기 때문이다. 코드를 읽는 사람은 `done`이라는 변수를 만나는 순간 머릿속에 질문이 생긴다. "이거 어디서 `true`로 바뀌지?" 그래서 루프 본문을 위아래로 훑으며 대입문을 찾아 헤맨다. 조건이 두세 군데서 플래그를 건드리면, 그 모든 지점을 머릿속에 펼쳐놓고 OR로 합쳐야 비로소 "아, 이 루프는 이러이러할 때 끝나는구나"가 보인다. 탈출 조건이 코드 한 줄에 모여 있지 않고 **여기저기 흩어져 있는 것**, 이게 제어 플래그의 본질적 비용이다.

그래서 이 리팩토링이 답인 냄새는 대체로 이렇다.

- 루프 안에 `done`, `found`, `notFinished`, `keepGoing` 같은 boolean 변수가 있고, 본문 어딘가에서 거기에 값을 대입한다.
- 루프 조건은 그 플래그만 검사한다 (`WHILE done = FALSE`). 정작 "왜 끝나는지"는 본문을 읽어야 안다.
- 같은 플래그를 두 군데 이상에서 `true`로 바꾼다. 탈출 사유가 흩어져 있다.
- 플래그를 세팅한 직후에도 루프 본문이 계속 흘러서, "이미 끝내기로 했는데 왜 아래 코드가 더 도는 거지?" 하는 미세한 의심이 든다.

특히 마지막 항목이 은근히 버그의 온상이다. 플래그를 `true`로 바꿔놓고도 루프 본문 나머지가 그대로 실행되면, 정작 멈추는 건 다음 회차다. 그 "한 박자 늦은 탈출" 사이에 부수효과가 한 번 더 일어나면 묘한 결함이 생긴다.

### 시나리오: 이런 적 있을 거임

은행 시스템의 저장 프로시저를 하나 물려받았다고 치자. 한 고객(Customer)의 계좌(Account) 목록을 순회하면서, **마이너스 잔액(Balance)인 계좌가 하나라도 있으면** 그 고객을 "주의 고객"으로 플래그하고 멈추는 로직이다. PL/SQL로 대충 이렇게 생겼다.

```sql
CREATE OR REPLACE PROCEDURE FlagOverdrawnCustomer (
  p_CustomerID IN NUMBER,
  p_Result     OUT VARCHAR2
) AS
  v_Found   BOOLEAN := FALSE;   -- 제어 플래그
  v_Balance NUMBER;
  CURSOR c_Accounts IS
    SELECT Balance FROM Account WHERE CustomerID = p_CustomerID;
BEGIN
  p_Result := 'OK';
  FOR rec IN c_Accounts LOOP
    v_Balance := rec.Balance;
    IF v_Balance < 0 AND v_Found = FALSE THEN
      v_Found  := TRUE;          -- "찾았다"를 변수에 적어둠
      p_Result := 'OVERDRAWN';
    END IF;
    -- v_Found가 TRUE여도 루프는 계속 돈다...
  END LOOP;
END;
```

이 코드, 동작은 한다. 근데 읽는 사람은 세 번 멈칫한다. 첫째, `v_Found`가 대체 왜 있는지. 둘째, `AND v_Found = FALSE`가 무슨 방어 코드인지 (이미 찾았으면 또 안 덮어쓰려는 거다). 셋째, 마이너스 계좌를 찾았는데도 **커서가 끝까지 다 돈다**는 점. 고객이 계좌를 300개 가지고 있고 첫 번째가 이미 마이너스면, 나머지 299개를 헛돈다. 답은 첫 줄에서 이미 나왔는데.

`v_Found`는 "찾았다"를 코드 흐름 대신 변수에 기록하는 포스트잇이다. 이거 떼면 코드가 짧아지고, 빨라지고, 무엇보다 의도가 한눈에 들어온다.

## 주의할 점

<Callout type="warning" title="플래그가 진짜로 일하고 있는 건 아닌지 먼저 확인">
모든 제어 플래그가 군더더기인 건 아니다. 떼기 전에 이 플래그가 **단순 탈출 신호** 이상의 일을 하고 있지 않은지 봐야 한다.

- **루프가 끝난 뒤 그 값을 다시 쓰는 경우**: `v_Found`를 루프 후에 `IF v_Found THEN ...`으로 또 검사한다면, 이건 탈출 플래그가 아니라 **결과를 담는 상태 변수**다. 함부로 떼면 그 분기가 깨진다. 이 경우 탈출은 직접 조건으로 빼되, 결과 변수는 남겨두거나 다른 방식으로 표현해야 한다.
- **여러 조건이 같은 플래그에 OR로 모이는 경우**: 탈출 사유가 세 가지인데 전부 `done = true`로 수렴한다면, 단순히 플래그만 떼면 탈출 조건식이 비대해질 수 있다. 이럴 땐 `Consolidate Conditional Expression`(조건식 통합)이나 `Extract Method`(메서드 추출)를 먼저 적용해 조건에 이름을 붙인 다음 플래그를 제거하는 게 깔끔하다.
- **`EXIT`/`RETURN`이 부수효과를 건너뛰는 경우**: 루프 본문 끝에 카운터 증가나 로그 기록 같은 후처리가 있고, 그게 매 회차 반드시 일어나야 한다면, 중간에서 `EXIT`로 빠져나가는 순간 그 후처리를 건너뛴다. 의미가 바뀌는지 확인할 것.

요컨대 "탈출 신호 전용 플래그"만 안전하게 제거 대상이다.
</Callout>

또 하나. 이 리팩토링은 **인터페이스를 안 바꾸므로** 호출부 변경이 필요 없다는 게 큰 장점이지만, 동시에 그게 함정이기도 하다. "어차피 내부니까 테스트 없이 슥 고치자"는 유혹이 생긴다. 인터페이스가 같아도 **동작이 바뀌면 그건 리팩토링이 아니라 버그**다. 들어가는 입력에 대해 나오는 결과가 똑같다는 걸 보증할 장치가 있어야 마음 놓고 손댈 수 있다.

## 이렇게 한다

핵심 절차는 단순하다. 플래그를 검사하던 자리에 **진짜 조건을 직접 박고**, 찾은 즉시 루프를 빠져나가고, 플래그 선언과 대입문을 지운다.

<Steps>
<Step title="플래그가 표현하던 탈출 조건을 식별한다">
`v_Found`가 `TRUE`로 바뀌는 지점을 찾아, 그게 "어떤 조건이 충족됐을 때"인지를 한 줄짜리 조건식으로 적는다. 위 예시에선 `v_Balance < 0`이 그 조건이다.
</Step>
<Step title="그 조건을 루프 탈출 지점에 직접 쓰고 즉시 EXIT한다">
플래그를 갱신하고 다음 회차를 기다리는 대신, 조건이 참이면 그 자리에서 `EXIT`(또는 `RETURN`)한다. 탈출 사유가 루프 안 한 곳에 모인다.
</Step>
<Step title="플래그 선언·대입·검사를 전부 제거한다">
`v_Found` 변수 선언, `:= TRUE` 대입, `WHILE v_Found = FALSE` 류의 조건 검사를 모두 지운다. 컴파일러/린터가 "안 쓰는 변수"라고 짚어주면 깨끗이 떨어진 것이다.
</Step>
<Step title="동작 동일성을 회귀 테스트로 확인한다">
같은 입력에 같은 출력이 나오는지 검증한다. 인터페이스가 안 바뀌었으니 기존 테스트가 그대로 통과해야 정상이다.
</Step>
</Steps>

### Before / After

위 시나리오를 리팩토링하면 이렇게 된다. 커서를 끝까지 돌릴 필요 없이, 마이너스 계좌를 만나는 즉시 빠져나온다.

```sql
-- Before: 제어 플래그 v_Found로 탈출 의도를 변수에 기록
CREATE OR REPLACE PROCEDURE FlagOverdrawnCustomer (
  p_CustomerID IN NUMBER,
  p_Result     OUT VARCHAR2
) AS
  v_Found   BOOLEAN := FALSE;
  v_Balance NUMBER;
  CURSOR c_Accounts IS
    SELECT Balance FROM Account WHERE CustomerID = p_CustomerID;
BEGIN
  p_Result := 'OK';
  FOR rec IN c_Accounts LOOP
    v_Balance := rec.Balance;
    IF v_Balance < 0 AND v_Found = FALSE THEN
      v_Found  := TRUE;
      p_Result := 'OVERDRAWN';
    END IF;
  END LOOP;
END;
```

```sql
-- After: 플래그 없이, 조건을 직접 검사하고 즉시 탈출
CREATE OR REPLACE PROCEDURE FlagOverdrawnCustomer (
  p_CustomerID IN NUMBER,
  p_Result     OUT VARCHAR2
) AS
  CURSOR c_Accounts IS
    SELECT Balance FROM Account WHERE CustomerID = p_CustomerID;
BEGIN
  p_Result := 'OK';
  FOR rec IN c_Accounts LOOP
    IF rec.Balance < 0 THEN
      p_Result := 'OVERDRAWN';
      EXIT;                 -- 찾았으면 그 자리에서 끝
    END IF;
  END LOOP;
END;
```

변수 `v_Found`, `v_Balance`가 통째로 사라졌다. `AND v_Found = FALSE` 같은 방어 조건도 필요 없다. 한 번 찾으면 바로 나가니 중복으로 덮어쓸 일 자체가 없기 때문이다. 코드는 짧아졌고, "마이너스 계좌가 하나라도 있으면 OVERDRAWN"이라는 의도가 그대로 읽힌다.

`WHILE` 루프로 짠 코드도 결은 같다. 플래그를 조건으로 끌어올리거나 탈출문으로 바꾸면 된다.

```sql
-- Before: WHILE NOT done 형태
v_Done := FALSE;
WHILE v_Done = FALSE LOOP
  FETCH c_Accounts INTO v_Balance;
  IF c_Accounts%NOTFOUND THEN
    v_Done := TRUE;
  ELSIF v_Balance < 0 THEN
    p_Result := 'OVERDRAWN';
    v_Done := TRUE;
  END IF;
END LOOP;
```

```sql
-- After: 탈출 조건을 직접 검사
LOOP
  FETCH c_Accounts INTO v_Balance;
  EXIT WHEN c_Accounts%NOTFOUND;
  IF v_Balance < 0 THEN
    p_Result := 'OVERDRAWN';
    EXIT;
  END IF;
END LOOP;
```

### 현대 도구로 강화하기

2006년의 책은 이런 코드를 사람이 눈으로 읽으며 손으로 고친다. 지금은 같은 일을 더 안전하고 자동으로 할 수단이 있다.

**1. 정적 분석으로 플래그를 자동 색출.** 사람이 일일이 `done`, `found`를 찾아다닐 필요 없다. PL/SQL은 `PL/SQL Cop`(Trivadis plsql-cop) 같은 린터가, T-SQL은 `SQL Prompt`/`tSQLt` 생태계의 정적 분석이 "안 쓰는 변수", "한 번만 대입되는 boolean" 류를 짚어준다. PostgreSQL `plpgsql_check` 익스텐션도 죽은 변수와 도달 불가 코드를 잡아낸다. 컴파일 경고를 켜두면 플래그를 떼고 남은 잔재(미사용 변수)가 바로 빨갛게 뜬다.

```text
plpgsql_check:
  warning: unused variable "v_found"
  warning: unused variable "v_balance"
```

**2. 회귀 테스트로 동작 동일성 보증.** 내부 리팩토링의 전제 조건은 결국 **프로시저용 자동화 테스트**다. 저장 프로시저 단위 테스트 프레임워크를 쓰면 된다. SQL Server는 `tSQLt`, Oracle은 `utPLSQL`, PostgreSQL은 `pgTAP`. 리팩토링 전후로 같은 픽스처에 같은 결과가 나오는지 박아두면, 플래그를 떼다 의미를 바꿔버리는 사고를 잡는다.

```sql
-- utPLSQL 예시: 마이너스 계좌가 있으면 OVERDRAWN
PROCEDURE test_overdrawn_customer IS
  v_result VARCHAR2(20);
BEGIN
  -- given: 고객 42에게 잔액 -100짜리 계좌가 있다
  FlagOverdrawnCustomer(42, v_result);
  ut.expect(v_result).to_equal('OVERDRAWN');
END;
```

**3. 애초에 루프를 SQL로 내려라 (제일 강한 강화).** 사실 위 프로시저는 더 근본적인 질문을 부른다. **굳이 행을 하나씩 가져와 루프를 돌 일인가?** "마이너스 계좌가 하나라도 있나"는 집합 연산 한 방으로 끝난다. 제어 플래그를 없애는 가장 확실한 방법은, 플래그가 살던 절차적 루프 자체를 선언적 SQL로 치환하는 것이다. 이건 책의 `Substitute Algorithm`(알고리즘 치환)과 맞닿는다.

```sql
-- 루프도 플래그도 통째로 제거: 존재 여부를 SQL에게 묻는다
CREATE OR REPLACE PROCEDURE FlagOverdrawnCustomer (
  p_CustomerID IN NUMBER,
  p_Result     OUT VARCHAR2
) AS
  v_HasOverdrawn NUMBER;
BEGIN
  SELECT CASE WHEN EXISTS (
           SELECT 1 FROM Account
           WHERE CustomerID = p_CustomerID AND Balance < 0
         ) THEN 1 ELSE 0 END
    INTO v_HasOverdrawn
    FROM dual;

  p_Result := CASE WHEN v_HasOverdrawn = 1 THEN 'OVERDRAWN' ELSE 'OK' END;
END;
```

`EXISTS`는 첫 매칭에서 멈추므로, 우리가 `EXIT`로 흉내 내려던 "하나라도 있으면 그만"을 옵티마이저가 알아서 해준다. 인덱스(`Account(CustomerID, Balance)`)가 있으면 계좌가 300개든 3만 개든 즉시 답이 나온다. 제어 플래그 제거의 종착점은 종종 "루프를 SQL에게 떠넘기기"다.

<Callout type="note" title="그래도 절차형이 필요할 때가 있다">
모든 루프를 집합 연산으로 접을 수 있는 건 아니다. 각 행마다 외부 호출, 복잡한 누적 계산, 순서 의존 상태 변경이 끼면 커서 루프가 정당하다. 그럴 땐 루프는 남기되 **제어 플래그만** 직접 조건/`EXIT`로 바꾸면 된다. "플래그 제거"와 "루프 자체 제거"는 별개의 결정이다 — 전자는 거의 항상 옳고, 후자는 케이스 바이 케이스다.
</Callout>

## 정리

제어 플래그는 "멈출지 말지"를 코드 흐름 대신 변수에 적어두는 포스트잇이다. 읽는 사람은 그 포스트잇이 어디서 동그라미 쳐지는지 찾아 헤매야 하고, 탈출 사유는 본문 여기저기 흩어진다.

> **탈출하고 싶으면 탈출 조건을 거기에 직접 적어라. 변수에 적어두고 나중에 그 변수를 검사하지 마라.**

리팩토링은 단순하다. 플래그가 표현하던 조건을 루프 탈출 지점에 직접 박고, 즉시 `EXIT`하고, 플래그 선언·대입·검사를 지운다. 인터페이스를 안 건드리니 호출부는 그대로고, 외부 영향은 없다. 안전벨트는 회귀 테스트(`utPLSQL`/`tSQLt`/`pgTAP`)와 정적 분석(`plpgsql_check`/PL/SQL Cop)이 매 준다. 그리고 한 발 더 나아가면, 그 절차적 루프 자체가 `EXISTS` 한 줄로 접히는 경우가 의외로 많다. 플래그를 떼다 보면 루프가 통째로 사라지는, 가장 기분 좋은 종류의 청소다.
