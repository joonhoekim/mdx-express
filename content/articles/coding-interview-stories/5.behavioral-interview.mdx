행동 면접(Behavioral Interview). 개발자들이 가장 **과소평가하는** 면접 유형임.

"나는 코딩 잘하니까 행동 면접은 대충 해도 되겠지?"
→ 탈락.

"기술 면접은 완벽했는데 왜 떨어졌지?"
→ 행동 면접에서 빨간불 켜졌기 때문.

아무리 코딩을 잘해도 **같이 일하기 힘든 사람**은 안 뽑음.
그리고 행동 면접은 "같이 일하기 힘든 사람"을 걸러내는 필터임.

---

## 행동 면접이 뭔가

기술 면접이 "뭘 할 수 있는가"를 보는 거라면,
행동 면접은 "어떤 사람인가"를 보는 거임.

과거의 행동이 미래의 행동을 예측한다는 전제 하에,
**실제 경험**을 물어봄.

```
기술 면접:  "이진트리를 뒤집어보세요"
행동 면접:  "팀원과 의견이 달랐던 경험을 말해보세요"

기술 면접:  정답이 있음
행동 면접:  정답이 없음 (하지만 오답은 있음)
```

<Callout type="warning" title="행동 면접의 무서운 점">
기술 면접은 모르면 "모르겠습니다" 하면 됨.
행동 면접은 **"없었습니다"라고 하면 안 됨.**

"갈등이 있었던 경험?" → "없었습니다" → 🚩 빨간불
(갈등이 한 번도 없었다는 건 a) 거짓말이거나 b) 소통을 안 했다는 뜻)

"실패한 프로젝트?" → "없었습니다" → 🚩 빨간불
(실패 경험이 없다는 건 도전을 안 했다는 뜻)

"피드백을 받은 경험?" → "없었습니다" → 🚩 빨간불
(피드백을 안 받았다는 건 성장에 관심이 없다는 뜻)
</Callout>

## STAR 메소드 — 행동 면접의 프레임워크

행동 면접에는 공식이 있음. **STAR 메소드.**

```
S - Situation (상황): 어떤 상황이었는가?
T - Task (과제): 뭘 해야 했는가?
A - Action (행동): 뭘 했는가?
R - Result (결과): 결과가 어땠는가?
```

### 예시: "팀원과 의견이 달랐던 경험"

**나쁜 답변:**
```
"네, 한번 팀원이랑 기술 스택 때문에 의견이 달랐었는데,
제가 설득해서 제 의견대로 했습니다. 결과도 좋았습니다."
```
→ 구체적이지 않음, 자기 자랑만 함, 팀워크 안 보임

**STAR로 구조화한 답변:**
```
[Situation]
"작년에 신규 프로젝트의 프론트엔드 프레임워크를 선택해야 했습니다.
3명의 프론트엔드 개발자가 있었고, 마감은 3개월이었습니다."

[Task]
"저는 Next.js를 원했고, 다른 팀원은 Remix를 주장했습니다.
팀 리드로서 합리적인 결정을 내려야 했습니다."

[Action]
"각자의 주장을 문서로 정리하자고 제안했습니다.
평가 기준을 미리 합의했고 (학습 곡선, 성능, 커뮤니티 지원, 우리 요구사항 적합도),
각 프레임워크로 간단한 프로토타입을 만들어 비교했습니다.
최종 결정은 투표가 아니라 기준에 따른 점수로 했습니다."

[Result]
"결과적으로 Next.js를 선택했습니다. Remix를 주장한 팀원은
결과 자체는 받아들였지만, 솔직히 한동안 '아쉬움이 남는다'는 톤이 있었습니다.
그래도 *프로세스가 공정했다* 는 데는 동의했고,
다음 프로젝트에서 그 팀원이 기술 선택을 주도할 때
같은 방식(기준 합의 → 프로토타입 비교)을 먼저 제안해줬습니다.
프로토타입을 만들며 서로 다른 프레임워크를 써본 게
기술 폭을 넓힌 부수 효과도 있었고요."
```

<Callout type="success" title="STAR 답변의 핵심 포인트">
1. **구체적 숫자/사실** — "팀원 3명", "마감 3개월" (추상적이면 믿음이 안 감)
2. **갈등의 실체** — 뭐 때문에 의견이 달랐는지 명확히
3. **내가 한 행동** — "팀이 했다"가 아니라 "내가 했다"에 초점
4. **학습/성장** — 결과에서 뭘 배웠는지
5. **팀 관점** — 나만 이기는 결론이 아니라 팀이 이기는 결론
6. **너무 완벽한 결말은 오히려 의심받음** — "모두가 만족했다" 보다 "한 명은 끝까지 아쉬워했지만 프로세스엔 동의했다" 가 더 진짜처럼 들림. 면접관은 *디테일의 질감* 으로 진위를 가늠함.
</Callout>

## 자주 나오는 행동 면접 질문 TOP 10

### 1. "가장 어려웠던 기술적 문제는?"

이 질문의 **진짜 의도**: 문제 해결 능력, 끈기, 기술적 깊이

```
좋은 답변의 구조:
- 구체적인 기술 문제 (메모리 누수, 성능 병목, 데이터 마이그레이션 등)
- 시도한 방법들 (여러 가지 시도 → 실패 → 다른 접근)
- 최종 해결 방법
- 배운 점

나쁜 답변:
"CSS 센터링이 안 됐는데 StackOverflow에서 찾아서 해결했습니다"
→ 너무 사소함, 학습 없음
```

### 2. "데드라인을 못 맞출 것 같았던 경험은?"

진짜 의도: 우선순위 설정, 커뮤니케이션, 압박 대처

```
좋은 답변 포인트:
- 일찍 위험 신호를 감지하고 보고했다
- 기능의 우선순위를 재조정했다 (MVP 접근)
- 팀과 소통하며 대안을 찾았다
- 결과적으로 핵심 기능은 제때 출시했다

나쁜 답변:
"야근해서 다 했습니다"
→ 문제 해결이 아니라 노동력 투입. 지속 가능하지 않음.
```

### 3. "실수한 경험은?"

진짜 의도: 자기 인식, 책임감, 학습 능력

<Callout type="error" title="절대 하면 안 되는 답변들">
**"실수한 적 없습니다"**
→ 거짓말이거나 자기 인식이 없는 사람

**"팀원이 실수해서 제가 수습했습니다"**
→ 남 탓. 질문은 "본인의 실수"를 물어본 거임

**"프로덕션 DB를 날렸습니다 ㅋㅋ"**
→ 솔직한 건 좋은데 이건 좀... 심각한 실수는 "어떻게 방지 시스템을 구축했는지"까지 말해야 함

**좋은 답변 구조:**
1. 실수를 구체적으로 인정 (축소 X, 과장 X)
2. 발견 과정 (스스로 발견? 다른 사람이 알려줌?)
3. 수습 과정 (빠르게 대응했는지)
4. 재발 방지책 (같은 실수 안 하려고 뭘 했는지)
5. 배운 점
</Callout>

### 4. "피드백을 받고 개선한 경험은?"

진짜 의도: 성장 마인드셋, 피드백 수용 능력

```
좋은 답변 예시:
"코드 리뷰에서 '변수명이 너무 축약되어 있다'는 피드백을 받았습니다.
처음에는 '코드 길어지잖아'라고 생각했는데,
3개월 뒤 제가 짠 코드를 보니 진짜 뭔지 모르겠더라고요.
그 이후로 변수명을 명확하게 짓는 습관을 들였고,
팀의 코딩 컨벤션 문서에도 이 내용을 추가했습니다."
```

### 5. "리더십을 발휘한 경험은?"

진짜 의도: 팀 리드/시니어 역할 가능성

```
주의: "리더십 = 팀장/리드 경험"이 아님

리더십 예시:
- 새로운 프로세스를 제안하고 도입한 경험
- 주니어를 멘토링한 경험
- 기술 부채를 해결하자고 설득한 경험
- 팀 내 지식 공유 세션을 시작한 경험
- 위기 상황에서 주도적으로 문제를 해결한 경험

직급이 낮아도 리더십은 보여줄 수 있음.
```

### 6. "왜 이직하려고 하세요?"

이 질문은 **지뢰밭**임.

```
절대 하면 안 되는 답변:
❌ "현재 회사 사람들이 싫어서요"
❌ "연봉이 너무 낮아서요" (솔직하지만 너무 직접적)
❌ "야근이 많아서요"
❌ "우리 회사 기술 스택이 구려서요"
❌ "상사가 미워서요"

OK인 답변:
✅ "더 큰 규모의 서비스를 경험하고 싶습니다"
✅ "특정 기술 분야에서 더 깊이 성장하고 싶습니다"
✅ "새로운 도메인에 도전하고 싶습니다"
✅ "지금까지의 경험을 바탕으로 더 큰 임팩트를 내고 싶습니다"

핵심: "현재 회사에서 도망치는 것"이 아니라
      "새로운 기회를 향해 가는 것"으로 프레이밍
```

<Callout type="info" title="이직 사유 — 현실과 포장의 간격을 직시하기">
**현실:**
"연봉이 낮고, 야근이 많고, 기술 부채가 심하고, 성장이 안 되는 느낌"

**포장:**
"현재 역할에서 많이 배웠지만, 이제는 더 도전적인 환경에서
기술적으로 성장하면서 사용자에게 더 큰 가치를 전달하고 싶습니다.
귀사의 [구체적 기술/문화/서비스]에 매력을 느꼈습니다."

**여기서 솔직해질 부분:** 면접관도 진짜 이유의 상당 부분이 연봉·사람·번아웃인 걸 안다. 수백 명을 봤으니까. 그래서 이 질문은 포장을 *간파당하지 않으려는* 게임이 아니라, **그 현실을 어떤 언어로 재구성하는지** 를 보는 것에 가까움. "성장이 안 됨" → "성장의 *방향* 이 회사와 어긋남" 으로 바꾸는 건 거짓말이 아니라 *해석* 임. 그 해석의 성숙도가 곧 직업적 성숙도의 신호가 됨.

**포인트:**
- 현재 회사를 깎아내리지 않기 (뒷담화 = 빨간불). 단 *지나치게 미화* 해도 "그럼 왜 떠나요?" 라는 역공을 받음.
- 지원 회사에 대한 구체적 관심 — *링크드인 5분 본 수준* 의 칭찬은 티가 남.
- 진짜 이유 한 조각을 *전략적으로* 노출하는 게 100% 포장보다 신뢰를 삼. (예: "솔직히 보상도 고려 요소지만, 더 큰 건…")
</Callout>

### 7. "5년 후에 어디에 있고 싶나요?"

솔직히 5년 후를 정확히 아는 사람은 없음. 면접관도 이 답변이 *면접용* 인 걸 안다. 그래서 이 질문은 *예언의 정확도* 를 보는 게 아니라 두 가지를 봄: (1) 말이 안 되는 비전인가 (개발 면접인데 "3년 안에 창업하겠다"), (2) *지금 지원한 회사의 방향* 과 일관되는가.

```
솔직한 답변: "5년 후에 여전히 개발하고 있을지도 모르겠는데요..."
→ 인간적이지만, *방향이 없어 보이는* 게 문제.
  회사는 "이 사람은 뭘 하고 싶은지 모른다"고 읽음.

좋은 답변 구조:
1. 기술적 성장 방향 (어떤 분야의 전문가가 되고 싶은지)
2. 역할 변화 (IC → 시니어? 매니저? 아키텍트?) — 단, 지원 포지션과 모순되면 안 됨
3. 지원 회사와의 연결 (이 회사에서 그 방향이 가능한지)

예시:
"5년 후에는 프론트엔드 아키텍처 수준에서
의사결정을 할 수 있는 시니어 엔지니어가 되고 싶습니다.
특히 대규모 서비스의 성능 최적화와 DX 개선에 전문성을 갖추고 싶고,
귀사의 서비스 규모가 그런 경험을 쌓기 좋은 환경이라고 봤습니다."

→ 이 답변이 *완벽해서* 좋은 게 아님.
  지원 포지션(프론트엔드)과 회사 특성(대규모 서비스)에 *일관* 되기 때문에 좋은 것.
  같은 구조에 본인 상황을 갈아끼우면 됨.
```

### 8. "스트레스를 어떻게 관리하나요?"

```
나쁜 답변:
"저는 스트레스를 안 받는 편입니다"
→ 로봇인가? 또는 자기 인식이 없거나

좋은 답변:
"온콜 대응이나 긴급 배포 같은 상황에서 스트레스를 받는 편인데,
그럴 때는 먼저 문제를 작은 단위로 쪼개서 하나씩 해결하려고 합니다.
그리고 주말에는 운동이나 게임으로 리프레시하고 있습니다.
번아웃이 올 것 같으면 팀 리드에게 미리 말씀드리고 조정하는 편입니다."
```

### 9. "다른 사람에게 어려운 피드백을 준 경험은?"

```
이 질문이 나오면 시니어급 평가임.

좋은 답변 구조:
- 피드백이 필요했던 상황 (코드 품질? 커뮤니케이션? 성과?)
- 어떻게 전달했는지 (1:1, 구체적 예시, 감정이 아닌 사실 기반)
- 상대의 반응
- 결과 (개선됐는지, 관계는 유지됐는지)

핵심: "직접적이되 존중을 유지했다"를 보여줘야 함
```

### 10. "마지막으로 질문 있으신가요?"

<Callout type="success" title="면접 끝의 역질문 — 이걸 잘하면 인상이 확 바뀜">
**좋은 질문:**
- "이 팀에서 가장 큰 기술적 도전은 뭔가요?"
- "신규 입사자의 온보딩 프로세스는 어떻게 되나요?"
- "코드 리뷰 문화는 어떤가요?"
- "이 역할의 성공을 어떻게 정의하나요?"
- "팀의 향후 6개월 로드맵은 어떻게 되나요?"

**나쁜 질문:**
- "연봉은 얼마인가요?" (첫 면접에서는 ㅋㅋ)
- "재택근무 되나요?" (이것만 물어보면 인상 안 좋음)
- "야근 많나요?" (불안해 보임)
- "질문 없습니다" (관심 없어 보임)

**역질문은 최소 2-3개 준비해가기.**
면접관의 답변에 따라 후속 질문까지 하면 더 좋음.
</Callout>

## 개발자가 특히 약한 소프트 스킬 질문들

### "비개발자에게 기술적 내용을 설명한 경험은?"

이건 진짜 개발자들이 약한 부분임.

```
나쁜 예:
PM: "왜 이 기능이 3주나 걸려요?"
개발자: "마이크로서비스 간 이벤트 소싱 파이프라인을 구축하고
         CQRS 패턴으로 읽기/쓰기를 분리해야 하는데,
         기존 모놀리스의 데이터 모델을 마이그레이션하려면..."
PM: "..."

좋은 예:
PM: "왜 이 기능이 3주나 걸려요?"
개발자: "비유하자면, 기존 시스템은 모든 물건을 한 창고에 보관하는 방식이에요.
         새 기능은 물건을 종류별로 다른 창고에 나눠 보관해야 해서,
         이사하는 데 시간이 걸립니다.
         이사 중에도 서비스는 정상 운영해야 하니까 조심스럽게 해야 하고요."
PM: "아 이해됐어요!"
```

### "우선순위가 충돌할 때 어떻게 하나요?"

```
현실:
- PM: "이 기능 이번 주 안에 필요해요"
- 테크 리드: "기술 부채 먼저 해결해야 합니다"
- 디자이너: "이 UI 개선도 급해요"
- QA: "버그 100개가 밀려있어요"

좋은 답변:
"먼저 각 요청의 임팩트와 긴급도를 파악합니다.
비즈니스에 직접적 영향이 있는 건(매출, 사용자 경험 심각한 버그) 최우선이고,
그 다음은 팀과 협의해서 결정합니다.
모든 걸 동시에 할 수 없으니까 투명하게 소통하는 게 중요하다고 생각합니다.
'이건 다음 스프린트로 미루겠다'고 명확히 말하는 것도 역량이라고 봅니다."
```

### "완벽하지 않은 상태에서 출시해야 했던 경험은?"

```
이 질문은 실용주의 vs 완벽주의를 봄.

완벽주의 답변 (빨간불):
"저는 완벽하지 않은 코드를 출시하는 것에 반대합니다.
품질은 타협할 수 없습니다."
→ 현실 감각이 없어 보임

실용주의 답변 (그린불):
"MVP 개념을 이해하고 있습니다.
핵심 기능은 안정적으로 동작하되, 부가 기능은 후속 릴리즈로 미루는 결정을 한 적 있습니다.
다만 '기술 부채 목록'을 만들어서 추후 반드시 해결하도록 트래킹했습니다.
출시 후에도 모니터링을 강화해서 문제가 생기면 빠르게 대응했습니다."
```

## 행동 면접 준비 방법

### 1. 에피소드 뱅크 만들기

면접 전에 **8-10개의 에피소드**를 미리 준비해두면 대부분의 질문에 대응 가능.

```
에피소드 카테고리:

1. 기술적 도전 + 해결   → "어려운 문제" 질문에 사용
2. 팀 갈등 + 해결       → "갈등" 질문에 사용
3. 실수 + 학습          → "실수" 질문에 사용
4. 리더십 발휘          → "리더십" 질문에 사용
5. 데드라인 압박        → "압박" 질문에 사용
6. 피드백 수용          → "성장" 질문에 사용
7. 크로스팀 협업        → "협업" 질문에 사용
8. 주도적 개선          → "주도성" 질문에 사용
```

### 2. STAR 형식으로 정리하기

```
각 에피소드를 이렇게 정리:

제목: [프로덕션 메모리 누수 해결]

S: 서비스 오픈 2주 후 서버 메모리가 매일 증가하는 현상 발견
T: 원인을 찾고 메모리 누수를 해결해야 했음
A: 1) heap dump 분석 도구(Chrome DevTools) 사용
   2) 이벤트 리스너 미해제가 원인임을 발견
   3) 컴포넌트 언마운트 시 클린업 로직 추가
   4) 메모리 모니터링 대시보드 구축
R: 메모리 사용량 60% 감소, 이후 같은 문제 발생 시 5분 내 감지 가능

사용 가능한 질문:
- "어려운 기술적 문제를 해결한 경험"
- "주도적으로 문제를 발견하고 해결한 경험"
- "모니터링/운영 경험"
```

### 3. 모의 면접하기

```
혼자 연습하는 방법:
1. 질문 리스트에서 랜덤하게 하나 뽑기
2. 타이머 2분 설정
3. 소리 내어 답변하기 (녹음하면 더 좋음)
4. 녹음 다시 듣고 개선점 찾기

체크포인트:
- 2분 이내에 답변했는가? (너무 길면 감점)
- 구체적 숫자/사실이 포함되었는가?
- "나"의 역할이 명확한가?
- 결과와 학습이 있는가?
- 긍정적 인상을 남기는가?
```

### 4. 준비한 에피소드 *밖* 을 대비하기

여기서 STAR의 한계가 드러남. 준비한 8~10개로 대부분 막을 수 있지만, *노련한 면접관은 일부러 준비 안 된 영역을 파고듦*. "방금 그건 잘 정리돼 있네요. 그럼 *반대로*, 실패해서 끝내 회복 못 한 경험은요?" 같은 식.

이때 억지로 준비된 에피소드를 끼워 맞추면 티가 남. 차라리:

- "그 정확한 경험은 없는데, 결이 비슷한 걸로는…" 하고 *솔직하게 인접 경험* 으로 빠지기
- 진짜 없으면 "지금 바로는 안 떠오르는데, 떠오르면 다시 말씀드려도 될까요?" 하고 *나중에 돌아오기* (실제로 돌아오면 인상이 좋음)

에피소드 뱅크는 *대본* 이 아니라 *재료* 임. 준비의 목적은 "모든 질문에 답을 갖는 것" 이 아니라 "어떤 질문이 와도 *내 경험 어딘가로 연결* 할 수 있는 상태" 가 되는 것.

<Callout type="note" title="행동 면접의 문화적 차이">
**미국/글로벌 기업:**
- STAR 메소드를 거의 공식적으로 사용
- 개인의 기여를 강조 ("I did...")
- 리더십 원칙(Leadership Principles)에 맞추기 (아마존 14개 등)

**한국 기업:**
- 형식이 덜 구조적
- 팀 성과를 강조하는 게 더 좋을 수 있음 ("저희 팀이...")
- 겸손한 톤이 더 먹힐 수 있음
- "인성 면접"이라고도 부름

지원 회사의 문화에 맞춰서 답변 톤을 조절하는 것도 전략임.
</Callout>

## 행동 면접에서 빨간불이 켜지는 신호

면접관 입장에서 이런 답변을 들으면 불합격 쪽으로 기울어짐. 다만 알아둘 게 있음 — 이 빨간불들은 *나쁜 사람이라서* 켜지는 게 아니라 **평가 상황의 압박이 만드는 자동 반응** 인 경우가 대부분임. 그래서 의식적으로 연습해서 눌러야 함.

### 1. 남 탓만 하는 사람

```
"프로젝트가 실패한 건 PM이 요구사항을 계속 바꿔서입니다."
"버그가 많았던 건 QA 팀이 테스트를 안 해서입니다."
"서비스가 느린 건 인프라 팀이 서버를 적게 줘서입니다."

→ 이 사람과 일하면 문제가 생길 때마다 남 탓을 할 거라는 인상
```

> *왜 그렇게 되는가*: 평가 상황에선 본능적으로 *방어 자세* 가 됨. "내 책임이 아니다" 가 자기 보호로 튀어나오는 것. 의식적으로 *내가 통제할 수 있었던 한 조각* 을 찾아 "그 상황에서 내가 더 했어야 한 건…" 으로 프레이밍하는 연습이 필요. 남 탓 0%가 아니라, 외부 요인을 인정하되 그 안에 *내 몫* 을 끼워 넣는 것.

### 2. 과장하는 사람

```
"제가 혼자서 프로젝트를 처음부터 끝까지 다 했습니다."
"제가 합류한 후 팀 생산성이 300% 올랐습니다."
"제 아키텍처 제안 덕분에 회사가 살았습니다."

→ 면접관도 개발자임. 혼자서 다 한다는 건 불가능하다는 걸 앎.
   과장이 보이면 신뢰가 무너짐.
```

> *왜 그렇게 되는가*: 과장은 "잘 보이고 싶어서" 보다 *시간 압박* 때문인 경우가 많음. 2분 안에 임팩트를 만들려다 디테일이 날아가고 결과 수치만 부풀려짐. 차라리 *작지만 구체적인* 성과 (숫자 + 내가 한 정확한 행동) 가 신뢰를 삼. "생산성 300%" 보다 "p95 응답을 1.2초에서 0.4초로" 가 강함.

### 3. 자기 인식이 없는 사람

```
"단점이요? 글쎄요... 일을 너무 열심히 하는 것?"
"개선할 점? 딱히 없는 것 같은데요."

→ 자기 약점을 모르는 사람은 성장하지 않는다는 인상
```

> *왜 그렇게 되는가*: "단점을 말하면 감점" 이라는 오해에서 옴. 실제론 *약점을 알고 관리하는 모습* 이 가점 요소이고, 약점 없는 척이 더 큰 빨간불. 단 "완벽주의라 일을 너무 오래 붙들어요" 같은 *위장된 자랑* 도 면접관은 다 앎. 진짜 약점 + 그걸 다루는 장치를 함께 말해야 함.

### 4. 커뮤니케이션이 안 되는 사람

```
질문: "프로젝트 경험을 말해주세요"
답변: (10분 동안 끊임없이 말함, 핵심 없이 TMI)

또는

답변: "네, 했습니다." (끝)

→ 적절한 양의 정보를 적절한 시간 내에 전달하는 능력 = 커뮤니케이션
```

> *왜 그렇게 되는가*: 긴장하면 *과잉(TMI)* 아니면 *과소(단답)* 로 쏠림. 둘 다 불안의 표현임. STAR가 분량 조절 도구가 되는 이유가 여기 있음 — 구조가 있으면 *어디까지 말하고 멈출지* 가 정해짐.

<Callout type="warning" title="답변 길이의 황금비">
행동 면접 답변의 적정 길이: **1.5분 ~ 3분**

1분 미만 → 너무 짧음. 구체성 부족.
3분 이상 → 너무 김. 면접관 집중력 떨어짐.

STAR 각 단계에 30-45초씩 배분하면 딱 맞음.
S(30초) + T(15초) + A(60초) + R(30초) = 약 2분 15초

Action 파트에 가장 많은 시간을 쓰는 게 포인트.
**"뭘 했는가"가 가장 중요하니까.**
</Callout>

## 행동 면접 실전 시뮬레이션

면접관과 지원자의 대화를 시뮬레이션해볼게:

```
면접관: "최근에 기술적으로 어려운 문제를 해결한 경험을 말씀해주세요."

[좋은 답변]
지원자: "최근에 저희 서비스에서 특정 페이지의 로딩 속도가 5초 이상
걸리는 문제가 있었습니다. (S)

프론트엔드 성능 최적화를 맡게 되었는데, 사용자 이탈률이
로딩 3초 이상에서 급격히 증가하는 데이터가 있어서
2초 이내로 줄이는 것이 목표였습니다. (T)

먼저 Chrome DevTools의 Performance 탭으로 프로파일링을 했더니,
번들 크기가 2.3MB로 거대했고, 첫 렌더링에 불필요한 라이브러리가
많이 로드되고 있었습니다.

세 가지 조치를 취했는데요.
첫째, React.lazy와 Suspense로 코드 스플리팅을 적용해서
초기 번들을 800KB로 줄였습니다.
둘째, lodash 전체 대신 개별 함수만 import하도록 변경했습니다.
셋째, 이미지를 WebP로 변환하고 lazy loading을 적용했습니다. (A)

결과적으로 LCP가 5.2초에서 1.8초로 줄었고,
사용자 이탈률도 35%에서 22%로 감소했습니다.
이 경험을 통해 '번들 크기 모니터링'을 CI 파이프라인에 추가하게 되었고,
이후에는 번들 크기가 임계값을 넘으면 PR에서 경고가 뜨도록 했습니다. (R)"
```

```
면접관: "lodash 대신 뭘 쓰셨나요?" (후속 질문)
지원자: "lodash/debounce 같이 개별 함수만 import했고,
일부 유틸은 직접 구현했습니다.
예를 들어 deepClone은 structuredClone API를 활용했습니다."

면접관: "팀원들의 반응은 어땠나요?" (협업 확인)
지원자: "처음엔 코드 스플리팅 도입에 우려가 컸습니다.
'로딩 중에 화면이 깜빡이면 더 안 좋다'는 반대가 있어서,
사실 이거 합의하는 데만 회의를 두 번 했어요.
결국 Skeleton UI를 같이 만들어서 *깜빡임이 아니라 로딩처럼 보이게* 한 뒤에야
동의를 받았습니다. 코드 자체보다 이 합의에 시간이 더 들었습니다."
```

완벽하게 답하는 게 포인트가 아님. 위 답변에서 *진짜처럼 들리는 부분* 은 "회의를 두 번 했다", "코드보다 합의에 시간이 더 들었다" 같은 **비효율의 디테일** 임. 매끄럽게 다 잘 풀린 이야기는 오히려 의심을 삼. 후속 질문에 *약간 버벅이더라도 구체적인* 답이, 매끈하지만 추상적인 답을 이김.

## 결론: 행동 면접은 "준비된 이야기"가 아니라 "해석된 경험"을 본다

행동 면접이 어려운 이유는 정답이 없어서가 아님. **자기 경험을 해석해본 적이 없어서** 임.

대부분은 자기가 한 일을 *사건 단위* 로만 기억함. "그때 메모리 누수 잡았지", "그 프로젝트 갈등 있었지." 그런데 면접관이 보는 건 사건이 아니라 *그 사건에서 당신이 무엇을 끌어냈는가* 임. 같은 경험을 두고도, 한 번도 곱씹어본 적 없는 사람과 *몇 번 해석해본 사람* 은 말의 밀도가 다름. 면접관은 그 밀도로 사람을 가늠함.

그래서 에피소드 뱅크의 진짜 가치는 *대본 암기* 가 아니라 **자기 경력을 회고하는 작업** 에 있음. 8~10개를 STAR로 정리하다 보면 "아, 그때 내가 사실 이걸 배웠구나" 를 처음 깨닫는 경우가 많음. 면접 준비가 *경력 정리* 의 기회가 되는 셈.

준비 방법 요약:

1. **에피소드 뱅크** — 8~10개를 STAR로 정리하며 *자기 경험을 해석*
2. **소리 내어 연습** — 머릿속으로만 하면 *말의 밀도* 가 안 올라감
3. **녹음 + 리뷰** — 자기 답변의 *침묵·횡설수설* 구간 확인
4. **모의 면접** — 친구·동료·Pramp 같은 플랫폼
5. **지원 회사 리서치** — 같은 에피소드라도 *회사 방향에 맞는 측면* 을 골라 강조

그리고 가장 중요한 건 **솔직하되 전략적으로 솔직하기.** 모든 걸 말할 필요는 없지만, 말하는 건 진짜여야 함. 면접관은 수백 명을 봤기 때문에 *지어낸 이야기의 매끄러움* 과 *진짜 경험의 거칠음* 을 구분함. 역설적으로, *약간 못나도 진짜인 이야기* 가 *완벽하지만 지어낸 이야기* 를 이김.

"팀원과 갈등이 있었던 경험을 말해보세요" 에 진짜 갈등을, 어떻게 풀었는지를, 뭘 배웠는지를 — *그리고 무엇이 끝내 안 풀렸는지까지* 자연스럽게 말할 수 있으면 행동 면접은 이긴 거임.
