작년에 [TypeScript 풀스택 개발에서 피해야 할 것들](/docs/diary/unrecommended-actions)이라는 회고를 쓰면서, 첫 항목으로 "데이터베이스 뷰를 사용하지 마라"라고 적었다. 명령형 제목이었고, 단일 책임 원칙·성능 오해·ORM 불협화음 같은 일반론을 근거로 삼았다.

그 글은 빠른 체크리스트로는 쓸모가 있었지만, 다시 읽어보면 두 가지가 마음에 걸렸다. *왜 우리 스택에서 특히* 비용이 큰지에 대한 구체적 이유가 없었고, view의 비용을 잘못 이해하고 있었다. 이번 글은 그때 못 본 것을 채우는 자기-수정이다. 결론은 비슷하다 — view를 기본 미사용으로 둔다. 다만 그 결론에 도달하는 길이 더 정직해졌다.

<Callout type="info" title="TL;DR">
view는 나쁜 기술이 아니다. 지금 스택(Drizzle + PostgreSQL + Next.js + 소규모 팀, 2026-05 기준)에서 **비용 대비 이점이 크지 않을 뿐**이다. 그래서 일반 조회 추상화 목적의 view는 **기본 미사용**으로 둔다.

진짜 비용은 흔한 직관과 다르다. **SQL 실행 비용은 view가 있든 없든 같다.** 진짜로 잃는 건 두 가지 — 분리 쿼리 사이의 MVCC 스냅샷 일관성, 그리고 합성 컬럼의 ORDER BY 가능성. 이 둘이 의미 있는 곳(회계·합산·강한 일관성)에서만 추가 장치(트랜잭션 / view 잔존)가 정당하다.

예외 영역(Materialized View, RLS 보조, BI/리포팅, 레거시 호환, 다중 서비스 공통 projection)은 의식적으로 좁게 유지한다. 그리고 도구가 진화하면 이 정책은 재검토한다.
</Callout>

<Callout type="warning" title="이 글이 다루지 않는 것">
view 제거의 *방법*(마이그레이션 패턴, 진단 체크리스트, 위험 지점)은 별도의 작업 가이드에 있다. 이 글은 *왜 그 방향을 권하는가*에 대한 정책 에세이다. 결론(미사용 권고)에는 동의하지 않더라도 비용 분석 부분은 view를 *유지*하는 결정에도 그대로 쓸 수 있다.
</Callout>

## 1. 작년에 단정했던 것의 결함

"View를 사용하지 마라"고 쓰던 시점의 나는 두 가지를 못 보고 있었다.

첫 번째는, *왜 우리 스택에서* 비용이 큰지에 대한 구체적 이유가 없었다는 점이다. "ORM과의 불협화음"이라는 일반론은 어느 ORM이든 적용된다. Hibernate에서 만나는 view 비용과 Drizzle에서 만나는 view 비용은 다른 모양인데, 그 다름을 짚지 못했다. 일반론은 광범위하게 공감을 얻지만 실질적인 결정에는 도움이 되지 않는다.

두 번째는, view의 비용을 잘못 이해하고 있었다는 점이다. 작년 글에서 "뷰를 쓰면 성능이 좋아진다는 흔한 오해는 틀렸다"고 적었다. 그 진술은 절반만 맞는다. 정확한 형태는 다음과 같다.

<Callout type="note" title="흔한 직관의 정정">
**PostgreSQL은 일반 view를 parent query에 inline해서 같은 plan으로 실행한다.** 즉 view가 있든 없든 SQL 실행 비용은 같다. "view를 빼면 쿼리 수가 늘어서 느려질 것"이라는 흔한 반박은 틀렸고, 동시에 "view를 쓰면 빨라질 것"이라는 흔한 기대도 틀렸다.

(여기서 말하는 view는 일반 view에 한정된다. Materialized View는 결과를 저장하므로 다른 이야기다.)
</Callout>

이걸 정확히 말하지 못하면 view 정책에 관한 어떤 토론도 헛돈다. 양쪽 모두 환상의 비용을 두고 다투게 된다.

## 2. 그렇다면 진짜 비용은 어디 있는가

view를 application 레이어 쿼리로 옮길 때 자주 두려워하는 비용은 두 가지다. "쿼리 수가 늘어서 느려진다." "JOIN 코드가 산재해서 가독성이 떨어진다."

첫 번째는 부분적으로만 사실이다. 1 view → 1 JOIN으로 옮기면 쿼리 수도 같고 실행 비용도 같다. 1 view → N개 분리 쿼리(메인 + 서브 projection)로 옮기면 roundtrip이 늘긴 한다. 그런데 이건 **"view 제거"의 비용이 아니라 "분리 쿼리로 합성하기로 한 선택"의 비용**이다. view를 두든 빼든, 합성 전략은 어차피 선택해야 한다.

두 번째 — 가독성 — 는 취향과 도구의 문제다. view에 SQL을 묻어두는 것이 더 읽기 좋다는 사람도 있고, TypeScript 한 곳에서 쿼리 빌더로 표현하는 것이 더 읽기 좋다는 사람도 있다. 이건 정답이 없다.

진짜 비용은 따로 있다. 잘 안 보이는 두 곳이다.

### 2.1 MVCC 스냅샷 일관성

view 1쿼리는 PostgreSQL이 단일 MVCC 스냅샷에서 모든 sub-projection을 일관되게 읽어준다. 메인 + 서브 쿼리로 분리하면 그 사이에 base 데이터가 변할 수 있다.

read-only 페이지에서 보조 정보가 잠깐 다른 시점이어도 의미 없는 경우가 대부분이다. 그러나 합산·통계·회계 reporting처럼 **"메인과 보조 데이터가 같은 시점이어야 의미가 있는"** 경우에는 결정적이다. 잔액이 100원인 시점의 거래 목록과, 거래 목록이 잡힌 후 잔액이 다시 조회된 시점이 섞여 보이면 그 화면은 거짓말을 하고 있는 거다.

이 비용은 관리할 수 있다. 정합성이 필요한 합성을 `db.transaction()`으로 감싸고, 필요하면 `REPEATABLE READ` 격리 수준까지 끌어올리면 된다.

```typescript
await db.transaction(async (tx) => {
  const main = await tx.select().from(orders).where(eq(orders.id, id));
  const items = await tx.select().from(orderItems).where(eq(orderItems.orderId, id));
  return { ...main[0], items };
});
```

판별 기준 한 줄: **"메인 데이터와 보조 데이터가 같은 시점이어야 의미가 있는가?"** Yes면 트랜잭션, No면 분리 쿼리 그대로.

### 2.2 합성 컬럼의 ORDER BY 한계

view는 SQL 안에 합성 컬럼(예: `last_event_at`, `total_count`)을 만들어 두고 그 컬럼으로 페이지 정렬을 시킬 수 있다. 분리 쿼리 합성에서는 메인 쿼리가 모르는 컬럼이라 DB 레벨 ORDER BY가 불가능하다. 페이지 전체를 메모리로 끌어와 정렬하든가, 그 컬럼은 정렬 비활성화로 남기든가, 둘 중 하나를 선택해야 한다.

이걸 회피할 마법은 없다. 단지 사용자에게 "이 컬럼은 정렬되지 않습니다"를 정직하게 노출하는 게 합리적이다.

이 두 비용이 view 제거의 *실제* 비용이다. 흔한 직관이 가리키던 곳(쿼리 수, 가독성)이 아니라, 잘 안 보이는 곳에 비용이 있다.

## 3. 그래도 기본 미사용을 권하는 이유

진짜 비용을 보고도 여전히 view를 기본 미사용으로 두는 게 나은 이유는 네 가지다. 우리 스택에 한정해서 말한다.

### 3.1 운영 파이프라인 비용

`drizzle-kit generate`(2026-04 기준)는 view 의존 관계를 충분히 추적하지 못하고, 테이블 스키마 변경 시 view 재생성 쿼리를 자동으로 안전하게 생성하지 않는다. 결과적으로 마이그레이션 수동 수정, `DROP VIEW → CREATE VIEW` 순서 관리, 배포 시 의존성 검증을 사람이 떠안게 된다.

작은 팀에서는 이런 휴먼 에러 표면을 의도적으로 줄이는 게 장기적으로 유지 비용을 더 줄인다. "운영 자동화 신뢰성이 낮은 기능은 의식적으로 제한한다"는 판단.

이 근거는 도구 성숙도에 묶여 있다. drizzle-kit이 view 추적을 강화하면 무게가 약해진다 — 정책 재검토 시 가장 먼저 다시 평가할 항목이다.

### 3.2 Source of Truth 분산

view를 풀-정의(`pgView(...).as(sql\`...\`)`)로 두면 다음이 동시에 존재한다.

- 테이블 스키마
- DB View SQL 정의
- Drizzle View 타입 정의

세 곳을 함께 수정해야 하고, 누락 시 마이그레이션 실패 / 런타임 오류 / 타입과 실제 스키마 불일치가 발생한다. *지난 글에서 단일 책임 원칙(SRP) 위반이라고 적었던 항목의 정확한 형태가 이것이다.* "변경 지점이 두 곳"이 아니라 "권위 있는 표현이 세 곳에 분산"인 것.

`pgView(...).existing()` 모드는 SoT가 DB 단일이라 이 비판은 약하게 적용된다. 그래도 (3.3)·(3.4) 근거가 여전히 적용되므로 일반 조회 추상화는 application 레이어로 이관하는 쪽을 권한다.

### 3.3 Drizzle의 성격상 view의 표현력 공백이 작다

이게 작년 글에서 가장 빠져 있던 자리다.

전통적인 ORM(TypeORM, Hibernate)은 엔티티-테이블 매핑 중심이라 복잡한 조인/집계를 ORM 레이어에서 표현하기 어려웠다. 그 표현력 공백을 view가 실용적 우회 수단으로 채워왔다. "ORM에서 표현 안 되는 SQL을 view에 묻어두고, ORM은 그 view를 한 테이블처럼 다룬다." 이게 view가 채우던 자리였다.

**Drizzle은 ORM이라기보다 "TypeScript로 표현된 DB"에 가까운 도구**다. `with` / `sql` 템플릿 / subquery / `$with`(CTE builder) 조합으로 복잡 쿼리를 그대로 표현할 수 있다. 표현력 공백 자체가 작다.

view의 필요성이 느껴지면, 먼저 **Drizzle 쿼리 빌더 기능을 충분히 활용하지 않고 있다는 신호**로 해석하는 게 맞다. 재사용이 필요하면 view 대신 reusable query factory / service 함수 / `sql` 템플릿으로 풀 수 있다. 권한 처리가 필요하면 RLS 또는 application 레이어로 푼다.

### 3.4 책임과 관찰 가능성을 application 레이어에 집중

로그·트레이싱·에러 추적·테스트·쿼리 탐색 흐름이 모두 TypeScript 레이어 중심이다. 조회 정의 일부를 DB 메타데이터로 옮기면 로직 위치 탐색 비용, 디버깅 경로 분산, application과 DB 간 책임 경계 증가가 발생한다.

이건 도구 성숙도와 무관한, **팀 워크플로우의 무게중심** 문제다. 무게중심이 application 레이어에 있는 한, 데이터 페칭과 조회 조합 책임도 거기에 두는 게 일관된다.

## 4. 예외는 의식적으로 좁게

view가 분명히 유효한 영역이 있다. 본 정책의 미사용 대상이 아니다.

- **Materialized View** — 결과를 물리적으로 저장한다. 실행 비용을 실제로 줄인다. 갱신 정책만 정해지면 강력한 도구.
- **RLS 보조 view** — Row-Level Security와 결합해 권한 경계를 SQL 레이어에서 처리하는 경우.
- **BI / 리포팅** — 분석가가 직접 SQL을 짜는 환경에서, 안정된 컬럼 셋을 view로 노출하는 것은 합리적인 계약이다.
- **레거시 호환 계층 / DB 계약 안정화** — 외부에 노출되는 인터페이스를 view로 고정하고 그 뒤를 자유롭게 리팩터링하는 패턴.
- **다중 서비스 간 공통 projection** — 여러 서비스가 같은 모양으로 읽어야 할 때, view가 자연스러운 공유 단위가 된다.

일반 조회 로직까지 예외로 늘리면 기준이 흐려지고 view와 application 쿼리가 혼재된다. 예외는 의식적으로 좁게 유지한다.

---

## 5. 정책 재검토 트리거

이 정책은 시점과 도구 성숙도에 묶여 있다. 다음 중 하나가 바뀌면 재평가한다.

- drizzle-kit이 view 의존성 추적과 안전한 마이그레이션 자동 생성을 지원하면 (3.1)의 무게가 크게 약해진다.
- 팀 규모/구성이 바뀌어 DBA-개발자 분리가 의미 있게 작동하면 (3.4)의 전제가 흔들린다.
- BI나 분석가 협업 비중이 커지면 예외 영역이 넓어진다.

이걸 미리 적어두는 이유는, *왜 미사용인가*를 잊어버린 채 정책만 남으면 컨텍스트가 바뀐 뒤에도 같은 정책을 고수하게 되기 때문이다. 정책의 유효 기간을 정책과 함께 적어두는 편이 정직하다.

## 마치며: "사용하지 마라"와 "기본 미사용"의 무게 차이

작년에 적었던 "사용하지 마라"와 이번에 적은 "기본 미사용"은 결론은 비슷하지만 무게가 다르다. 첫 번째는 컨텍스트가 바뀌어도 그대로 남는 명령이고, 두 번째는 컨텍스트가 바뀌면 재평가하기로 약속한 정책이다.

[지난번 ORM 논쟁 글](/docs/dev/database/reading-two-orm-debates)에서 좋은 기술 글쓰기의 4원칙으로 컨텍스트 명시 / 반론 진지하게 다루기 / 한계 인정 / 은탄환 부정을 뽑아냈다. 작년의 나는 마지막 항목을 어겼다. "사용하지 마라"라고 단정하고, 그 단정이 어떤 컨텍스트에서 유효한지를 적지 않았다.

view에 관한 정책 하나에 이렇게 길게 쓴 이유는 단순하다. 단정은 빠르게 쓸 수 있지만 시간을 견디지 못한다. 정직한 정책은 길어지지만 컨텍스트가 바뀐 뒤에도 살아남는다. 작년의 한 줄짜리 명령이 1년 만에 같은 결론에 도달하기 위해 이만큼의 글을 다시 쓰게 한 셈이다.

<Callout type="success" title="요약 한 줄">
view를 기본 미사용으로 두는 이유는 "view가 나빠서"가 아니라 "지금 우리 스택에서 view가 채우던 자리를 다른 도구가 더 잘 채워서"다. 컨텍스트가 바뀌면 결론도 바뀐다.
</Callout>
