## 이게 뭔데

non-sargable. 처음 보면 무슨 외계어 같은데, 풀어 보면 별거 없다. **sargable = Search ARGument able**, 직역하면 "검색 인자로 쓸 수 있는". 즉 `WHERE` 절의 조건을 인덱스가 곧장 검색 인자로 받아 처리할 수 있느냐를 묻는 단어다. 그게 안 되면 non-sargable, 인덱스를 못 타는 쿼리가 된다.

핵심은 이거 하나다. **인덱스는 컬럼의 "원본 값"에 걸린다.** `created_at`에 인덱스를 만들면, DB는 `created_at`의 실제 값들(`2024-03-11 09:21:00` 같은)을 정렬해서 B-tree에 차곡차곡 쌓아둔다. 정렬돼 있으니까 "이 값 이상, 저 값 미만" 같은 범위를 이진 탐색으로 콕 집어 찾을 수 있다. 이게 인덱스가 빠른 이유다.

그런데 조건을 `WHERE YEAR(created_at) = 2024`처럼 쓰는 순간, 우리는 인덱스가 알고 있는 "원본 값"이 아니라 **거기에 함수를 씌운 가공값**을 묻고 있다. 인덱스에는 `YEAR(created_at)` 값이 정렬돼 있지 않다. `created_at` 원본만 정렬돼 있을 뿐이다. 그래서 DB는 별수 없이 모든 행을 하나씩 꺼내, 매 행마다 `YEAR()`를 돌려보고, 결과가 2024인지 비교한다. 그게 바로 풀스캔(Seq Scan)이다.

전형적인 형태는 둘이다.

- **인덱스 컬럼을 함수로 감싸기** — `YEAR(created_at) = 2024`, `LOWER(email) = ...`, `SUBSTR(phone, 1, 3) = '010'`. 컬럼이 함수 안에 들어가는 순간 그 인덱스는 죽는다.
- **선행 와일드카드 LIKE** — `LIKE '%검색어%'`처럼 `%`로 시작하는 패턴. B-tree는 "앞에서부터" 정렬돼 있어서 시작점을 못 잡는다.

<Callout type="warning" title="한 줄 요약">
인덱스는 컬럼 원본 값에 걸린다. 조건에서 컬럼을 함수로 감싸거나 가공하는 순간, 인덱스가 알던 값과 달라져서 옵티마이저는 인덱스를 버리고 풀스캔으로 추락한다.
</Callout>

## 시나리오: 이런 적 있을 거임

대시보드에 "올해 가입자 수"를 띄운다. `created_at`에는 인덱스가 잘 걸려 있다. 자신 있게 친다.

```sql
SELECT COUNT(*) FROM users WHERE YEAR(created_at) = 2024;
```

로컬에선 0.01초. 인덱스도 있으니 당연히 빠를 줄 안다. PR도 무사히 머지된다. 화면도 멀쩡히 잘 뜬다.

그런데 운영에 올라가니까 이 대시보드 한 칸이 3초씩 걸린다. "인덱스 걸려 있는데 왜 느리지?" 하고 `EXPLAIN`을 찍어본다.

```text
                          QUERY PLAN
----------------------------------------------------------------
 Aggregate  (cost=185000.00..185000.01 rows=1)
   ->  Seq Scan on users  (cost=0.00..182500.00 rows=1000000)
         Filter: (EXTRACT(year FROM created_at) = 2024)
```

**Seq Scan.** 인덱스가 분명히 있는데도 안 탄다. 이유는 위에서 본 그대로다. 인덱스는 `created_at` **원본 값**으로 정렬돼 있는데, `YEAR(created_at) = 2024`를 판단하려면 **모든 행을 꺼내 함수를 돌려봐야** 알 수 있으니, 옵티마이저가 인덱스를 포기하고 600만 행을 통째로 긁은 거다.

그리고 이게 한 군데서만 나는 사고가 아니다. 로그인 쿼리에서도 똑같은 일이 벌어진다. 대소문자 구분 없이 이메일을 찾으려고 이렇게 친다.

```sql
SELECT * FROM users WHERE LOWER(email) = LOWER('Hong@example.com');
```

`email`에 유니크 인덱스가 떡하니 걸려 있어도 못 탄다. `LOWER(email)`로 감쌌으니까. 인덱스는 `Hong@example.com`을 알지 `hong@example.com`을 모른다. 로그인은 모든 사용자가 매번 때리는 길목인데, 거기서 풀스캔이 돌고 있는 거다. 트래픽 몰리는 순간 커넥션 풀이 말라붙는다.

풀스캔을 부른 건 쿼리의 목적이 아니라, 그걸 짠 손버릇이다.

- "created_at에 인덱스 걸었으니 created_at 조건은 다 빠르겠지" — 인덱스는 **원본 값**에 걸린다는 걸 잊음. 컬럼 이름만 들어가면 인덱스를 탄다고 착각한다.
- "YEAR()로 감싸는 게 읽기 편하잖아" — **가독성을 위해 컬럼을 가공**하는 손버릇. `YEAR(x) = 2024`가 범위 두 줄보다 한눈에 읽히긴 한다. 근데 읽기 편한 만큼 인덱스가 죽는다.
- "느리면 인덱스를 더 추가하지 뭐" — **진단 없이 인덱스만 늘림**. 정작 원인은 멀쩡히 있는 인덱스를 함수로 무력화한 건데, EXPLAIN 한 번 안 찍고 인덱스만 자꾸 더 만든다. 안 타는 인덱스를 하나 더 만들어봐야 디스크만 먹는다.

이 셋 중 하나라도 짚이는 게 있다면, 당신은 이미 non-sargable 쿼리를 한 번쯤 운영에 올려봤을 가능성이 높다.

## 무슨 일이 벌어지나 (비용)

왜 인덱스가 죽는지, 세 갈래로 뜯어보자.

**1. 함수로 감싼 컬럼 → 인덱스 무시, 풀스캔으로 추락.** 위에서 본 메인 케이스다. B-tree 인덱스는 `created_at` 원본 값을 정렬해 둔다. 정렬돼 있으니 "이 값부터 저 값까지"를 이진 탐색으로 범위 스캔할 수 있다. 그런데 `YEAR(created_at)`은 인덱스에 없는 값이다. 옵티마이저는 이 함수의 결과를 미리 알 방법이 없으니, **모든 행을 읽어서 한 행씩 `YEAR()`를 계산**하는 수밖에 없다. 행 100만이면 함수 호출 100만 번 + 디스크 풀스캔이다. 인덱스를 잘 걸어놓고 스스로 끈 셈이다.

**2. 선행 와일드카드 `LIKE '%x'` → 시작점을 못 잡음.** B-tree는 사전 순으로 정렬돼 있어서 `LIKE 'abc%'`(앞이 고정)는 인덱스를 탄다. "abc로 시작하는 구간"의 시작점을 찾아 거기서부터 쭉 읽으면 되니까. 하지만 `LIKE '%abc'`나 `LIKE '%abc%'`처럼 `%`가 앞에 붙으면, "어디서부터 읽기 시작할지"를 정할 수가 없다. 사전에서 "끝이 -son인 단어"를 찾으라는 거랑 같다. 처음부터 끝까지 다 봐야 한다. 그래서 선행 와일드카드 검색은 풀스캔이다. (텍스트 검색 자체를 LIKE로 때우는 더 큰 문제는 시리즈 다른 편 주제고, 여기선 "인덱스를 못 탄다"는 각도만 본다.)

**3. 암묵적 형변환도 같은 함정.** 이건 더 음흉하다. 함수를 눈에 보이게 쓴 것도 아닌데 인덱스가 죽는다. 예를 들어 `phone`이 `VARCHAR` 컬럼인데 `WHERE phone = 01012345678`처럼 **숫자로** 비교하면, DB는 타입을 맞추려고 내부적으로 컬럼 쪽을 숫자로 캐스팅한다. 즉 `WHERE CAST(phone AS BIGINT) = 01012345678`이 돼버린다. 컬럼에 보이지 않는 함수가 씌워진 거라, 결과는 똑같이 풀스캔. 게다가 `01012345678`은 숫자로 해석되면 앞의 0이 날아가는 부작용까지 따라온다.

<Callout type="error" title="뭐가 문제냐면">
- **인덱스가 멀쩡한데도 풀스캔이다**: "인덱스 없어서 느린 거 아냐?" 하고 인덱스를 확인하면 멀쩡히 있다. 그래서 원인을 못 찾고 헤맨다. 문제는 인덱스가 아니라 **쿼리가 그 인덱스를 못 타게 짜여 있다**는 거다.
- **로컬에선 안 보인다**: 개발 DB는 데이터가 적어서 풀스캔이든 인덱스 스캔이든 0.01초다. 운영에 600만 행 쌓여야 3초로 터진다.
- **인덱스를 더 걸어도 안 낫는다**: 원인이 함수 래핑이라, 컬럼에 인덱스를 하나 더 추가해도 똑같이 못 탄다. 진짜 약은 쿼리를 바꾸거나 함수 기반 인덱스를 거는 거다.
</Callout>

## 정석: 컬럼을 가공하지 말고 조건을 옮긴다

원칙은 하나로 요약된다. **컬럼을 건드리지 말고, 가공을 반대쪽(상수쪽)으로 옮겨라.** `함수(컬럼) = 상수`를 `컬럼 = 역함수(상수)`나 `컬럼 BETWEEN ... AND ...` 형태로 바꾸는 거다. 그러면 컬럼이 원본 그대로 남으니 인덱스가 살아난다.

**날짜 — 함수 대신 범위로.** `YEAR(created_at) = 2024`는 결국 "2024년 1월 1일 0시부터 2025년 1월 1일 0시 직전까지"라는 범위다. 그걸 그대로 범위 조건으로 풀어 쓰면 된다.

```sql
-- 컬럼을 가공하지 않는다 → 인덱스가 산다
WHERE created_at >= '2024-01-01'
  AND created_at <  '2025-01-01'
```

`<= '2024-12-31'`이 아니라 `< '2025-01-01'`을 쓰는 게 포인트다. 끝을 "다음 구간의 시작 미만"으로 잡으면 시/분/초나 타임존, 윤초 같은 경계 문제를 통째로 피해 간다. `BETWEEN`은 양 끝이 포함이라 마지막 날 자정~23:59:59 사이 데이터를 놓치기 쉬우니, 날짜 범위에는 `>= AND <` 패턴이 안전하다.

**대소문자 — citext 또는 함수 기반 인덱스.** 매번 `LOWER(email)`로 감싸는 게 손버릇이라면, 두 가지 정공법이 있다.

```sql
-- 방법 A) 컬럼 자체를 대소문자 무시 타입으로 (PostgreSQL)
CREATE EXTENSION IF NOT EXISTS citext;
ALTER TABLE users ALTER COLUMN email TYPE citext;
-- 이제 WHERE email = 'Hong@example.com' 이 알아서 대소문자 무시 + 인덱스 사용

-- 방법 B) 가공값에 인덱스를 직접 건다 (함수 기반 인덱스)
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
-- 이제 WHERE LOWER(email) = LOWER('Hong@example.com') 이 이 인덱스를 탄다
```

방법 B의 발상이 중요하다. **인덱스를 못 타는 게 함수 때문이라면, 그 함수 결과에 인덱스를 걸어버리면 된다.** `LOWER(email)`로 검색할 거면 `LOWER(email)`에 인덱스를 만들어 두는 거다. 그러면 쿼리는 그대로 둬도 인덱스를 탄다. 마찬가지로 "올해 가입자" 같은 게 정말 자주 필요하고 범위로 못 바꾸는 사정이 있다면 `CREATE INDEX ... ON users (EXTRACT(year FROM created_at))`도 가능하다. 다만 함수 기반 인덱스는 쿼리에 쓴 함수 표현식과 **정확히 일치**해야 타니까, 양쪽 표현을 맞춰 줘야 한다.

**접미사/부분 검색 — trigram GIN.** `LIKE '%검색어%'`나 `LIKE '%suffix'`처럼 앞이 안 고정된 검색이 진짜로 필요하면, B-tree로는 답이 없다. PostgreSQL이라면 `pg_trgm` 확장의 trigram GIN 인덱스를 쓴다.

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING gin (name gin_trgm_ops);
-- 이제 WHERE name LIKE '%hong%' 도 인덱스를 탄다
```

trigram은 문자열을 세 글자 단위 조각으로 쪼개 색인하기 때문에, 중간이나 끝에 걸린 패턴도 후보를 좁혀서 찾을 수 있다. 접미사만 필요하다면 컬럼을 뒤집어(`REVERSE(email)`) 함수 기반 인덱스를 걸고 `LIKE` 패턴도 뒤집어 거는 고전적인 트릭도 있다.

<Callout type="info" title="순서: 쿼리 고치기 → 함수 인덱스 → 특수 인덱스">
대부분의 non-sargable은 쿼리를 범위 조건으로 바꾸는 것만으로 공짜로 해결된다(인덱스 추가 비용 0). 그게 안 될 때만 함수 기반 인덱스를 고려하고, LIKE 부분검색처럼 구조적으로 B-tree가 못 하는 경우에만 trigram/GIN 같은 특수 인덱스로 간다. 인덱스를 늘리기 전에 쿼리부터 의심하자.
</Callout>

## 이렇게 하면 됨

습관 하나만 들이면 된다. **"인덱스 걸었는데 느리다" 싶으면 인덱스를 더 만들기 전에 `EXPLAIN`부터 찍는다.** Seq Scan이 보이면 그 줄의 `Filter`를 보라. 거기에 `EXTRACT`, `LOWER`, `CAST` 같은 함수가 컬럼을 감싸고 있으면 그게 범인이다.

날짜 케이스를 Before/After로 비교해 보자.

```sql
-- Before: 컬럼을 YEAR()로 감쌈 → non-sargable
EXPLAIN
SELECT COUNT(*) FROM users WHERE YEAR(created_at) = 2024;
```

```text
 Aggregate  (cost=185000.00..185000.01 rows=1)
   ->  Seq Scan on users  (cost=0.00..182500.00 rows=1000000)
         Filter: (EXTRACT(year FROM created_at) = 2024)
 Execution Time: 2980.412 ms
```

```sql
-- After: 범위 조건으로 → sargable, 컬럼이 원본 그대로
EXPLAIN
SELECT COUNT(*) FROM users
WHERE created_at >= '2024-01-01'
  AND created_at <  '2025-01-01';
```

```text
 Aggregate  (cost=4210.00..4210.01 rows=1)
   ->  Index Only Scan using idx_users_created_at on users
         (cost=0.43..4100.00 rows=120000)
         Index Cond: (created_at >= '2024-01-01' AND created_at < '2025-01-01')
 Execution Time: 31.880 ms
```

`Seq Scan` → `Index Scan`(여기선 Index Only Scan)으로 바뀌고, `Filter`였던 조건이 `Index Cond`로 올라간 게 보인다. **Filter는 "행을 다 꺼낸 뒤 거른다", Index Cond는 "인덱스로 범위를 콕 집는다"**는 뜻이다. 실행 시간은 2980ms에서 31ms로, 거의 100배다. 쿼리 한 줄을 두 줄로 풀어 쓴 것 말고는 인덱스도 그대로, 데이터도 그대로다.

로그인 케이스도 같다. 쿼리를 못 바꾸면(이미 코드 여기저기서 `LOWER(email)`로 부른다면) 함수 기반 인덱스로 막는다.

```sql
-- Before: LOWER(email)에 인덱스가 없어서 Seq Scan
SELECT * FROM users WHERE LOWER(email) = LOWER('Hong@example.com');
--   ->  Seq Scan on users   Filter: (lower(email) = 'hong@example.com')

-- 인덱스를 가공값에 건다
CREATE INDEX idx_users_email_lower ON users (LOWER(email));

-- After: 같은 쿼리가 이제 인덱스를 탄다
SELECT * FROM users WHERE LOWER(email) = LOWER('Hong@example.com');
--   ->  Index Scan using idx_users_email_lower   Index Cond: (lower(email) = 'hong@example.com')
```

핵심은 EXPLAIN을 진단 도구로 쓰는 거다. "느리다 → 인덱스 추가"라는 반사신경 대신, "느리다 → EXPLAIN → 왜 인덱스를 안 타는지 확인 → 원인에 맞게 처방"으로 가야 한다. 안 타는 인덱스를 백 개 만들어봐야 운영 DB만 무거워진다.

<Callout type="note" title="설계 단계에서 막는 습관">
조건에 컬럼이 들어가는 자리에 함수를 씌우려는 손이 올라가면, 잠깐 멈추고 "이걸 상수 쪽으로 옮길 수 있나?"를 먼저 묻자. 날짜는 범위로, 형변환은 리터럴 타입을 컬럼에 맞춰서, 대소문자는 데이터 모델(citext)이나 함수 인덱스로. 가공은 가능한 한 컬럼이 아니라 비교 대상 쪽에서 한다 — 이 한 문장이 non-sargable 사고의 대부분을 예방한다.
</Callout>

## 정리

non-sargable 쿼리는 악의로 만들어지지 않는다. 인덱스는 멀쩡히 걸려 있고, 쿼리도 "읽기 편하게" 짠 것뿐이다. `YEAR(created_at) = 2024`는 누가 봐도 한눈에 읽힌다. 문제는 그 한눈에 읽히는 가공이 인덱스가 알던 원본 값을 비틀어버린다는 거다.

> **인덱스는 컬럼 원본에 걸린다. 컬럼을 건드리는 순간 인덱스도 같이 죽는다.**

그러니 `WHERE` 절에서 컬럼을 함수로 감싸고 싶어지면, 그 가공을 반대편(상수/리터럴 쪽)으로 옮길 수 있는지부터 보자. 날짜는 범위로, 대소문자는 citext나 함수 기반 인덱스로, 부분 검색은 trigram으로. 그리고 "인덱스 있는데 느리다"는 순간 인덱스를 더 만들기 전에 `EXPLAIN`을 찍어 Seq Scan과 Filter를 확인하는 습관. 이 두 가지면 못 타는 쿼리의 대부분이 사라진다.

다음 편에서는 방향이 정반대인 실수를 본다. 성능을 미리 챙긴답시고 멀쩡한 정규화를 풀어헤치는 일 — 데이터를 여기저기 복사해 두고는 그 사본들이 서로 어긋나는 순간 지옥문이 열리는, 성급한 비정규화 얘기다.
