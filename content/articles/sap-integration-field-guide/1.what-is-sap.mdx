# SAP이 뭔지 최소한만 알기

*"SAP"이라는 이름 뒤에 뭐가 있는지부터 시작하자*

---

SAP 연동 프로젝트에 투입된 첫날, 회의에서 이런 문장이 날아온다.

> "ECC 쪽 BAPI로 PR 받고, MDG에서 자재 마스터 내려오면 PI로 중계해서 이쪽 SOAP 엔드포인트에 INBOUND로 쏠게요."

문장의 99%는 고유명사 아니면 약어다. 이걸 해석하려면 먼저 **SAP이라는 이름이 가리키는 것이 단일 시스템이 아니라는 사실**을 받아들여야 한다. SAP은 여러 레이어가 조합된 덩어리이고, 회의에서 등장하는 약어들은 대부분 그 레이어 중 하나의 이름이다.

이 글의 목표는 단 하나. **회의실에서 들리는 SAP 용어가 어느 레이어를 가리키는지 구분할 수 있을 정도의 지도**를 머릿속에 심는 것이다.

---

## ERP가 뭔지 (1분 복습)

SAP은 **ERP(Enterprise Resource Planning)** 소프트웨어의 대표 제품이다. ERP는 기업의 재무·구매·생산·물류·인사 등 거의 모든 업무 프로세스를 하나의 데이터 기반 위에서 관리하는 시스템을 가리킨다.

외부 개발자 관점에서 중요한 건 다음 두 가지다.

- **ERP 내부의 데이터는 강하게 연결돼 있다.** "구매 발주를 넣는다"는 행위 하나가 재무 장부, 재고, 원가 계산, 세금 신고, 예산 집계에 동시에 영향을 준다. 필드 하나만 잘못 보내도 여러 모듈이 어긋나는 이유가 여기 있다.
- **ERP는 태생적으로 독점 시스템이다.** 회사의 핵심 데이터가 전부 들어 있으므로 외부 시스템은 ERP에 직접 붙을 수 없고, 중간 계층을 경유하게 돼 있다. 그 중간 계층이 우리가 앞으로 매일 만나게 될 **PI/PO**다.

<Callout type="info" title="SAP이라는 회사와 SAP이라는 제품">
SAP은 독일 회사 이름이자 그 회사의 주력 제품군 전체를 부르는 이름이다. 1972년 독일 발도르프에서 창업했고, 전 세계 대기업 ERP 시장의 상당 부분을 차지한다. 회의에서 "SAP"이라고 하면 보통 "우리 회사가 쓰는 SAP ERP 시스템"을 가리킨다.
</Callout>

## 외부 개발자가 실제로 만나는 3개 레이어

우리 관점에서 SAP은 다음 3개로 나눠서 이해하면 된다. 이 셋의 역할 분리를 혼동하면 대화가 꼬인다.

```
┌────────────────────── SAP 전체 ──────────────────────┐
│                                                      │
│   ┌─────────┐     ┌─────────┐     ┌─────────────┐    │
│   │  ECC    │     │  MDG    │     │  PI/PO      │    │
│   │  (업무) │     │ (마스터)│     │  (미들웨어) │    │
│   └────┬────┘     └────┬────┘     └──────┬──────┘    │
│        │               │                 │           │
└────────┼───────────────┼─────────────────┼───────────┘
         │               │                 │
         └───────────────┴─────────────────┘
                         │
                         │  SOAP/XML over HTTPS
                         ▼
                  ┌─────────────┐
                  │  외부 시스템 │
                  │   (우리)    │
                  └─────────────┘
```

### ECC (ERP Central Component) — 업무 시스템

실제 업무 트랜잭션이 돌아가는 핵심 시스템이다. 발주를 일으키고, 송장을 처리하고, 재고를 이동시키는 일이 전부 여기서 일어난다. 대화에서 "ECC에서 PO를 생성해주세요" 하면 "실제 ERP 테이블에 발주 레코드를 만들어달라"는 뜻이다.

외부 개발자가 ECC에 직접 붙는 일은 거의 없다. 중간에 PI/PO가 서 있다.

### MDG (Master Data Governance) — 마스터 데이터 시스템

자재·벤더·직원·프로젝트·조직 같은 **기준 정보**를 중앙에서 관리하는 시스템이다. 이름에 "Governance"가 들어간 이유는 마스터 데이터의 생성·변경·폐기를 통제하는 워크플로가 여기 달려 있어서다.

회사에 따라 MDG를 도입하지 않고 ECC 내부에서 마스터를 관리하기도 한다. 우리 관점에서 중요한 건 "**마스터 데이터가 나오는 출처가 어디인지**"를 파악하는 것이다. 자재 마스터를 누가 보내주는지에 따라 연동 인터페이스의 소속 시스템이 달라진다.

### PI/PO (Process Integration / Process Orchestration) — 미들웨어

SAP과 외부 시스템 사이를 중계하는 **메시지 브로커**다. ECC 내부의 ABAP 코드나 RFC 호출을 외부에 그대로 노출하지 않고, PI/PO가 받아서 SOAP/XML로 바꿔서 보내준다. 반대 방향도 마찬가지다.

**외부 개발자가 실제로 통신하는 상대는 99% PI/PO다.** ECC 서버에 직접 HTTP 요청을 보내는 일은 없다. WSDL을 주고받고, SOAP Envelope를 조립하고, Basic Auth 헤더를 붙이는 모든 작업은 PI/PO를 상대로 한다.

같은 미들웨어 제품이 세대마다 이름을 바꿔 왔다. 운영자가 부르는 이름(XI, PI, PO, CPI 등)만 듣고는 그 회사가 실제로 어느 세대를 돌리고 있는지 추정이 어렵다. 아래 **SAP 미들웨어 이름의 계보** 섹션에서 계보를 정리한다.

## SAP 미들웨어 이름의 계보

SAP 미들웨어는 20여 년 동안 세 번 리브랜딩됐고, 그 사이에 기술 스택도 바뀌었다. **운영자가 쓰는 이름이 실제 시스템 세대와 일치하지 않는 경우가 많아서** 외부 개발자 입장에서는 혼란의 원인이 된다. "우리 XI 붙이면 됩니다"라는 말을 들었는데 실제 돌아가는 건 PO 7.5였던 식의 경험이 드물지 않다.

### 세대별 계보

| 이름 | 정식 | 출시 | 기술 스택 | 특징 |
|------|------|------|-----------|------|
| **XI** | Exchange Infrastructure | 2003 (NetWeaver '04) | Dual-stack (ABAP + Java) | SAP 최초의 통합 미들웨어. ESA(Enterprise Service Architecture) 비전의 핵심. |
| **PI** | Process Integration | 2006 (NetWeaver 7.0) | Dual-stack | 사실상 XI의 이름 변경. 기술 구조는 거의 동일. |
| **PI 7.3** | Process Integration 7.3 | 2011 | Dual-stack + AEX 옵션 | Java 전용 경량 구성(AEX, Advanced Adapter Engine eXtended) 도입. |
| **PO** | Process Orchestration | 2012 (NetWeaver 7.31) | **Single-stack** (Java only) | BPM·BRM(비즈니스 룰 엔진) 통합. ABAP 스택 제거. 많은 기업의 현재 버전. |
| **PI 7.5 Single Stack** | PI 7.5 (Java only) | 2015 | Single-stack (Java only) | PO와 기능적으로 유사. 이름은 PI지만 실체는 Java 전용. |
| **CPI** | Cloud Platform Integration | 2016 | 클라우드 네이티브 (HCI에서 리브랜딩) | PI/PO의 클라우드 후계자. iFlow 기반 설계. |
| **Integration Suite** | SAP Integration Suite | 2020 | 클라우드 (CPI 재리브랜딩) | BTP(Business Technology Platform) 산하로 편입. API Management·Open Connectors 등 통합. |

### 운영자의 말로 세대 추측하기

- **"XI"**: 2010년 이전에 도입한 회사가 관성으로 부르는 이름. 실제로는 PI나 PO로 업그레이드됐을 가능성이 높다.
- **"PI"**: 2006~2015년 어느 시점의 시스템. Dual-stack일 수도, Single-stack일 수도 있다.
- **"PO"**: Java 전용 single-stack. BPM/BRM 기능을 쓰는 경우가 많다. 2026년 현재 온프레미스 환경에서 가장 흔한 선택.
- **"CPI"**: 클라우드로 이미 넘어감. iFlow라는 단어가 같이 나오면 거의 확실.
- **"Integration Suite"** 또는 **"CI"**: CPI의 새 이름. 2020년 이후 도입 프로젝트.

<Callout type="note" title="이름이 중요한가">
외부 개발자 입장에서 **실제 통신 프로토콜은 XI·PI·PO·CPI 모두 SOAP over HTTPS로 동일**하다. 따라서 "어느 세대인지"가 코드 작성에 직접 영향을 주지는 않는다.

그래도 세대를 아는 것이 쓸모 있는 순간은:

- **담당자와의 소통**: "CPI iFlow에서 확인해주세요"와 "PO의 ICO(Integrated Configuration)에서 확인해주세요"는 전혀 다른 화면이다. 담당자가 어느 도구로 일하는지 파악하면 요청이 더 구체적으로 가능.
- **장애 진단 명령**: `SXI_MONITOR`는 dual-stack 시절의 ABAP T-Code다. Single-stack PO나 CPI에서는 웹 기반 Message Monitor를 쓴다. 7편에서 언급한 T-Code가 안 통하는 경우가 있다.
- **인증 방식**: CPI/Integration Suite는 Basic Auth 외에 OAuth2·인증서 기반 방식을 지원한다. 온프레미스 PI/PO는 Basic Auth가 기본.
</Callout>

### S/4HANA 시대의 미들웨어 선택

SAP은 ECC에서 **S/4HANA**로의 이관을 밀어붙이고 있고, 그와 함께 미들웨어의 기본 권장도 바뀌는 중이다.

- 온프레미스 레거시: 여전히 **PO 7.5**를 쓰는 기업이 많다. 2030년까지 지원 예정.
- 하이브리드: 온프레미스 PO + 클라우드 Integration Suite를 **병행 운영**하는 전환기 구성이 흔하다.
- 신규 S/4HANA Cloud: **Integration Suite를 기본 권장**. 온프레미스 PI/PO는 선택지에서 빠진다.

외부 시스템 개발자 입장에서 새 프로젝트라면 Integration Suite와 붙을 가능성이 높지만, 유지보수성 프로젝트는 여전히 PI/PO가 대부분이다. 이 시리즈의 내용은 **PI/PO/Integration Suite 어느 세대에도 적용**된다 — SOAP over HTTPS라는 공통 분모가 유지되기 때문.

## "모듈"이라는 SAP 고유 개념

SAP은 업무 영역별로 **모듈(Module)**이라는 단위로 나뉜다. 이 분할은 SAP 내부의 코드 구조이기도 하지만, 더 중요하게는 **담당자 조직**이기도 하다. "MM 담당자", "FI 담당자"라는 말은 "구매/자재 업무를 담당하는 SAP 컨설턴트", "재무 회계를 담당하는 SAP 컨설턴트"를 뜻한다.

외부 개발자가 가장 자주 만나게 되는 모듈은 다음과 같다.

| 모듈 | 정식 이름 | 역할 | 외부 연동 접점 |
|------|-----------|------|----------------|
| **MM** | Materials Management | 구매, 자재 관리 | 입찰, RFQ, 발주(PO), 자재 마스터 |
| **FI** | Financial Accounting | 재무 회계 | 세금 코드, 계정, 송장 |
| **CO** | Controlling | 관리 회계 | 코스트센터, 원가 배부 |
| **PS** | Project System | 프로젝트 관리 | WBS, 프로젝트 번호 |
| **SD** | Sales & Distribution | 영업, 유통 | 고객 마스터, 판매 오더 |
| **PP** | Production Planning | 생산 계획 | BOM, MRP, 생산 오더 |

대부분의 연동 프로젝트는 **MM 모듈 중심**이다. 구매·조달 업무가 외부 파트너와 데이터를 주고받을 일이 가장 많아서다. 회의에서 상대방이 "MM 쪽 I/F"라고 하면 구매 도메인이라는 뜻이다.

<Callout type="info" title="담당자 찾기 팁">
어떤 필드가 뭔지 모를 때 SAP 담당자에게 물어보려면 먼저 **그 필드가 어느 모듈 소속인지** 짐작해야 답을 받을 사람을 찾을 수 있다. `LIFNR`(벤더)·`MATNR`(자재)·`EBELN`(발주번호)는 MM, `MWSKZ`(세금 코드)·`SAKTO`(G/L 계정)는 FI, `KOSTL`(코스트센터)은 CO, `POSID`(WBS)는 PS다. 필드 접두사로 대강 모듈을 추론할 수 있다(2편에서 자세히).
</Callout>

## T-Code — SAP 담당자의 외계어

SAP의 모든 화면에는 **트랜잭션 코드(Transaction Code, 보통 T-Code)**가 붙어 있다. 4~5글자 코드로, 회의에서 SAP 담당자가 말하는 "ME21N 들어가서 확인했더니"의 `ME21N`이 바로 이것이다.

외부 개발자가 T-Code 화면에 접근할 일은 거의 없지만, **무슨 뜻인지 해석할 수 있어야 담당자와 대화가 된다.** 외워둘 값어치가 있는 것들만 추렸다.

| T-Code | 이름 | 설명 |
|--------|------|------|
| `ME51N` | PR 생성 | 구매요청(Purchase Requisition) 생성 화면 |
| `ME21N` | PO 생성 | 발주(Purchase Order) 생성 화면 |
| `ME23N` | PO 조회 | 발주 상세 조회 |
| `ME31K` / `ME33K` | 계약 생성/조회 | 단가계약(Contract) 처리 |
| `XK01` / `XK03` | 벤더 생성/조회 | Vendor Master 처리 |
| `MM03` | 자재 조회 | Material Master 조회 |
| `FTXP` | 세금 코드 관리 | Tax Code 설정 |
| `SE80` | ABAP Workbench | ABAP 개발 환경 |
| `SM50` | 워크프로세스 모니터 | 서버 프로세스 상태 조회 |
| `SXI_MONITOR` | PI 메시지 모니터 | 메시지 송수신 이력 (에러 디버깅에 필수) |
| `SPRO` | IMG | SAP 커스터마이징 전체 메뉴 |

<Callout type="success" title="담당자 소통 팁 — T-Code로 말하면 10배 빠르다">
에러 상황에서 "어제 들어온 구매요청이 이상한데 확인해주세요"보다는 **"ME53N에서 PR 번호 0010012345 봤는데 DISMM 값이 비어 있는 것 같다"**처럼 T-Code와 필드명을 명시해서 물으면 담당자 응답 속도가 체감 몇 배 빨라진다. 담당자는 그 코드로 화면을 바로 띄울 수 있으므로 메시지 한 번에 문제 지점을 본다.
</Callout>

## 약어 사전

회의실에서 들리는 SAP 약어의 정체를 정리했다. 처음 보면 다 똑같이 들리지만 구분 안 하면 대화가 꼬인다.

| 약어 | 정식 이름 | 정체 |
|------|-----------|------|
| **ABAP** | Advanced Business Application Programming | SAP 내부 프로그래밍 언어. ECC 코드는 전부 ABAP으로 작성됨. |
| **RFC** | Remote Function Call | SAP 내부의 함수 원격 호출 프로토콜. 보통 ECC ↔ PI 사이에서 사용. |
| **BAPI** | Business Application Programming Interface | SAP이 제공하는 표준 비즈니스 API. ABAP 함수 형태지만 외부 노출을 의도한 것. |
| **IDoc** | Intermediate Document | SAP 시스템 간 데이터 교환용 문서 포맷. 파일 기반 비동기 통신에 주로 사용. |
| **WSDL** | Web Services Description Language | 웹서비스 계약서. 외부 개발자가 PI/PO와 붙을 때 가장 먼저 받는 파일(5편에서 다룸). |
| **XI / PI / PO** | Exchange Infrastructure / Process Integration / Orchestration | SAP 미들웨어의 세대별 이름. 위 "미들웨어 이름의 계보" 섹션에서 자세히. |
| **CPI / Integration Suite** | Cloud Platform Integration → SAP Integration Suite | PI/PO의 클라우드 후계자. 2020년 리브랜딩. |
| **iFlow** | Integration Flow | CPI/Integration Suite에서 쓰는 통합 시나리오 정의 단위. PI/PO의 ICO에 대응. |
| **ICO** | Integrated Configuration | PI/PO에서 쓰는 통합 시나리오 설정. "PI 관리 화면에서 ICO 봐주세요"는 PI 담당자 말. |
| **ECC** | ERP Central Component | SAP의 전통 ERP 본체. |
| **S/4HANA** | SAP S/4HANA | ECC의 차세대 후계 제품. HANA DB 기반. |
| **MDG** | Master Data Governance | 마스터 데이터 관리 시스템. |
| **SPRO / IMG** | Implementation Guide | SAP 커스터마이징 설정 화면. |

<Callout type="note" title="ECC vs S/4HANA">
많은 기업이 여전히 레거시 ECC를 운영 중이지만, SAP은 2027년(+연장 옵션 2030년)까지 ECC 유지보수를 종료하고 S/4HANA로의 이관을 요구하고 있다. 외부 연동 관점에서 두 제품의 차이는 대부분 내부 데이터 모델과 성능이며, **SOAP/PI 경유 외부 인터페이스의 동작 방식은 거의 동일**하다. 이 시리즈의 내용은 양쪽에 공통으로 적용된다.
</Callout>

## 외부 개발자의 "실제 통신 그림"

위 레이어 설명을 우리 관점으로 단순화하면 이렇게 된다.

```
[ECC 모듈들: MM / FI / CO / PS / SD / PP]
                │
                │  RFC / IDoc / ABAP 호출
                ▼
          [PI/PO 미들웨어]
                │
                │  SOAP 1.1 over HTTPS + Basic Auth
                ▼
          [우리 시스템]
          - INBOUND API: PI/PO가 데이터를 보내옴
          - OUTBOUND 호출: 우리가 PI/PO에 데이터를 보냄
```

왼쪽 영역(ECC·모듈·RFC·ABAP)은 **SAP 담당자의 책임 영역**이다. 우리가 손댈 수 없고, 대부분의 경우 내부 동작을 알 필요도 없다. 필요한 건 "PI/PO까지 오는 SOAP 메시지가 어떻게 생겨야 하는가"뿐이다.

오른쪽 영역(PI/PO 이후)은 **우리의 책임 영역**이다. WSDL을 읽고, XML을 파싱하고, 응답을 해석하고, 에러를 처리한다. 이 시리즈의 나머지 7편은 전부 오른쪽 영역의 이야기다.

## 정리

- SAP은 단일 시스템이 아니라 **ECC / MDG / PI/PO** 같은 여러 레이어의 묶음이다.
- 외부 개발자가 통신하는 상대는 거의 항상 **PI/PO 미들웨어**다. ECC에 직접 붙는 일은 없다.
- **모듈**은 업무 영역이자 담당자 조직이다. 필드 질문을 할 때 어느 모듈 소속인지 짐작하면 답을 받을 사람을 찾을 수 있다.
- **T-Code**는 SAP 화면 식별자다. 담당자와 대화할 때 T-Code로 말하면 응답 속도가 압도적으로 빨라진다.
- 약어는 **XI/PI/PO는 같은 것의 세대 이름**이고, RFC·BAPI·IDoc은 SAP 내부 통신 수단, WSDL·SOAP은 외부 통신 수단이다.

다음 편은 SAP 필드 이름이 왜 이렇게 생겼는지를 다룬다. `LIFNR`, `MATNR`, `EBELN` 같은 암호 같은 문자열이 사실은 **독일어 약어**라는 이야기.
