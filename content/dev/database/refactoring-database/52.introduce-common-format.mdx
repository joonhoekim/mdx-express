## 이게 뭔데

Introduce Common Format은 **한 컬럼에 들어 있는 값들의 형식을 하나로 통일하는** 리팩토링이다. 같은 종류의 데이터인데 행마다 모양이 제각각인 컬럼을, 단 하나의 약속된 형식으로 맞춘다.

비유하자면 옷장 정리다. 양말이 한 서랍에 있긴 한데, 누구는 돌돌 말아 넣고, 누구는 반으로 접고, 누구는 그냥 던져 넣었다. 양말이 양말인 건 맞다. 근데 짝을 맞추려고 매번 서랍을 뒤집어야 한다. Introduce Common Format은 "이 서랍의 양말은 전부 돌돌 말아서 넣는다"로 규칙을 통일하는 일이다. 들어 있는 게 바뀌는 건 아니다. **들어가는 모양이 통일되는 거다.**

DB로 돌아오면, 가장 흔한 희생양은 전화번호다. `Customer.PhoneNumber` 컬럼을 열어보면 이런 게 한 서랍에 다 들어 있다.

```text
4163458899
905-777-8889
(990) 345-6666
+1 416 345 8899
416.345.8899
416 345 8899 (내선 102)
```

전부 멀쩡한 전화번호다. 그런데 형식이 죄다 다르다. 이러면 무슨 일이 벌어지냐면, 이 컬럼을 읽는 **모든 애플리케이션이 모든 형식을 다 파싱할 줄 알아야** 한다. 이게 바로 이 리팩토링이 노리는 통증이다.

<Callout type="info" title="한 줄 요약">
같은 데이터가 여러 형식으로 저장돼 있으면 그걸 읽는 모든 코드가 모든 형식을 파싱해야 한다. 저장 형식을 하나로 통일하면 검증·표시 로직이 단순해진다. 핵심은 "저장은 정규화 형식, 표시는 예쁜 형식"으로 둘을 분리하는 거다.
</Callout>

## 언제 쓰나

동기는 단순하다. **형식이 여러 개라서 다들 고생하고 있을 때**다. 구체적으로 어떤 냄새가 나면 이걸 꺼낼 때인지 보자.

첫째, **검증 코드가 형식마다 분기한다.** 전화번호가 유효한지 보려는데, 하이픈 버전도 받고, 괄호 버전도 받고, 점 버전도 받고, 국가코드 붙은 버전도 받아야 한다. 정규식이 점점 길어지고, 어느 날 누군가 "내선번호 붙은 건 왜 거부됨?" 하고 버그를 단다. 형식이 하나면 검증 패턴도 하나면 된다.

둘째, **표시할 때 매번 다시 가공해야 한다.** 화면에 `(416) 345-8899`로 예쁘게 보여주고 싶은데, 저장된 게 `416.345.8899`인지 `4163458899`인지 알 수가 없으니, 표시 코드가 먼저 "이게 무슨 형식이지?"부터 알아내야 한다. 저장 형식이 통일돼 있으면 표시 로직은 그냥 "이 형식을 받아 이렇게 꾸민다" 한 줄이다.

셋째, **조인이나 비교가 깨진다.** `Country.Code`가 `USA`인데 `Address.CountryCode`가 `US`이면 둘이 같은 나라인데도 조인이 안 붙는다. 전화번호도 마찬가지다. 같은 번호인데 `4163458899`와 `416-345-8899`는 문자열 비교에서 다른 값이라, "이 번호 중복인가?" 같은 질문에 DB가 거짓말을 한다.

넷째, **전사 표준이나 공식 데이터 소스를 따라야 할 때**도 이 리팩토링이 답이다. "전화번호는 E.164로 저장한다" 같은 규칙이 내려오면, 흩어진 형식을 그 규칙으로 수렴시키는 작업이 바로 이거다.

### 현실 시나리오

은행 고객센터 시스템을 떠올려 보자. `Customer` 테이블의 `PhoneNumber` 컬럼은 10년째 살아 있다. 그동안 입력 경로가 늘었다. 처음엔 콜센터 직원이 키보드로 직접 쳤고(`905-777-8889`), 나중엔 웹 가입 폼이 생겼고(`(905) 777-8889`), 그다음엔 모바일 앱이 붙었고(`+1 905 777 8889`), 어느 시점엔 외부 마케팅 업체 데이터를 일괄 임포트했다(`9057778889`). 입력 경로마다 형식 취향이 달랐고, 아무도 막지 않았다.

그러던 어느 날, 보험 상품 안내 문자(SMS)를 자동 발송하는 배치를 새로 만든다. SMS 게이트웨이는 번호를 `+19057778889` 같은 E.164 형식으로 달라고 한다. 배치 개발자는 별생각 없이 `SELECT PhoneNumber FROM Customer`를 긁어 게이트웨이로 넘긴다.

결과는 처참하다. 어떤 고객은 문자를 두 번 받고(`905-777-8889`와 `9057778889`가 다른 행으로 들어가 있어서), 어떤 고객은 아예 못 받는다(`(905) 777-8889`를 게이트웨이가 거부해서). 배치는 "성공"으로 끝났는데 발송 성공률이 절반이다. 원인을 추적하다 `PhoneNumber` 컬럼을 `SELECT DISTINCT`로 열어본 순간, 한 컬럼에 형식이 **15가지**가 섞여 있는 광경을 보게 된다. 책에 나오는 바로 그 장면이다.

이 순간 깨닫는다. 문제는 SMS 배치가 아니다. **컬럼 하나가 15개의 약속을 동시에 지키고 있던 게** 문제다. 그리고 그걸 푸는 게 Introduce Common Format이다.

## 주의할 점

작은 리팩토링처럼 보이지만, 함정이 두 개 있다.

<Callout type="warning" title="형식이 N가지면 변환기도 N가지를 알아야 한다">
컬럼에 형식이 15가지 섞여 있으면, "어떤 행이 어떤 형식인지"를 먼저 탐지하고 각각을 정규화하는 코드가 필요하다. 단순 `REPLACE` 한 방으로 안 끝나는 경우가 많다. 국가코드 `+1`이 붙은 행, 내선번호 `(내선 102)`가 붙은 행, 숫자가 7자리뿐인 지역번호 누락 행은 따로 처리해야 한다. **다행인 건, 이 변환 코드가 보통 이미 어딘가에 있다는 거다.** 기존 애플리케이션이 화면 표시나 외부 연동을 위해 이미 형식을 정리하는 코드를 갖고 있을 가능성이 높다. 그걸 끌어와 마이그레이션에 재사용하면 된다.
</Callout>

<Callout type="warning" title="정규화하다가 정보를 버리지 마라">
숫자만 남기겠다고 `REPLACE`로 비숫자를 다 날리면, 내선번호·국가코드 같은 의미 있는 정보가 같이 증발한다. `416 345 8899 (내선 102)`를 `4163458899102`로 만들면 내선이 번호에 합쳐져 완전히 망가진다. 어떤 행은 정규화 전에 "이건 내선이니까 별도 컬럼으로 빼자" 같은 판단이 필요하다. 즉 이 리팩토링은 종종 **데이터 정제(cleansing)와 한 묶음**으로 간다. 변환 규칙을 정하기 전에 `SELECT DISTINCT`로 실제 형식 분포를 반드시 먼저 확인해라.
</Callout>

한 가지 더. 형식을 통일한다고 **표시 형식까지 통일하라는 게 아니다.** 사람이 보는 화면에 `+14163458899`를 그대로 띄우면 누가 봐도 못생겼다. 핵심 원칙은 따로 있다.

> **저장 형식과 표시 형식은 다른 문제다. 저장은 기계가 좋아하는 정규화 형식으로, 표시는 사람이 좋아하는 형식으로.**

저장은 `4163458899`(혹은 E.164 `+14163458899`) 하나로 통일하되, 화면에 그릴 땐 `(416) 345-8899`로 포맷팅해서 보여준다. 저장 형식이 통일돼 있으니 표시 포맷팅은 단방향 함수 한 개로 끝난다. 이게 이 리팩토링의 진짜 이득이다.

## 이렇게 한다

전체 흐름은 네 단계다. 형식을 정하고, 데이터를 변환하고, 컬럼에 제약을 다시 걸고, 코드를 고친다.

<Steps>
<Step title="공식 형식을 확정하고 컬럼을 식별한다">

먼저 "이 컬럼은 앞으로 이 형식으로만 저장한다"를 합의해야 한다. 전화번호라면 보통 두 후보다.

- **숫자만(`4163458899`)** — 단순하고 비교가 빠르다. 국내 단일 국가만 다룰 때 충분.
- **E.164(`+14163458899`)** — 국가코드를 포함해 국제 표준. 국제 발송·다국가 서비스라면 이쪽이 정답.

은행 도메인이 캐나다·미국을 다 다룬다고 치면 E.164로 가는 게 미래에 덜 운다. 형식을 정했으면, **한 번에 한 컬럼씩** 처리한다. `Customer.PhoneNumber` 먼저, 그다음 릴리스에 `Customer.FaxNumber`, 또 그다음에 `Branch.Phone`. 작은 리팩토링이 배포하기도, 롤백하기도 쉽다.

</Step>
<Step title="현재 형식 분포를 파악한다">

변환 규칙을 짜기 전에, 적이 몇 종류인지부터 센다.

```sql
-- 어떤 형식들이 섞여 있나? 정규화 후 길이별로 묶어 분포를 본다
SELECT
  LENGTH(REGEXP_REPLACE(PhoneNumber, '[^0-9]', '')) AS digit_count,
  COUNT(*) AS cnt
FROM Customer
WHERE PhoneNumber IS NOT NULL
GROUP BY digit_count
ORDER BY digit_count;
```

여기서 `digit_count`가 10이면 표준 북미 번호, 11이면 국가코드 `1`이 붙은 번호, 7이면 지역번호 누락, 13 이상이면 내선번호가 끼어든 행일 가능성이 높다. 이 분포가 변환 코드의 분기 목록이 된다. 무턱대고 `REPLACE`부터 돌리지 말고, **먼저 적의 얼굴을 다 봐라.**

</Step>
<Step title="데이터를 변환한다 (DML)">

소량이면 SQL 스크립트 한 방, 대량·복잡하면 애플리케이션 코드나 ETL로 한 행씩 돌린다.

</Step>
<Step title="제약을 다시 걸고 접근 프로그램을 고친다">

형식이 통일됐으면 이제 "앞으로도 이 형식만 들어오게" 제약을 걸 수 있다. 그리고 코드의 정제·검증·표시 로직을 새 형식 기준으로 정리한다.

</Step>
</Steps>

### 데이터 마이그레이션 (DML)

가장 단순한 경우, 즉 형식이 "구분자만 다른 10자리 번호"들이라면 SQL `REPLACE`/`REGEXP_REPLACE`로 비숫자를 털어버리면 된다.

```sql
-- Before: '905-777-8889', '(905) 777-8889', '905.777.8889' 가 섞여 있음
-- After:  전부 '9057778889' (숫자만)

UPDATE Customer
SET PhoneNumber = REGEXP_REPLACE(PhoneNumber, '[^0-9]', '')
WHERE PhoneNumber IS NOT NULL;
```

하지만 현실은 이렇게 안 곱게 안 끝난다. 국가코드와 내선번호가 섞여 있으면 `REGEXP_REPLACE` 한 방은 위험하다(내선이 번호 뒤에 들러붙는다). 이럴 땐 변환을 **저장 프로시저나 애플리케이션 함수로 캡슐화**해서, 행마다 형식을 판단하고 변환한다. 책에서 `FormatPhoneNumber()`로 캡슐화하라고 권하는 게 이 맥락이다.

```sql
-- 형식별 분기를 담은 정규화 함수 (PostgreSQL 예시)
CREATE OR REPLACE FUNCTION normalize_phone(raw TEXT)
RETURNS TEXT AS $$
DECLARE
  digits TEXT;
BEGIN
  IF raw IS NULL THEN
    RETURN NULL;
  END IF;

  -- 내선번호 '(내선 102)' 같은 꼬리표는 정규화 대상에서 떼어낸다(여기선 버림)
  raw := REGEXP_REPLACE(raw, '\(내선.*\)', '', 'g');

  -- 숫자만 추출
  digits := REGEXP_REPLACE(raw, '[^0-9]', '', 'g');

  -- 10자리 북미 번호 → 국가코드 1 붙여 E.164
  IF LENGTH(digits) = 10 THEN
    RETURN '+1' || digits;
  -- 이미 국가코드 1이 붙은 11자리
  ELSIF LENGTH(digits) = 11 AND LEFT(digits, 1) = '1' THEN
    RETURN '+' || digits;
  ELSE
    -- 규칙에 안 맞는 행은 변환하지 말고 리포트 대상으로 남긴다
    RETURN NULL;
  END IF;
END;
$$ LANGUAGE plpgsql IMMUTABLE;
```

이렇게 두면 변환은 한 줄이 되고, **규칙에 안 맞는 행은 `NULL`로 떨어져 따로 잡힌다.** 자동 변환이 불가능한 쓰레기 데이터는 사람이 봐야 한다.

```sql
-- 1) 변환 불가 행을 먼저 리포트해서 사람이 판단
SELECT CustomerOID, PhoneNumber
FROM Customer
WHERE PhoneNumber IS NOT NULL
  AND normalize_phone(PhoneNumber) IS NULL;

-- 2) 변환 가능한 행만 안전하게 갱신
UPDATE Customer
SET PhoneNumber = normalize_phone(PhoneNumber)
WHERE normalize_phone(PhoneNumber) IS NOT NULL;
```

<Callout type="note" title="대량 데이터면 한 행씩 락 vs 전체 락">
데이터 품질 리팩토링은 갱신 전략을 골라야 한다. (1) 테이블 전체를 락 걸고 한 방에 `UPDATE` — 일관성은 완벽하지만 대량이면 그동안 서비스가 멈춘다. (2) 배치로 잘게 쪼개 한 청크씩(`WHERE CustomerOID BETWEEN ...`) 갱신 — 서비스는 살아 있지만 변환 도중엔 정규화된 행과 안 된 행이 공존한다. 운영 중인 테이블이면 거의 항상 (2)다. 그래서 이 시점엔 코드가 두 형식을 다 읽을 수 있어야 한다(아래 expand-contract).
</Callout>

### 무중단으로 하기 — expand-contract (parallel change)

운영 중인 컬럼을 통째로 `UPDATE`하는 동안, 그 컬럼을 읽는 코드가 옛날 형식을 가정하고 있으면 마이그레이션 중간에 깨진다. 그래서 무중단이 필요하면 한 번에 갈아엎지 않고 **expand-contract**로 단계를 나눈다.

<Steps>
<Step title="Expand — 읽기를 형식에 관대하게 만든다">

먼저 배포하는 건 스키마가 아니라 **코드**다. 컬럼을 읽는 모든 지점이 "옛 형식이든 새 형식이든 다 받아서 정규화해 쓰는" 상태로 만든다. 즉 읽을 때 항상 `normalize_phone()`를 한 번 통과시킨다. 이제 데이터가 어떤 형식이든 코드는 안 깨진다.

</Step>
<Step title="Migrate — 백필을 천천히 돌린다">

코드가 관대해진 뒤, 위의 청크 배치로 데이터를 새 형식으로 백필한다. 도중에 형식이 섞여 있어도 1단계 덕분에 읽기는 멀쩡하다. 새로 들어오는 데이터도 애플리케이션 쓰기 경로에서 정규화해 저장하게 바꿔 둔다.

</Step>
<Step title="Contract — 제약을 걸고 관대함을 거둔다">

모든 행이 새 형식이 된 걸 확인하면, CHECK 제약으로 형식을 못박고, 읽기 경로의 "옛 형식도 받아주던" 관대한 코드를 걷어낸다. 이제 컬럼은 단 하나의 형식만 약속한다.

마이그레이션 자체는 Flyway/Liquibase/Alembic 같은 도구로 버전 관리하는 게 좋다. 위의 `normalize_phone` 함수 생성과 백필 `UPDATE`를 각각 버전 매겨진 마이그레이션 스크립트로 두면, 어느 환경이 어디까지 정규화됐는지 추적된다.

### 스키마 변경 (DDL) — 제약으로 형식 못박기

데이터가 다 통일됐으면, 앞으로 다시 더러워지지 않게 CHECK 제약을 건다. PostgreSQL에서 운영 테이블에 제약을 거는 안전한 방법은 `NOT VALID`로 먼저 추가하고 나중에 검증하는 거다.

```sql
-- 1) 신규 쓰기에만 즉시 적용, 기존 행 전수 검사는 미룬다 (락 짧음)
ALTER TABLE Customer
  ADD CONSTRAINT Check_Phone_Format
  CHECK (PhoneNumber ~ '^\+1[0-9]{10}$') NOT VALID;

-- 2) 백필이 끝난 뒤, 기존 행을 전수 검증 (이때 못 통과하는 행이 있으면 잡힌다)
ALTER TABLE Customer
  VALIDATE CONSTRAINT Check_Phone_Format;
```

이렇게 하면 1단계에서 테이블을 길게 락 걸지 않고도 "지금부터 들어오는 값"에 형식을 강제할 수 있고, 2단계에서 과거 데이터를 안전하게 확인한다.

저장 형식을 믿을 수 없으니, 읽을 때마다 형식을 추측하고, 검증 정규식은 모든 변형을 받아줘야 한다.

```typescript
// 저장값이 무슨 형식인지 모르니, 표시 전에 매번 형식부터 알아내야 한다
function displayPhone(stored: string): string {
  const digits = stored.replace(/[^0-9]/g, '');
  if (digits.length === 11 && digits.startsWith('1')) {
    return formatNorthAmerican(digits.slice(1));
  }
  if (digits.length === 10) {
    return formatNorthAmerican(digits);
  }
  if (digits.length === 7) {
    return digits.slice(0, 3) + '-' + digits.slice(3); // 지역번호 없음...
  }
  return stored; // 모르겠으면 날것 그대로 (못생김)
}

// 검증도 모든 변형을 다 받아줘야 한다
const VALID = /^(\+?1[-.\s]?)?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}(\s*\(내선.*\))?$/;
```

저장이 `+14163458899` 하나로 통일됐으니, 검증은 패턴 하나, 표시는 단방향 함수 하나로 끝난다.

```typescript
// 저장 형식이 E.164 하나로 보장되므로 검증은 한 줄
const VALID = /^\+1\d{10}$/;

// 표시: 저장 형식 -> 사람용 형식 단방향 변환
function displayPhone(stored: string): string {
  // stored === '+14163458899'
  const d = stored.slice(2); // 국가코드 제거 -> '4163458899'
  return `(${d.slice(0, 3)}) ${d.slice(3, 6)}-${d.slice(6)}`;
  // -> '(416) 345-8899'
}
```

저장 형식이 통일되니 "이게 무슨 형식이지?"라는 질문 자체가 사라졌다. 코드가 추측을 멈춘다.

</TabsContent>

표시 로직은 한 곳(예: `PhoneNumber` 값 객체나 공유 유틸)에 모아두는 게 좋다. 저장은 정규화 형식, 화면 직전에 `displayPhone()` 한 번. 새 입력 폼은 사용자가 `(416) 345-8899`로 치든 `416-345-8899`로 치든, 쓰기 직전에 `normalize_phone` 상당의 함수로 정규화해 저장한다. **입력은 관대하게 받고, 저장은 엄격하게, 표시는 예쁘게.**

<Callout type="success" title="현대적 보강 — DB가 표시 형식을 대신 계산해줄 수도">
PostgreSQL이라면 표시 형식을 generated column으로 둘 수도 있다. 저장은 정규화 컬럼에, 자주 쓰는 표시 형식은 DB가 파생해 들고 있게 하는 식이다.

```sql
ALTER TABLE Customer
  ADD COLUMN PhoneDisplay TEXT
  GENERATED ALWAYS AS (
    '(' || SUBSTRING(PhoneNumber FROM 3 FOR 3) || ') ' ||
    SUBSTRING(PhoneNumber FROM 6 FOR 3) || '-' ||
    SUBSTRING(PhoneNumber FROM 9 FOR 4)
  ) STORED;
```

다만 표시 규칙이 자주 바뀌거나 로케일을 타면, 표시는 애플리케이션 계층에 두는 게 보통 더 유연하다. 무엇을 DB에 박고 무엇을 코드에 둘지는 "이 규칙이 얼마나 자주 변하나"로 판단해라.
</Callout>

## 정리

Introduce Common Format은 작은 리팩토링이다. 컬럼 하나의 값 모양을 통일할 뿐이다. 그런데 그 작은 통일이 위쪽 코드 전부를 가볍게 만든다.

> **형식이 N가지면 그걸 읽는 모든 코드가 N가지를 알아야 한다. 형식이 하나면 코드도 하나면 된다.**

기억할 건 세 가지다. 첫째, 변환 규칙을 짜기 전에 `SELECT DISTINCT`로 실제 형식 분포부터 본다 — 적의 얼굴을 다 보고 시작해라. 둘째, 정규화하다 내선번호·국가코드 같은 정보를 버리지 마라 — 자동 변환이 안 되는 행은 사람이 본다. 셋째, **저장 형식과 표시 형식을 분리해라** — 저장은 기계가 좋아하는 형식 하나로, 표시는 화면 직전에 예쁘게. 운영 테이블이면 expand-contract로 읽기를 먼저 관대하게 만든 뒤 백필하고, 끝에 `CHECK` 제약으로 못박는다.

컬럼 하나가 15개의 약속을 동시에 지키고 있다면, 그건 사실 아무 약속도 안 지키는 거다. 약속을 하나로 줄이는 게 이 리팩토링이다.
