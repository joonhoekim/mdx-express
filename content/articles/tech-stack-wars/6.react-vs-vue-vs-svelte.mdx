# React vs Vue vs Svelte — 프론트엔드 삼국지

*"내가 쓰는 프레임워크가 최고야" — 세 진영 모두의 공통된 믿음*

---

프론트엔드 개발자에게 "어떤 프레임워크 써?"라고 물어보면
그 답변으로 그 사람의 성격, 커리어, 취향을 대략 파악할 수 있음.

React: "업계 표준이니까요" (현실주의자)
Vue: "개발 경험이 좋아서요" (DX 중시)
Svelte: "컴파일 타임이 미래예요" (혁신 추구)

이 세 프레임워크의 전쟁은 2013년부터 지금까지 이어지고 있고,
매년 새로운 버전, 새로운 기능, 새로운 떡밥으로 전선이 재편됨.

<Callout type="warning" title="Angular는 어디 갔냐">
Angular는 이 글에서 제외함. 왜?
Angular는 "프레임워크"가 아니라 "플랫폼"임. 비교 대상이 다름.
그리고 솔직히 2025년에 새 프로젝트를 Angular로 시작하는 건... 음...
(Angular 유저분들 죄송합니다. 하지만 npm trends가 말해주고 있습니다.)
</Callout>

## 세 프레임워크의 탄생

### React (2013)

```
탄생: 2013년, Facebook (현 Meta)
만든 사람: Jordan Walke
핵심 아이디어: "UI = f(state)" — UI는 상태의 함수

React의 혁명:
1. Virtual DOM — DOM 조작 추상화
2. JSX — JavaScript 안에 HTML을 쓴다
3. Component — UI를 재사용 가능한 조각으로 나눈다
4. One-way data flow — 데이터는 위에서 아래로

React가 나왔을 때의 반응:
"JavaScript 안에 HTML? 이건 미친 짓임" (2013년)
"오 이거 괜찮은데?" (2014년)
"React 안 쓰면 취업 안 됨" (2016년~)
```

### Vue (2014)

```
탄생: 2014년, Evan You (개인 프로젝트)
전직: Google Angular 팀
핵심 아이디어: "Angular의 좋은 점만 가져오고, 나쁜 점은 버리자"

Vue의 매력:
1. Template syntax — HTML 기반이라 친숙
2. Reactivity — 자동 반응성 (getter/setter, 후에 Proxy)
3. SFC — Single File Component (.vue 파일)
4. Progressive — 점진적으로 도입 가능

Evan You의 전설:
Angular 팀에서 일하다가 "이거 너무 복잡한데?"라는 생각에
퇴사하고 혼자서 프레임워크를 만듦.
그리고 그게 GitHub Star 20만 개를 넘김.
개인 프로젝트의 끝판왕.
```

### Svelte (2016)

```
탄생: 2016년, Rich Harris (현 Vercel)
전직: New York Times 시각화 개발자
핵심 아이디어: "프레임워크는 빌드 타임에 사라져야 한다"

Svelte의 혁명:
1. No Virtual DOM — 런타임 오버헤드 제거
2. Compiler — 프레임워크가 아니라 컴파일러
3. Less boilerplate — 코드가 짧음
4. Built-in animations — 애니메이션 내장

Rich Harris의 명언:
"Virtual DOM은 overhead다. 순수한 overhead."
이 한마디로 React와 Vue를 동시에 도발함.
```

## 같은 컴포넌트, 3가지 방식

### 카운터 (기본 중의 기본)

#### React

```tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(count - 1)}>-1</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}
```

#### Vue (Composition API)

```vue
<script setup lang="ts">
import { ref } from 'vue';

const count = ref(0);
</script>

<template>
  <div>
    <p>Count: {{ count }}</p>
    <button @click="count++">+1</button>
    <button @click="count--">-1</button>
    <button @click="count = 0">Reset</button>
  </div>
</template>
```

#### Svelte

```svelte
<script lang="ts">
  let count = 0;
</script>

<div>
  <p>Count: {count}</p>
  <button on:click={() => count++}>+1</button>
  <button on:click={() => count--}>-1</button>
  <button on:click={() => count = 0}>Reset</button>
</div>
```

<Callout type="note" title="코드량 비교">
같은 기능을 구현하는 데 필요한 코드:
- React: 13줄 (useState import, 함수 선언, JSX)
- Vue: 14줄 (script setup, template)
- Svelte: 11줄 (그냥 변수 선언하면 됨)

Svelte의 "Just write less code" 철학이 잘 드러남.
하지만 카운터 같은 간단한 예시에서는 차이가 크지 않음.
진짜 차이는 복잡한 컴포넌트에서 나타남.
</Callout>

### Todo 앱 (좀 더 실전적)

#### React

```tsx
import { useState, useCallback } from 'react';

interface Todo {
  id: number;
  text: string;
  completed: boolean;
}

function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [input, setInput] = useState('');

  const addTodo = useCallback(() => {
    if (!input.trim()) return;
    setTodos(prev => [
      ...prev,
      { id: Date.now(), text: input.trim(), completed: false },
    ]);
    setInput('');
  }, [input]);

  const toggleTodo = useCallback((id: number) => {
    setTodos(prev =>
      prev.map(todo =>
        todo.id === id ? { ...todo, completed: !todo.completed } : todo
      )
    );
  }, []);

  const deleteTodo = useCallback((id: number) => {
    setTodos(prev => prev.filter(todo => todo.id !== id));
  }, []);

  const remaining = todos.filter(t => !t.completed).length;

  return (
    <div>
      <h1>Todo ({remaining} remaining)</h1>
      <form onSubmit={(e) => { e.preventDefault(); addTodo(); }}>
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          placeholder="What needs to be done?"
        />
        <button type="submit">Add</button>
      </form>
      <ul>
        {todos.map(todo => (
          <li key={todo.id}>
            <input
              type="checkbox"
              checked={todo.completed}
              onChange={() => toggleTodo(todo.id)}
            />
            <span style={{
              textDecoration: todo.completed ? 'line-through' : 'none',
            }}>
              {todo.text}
            </span>
            <button onClick={() => deleteTodo(todo.id)}>Delete</button>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

#### Vue

```vue
<script setup lang="ts">
import { ref, computed } from 'vue';

interface Todo {
  id: number;
  text: string;
  completed: boolean;
}

const todos = ref<Todo[]>([]);
const input = ref('');

const remaining = computed(() =>
  todos.value.filter(t => !t.completed).length
);

function addTodo() {
  if (!input.value.trim()) return;
  todos.value.push({
    id: Date.now(),
    text: input.value.trim(),
    completed: false,
  });
  input.value = '';
}

function toggleTodo(id: number) {
  const todo = todos.value.find(t => t.id === id);
  if (todo) todo.completed = !todo.completed;
}

function deleteTodo(id: number) {
  todos.value = todos.value.filter(t => t.id !== id);
}
</script>

<template>
  <div>
    <h1>Todo ({{ remaining }} remaining)</h1>
    <form @submit.prevent="addTodo">
      <input v-model="input" placeholder="What needs to be done?" />
      <button type="submit">Add</button>
    </form>
    <ul>
      <li v-for="todo in todos" :key="todo.id">
        <input type="checkbox" v-model="todo.completed" />
        <span :style="{ textDecoration: todo.completed ? 'line-through' : 'none' }">
          {{ todo.text }}
        </span>
        <button @click="deleteTodo(todo.id)">Delete</button>
      </li>
    </ul>
  </div>
</template>
```

#### Svelte

```svelte
<script lang="ts">
  interface Todo {
    id: number;
    text: string;
    completed: boolean;
  }

  let todos: Todo[] = [];
  let input = '';

  $: remaining = todos.filter(t => !t.completed).length;

  function addTodo() {
    if (!input.trim()) return;
    todos = [...todos, { id: Date.now(), text: input.trim(), completed: false }];
    input = '';
  }

  function deleteTodo(id: number) {
    todos = todos.filter(t => t.id !== id);
  }
</script>

<div>
  <h1>Todo ({remaining} remaining)</h1>
  <form on:submit|preventDefault={addTodo}>
    <input bind:value={input} placeholder="What needs to be done?" />
    <button type="submit">Add</button>
  </form>
  <ul>
    {#each todos as todo (todo.id)}
      <li>
        <input type="checkbox" bind:checked={todo.completed} />
        <span style:text-decoration={todo.completed ? 'line-through' : 'none'}>
          {todo.text}
        </span>
        <button on:click={() => deleteTodo(todo.id)}>Delete</button>
      </li>
    {/each}
  </ul>
</div>
```

<Callout type="info" title="코드 비교 분석">
**React**: useState + useCallback + 불변성 패턴 (spread operator)
→ 함수형 프로그래밍에 익숙하면 자연스럽지만, 보일러플레이트가 많음

**Vue**: ref + 직접 mutation 가능 (push, 프로퍼티 변경)
→ "직관적"이라는 평가. 양방향 바인딩(v-model)이 편리함

**Svelte**: 그냥 변수 선언 + `$:` 로 반응성
→ 코드가 가장 짧고, "프레임워크 없이 쓰는 것 같은" 느낌
</Callout>

## 핵심 차이점 심층 비교

### 반응성 (Reactivity)

```tsx
// React: "상태가 바뀌면 컴포넌트 함수를 다시 실행"
function Component() {
  const [count, setCount] = useState(0);
  // setCount 호출 → Component 함수 전체 재실행
  // Virtual DOM diff → 변경된 부분만 실제 DOM 업데이트

  console.log('렌더링!'); // setCount 할 때마다 출력됨

  // 비싼 계산은 useMemo로 캐싱해야 함
  const expensive = useMemo(() => {
    return heavyCalculation(count);
  }, [count]);

  // 함수도 useCallback으로 캐싱해야 함
  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);

  return <button onClick={handleClick}>{expensive}</button>;
}
```

```vue
<!-- Vue: "반응형 데이터가 바뀌면 관련된 부분만 업데이트" -->
<script setup>
import { ref, computed } from 'vue';

const count = ref(0);
// Proxy 기반 반응성: count.value가 바뀌면 관련 템플릿만 업데이트

// computed는 자동 캐싱
const expensive = computed(() => heavyCalculation(count.value));
// count가 바뀔 때만 재계산. useMemo 같은 명시적 의존성 선언 불필요.

function handleClick() {
  count.value++;
  // 이 함수는 재생성되지 않음. useCallback 필요 없음.
}
</script>
```

```svelte
<!-- Svelte: "컴파일러가 반응성 코드를 생성" -->
<script>
let count = 0;
// 컴파일러가 count 변경을 감지하는 코드를 자동 생성
// 런타임 Proxy도 Virtual DOM도 없음

// $: 로 파생 상태 선언
$: expensive = heavyCalculation(count);
// 컴파일러가 count 의존성을 분석하고
// count가 바뀔 때만 expensive를 재계산하는 코드를 생성

function handleClick() {
  count++;
  // Svelte 컴파일러가 이 할당을 감지하고
  // $$invalidate(0, count, count++) 같은 코드를 생성
}
</script>
```

<Callout type="warning" title="React의 re-render 문제">
React에서 가장 많이 발생하는 성능 문제: 불필요한 리렌더링.

```tsx
// 부모가 리렌더되면 자식도 전부 리렌더됨
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <HeavyChild />  {/* count와 무관한데 리렌더됨! */}
    </div>
  );
}

// 해결: React.memo
const HeavyChild = React.memo(function HeavyChild() {
  return <div>무거운 컴포넌트</div>;
});
```

Vue와 Svelte는 이런 문제가 구조적으로 적음.
Vue는 Proxy가 정확히 어떤 데이터가 쓰이는지 추적하고,
Svelte는 컴파일러가 어떤 변수가 어디서 쓰이는지 정적 분석함.

React의 "함수 전체 재실행" 모델은 단순하지만, 최적화가 개발자 책임임.
</Callout>

### 상태 관리

```tsx
// React: 선택지가 너무 많음 (이것도 문제)
// useState, useReducer, useContext,
// Redux, MobX, Zustand, Jotai, Recoil, Valtio...

// Zustand (2025년 가장 인기 있는 상태 관리 라이브러리)
import { create } from 'zustand';

const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  reset: () => set({ count: 0 }),
}));

function Counter() {
  const { count, increment, decrement, reset } = useStore();
  return (
    <div>
      <p>{count}</p>
      <button onClick={increment}>+</button>
      <button onClick={decrement}>-</button>
      <button onClick={reset}>Reset</button>
    </div>
  );
}
```

```vue
<!-- Vue: Pinia (공식 상태 관리) -->
<script setup>
// stores/counter.ts
import { defineStore } from 'pinia';

export const useCounterStore = defineStore('counter', () => {
  const count = ref(0);
  const increment = () => count.value++;
  const decrement = () => count.value--;
  const reset = () => count.value = 0;
  return { count, increment, decrement, reset };
});

// 컴포넌트에서 사용
const store = useCounterStore();
</script>

<template>
  <div>
    <p>{{ store.count }}</p>
    <button @click="store.increment">+</button>
    <button @click="store.decrement">-</button>
    <button @click="store.reset">Reset</button>
  </div>
</template>
```

```svelte
<!-- Svelte: stores (내장) -->
<script>
// stores/counter.ts
import { writable, derived } from 'svelte/store';

export const count = writable(0);
export const doubled = derived(count, $count => $count * 2);

// 컴포넌트에서 사용
import { count, doubled } from './stores/counter';
</script>

<div>
  <p>{$count} (doubled: {$doubled})</p>
  <button on:click={() => $count++}>+</button>
  <button on:click={() => $count--}>-</button>
  <button on:click={() => $count = 0}>Reset</button>
</div>

<!-- $ 접두사로 스토어 자동 구독/해제 -->
<!-- 별도 라이브러리 없이 내장 기능으로 충분 -->
```

## 성능 비교

### 번들 크기

```
프레임워크 런타임 크기 (gzipped, 2025년 기준):

React 19 + ReactDOM:    ~45 KB
Vue 3:                  ~33 KB
Svelte 5:               ~2 KB (컴파일 결과에 포함)

Svelte가 압도적으로 작은 이유:
프레임워크 런타임이 거의 없음.
컴파일 타임에 필요한 코드만 생성하니까.

하지만!
앱이 커지면 Svelte 컴파일 결과가 커지고,
React/Vue는 런타임을 공유하니까 상대적으로 효율적.
→ 작은 앱: Svelte 유리
→ 큰 앱: 차이가 줄어듦
```

### 벤치마크 (js-framework-benchmark)

```
크롬에서 1,000행 테이블 조작 벤치마크 (2024 기준):
(낮을수록 좋음, 단위: ms)

                    Create 1K   Update 10K   Swap rows   Remove row
Svelte 5            41.5        232.1        17.8        14.2
Vue 3               49.2        261.5        20.1        16.8
React 19            55.8        298.3        22.4        18.1
vanilla JS          36.2        198.7        15.1        12.3

관찰:
1. vanilla JS가 당연히 가장 빠름 (프레임워크 오버헤드 없으니까)
2. Svelte가 React/Vue보다 빠름 (Virtual DOM 없으니까)
3. 하지만 차이가 극적이진 않음
4. 실무에서 이 차이가 체감되는 경우는 거의 없음
```

<Callout type="note" title="성능 벤치마크의 함정">
마이크로 벤치마크 결과를 실무 성능으로 직결하면 안 됨.

실무에서 성능 병목은 보통:
- 네트워크 요청 (API 호출)
- 이미지/미디어 로딩
- 비효율적인 데이터 구조/알고리즘
- 불필요한 리렌더링 (React)

프레임워크 런타임 성능 차이: 5~20%
비효율적인 코드 패턴: 100~1000%

프레임워크 바꾸는 것보다 코드 최적화가 100배 효과적임.
</Callout>

## 메타 프레임워크 전쟁

각 프레임워크의 메타 프레임워크 (SSR/SSG 지원):

### Next.js (React)

```typescript
// Next.js — React의 사실상 표준 메타 프레임워크
// Server Components, Server Actions, App Router

// app/users/page.tsx
export default async function UsersPage() {
  // 서버에서 직접 DB 조회 (API 호출 없이!)
  const users = await db.users.findMany();

  return (
    <div>
      <h1>Users</h1>
      {users.map(user => (
        <UserCard key={user.id} user={user} />
      ))}
      <CreateUserForm />
    </div>
  );
}

// Server Action
async function createUser(formData: FormData) {
  'use server';
  const name = formData.get('name') as string;
  await db.users.create({ data: { name } });
  revalidatePath('/users');
}
```

### Nuxt (Vue)

```typescript
// Nuxt — Vue의 메타 프레임워크
// Auto-imports, File-based routing, Server API

// pages/users.vue
<script setup>
// useFetch가 서버/클라이언트 양쪽에서 동작
const { data: users } = await useFetch('/api/users');
</script>

<template>
  <div>
    <h1>Users</h1>
    <UserCard v-for="user in users" :key="user.id" :user="user" />
    <CreateUserForm />
  </div>
</template>

// server/api/users.get.ts — 서버 API
```

## 2024-2025 트렌드

### React: Server Components 혁명

```tsx
// React Server Components (RSC)
// 서버에서 렌더링되고 클라이언트 JS 번들에 포함되지 않는 컴포넌트

// 이건 서버 컴포넌트 (기본값)
async function ServerComponent() {
  const data = await fetch('https://api.example.com/data');
  // 이 코드는 서버에서만 실행됨
  // 클라이언트 번들에 포함 안 됨
  return <div>{data}</div>;
}

// 이건 클라이언트 컴포넌트
'use client';
function ClientComponent() {
  const [count, setCount] = useState(0);
  // 이 코드는 클라이언트에서 실행됨
  return <button onClick={() => setCount(c + 1)}>{count}</button>;
}

// RSC가 중요한 이유:
// 1. 서버에서만 필요한 로직이 클라이언트 번들에 안 들어감
// 2. DB 직접 접근, 파일 시스템 접근 등 서버 기능 사용 가능
// 3. 초기 로딩 속도 개선
```

### Vue: Vapor Mode

```
Vue Vapor Mode (실험적):
- Virtual DOM 없이 컴파일 타임 최적화
- Svelte와 비슷한 접근
- 번들 크기 더 작아짐
- 성능 개선

```

### Svelte 5: Runes

```svelte
<!-- Svelte 5의 Runes — 새로운 반응성 시스템 -->
<script>
  // 기존 Svelte (3, 4)
  let count = 0;            // 반응형
  $: doubled = count * 2;   // 파생 상태

  // Svelte 5 (Runes)
  let count = $state(0);                // 명시적 반응형 선언
  let doubled = $derived(count * 2);    // 명시적 파생 상태

  // 왜 바꿨냐?
  // 1. 암묵적 반응성 → 명시적 반응성 (더 예측 가능)
  // 2. 클래스/함수 안에서도 반응성 사용 가능
  // 3. TypeScript 지원 개선
</script>
```

<Callout type="info" title="프레임워크의 수렴">
재미있는 현상: 세 프레임워크가 서로를 닮아가고 있음.

- React가 Server Components로 서버 렌더링을 강화 (Vue/Svelte 이미 있던 것)
- Vue가 Vapor Mode로 컴파일 최적화 (Svelte의 접근)
- Svelte가 Runes로 명시적 반응성 (React의 useState 같은)
- 모두가 Signal 패턴을 도입하거나 검토 중

결국 "좋은 아이디어"는 어디서든 채택됨.
경쟁이 있기 때문에 모든 프레임워크가 발전하는 것임.
</Callout>

## 선택 가이드

```
React를 선택해야 할 때:
✅ 취업 / 구인이 중요한 경우
✅ 대규모 팀, 대규모 프로젝트
✅ 생태계와 서드파티 라이브러리가 중요한 경우
✅ React Native로 모바일도 할 예정
✅ Next.js를 쓰고 싶은 경우

Vue를 선택해야 할 때:
✅ 빠른 프로토타이핑
✅ HTML 기반 템플릿을 선호
✅ 작~중간 규모 프로젝트
✅ 팀의 학습 곡선을 최소화하고 싶을 때
✅ Nuxt를 쓰고 싶은 경우

Svelte를 선택해야 할 때:
✅ 번들 크기가 극도로 중요한 경우 (임베디드, 위젯)
✅ 최신 기술을 좋아하는 소규모 팀
✅ DX를 최우선으로 할 때
✅ SvelteKit을 쓰고 싶은 경우
✅ 기존 사이트에 인터랙티브 요소를 추가할 때
```

<Callout type="success" title="최종 결론">
이 삼국지의 승자는 **React**임. 숫자적으로.
시장 점유율, 구인 공고, 생태계 규모 모두 압도적.

하지만 "최고의 프레임워크"가 아니라 "가장 인기 있는 프레임워크"임.
QWERTY 키보드가 최고의 키보드 레이아웃이 아닌 것처럼,
시장 점유율이 기술적 우위를 의미하진 않음.

Vue는 DX에서 React를 이기고,
Svelte는 성능과 코드량에서 둘 다 이김.
React는 생태계와 시장에서 이김.

결국 "어떤 가치를 우선하느냐"의 문제임.
