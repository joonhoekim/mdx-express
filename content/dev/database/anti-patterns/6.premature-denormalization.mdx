## 이게 뭔데

성급한 비정규화. 한마디로 **측정도 안 해놓고 "이거 빠를 것 같은데?"라는 직감 하나로 데이터를 복제하거나 미리 계산해서 박아두는 짓**이다. JOIN이 느릴 것 같아서 같은 값을 두 군데 저장하고, 집계가 무거울 것 같아서 합계를 컬럼에 미리 적어두고. 다 좋은 의도다. 문제는 그 "빠를 것 같은데"가 추측이라는 거다.

먼저 정규화가 왜 있는지부터 보자. 정규화는 무슨 학술적 미덕이라서 있는 게 아니다. **갱신 이상(update anomaly)을 막으려고** 존재한다. 같은 사실을 한 군데에만 적어두면, 그 사실이 바뀔 때 한 군데만 고치면 된다. 그런데 같은 값을 세 군데에 복제해두면, 바뀔 때 세 군데를 다 고쳐야 하고, **한 군데라도 빠뜨리는 순간 DB가 서로 모순되는 두 가지 진실을 동시에 들고 있게 된다.** 정규화는 이 모순이 애초에 생길 수 없게 만드는 설계다.

비유를 하나 들어보자. 친구 전화번호를 폰에 저장했는데, 그게 못 미더워서 수첩에도 적고, 책상 메모지에도 적고, 손등에도 적어뒀다고 치자. 평소엔 어디서든 번호를 빨리 찾으니까 좋다. 근데 친구가 번호를 바꾸는 날, 폰만 고치고 수첩·메모지·손등은 깜빡한다. 그다음부터 어느 걸 보고 거느냐에 따라 연결이 되기도 하고 안 되기도 한다. **사본이 늘수록 "맞는 번호"가 뭔지 아무도 확신 못 하게 된다.** 비정규화가 이거다.

<Callout type="warning" title="한 줄 요약">
비정규화 자체가 죄는 아니다. 죄는 "측정 안 하고 미리 했다"는 거다. 빠를 것 같다는 직감으로 데이터를 복제하면, 얻은 게 진짜 있는지도 모른 채 정합성이라는 비싼 값을 먼저 지불한다.
</Callout>

## 시나리오: 이런 적 있을 거임

주문 목록 화면을 만든다. 1편에서 데인 JOIN/N+1 기억이 떠오른다. "주문마다 유저 테이블 JOIN하면 느릴 것 같은데... 아예 `orders`에 `user_name`을 같이 저장해두면 JOIN 없이 끝나잖아?"

그래서 이렇게 한다.

```sql
ALTER TABLE orders ADD COLUMN user_name VARCHAR(100);

-- 주문 생성 시점에 유저 이름을 복사해 박아넣는다
INSERT INTO orders (user_id, user_name, total)
VALUES (42, '김코딩', 35000);
```

목록 쿼리가 깔끔해진다. JOIN이 사라졌다.

```sql
-- JOIN 없이 끝. 만족.
SELECT id, user_name, total FROM orders ORDER BY created_at DESC LIMIT 100;
```

배포하고 흐뭇하게 퇴근한다. 여기까지는 아무 일도 안 일어난다. 함정은 항상 **나중에** 터진다.

몇 달 뒤, 유저가 닉네임을 바꾼다. `users.name`은 '김코딩'에서 '김배포'로 갱신됐는데, `orders.user_name`은 **과거 이름 그대로** '김코딩'이다. CS에 "제 주문내역에 옛날 닉네임이 떠 있어요"라는 문의가 들어온다. 처음엔 한 건이라 그냥 직접 UPDATE 한 방 날리고 넘어간다.

근데 문의가 계속 들어온다. 그제야 부랴부랴 "유저 이름 바뀌면 orders도 전부 따라 업데이트"하는 트리거를 단다.

```sql
CREATE TRIGGER sync_order_user_name
AFTER UPDATE OF name ON users
FOR EACH ROW
EXECUTE FUNCTION update_orders_user_name();
-- 이제 이름 한 번 바꾸면 그 유저의 주문 수천 행이 통째로 갱신된다
```

이제 닉네임 한 번 바꾸는 데 수천 행이 UPDATE된다. 헤비유저는 주문이 수만 건이라 닉 한 번 바꾸면 락이 한참 걸린다. 그리고 코드를 뒤지다가 깨닫는다. **`user_name`만 복제한 게 아니었다.** 배송지 주소도, 상품명도, 카테고리명도 "JOIN 줄이려고" 똑같은 식으로 여기저기 박아놨다. 갱신해야 할 사본이 사방에 흩어져 있고, 어떤 건 트리거가 있고 어떤 건 없다. 어느 게 최신인지 이제 코드 짠 본인도 모른다.

정합성을 깬 건 성능 요구가 아니다. 측정 없이 먼저 움직인 **습관**이다.

- "JOIN은 느릴 거야, 미리 펼쳐두면 좋잖아" — **측정 안 한 추측**으로 최적화. 1편/JOIN 공포증과 정확히 한 뿌리다. 한쪽은 JOIN이 무서워 코드로 합치고, 이쪽은 JOIN이 무서워 미리 복제해 둔다.
- "지금 빠르게 해두면 나중에 편하지" — **미래의 정합성 비용을 장부에 안 적은** 셈. 지금 5분 아끼려고 6개월 뒤의 데이터 모순을 외상으로 긋는다.
- "사본은 그냥 트리거로 맞추면 되지" — 동기화 로직이 **새 버그의 온상**이 된다. 트리거가 도는 조건, 안 도는 조건, 배치가 놓치는 케이스가 전부 새로운 장애 표면이다.
- "다른 서비스도 다 비정규화하던데?" — 그쪽은 **측정하고** 깬 거다. 우리는 측정 안 하고 흉내만 냈다.

## 무슨 일이 벌어지나 (비용)

성급한 비정규화의 청구서는 셋으로 나뉜다.

**1. 갱신 이상(update anomaly).** 이게 본체다. 원본이 바뀌면 사본 **전부**가 따라 바뀌어야 정합성이 유지된다. 사본이 N개면 갱신 지점도 N개고, 그중 하나라도 빠지면 DB는 서로 모순되는 진실을 들고 있게 된다. 더 고약한 건 이게 **조용히** 깨진다는 거다. 에러가 안 난다. 그냥 어떤 화면엔 새 이름, 어떤 화면엔 옛 이름이 뜬다. 한참 뒤 CS 문의로야 발견되고, 그때쯤이면 이미 틀어진 데이터가 한참 쌓여 있다.

**2. 동기화 장치가 새로운 부하원·버그원.** 사본을 맞추려고 단 트리거·배치·이벤트 핸들러는 공짜가 아니다. 트리거는 이름 한 번 바뀔 때 수천 행을 락 걸고 갱신한다. 배치는 주기 사이에 데이터가 틀어진 채로 떠 있는 구간을 만든다. 이벤트는 유실되거나 순서가 뒤집힌다. **정합성을 지키려고 만든 장치가 그 자체로 장애의 원인**이 된다. 안 만들었으면 없었을 코드가, 빠르게 하려다 생겼다.

```text
[원래 의도]   JOIN 한 번 줄이기
      ↓
[실제 결과]   복제 컬럼 + 동기화 트리거 + 트리거 부하 + 정합성 버그 + 디버깅
```

**3. 가정이 틀리면 위험만 떠안고 이득은 0.** 가장 허무한 결말이다. "JOIN이 느릴 것 같아서" 복제했는데, 막상 측정해보니 그 JOIN은 인덱스 잘 타서 0.2ms였다. 이득은 없었고, 정합성 위험만 고스란히 안았다. **측정 안 한 최적화의 절반 이상은 이런 식으로 끝난다.** 빨라지지도 않았는데 데이터만 위태로워진 거다.

<Callout type="error" title="뭐가 문제냐면">
- **정합성이 조용히 깨진다**: 에러가 아니라 "틀린 값이 그냥 떠 있는" 형태라, 슬로우 쿼리 로그에도 예외 추적에도 안 걸린다. CS 문의로 발견된다.
- **고칠 곳이 점점 늘어난다**: 사본이 늘수록 갱신 지점도 늘고, 동기화 코드도 늘고, 그 코드의 버그도 는다. 비정규화는 한 번 시작하면 계속 번진다.
- **이득이 있었는지조차 모른다**: 측정을 안 했으니 비교 대상이 없다. 빨라졌는지 아닌지 영원히 알 수 없는 채로 위험만 진다.
</Callout>

## 정석: 일단 정규화, 측정 후 신중히 깬다

원칙은 단순하다. **기본은 정규화. 비정규화는 실측 병목이 잡힌 다음의 최후 수단이다.**

순서가 중요하다. 정규화된 스키마로 시작해서, 실제로 느린 쿼리가 잡히고, EXPLAIN으로 원인을 확인하고, 인덱스나 쿼리 튜닝 같은 **싼 수단을 먼저 다 써보고**, 그래도 안 되면 그제야 비정규화를 검토한다. 비정규화는 "혹시 몰라서 미리"가 아니라 "측정해보니 여기가 진짜 병목이라서 어쩔 수 없이"여야 한다.

그리고 깰 때는 **반드시 사본 동기화 전략을 함께 설계**한다. 비정규화의 본질적 비용은 "사본을 누가, 언제, 어떻게 맞추느냐"다. 이걸 안 정하고 컬럼부터 추가하면 위 시나리오의 트리거 지옥으로 직행한다. 선택지는 대략 셋이다.

- **머티리얼라이즈드 뷰(Materialized View)** — DB가 정규화 원본으로부터 사본을 만들어 관리한다. 사본 갱신 책임이 애플리케이션이 아니라 DB에 있다는 게 핵심. `REFRESH` 시점만 정하면 된다.
- **이벤트/CDC 기반 동기화** — 원본 변경 이벤트를 흘려보내 사본을 갱신한다. 분산 환경에 맞지만 유실·순서·지연을 직접 책임져야 한다.
- **트리거** — 가장 손쉽지만 가장 위험하다. 위 시나리오처럼 한 번의 변경이 수천 행 갱신으로 번지므로, 사본 행 수가 작을 때만.

어느 쪽이든 공통된 사고방식은 하나다. **사본은 원본의 "캐시"다.** 캐시는 무효화(invalidation) 전략이 있을 때만 캐시고, 없으면 그냥 틀린 데이터다.

<Callout type="info" title="저장할까, 계산할까">
사실 많은 경우, 데이터를 복제해 저장할 게 아니라 **그때그때 계산**하면 된다. 주문 합계, 게시글 댓글 수처럼 원본에서 파생되는 값은 저장해두면 갱신 이상의 위험을 떠안지만, 필요할 때 집계하면 항상 정확하다. "저장 대신 계산"이라는 같은 결의 이야기는 <a href="/docs/dev/database/derived-state-should-be-computed">파생 상태는 계산하라</a> 글에서 더 깊게 다룬다. 비정규화를 검토하기 전에, "이거 그냥 계산하면 되는 값 아닌가?"부터 물어보자.
</Callout>

## 이렇게 하면 됨

핵심은 두 단계다. **(1) 먼저 측정하고, (2) 깨야 한다면 동기화 책임을 DB에 넘긴다.**

**1단계 — 먼저 측정하라.** 비정규화를 떠올렸다는 건 "이 JOIN/집계가 느릴 것 같다"는 가정을 했다는 거다. 그 가정을 코드로 옮기기 **전에** EXPLAIN으로 확인한다.

```sql
-- "이 JOIN 느릴 것 같은데"의 진위를 먼저 확인
EXPLAIN ANALYZE
SELECT o.id, u.name, o.total
FROM orders o
JOIN users u ON u.id = o.user_id
ORDER BY o.created_at DESC
LIMIT 100;
```

여기서 `Seq Scan`이 보이고 실행 시간이 정말 문제라면, **비정규화 이전에 인덱스부터** 본다. 십중팔구 조인 키나 정렬 키 인덱스 하나로 해결된다.

```sql
-- 비정규화보다 먼저 시도할, 훨씬 싼 수단
CREATE INDEX idx_orders_user_id ON orders (user_id);
CREATE INDEX idx_orders_created_at ON orders (created_at DESC);
```

APM(슬로우 쿼리 대시보드, p95 응답시간)으로 **이 쿼리가 실제 트래픽에서 상위 병목인지**도 확인한다. 하루에 열 번 도는 쿼리를 위해 정합성을 거는 건 손해다. 측정의 결론이 "인덱스로 충분", "병목 아님"이면 거기서 끝. 비정규화는 꺼낼 필요도 없다.

**2단계 — 정말 깨야 한다면, 사본 동기화를 DB에 맡긴다.** 측정 결과 정규화로는 도저히 안 되는 진짜 병목이라면, 그때 비정규화한다. 단 맹목적 복제 컬럼이 아니라 머티리얼라이즈드 뷰로.

```sql
-- Before (안티패턴): orders에 user_name을 맹목적으로 복제
--   → 이름 바뀔 때마다 직접 동기화해야 하고, 빠뜨리면 정합성 깨짐
ALTER TABLE orders ADD COLUMN user_name VARCHAR(100);

-- After: 원본은 정규화 그대로 두고, 읽기용 사본은 머티리얼라이즈드 뷰로
CREATE MATERIALIZED VIEW order_list_view AS
SELECT o.id, o.total, o.created_at, u.name AS user_name
FROM orders o
JOIN users u ON u.id = o.user_id;

CREATE UNIQUE INDEX ON order_list_view (id);
```

`orders`와 `users`는 정규화된 상태 그대로다. 원본엔 사본이 없으니 갱신 이상이 원천적으로 안 생긴다. 읽기용 사본은 뷰가 들고 있고, 갱신 책임은 DB에 있다.

```sql
-- 사본 동기화 = REFRESH. 시점만 정하면 된다 (CONCURRENTLY로 읽기 무중단)
REFRESH MATERIALIZED VIEW CONCURRENTLY order_list_view;
```

이름이 바뀌어도 `users.name` 한 곳만 고치면 되고, 뷰는 `REFRESH` 한 번에 정합성을 맞춘다. 트리거로 수천 행을 즉시 갱신하던 모델과 달리, **"언제 동기화할지"를 의도적으로 선택**할 수 있다. 실시간성이 필요 없는 목록 화면이라면 이 트레이드오프가 깔끔하게 맞는다.

<Callout type="note" title="그럼 비정규화는 항상 나쁜가?">
아니다. 대규모 읽기 위주 시스템에서 비정규화는 정당한 무기다. 머티리얼라이즈드 뷰, 읽기 모델(CQRS), 검색 인덱스 전부 의도적 비정규화다. 차이는 딱 하나다. **그들은 측정하고, 동기화 전략을 설계하고, 깼다.** 안티패턴은 비정규화 자체가 아니라 "측정 없이, 동기화 계획 없이, 미리" 깬 것이다. 순서를 지키면 비정규화는 도구고, 순서를 뒤집으면 함정이다.
</Callout>

## 정리

성급한 비정규화는 좋은 의도에서 출발한다. 누구도 데이터를 망치려고 복제하지 않는다. 그냥 "JOIN 느릴 것 같아서", "미리 해두면 편할 것 같아서" 한 발 먼저 움직였을 뿐이다. 하지만 그 한 발이 측정 없는 추측이라면, 얻은 것도 모르는 채 정합성이라는 비싼 값을 먼저 치르게 된다.

> **비정규화는 측정이 끝난 뒤의 트레이드오프지, 시작점이 아니다.**

기본은 정규화다. 느릴 것 같으면 EXPLAIN으로 확인하고, 인덱스 같은 싼 수단을 먼저 쓰고, 그래도 안 되는 진짜 병목이 잡혔을 때만 비정규화한다. 그리고 깰 때는 사본을 누가 언제 어떻게 맞출지를 컬럼 추가보다 먼저 정한다. 이 순서만 지키면 비정규화는 무서운 게 아니다. 순서를 뒤집는 게 무서운 거다.

다음 편에서는 시선을 바깥으로 돌린다. 정규화가 답답하다며 "차라리 NoSQL로 갈아타면 다 해결되지 않을까?" 하는 마음 — 남의 떡이 커 보이는 그 충동을 뜯어본다.
