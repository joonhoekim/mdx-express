# REST vs GraphQL — API 설계 철학의 충돌

*"프론트엔드 개발자가 백엔드 API에 불만을 품으면 GraphQL이 탄생한다"*

---

2015년, Facebook이 GraphQL을 오픈소스로 공개했을 때
개발자 세계는 둘로 나뉘었음.

"드디어 REST의 문제를 해결할 기술이 나왔다!" vs "뭘 해결한다는 거? REST가 뭐가 문제인데?"

이 전쟁은 단순한 기술 선호도를 넘어서 **API 설계 철학의 근본적인 충돌**이었음.
서버가 주도권을 가질 것인가, 클라이언트가 주도권을 가질 것인가.

<Callout type="info" title="배경 지식">
이 글을 100% 이해하려면 REST API를 실무에서 써본 경험이 있으면 좋음.
하지만 "API가 뭔지는 아는데 REST/GraphQL 차이는 잘 모르겠다" 수준이어도
충분히 읽을 수 있게 썼음. 코드 예시가 많으니까.
</Callout>

## REST: 웹의 기본 원칙

### REST의 탄생

REST (Representational State Transfer)는 2000년 Roy Fielding의 박사 논문에서 정의됨.
HTTP 프로토콜의 설계자 중 한 명이 "HTTP를 제대로 쓰는 방법"을 정리한 것임.

REST의 핵심 원칙:

```
1. 리소스 중심: 모든 것은 리소스이고, URL로 식별된다
2. HTTP 메서드 활용: GET, POST, PUT, DELETE
3. 무상태: 각 요청은 독립적 (세션 X)
4. 표현의 분리: 같은 리소스를 JSON, XML 등으로 표현 가능
```

### REST API의 이상적인 모습

```
# 유저 관련 API
GET    /api/users          → 유저 목록
GET    /api/users/123      → 유저 상세
POST   /api/users          → 유저 생성
PUT    /api/users/123      → 유저 수정
DELETE /api/users/123      → 유저 삭제

# 유저의 게시글
GET    /api/users/123/posts        → 유저 123의 게시글 목록
GET    /api/users/123/posts/456    → 특정 게시글
POST   /api/users/123/posts        → 게시글 생성

# 깔끔하고, 직관적이고, 예측 가능함
```

### REST의 현실

<Callout type="error" title="BFF 패턴의 등장">
이 문제를 해결하려고 BFF (Backend For Frontend) 패턴이 등장함.
프론트엔드별로 전용 백엔드 API를 만드는 것.
웹용 BFF, 모바일용 BFF, 어드민용 BFF...

결국 백엔드 개발자가 프론트엔드 요구사항마다 API를 만들어주는 인간 API 공장이 됨.
"이 필드도 추가해주세요" "이거 빼주세요" "이거 합쳐주세요"
이런 요청이 매주 수십 개씩 들어옴.

GraphQL은 이 문제를 해결하려고 태어난 것임.
</Callout>

## GraphQL: 클라이언트가 주도하는 세상

### GraphQL의 탄생

Facebook은 2012년에 모바일 앱을 네이티브로 전환하면서 이 문제에 직면함.
수천 개의 REST 엔드포인트를 관리하는 게 불가능해진 것.

그래서 내부적으로 GraphQL을 만들었고, 2015년에 오픈소스로 공개함.

### GraphQL의 핵심 개념

```graphql
# GraphQL 스키마 정의
type User {
  id: ID!
  name: String!
  email: String!
  avatar: String
  posts(first: Int, after: String): PostConnection!
  followers: [User!]!
  followersCount: Int!
}

type Post {
  id: ID!
  title: String!
  content: String!
  author: User!
  comments: [Comment!]!
  likes: Int!
  createdAt: DateTime!
}

type Comment {
  id: ID!
  content: String!
  author: User!
  createdAt: DateTime!
}

type Query {
  user(id: ID!): User
  users(first: Int, after: String): UserConnection!
  post(id: ID!): Post
  posts(filter: PostFilter): [Post!]!
}

type Mutation {
  createPost(input: CreatePostInput!): Post!
  updatePost(id: ID!, input: UpdatePostInput!): Post!
  deletePost(id: ID!): Boolean!
}
```

```graphql
# 클라이언트가 필요한 것만 요청
# 프로필 카드에 이름과 아바타만 필요하면:

query ProfileCard {
  user(id: "123") {
    name
    avatar
  }
}

# 응답:
{
  "data": {
    "user": {
      "name": "김개발",
      "avatar": "https://..."
    }
  }
}

# 끝. 딱 필요한 것만 옴.
```

```graphql
# 프로필 페이지 전체가 필요하면? 한 번의 요청으로:

query ProfilePage {
  user(id: "123") {
    name
    email
    avatar
    bio
    followersCount
    posts(first: 10) {
      edges {
        node {
          id
          title
          content
          likes
          createdAt
          comments {
            id
            content
            author {
              name
              avatar
            }
          }
        }
      }
    }
    followers {
      name
      avatar
    }
  }
}

# REST에서 5번 요청할 걸 1번으로 해결
```

<Callout type="success" title="GraphQL의 강점">
1. **필요한 것만 가져옴**: Over-fetching 해결
2. **한 번에 가져옴**: Under-fetching 해결
3. **스키마가 문서**: 타입 시스템이 곧 API 문서
4. **프론트엔드 자율성**: 백엔드에 API 추가 요청 안 해도 됨
5. **강타입**: 컴파일 타임에 쿼리 유효성 검사 가능
</Callout>

## 같은 기능, 다른 구현

### 시나리오: 블로그 메인 페이지

최신 게시글 5개를 가져오되, 각 게시글의 작성자 이름과 댓글 수를 포함.

#### REST 방식

```typescript
// REST: 여러 번의 요청이 필요할 수 있음

// 방법 1: 순차 요청 (느림)
async function fetchBlogMain() {
  // 게시글 목록
  const postsRes = await fetch('/api/posts?limit=5&sort=-createdAt');
  const posts = await postsRes.json();

  // 각 게시글의 작성자 정보 (N+1 문제!)
  const postsWithAuthors = await Promise.all(
    posts.map(async (post) => {
      const authorRes = await fetch(`/api/users/${post.authorId}`);
      const author = await authorRes.json();

      const commentsRes = await fetch(`/api/posts/${post.id}/comments?count=true`);
      const commentsData = await commentsRes.json();

      return {
        ...post,
        author: { name: author.name, avatar: author.avatar },
        commentCount: commentsData.total,
      };
    })
  );

  return postsWithAuthors;
}
// 총 요청 수: 1 + 5 + 5 = 11번 ㅋㅋ

// 방법 2: 커스텀 엔드포인트 (현실적)
async function fetchBlogMainV2() {
  const res = await fetch('/api/posts?limit=5&sort=-createdAt&include=author,commentCount');
  return res.json();
}
// 1번 요청, 하지만 백엔드에서 include 파라미터를 구현해야 함
// 프론트: "author.followers도 include에 추가해주세요"
// 백엔드: "...또?"
```

#### GraphQL 방식

```typescript
// GraphQL: 한 번의 요청

const BLOG_MAIN_QUERY = gql`
  query BlogMain {

function BlogMain() {

}

// 프론트엔드 요구사항이 바뀌면?

```

### 시나리오: 유저 설정 변경

#### REST 방식

```typescript
// REST: PUT or PATCH
async function updateUserProfile(userId: string, data: Partial<UserProfile>) {
  const res = await fetch(`/api/users/${userId}`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data),
  });

  if (!res.ok) {
    const error = await res.json();
    throw new Error(error.message);
  }

  return res.json();
}

// 사용
await updateUserProfile('123', {
  name: '새이름',
  bio: '새 소개글',
});

// 간단하고 직관적임. REST의 장점.
```

#### GraphQL 방식

```typescript
// GraphQL: Mutation
const UPDATE_USER = gql`
  mutation UpdateUser($id: ID!, $input: UpdateUserInput!) {
    updateUser(id: $id, input: $input) {
      id
      name
      bio
      updatedAt
    }
  }
`;

function useUpdateUser() {
  const [updateUser, { loading, error }] = useMutation(UPDATE_USER, {
    // 캐시 자동 업데이트
    update(cache, { data: { updateUser } }) {
      cache.modify({
        id: cache.identify(updateUser),
        fields: {
          name: () => updateUser.name,
          bio: () => updateUser.bio,
        },
      });
    },
  });

  return { updateUser, loading, error };
}

// 사용
const { updateUser } = useUpdateUser();
await updateUser({
  variables: { id: '123', input: { name: '새이름', bio: '새 소개글' } },
});

// 더 복잡하지만, 캐시 관리가 자동화됨
```

<Callout type="warning" title="솔직한 비교">
CRUD 같은 단순한 작업에서는 REST가 GraphQL보다 훨씬 간단함.
GraphQL의 진가는 **복잡한 데이터 관계**를 다룰 때 드러남.

"유저 → 게시글 → 댓글 → 댓글 작성자" 같은 중첩 관계를
한 번의 요청으로 가져올 수 있는 건 GraphQL의 압도적 장점임.

하지만 "게시글 하나 CRUD"할 때 GraphQL을 쓰는 건 오버엔지니어링임.
</Callout>

## GraphQL의 그림자

GraphQL이 모든 걸 해결하는 은총알은 아님. 심각한 문제들이 있음.

### 1. N+1 문제 (서버 사이드)

```typescript
// GraphQL 리졸버에서 발생하는 N+1 문제

const resolvers = {
  Query: {
    posts: () => db.posts.findMany({ take: 10 }),
  },
  Post: {
    // 각 게시글마다 author를 개별 조회 → N+1!
    author: (post) => db.users.findUnique({ where: { id: post.authorId } }),
    comments: (post) => db.comments.findMany({ where: { postId: post.id } }),
  },
};

// 게시글 10개 조회 시:
// 1번: SELECT * FROM posts LIMIT 10
// 10번: SELECT * FROM users WHERE id = ? (각 게시글의 author)
// 10번: SELECT * FROM comments WHERE post_id = ? (각 게시글의 comments)
// 총 21번의 DB 쿼리!
```

```typescript
// DataLoader로 해결 (Facebook이 만든 배칭 라이브러리)
import DataLoader from 'dataloader';

const userLoader = new DataLoader(async (userIds: string[]) => {
  const users = await db.users.findMany({
    where: { id: { in: userIds } },
  });
  // 순서 보장
  return userIds.map(id => users.find(u => u.id === id));
});

const resolvers = {
  Post: {
    author: (post) => userLoader.load(post.authorId),
    // DataLoader가 자동으로 배칭해서
    // SELECT * FROM users WHERE id IN (1, 2, 3, ...) 한 번으로 처리
  },
};
```

### 2. 보안 문제: 악의적 쿼리

```graphql
# 클라이언트가 이런 쿼리를 보내면?
query EvilQuery {
  user(id: "1") {
    posts {
      comments {
        author {
          posts {
            comments {
              author {
                posts {
                  comments {
                    # 무한 중첩... 서버 죽음
                  }
                }
              }
            }
          }
        }
      }
    }
  }
}
```

```typescript
// 해결책: 쿼리 깊이 제한
import depthLimit from 'graphql-depth-limit';

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [depthLimit(7)], // 깊이 7 이상 거부
});

// 쿼리 복잡도 제한
import { createComplexityLimitRule } from 'graphql-validation-complexity';

const server = new ApolloServer({
  validationRules: [
    createComplexityLimitRule(1000, {
      scalarCost: 1,
      objectCost: 10,
      listFactor: 20,
    }),
  ],
});
```

### 3. 캐싱의 어려움

```typescript
// REST: HTTP 캐싱이 자연스러움
// GET /api/users/123 → Cache-Control: max-age=3600
// CDN에서 캐시 가능, 브라우저 캐시 가능

// GraphQL: 모든 요청이 POST /graphql
// URL이 같으니까 HTTP 레벨 캐싱이 안 됨
// 쿼리 내용이 다 다르니까 CDN 캐시도 안 됨

// 해결책: 클라이언트 사이드 캐싱
// Apollo Client의 InMemoryCache
const client = new ApolloClient({
  cache: new InMemoryCache({
    typePolicies: {
      User: {
        keyFields: ["id"],
      },
      Post: {
        keyFields: ["id"],
        fields: {
          comments: {
            merge(existing = [], incoming) {
              return [...existing, ...incoming];
            },
          },
        },
      },
    },
  }),
});

// 이거 설정하는 데 시간이 꽤 걸림...
// REST는 그냥 HTTP 캐시 헤더 하나면 되는데...
```

<Callout type="error" title="GraphQL의 학습 곡선">
GraphQL을 "제대로" 쓰려면 배워야 할 것:
- GraphQL 스키마 언어
- 리졸버 패턴
- DataLoader (N+1 해결)
- 쿼리 깊이/복잡도 제한 (보안)
- 클라이언트 캐싱 (Apollo/Relay)
- 에러 핸들링 (부분 응답)
- 구독 (WebSocket)
- Persisted Queries (성능)
- Fragment, Interface, Union 타입

REST는 HTTP만 알면 시작할 수 있는데,
GraphQL은 새로운 세계를 배워야 함. 이 비용은 무시할 수 없음.
</Callout>

## 새로운 도전자들

### tRPC: TypeScript 풀스택의 꿈

```typescript
// tRPC: 타입 안전한 API, 스키마 정의 없이!
// 서버와 클라이언트가 같은 TypeScript 프로젝트에 있을 때 최적

// 서버 (router.ts)
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();

export const appRouter = t.router({
  user: t.router({
    getById: t.procedure
      .input(z.object({ id: z.string() }))
      .query(async ({ input }) => {
        const user = await db.users.findUnique({
          where: { id: input.id },
        });
        return user;
      }),

    update: t.procedure
      .input(z.object({
        id: z.string(),
        name: z.string().optional(),
        email: z.string().email().optional(),
      }))
      .mutation(async ({ input }) => {
        return db.users.update({
          where: { id: input.id },
          data: input,
        });
      }),
  }),

  post: t.router({
    list: t.procedure
      .input(z.object({
        limit: z.number().min(1).max(100).default(10),
        cursor: z.string().optional(),
      }))
      .query(async ({ input }) => {
        return db.posts.findMany({
          take: input.limit + 1,
          cursor: input.cursor ? { id: input.cursor } : undefined,
        });
      }),
  }),
});

export type AppRouter = typeof appRouter;
```

<Callout type="success" title="tRPC의 매력">
tRPC는 "REST의 단순함 + GraphQL의 타입 안전성"을 결합한 것임.

장점:
- 스키마 정의가 필요 없음 (TypeScript 타입이 곧 스키마)
- 런타임 오버헤드가 거의 없음
- 학습 곡선이 매우 완만
- Zod로 런타임 유효성 검사까지

단점:
- TypeScript 풀스택에서만 사용 가능 (다른 언어 클라이언트 X)
- 모노레포가 아니면 타입 공유가 번거로움
- 공개 API에는 부적합 (내부 API 전용)

### gRPC: 마이크로서비스의 선택

```protobuf
// gRPC: Protocol Buffers + HTTP/2
// 구글이 내부적으로 사용하는 RPC 프레임워크

// user.proto
syntax = "proto3";

package user;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc ListUsers(ListUsersRequest) returns (ListUsersResponse);
  rpc UpdateUser(UpdateUserRequest) returns (User);
  rpc StreamUsers(StreamUsersRequest) returns (stream User);
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  string avatar = 4;
  int64 created_at = 5;
}

message GetUserRequest {
  string id = 1;
}

message ListUsersRequest {
  int32 page_size = 1;
  string page_token = 2;
}

message ListUsersResponse {
  repeated User users = 1;
  string next_page_token = 2;
}
```

```typescript
// gRPC 클라이언트 (Node.js)
const client = new UserServiceClient('localhost:50051', grpc.credentials.createInsecure());

const user = await new Promise((resolve, reject) => {
  client.getUser({ id: '123' }, (err, response) => {
    if (err) reject(err);
    else resolve(response);
  });
});

// gRPC의 장점: 바이너리 프로토콜이라 JSON보다 훨씬 빠름
// gRPC의 단점: 브라우저에서 직접 호출 불가 (gRPC-Web 필요)
```

<Callout type="note" title="gRPC의 포지션">
gRPC는 브라우저 → 서버 통신보다는 **서버 → 서버** 통신에 최적화되어 있음.
마이크로서비스 간 통신, 내부 API에서 진가를 발휘함.

성능이 중요한 내부 서비스: gRPC
외부 공개 API: REST or GraphQL
풀스택 TypeScript: tRPC

이렇게 용도에 따라 선택하는 게 현실적인 답임.
</Callout>

## 실무 선택 가이드

### REST를 선택해야 할 때

```
✅ 단순한 CRUD API
✅ 공개 API (third-party 개발자용)
✅ 캐싱이 중요한 경우 (CDN 활용)
✅ 팀에 GraphQL 경험이 없는 경우
✅ 마이크로서비스 간 간단한 통신
✅ 파일 업로드가 많은 경우
✅ 빠르게 MVP를 만들어야 할 때
```

### GraphQL을 선택해야 할 때

```
✅ 복잡한 데이터 관계 (소셜 네트워크, 대시보드 등)
✅ 다양한 클라이언트 (웹, 모바일, TV 등)
✅ 프론트엔드 팀의 자율성이 중요한 경우
✅ 실시간 업데이트가 필요한 경우 (Subscriptions)
✅ 빠른 프론트엔드 개발 이터레이션이 필요한 경우
✅ 팀에 GraphQL 경험이 있는 경우
```

### tRPC를 선택해야 할 때

```
✅ TypeScript 풀스택 프로젝트
✅ 내부 API (공개 API 아님)
✅ 모노레포 구조
✅ 작은~중간 규모 프로젝트
✅ 타입 안전성을 최우선으로 할 때
✅ Next.js + React 스택
```

<Callout type="info" title="혼합 사용도 가능">
실무에서는 하나만 고집할 필요 없음.

- 외부 API: REST (호환성)
- 내부 프론트엔드: tRPC (타입 안전성)
- 복잡한 대시보드: GraphQL (유연한 쿼리)
- 마이크로서비스 간: gRPC (성능)

하나의 프로젝트에서 여러 방식을 혼합하는 것도 합리적인 선택임.
</Callout>

## 2024-2025 트렌드

```
트렌드 1: Server Components + Server Actions
→ React Server Components가 등장하면서
  "서버에서 직접 DB 쿼리하면 API 자체가 필요 없는 거 아닌가?"
  라는 패러다임 변화가 일어나고 있음

트렌드 2: tRPC + Next.js 조합의 부상
→ TypeScript 풀스택에서는 tRPC가 REST와 GraphQL을 대체하는 추세

트렌드 3: GraphQL은 대규모 서비스에서 안정적으로 자리 잡음
→ GitHub, Shopify, Hasura 등이 증명

트렌드 4: REST는 여전히 가장 보편적
→ 공개 API의 90% 이상이 REST
→ 새로운 기술이 나와도 REST는 사라지지 않음

트렌드 5: Edge Computing과 API
→ Cloudflare Workers, Vercel Edge Functions에서는
  가벼운 REST가 유리 (GraphQL 서버는 무거움)
```

<Callout type="success" title="최종 결론">
REST vs GraphQL 전쟁의 진짜 답:

**"뭘 만드냐에 따라 다르다."**

이게 외교적 답변이라고? 아님. 진짜로 그럼.
CRUD 앱에 GraphQL 쓰면 오버엔지니어링이고,
소셜 네트워크에 REST만 쓰면 엔드포인트 지옥에 빠짐.

도구는 목적에 맞게 쓰는 것임. 종교가 아니라.

...라고 말해도 트위터에서는 매주 이 논쟁이 반복됨.
개발자는 그런 존재임.
</Callout>
