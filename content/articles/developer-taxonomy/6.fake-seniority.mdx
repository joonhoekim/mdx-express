## 시니어의 진짜 의미

"시니어 개발자"는 아마 IT 업계에서 가장 오용되는 단어일 것임. 어떤 회사는 3년차면 시니어고, 어떤 회사는 10년차여도 시니어가 아님. 어떤 회사에선 시니어가 코딩만 하고, 어떤 회사에선 팀을 리드함.

<Callout type="info" title="시니어 개발자의 다양한 정의">
**회사 A (스타트업)**: 3년차 이상 + 혼자 기능 개발 가능 = 시니어
**회사 B (대기업)**: 7년차 이상 + 기술 리드 경험 = 시니어
**회사 C (FAANG)**: 연차 무관 + 조직에 대한 영향력 = 시니어 (L5/L6)
**회사 D (SI)**: 5년 지나면 자동으로 시니어 (시간이 해결)

이 혼란 속에서 "시니어인 척"과 "주니어인 척"이 탄생함.
</Callout>

혼란을 정리하는 한 문장이 있음. **시니어는 연차가 아니라 영향력의 범위로 정의됨.** 주니어는 *자기 태스크*에 영향을 미치고(주어진 문제를 해결할 수 있음), 미드레벨은 *팀 내 프로젝트*에 영향을 미치며(문제를 스스로 정의하고 해결함), 시니어는 *팀 전체*에 영향을 미치고(기술 방향을 설정하고 다른 사람을 성장시킴), 스태프+ 는 *조직 전체*에 영향을 미침(여러 팀에 걸친 전략을 수립함). 코딩을 더 잘하는 건 미드레벨까지의 성장이고, 시니어부터는 "팀을 어떻게 더 잘 만드느냐"가 핵심임. 시니어는 "더 어려운 코딩"이 아니라 "더 넓은 영향력"임.

## 시니어인 척하는 주니어

2~3년차 중 일부가 보이는 "시니어 코스프레" 패턴들임. 본인은 시니어라고 생각하지만 주변에서 보기엔... 글쎄.

**패턴 1: 용어 폭격기.** "우리 이거 DDD로 헥사고날 아키텍처 적용해서 CQRS로 이벤트 소싱하면서 사가 패턴으로 분산 트랜잭션 처리하면 되지 않나요?" 팀원이 "...그래서 구체적으로 어떻게 하자는 거예요?"라고 물으면 "아 그건 일단 도메인 레이어부터..."에서 막힘. (DDD는 적용해본 적 없고, 헥사고날은 유튜브 하나 봤고, CQRS는 블로그 글 읽었고, 이벤트 소싱은 단어만 앎.) 진짜 시니어는 이럴 때 이렇게 말함. "그 패턴들 다 좋지만, 우리 사용자 5천 명이고 개발자 3명이에요. 모놀리스로 충분합니다. 복잡한 아키텍처는 복잡한 문제가 생겼을 때 도입해도 늦지 않아요."

<Callout type="warning" title="용어 폭격의 위험">
기술 용어를 많이 아는 것과 실제로 적용할 수 있는 것은 완전히 다름. 오히려 진짜 시니어일수록 **쉬운 말로 설명**함.

"이건 헥사고날 아키텍처의 포트 앤 어댑터 패턴입니다" vs "외부 시스템이 바뀌어도 핵심 로직은 안 바뀌게 하려고 이렇게 나눈 거예요" — 같은 내용인데 두 번째가 더 시니어임.
</Callout>

**패턴 2: 실체 없는 아키텍처.** 사용자 5천 명, 개발자 3명, 월 매출 500만원짜리 모놀리스 서비스에 대고 "마이크로서비스로 가야 합니다!"를 외침. 제안서엔 API Gateway, 서비스 디스커버리, 서비스 메시, Kafka, Redis Cluster, Kubernetes, Prometheus + Grafana + Jaeger, 그리고 마이크로서비스 12개가 적혀 있음. 진짜 시니어: "이거 운영할 사람이 누구죠?" / "DevOps팀이..." / "DevOps팀 없는데요. 우리 3명인데요." / "..." 넷플릭스 아키텍처를 5명짜리 스타트업에 적용하는 건 아키텍처가 아니라 자살 행위임.

**패턴 3: 코드 리뷰 트집러.** `const isActive = user.status === 'active'` 한 줄에 "이거 enum으로 안 뽑나요? 매직 스트링이잖아요. 변수명도 isUserActive가 낫고, 유틸 함수로 빼서 재사용하고, 타입 가드 패턴도 적용하면..." 같은 코멘트를 줄줄이 닮. 정작 중요한 것 — 이 변경이 비즈니스 로직에 맞는지, 에러 케이스를 처리했는지, 성능 영향이 있는지 — 은 안 봄. 진짜 시니어는 나무가 아니라 숲을 봄. "로직은 맞는데, suspended 상태 사용자는 어떻게 처리하죠? 기획서에 없는 것 같은데 PM한테 확인해볼까요?"

**패턴 4: 과도한 추상화.** 이건 코드로 봐야 와닿음. 요구사항은 그냥 "사용자 이름 가져오기" 하나임.

```typescript
// 시니어인 척 주니어 버전: "나중에 확장할 수 있게"
interface IUserNameProvider { getUserName(id: string): Promise<string>; }
interface IUserNameProviderFactory { createProvider(type: string): IUserNameProvider; }

class DatabaseUserNameProvider implements IUserNameProvider { /* ... */ }
class CachedUserNameProvider implements IUserNameProvider { /* ... */ }
class UserNameProviderFactory implements IUserNameProviderFactory {
  createProvider(type: string): IUserNameProvider {
    switch (type) { /* database / cache / ... new 지옥 */ }
  }
}

// 진짜 시니어 버전
async function getUserName(userId: string): Promise<string> {
  const user = await db.users.findById(userId);
  return user.name;
}
```

<Callout type="error" title="과도한 엔지니어링의 비용">
위 예시에서 — 시니어인 척 버전: 파일 5개, 클래스 5개, 인터페이스 3개, 약 80줄. 진짜 시니어 버전: 함수 1개, 3줄. 둘 다 "사용자 이름 가져오기"를 함.

**YAGNI (You Ain't Gonna Need It).** "나중에 확장할 수 있게"는 좋은 의도지만, 지금 필요 없는 추상화는 복잡성만 늘림. 필요해지면 그때 추상화해도 늦지 않음. 미리 만든 확장성은 확장이 아니라 부채임.
</Callout>

이 패턴들을 한 단어로 묶으면 **더닝-크루거 효과**임. 능력이 부족한 사람일수록 자기 능력을 과대평가한다는 그 인지 편향. 아는 게 적을수록 "이 정도면 다 안다"는 착각이 커지고, 진짜로 깊이 알수록 "내가 모르는 게 이렇게 많구나"를 깨달아 오히려 겸손해짐. 용어 폭격도, 마이크로서비스 욕심도, 과잉 추상화도 전부 더닝-크루거 곡선의 첫 번째 봉우리(자칭 "전문가의 자신감")에서 벌어지는 일임.

## 주니어인 척하는 시니어

반대 방향의 문제인데, 의외로 흔함. 경력은 많은데 의도적으로 주니어처럼 행동하는 시니어들.

**패턴 1: 겸손을 가장한 회피.** "이번 아키텍처 어떻게 갈까요?"라는 질문에 "음... 저도 잘 모르겠는데요, 다들 어떻게 생각하세요?" (실제론 10년 경력에 비슷한 프로젝트 5번 해봤음. 답을 알지만 책임지기 싫어서 안 말함.) 그러면 경험 없는 주니어들이 경험 없이 결정을 내림 → 예측 가능한 실패. 진짜 겸손은 "제 경험으론 A가 좋았는데, 다른 의견 있으면 듣고 싶어요"고, 가짜 겸손은 "모르겠어요~"(알면서 안 말함)임.

**패턴 2: 의사결정 미루기.** "DB 뭐 쓸지"를 1주차엔 "좀 더 조사해보고", 3주차엔 "벤치마크 보고", 5주차엔 "팀원 의견 더 모아서", 7주차엔 "다음 주에" 결정하자고 미룸. 결국 9주차에 "일단 MySQL 쓰죠 (시간 없으니까)"로 끝남. 1주차에 내릴 수 있던 결정을 9주 끌어서 가장 안전하지만 최적은 아닌 선택으로 귀결시키고, 그동안 팀원들은 방향 없이 임시 코드만 짬. 시니어의 존재 의미: ???

**패턴 3: 위임이 아니라 방치.** 정상적인 위임은 "이 기능은 A가 맡아주세요. 이 부분은 주의가 필요하고, 막히면 저한테 물어보세요"임. 주니어인 척 시니어의 "위임"은 "이 기능 누가 맡을래요?"(방향 안 잡아줌) → 막혀서 물어보면 "음... 검색해보세요"임. 이건 위임이 아니라 방치고, 주니어가 삽질하는 시간 = 시니어가 가이드 안 해준 비용임.

**패턴 4: 기술 부채 방관.** "이거 리팩토링해야 할 것 같은데요" → "지금은 기능이 우선이라". "테스트 커버리지 10%인데요" → "시간 나면 해요". "이 라이브러리 2년 전 버전인데요" → "동작하는데 건드릴 필요 있나요?" 6개월 후엔 기술 부채 산더미, 개발 속도 반토막, 장애 3배 증가, 그리고 주니어들의 "시니어가 이러면 우리가 뭘 어쩌라고..." 진짜 시니어라면 "이번 스프린트에 부채 해소 20% 할애합시다. 이 3개는 우선순위 높으니 이번에"라고 방향을 잡아줌.

<Callout type="note" title="왜 시니어가 주니어인 척 할까?">
**1. 번아웃**: 오래 리드 하다 지쳐서 책임 회피
**2. 정치적 계산**: 잘못된 결정의 책임을 지기 싫음
**3. 조직 불신**: "어차피 내 의견 반영 안 될 건데"
**4. 이직 준비**: 곧 나갈 건데 깊이 관여하기 싫음
**5. 피터의 법칙**: 시니어까지 승진했지만 실제 능력은 미드레벨

어떤 이유든 결과는 같음 — 팀에 시니어가 있는데 시니어 역할은 아무도 안 하는 공백 상태.
</Callout>

## 진짜 성장은 무엇이 달라지는 것인가

그럼 진짜 시니어가 되려면 뭐가 달라져야 하나. 기술적으로 보면, 차이는 "더 잘한다"가 아니라 "다른 차원의 일을 한다"임.

| | 주니어 | 시니어 |
|---|--------|--------|
| 문제 해결 | 주어진 문제를 풂 | 올바른 문제를 정의하고, 안 풀어도 될 문제를 식별함 |
| 코드 품질 | 동작하는 코드를 씀 | 팀 전체의 품질 기준을 세우고 유지함 |
| 디버깅 | console.log로 추적 | 시스템적으로 원인을 찾고 재발 방지책을 세움 |
| 테스트 | 수동 테스트 | 테스트 전략을 설계하고 무엇을 테스트할지 판단함 |
| 아키텍처 | 프레임워크가 준 구조를 따름 | 비즈니스에 맞는 구조를 설계하고 진화시킴 |
| 소통 | 기술적 질문을 함 | 기술적 결정을 비개발자에게 설명함 |

근데 사실 시니어와 주니어의 가장 큰 차이는 기술이 아니라 **태도**임. 주니어가 "정답이 뭐예요?"라고 물을 때 시니어는 "정답은 없고, 지금 상황에선 이게 최선이에요. 근거는 A·B·C고 리스크는 X·Y예요"라고 답함(불확실성 내성). "A가 B보다 좋다" 대신 "A는 이럴 때, B는 저럴 때 좋은데 우리 상황은 이러니 A가 나을 것 같다"라고 생각함(트레이드오프 사고). 갈등이 나면 "내가 맞아!"가 아니라 "둘 다 일리 있는데 우리 제약을 고려하면 이게 현실적이에요"라고 중재함. 그리고 자기 성장보다 **다른 사람의 성장**을 도와 팀 전체의 역량을 올림.

<Callout type="success" title="시니어가 되는 현실적인 경로">
**1단계 (1~2년차) 자기 태스크 완수**: 주어진 업무를 독립적으로 수행, 기본기 다지기

**2단계 (2~4년차) 팀 기여**: 코드 리뷰에서 의미 있는 피드백, 기술 부채 인식·해소 제안, 주니어 질문에 답변

**3단계 (4~7년차) 팀 영향력**: 기술 방향 제시와 실행, 프로세스/문화 기여, 타 팀과의 기술 소통

**4단계 (7년차+) 조직 영향력**: 여러 팀에 걸친 전략, 시니어/리드 멘토링, 기술 조직 방향 설정

주의: 연차는 참고일 뿐. 2단계에서 10년 있는 사람도, 3년 만에 3단계 가는 사람도 있음.
</Callout>

## 양쪽 다 탈출하는 법

**시니어인 척 주니어**는 *실행*으로 탈출함. 용어만 늘어놓지 말고 사이드 프로젝트에서 직접 적용해서 이론과 실전의 갭을 체감하고, 실패 경험을 쌓고("마이크로서비스 도입했다가 운영 못 해서 모놀리스로 돌아간 경험" 같은 게 진짜 시니어를 만듦), "왜?"를 5번 물어보는 것. "마이크로서비스가 좋다" → "왜?" → "확장이 쉬우니까" → "왜 확장이 필요한데?" → "사용자가 늘 수 있으니까" → "지금 몇 명인데?" → "5천 명..." → 5번째 "왜?"에서 진실이 나옴.

**주니어인 척 시니어**는 *목소리*로 탈출함. "음... 잘 모르겠는데요" 대신 "제 경험으론 A를 추천합니다. 이유는..."이라고 의견을 먼저 내는 것. 틀려도 됨 — 시니어가 틀리는 건 자연스럽고, 의견 없는 시니어가 더 문제임. 그리고 "내가 추천한 거니 문제 생기면 내가 해결하겠다"고 책임을 받아들이는 것. 이 한마디가 팀에 주는 안도감이 엄청남. 번아웃이면 숨기지 말고 "요즘 지쳐서 리드를 잠시 내려놓고 싶다"고 솔직히 말하는 것도 필요함.

## 마무리: 레벨은 이름표가 아니라 행동이다

정리하면 — **시니어인 척 주니어**는 용어는 많지만 실행 경험이 없고, 과한 아키텍처를 제안하지만 운영 경험이 없고, 코드 리뷰에서 나무만 봄. 처방은 실행하고, 실패하고, 배우고, 겸손해지는 것. **주니어인 척 시니어**는 경험은 있지만 책임을 회피하고, 의견을 숨기고, 방향을 안 줌. 처방은 목소리 내고, 책임지고, 팀을 이끄는 것. **진짜 시니어**는 화려한 게 아니라 적절한 기술을 고르고, 방향을 정하고 책임지고, 다른 사람을 성장시키고, 모르는 건 모른다 하고 아는 건 공유하고, 실패를 두려워하지 않되 실패에서 배움.

<Callout type="note" title="가장 중요한 것">
시니어인 척하지도, 주니어인 척하지도 말고, **지금 자기 레벨에서 최선을 다하면** 됨.

성장은 직선이 아니라 계단형임. 한동안 정체된 것 같다가 갑자기 한 칸 올라가는 순간이 옴. 그 순간은 "남들이 시니어라고 불러줄 때"가 아니라 "내가 팀에 진짜 도움이 되고 있다고 느낄 때" 옴.

타이틀은 협상의 결과고, 실력은 행동의 결과임. 둘 다 중요하지만 순서가 있음. 실력 먼저, 타이틀은 따라옴.
</Callout>
