## 10x 엔지니어라는 전설

개발 업계에서 "10x 엔지니어"는 거의 신화적인 존재임. 무협지로 치면 절대고수, 게임으로 치면 만렙 히든 보스. 코드 한 줄로 회사를 구한다는 그 사람들 말임.

이 개념의 뿌리는 의외로 오래됐음. 1968년 Sackman, Erikson, Grant의 연구에서 "프로그래머 간 생산성 차이가 최대 10:1"이라는 숫자가 나왔거든. 문제는 이 연구가 좀... 허술했다는 거임.

<Callout type="warning" title="연구의 함정">
원래 연구는 경력 차이, 도구 차이, 문제 난이도 차이를 제대로 통제하지 않았음. F1 경주차랑 경운기를 같은 트랙에 올려놓고 "와 10배 차이 남!" 하는 거랑 비슷한 상황. 그런데 이 숫자가 실리콘밸리 VC들 입에 오르면서 하나의 종교가 되어버림.
</Callout>

### 채용 공고 번역기

2010년대 실리콘밸리 채용 공고에는 이런 단어들이 유행했는데, 솔직하게 번역하면 이런 뜻임.

- **"10x 엔지니어를 찾습니다"** → 1명 월급으로 10명분 일 시키겠다는 뜻
- **"락스타 개발자 구함"** → 새벽 3시까지 일할 사람 구한다는 뜻
- **"닌자 프로그래머 환영"** → 아무도 모르게 배포하고 사라질 사람 구한다는 뜻

2019년엔 한 VC가 "10x 엔지니어의 특징"이라며 트윗을 올렸다가 대논란이 났음. 그 특징이라는 게 *어두운 방에서 혼자 코딩하고, 회의 안 가고, 코드 리뷰 안 하고, 문서 안 쓴다*는 거였거든. 이건 10x 엔지니어가 아니라 그냥 "같이 일하기 싫은 사람"의 정의인데, 놀랍게도 일부는 이걸 진지하게 받아들였음.

## 진짜 10x는 뭐가 다른가

핵심부터 말하면, 코드를 10배 많이 치는 게 아님. 진짜 차이는 **의사결정**에서 나옴.

**첫째, 문제를 고르는 눈이 다름.** 0.1x는 버튼 색깔 하나를 3일 동안 고민하면서 명도·채도·대비를 계산하는 함수를 짠다. 10x는 디자이너한테 물어보고 1분 만에 끝낸다. 10x의 진짜 능력은 "이건 안 해도 된다"를 빠르게 판단하는 거임. 가장 좋은 코드는 작성되지 않은 코드라는 말이 괜히 있는 게 아님.

**둘째, 디버깅 방식이 다름.** 0.1x의 디버깅은 대충 이런 흐름임. `console.log`를 200개 박는다 → 랜덤으로 코드를 수정한다 → 어쩌다 되면 커밋한다(왜 되는지는 모름) → 3개월 후 같은 자리에서 다시 터진다. 반면 10x는 일단 **에러 메시지를 실제로 읽는다**(혁명적이다). 스택 트레이스를 따라가고, 가설을 세우고, 검증하고, 근본 원인을 고친 다음, 재발 방지 테스트까지 추가한다.

**셋째, 코드 리뷰 태도가 다름.** 0.1x의 PR은 제목 "fix stuff", 설명 공란, 변경 파일 47개, 리뷰어 없음(셀프 머지)임. 10x는 PR을 올리기 전에 스스로 한 번 리뷰하고, 설명에 **왜(WHY)** 이 변경이 필요한지를 적는다. "결제 처리의 레이스 컨디션을 막음 — checkInventory와 deductInventory 사이에 락이 없어서 동시 결제 시 재고가 음수가 됨" 같은 식으로. 리뷰어가 코드만 보고도 맥락을 이해할 수 있게.

<Callout type="info" title="핵심 인사이트">
10x 엔지니어의 진짜 능력은 "이걸 안 해도 된다"는 판단을 빠르게 내리는 것임. 코드를 많이 짜는 게 아니라, 짜지 않아도 되는 코드를 식별하는 능력. 다시 말하지만 — 가장 좋은 코드는 작성되지 않은 코드임.
</Callout>

## 0.1x 엔지니어의 생태학

0.1x 엔지니어는 단순히 느린 게 아님. **마이너스 생산성**을 가진 존재임. 즉, 이 사람이 안 왔으면 프로젝트가 오히려 빨랐을 거라는 뜻. 이게 가능하냐고? 가능함. 아주 잘.

<Callout type="error" title="위험 신호: 0.1x 패턴">
다음 중 3개 이상 해당되면 0.1x 위험군임:
- 코드 리뷰 코멘트를 resolve만 누르고 실제로 수정 안 함
- 테스트를 "나중에 추가할게요"라고 하고 절대 안 추가함
- 에러가 나면 try-catch로 감싸서 조용히 무시함
- PR에 "사소한 변경"이라고 쓰고 500줄 바꿈
- "내 로컬에서는 되는데요"가 입버릇
</Callout>

대표적인 패턴 몇 개를 보면:

**에러 삼키기.** "에러가 안 보이면 에러가 아니다"가 0.1x의 철학임. 결제 함수에서 예외가 나면 `catch`로 잡아서 `console.log("뭔가 에러남")` 찍고 `null` 리턴. 그 `null`은 다음 사람을 NPE로 고생시키는 지뢰가 됨. 3개월 후 CS팀이 "결제는 됐다는데 주문이 안 들어와요"라고 하면 0.1x는 당당하게 말함. "제 코드엔 에러 없는데요?" 기술적으로 맞는 말이긴 함. 에러를 삼켜버렸으니까.

**복붙 개발.** DB 연결 코드를 10개 서비스 파일에 그대로 복사함. 하드코딩된 비밀번호까지 사이좋게 10번 복붙됨. DB 비밀번호 바꿀 일이 생기면? "grep으로 찾으면 되잖아요" 하다가 하나 빠뜨려서 프로덕션 장애.

**테스트 회피의 예술.** `expect(true).toBe(true)`처럼 항상 통과하는 테스트를 짜놓거나, 스냅샷 테스트로 떡칠해놓고 뭐가 바뀌든 `--updateSnapshot` 치거나, 아예 빈 테스트 파일만 만들어놓고 "테스트 구조는 잡아놨습니다"라고 함.

## 같은 요구사항, 두 가지 결과

말로만 하면 와닿지 않으니 코드로 한 번 보자. **요구사항: 사용자 목록을 가져와서 활성 사용자만 필터링하고 이름순으로 정렬.** 진짜 코드가 의미 있는 대목이니 여기는 코드로 보는 게 맞음.

```typescript
// 0.1x 버전: any 남발, 에러 무시, == 사용, 직접 짠 버블 소트
async function getActiveUsers() {
  let users: any[] = [];
  try {
    users = await (await fetch('/api/users')).json();
  } catch (e) { /* 에러? 무시ㅋ */ }

  const active = users.filter((u) => u.status == 'active'); // == 등장
  // ...그리고 정렬은 굳이 버블 소트로 30줄 더 (왜?)
  return active;
}
```

```typescript
// 10x 버전: 타입 명시, 에러는 던지고, 내장 메서드 활용
interface User { id: string; name: string; status: 'active' | 'inactive' | 'suspended'; }

async function getActiveUsers(): Promise<User[]> {
  const res = await fetch('/api/users');
  if (!res.ok) throw new ApiError(`Failed: ${res.status}`, { endpoint: '/api/users' });

  const users: User[] = await res.json();
  return users
    .filter((u) => u.status === 'active')
    .sort((a, b) => a.name.localeCompare(b.name, 'ko')); // 한글 정렬까지 한 줄
}
```

<Callout type="success" title="차이점 정리">
10x 버전이 나은 이유: ① 타입 안전성(any 대신 명확한 인터페이스), ② 에러를 삼키지 않고 의미 있게 던짐, ③ 내장 메서드 활용으로 간결함, ④ `localeCompare`로 한글 정렬까지 처리, ⑤ 의도가 코드에 그대로 드러남.

그리고 10x 버전이 더 **짧음**. 라인 수가 적다고 게으른 게 아님. 오히려 그 반대임.
</Callout>

## 진짜 차이는 팀에서 드러남

개인의 코딩 실력보다 더 큰 차이를 만드는 건 **팀에 미치는 영향**임.

10x는 공통 API 클라이언트 같은 걸 하나 잘 만들어두고 팀 전체의 작업 시간을 절약시킴. "이거 쓰니까 API 연동이 10분이면 끝나요"라는 말이 나오면 그게 진짜 10x임. 본인이 10배가 아니라 **팀 전체를 10배로** 만드는 사람.

반대로 0.1x가 머지한 코드는 다른 팀원 3명이 각각 반나절씩 디버깅하게 만듦. `doStuff(data, flag, type, extra)` 같은 함수를 200줄짜리 if-else 스파게티로 짜놓으면, 6개월 후 누군가는 이렇게 중얼거림. "doStuff가 대체 뭔 stuff를 do하는 건데..." 결국 새로 짬. **0.1x가 쓴 시간 + 다시 짜는 시간 = 마이너스 생산성.**

## 자가 진단: 나는 어디쯤인가

<Callout type="note" title="냉정한 자가 진단">
10x 엔지니어라고 자칭하는 사람 중 진짜 10x인 비율: 약 0.1%. 나머지 99.9%는 그냥 "혼자 일하고 싶은 사람"임.

진짜 10x는 자기가 10x라고 말하지 않음. 자기가 10x라고 떠들고 다니면 그건 거의 확실히 0.1x임.
</Callout>

그리고 사실, 대부분의 개발자는 10x도 0.1x도 아님. 그냥 1x임. 그리고 그게 전혀 나쁜 게 아님.

```
생산성 스펙트럼:

0.1x ──────── 0.5x ──────── 1x ──────── 3x ──────── 10x
  │             │             │            │            │
 마이너스     배우는 중     평균 기여    팀 레버리지    전설
 생산성        성장 중                              (거의 없음)

대부분의 개발자는 여기 어딘가: [════ 0.7x ~ 2x ════]
그리고 그건 완전히 정상임.
```

10x가 되는 건 솔직히 비현실적인 목표임. 하지만 0.1x에서 벗어나는 건 누구나 가능함. 방법은 별거 없음. **에러를 삼키지 말고**(`catch {}`로 비워두지 말고 로깅하거나 다시 던지기), **`any`를 쓰지 말고**(타입을 명시하기), **코드 리뷰 코멘트를 진짜로 반영하기**(이해 안 되면 "이거 왜 이렇게 바꿔야 하나요?"라고 물어보기 — 그게 학습이고 성장임).

<Callout type="success" title="핵심 메시지">
10x 엔지니어가 되려고 하지 마셈. 그건 마케팅 용어임.

대신 이걸 목표로 해:
1. **팀에 마이너스가 되지 않기** (0.1x 탈출)
2. **다른 사람의 시간을 존중하기** (깨끗한 코드, 좋은 PR)
3. **배우는 걸 멈추지 않기** (어제의 나보다 나은 오늘의 나)

이것만 해도 상위 20%임. 진짜임.
</Callout>

## 10x 신화의 해체

결국 10x 엔지니어 개념은 **조직의 문제를 개인의 능력으로 해결하려는 판타지**임.

프레드 브룩스가 1986년에 쓴 그 유명한 에세이 제목이 「은탄환은 없다(No Silver Bullet)」임. 소프트웨어 개발의 본질적 복잡성을 한 방에 날려버리는 마법의 도구 같은 건 없다는 얘기. 10x 엔지니어 신화는 정확히 그 은탄환을 사람 형태로 믿는 거임. "천재 한 명 뽑으면 다 해결됨" — 미안하지만 그런 일은 안 일어남.

생각해보면 10x 엔지니어가 "필요한" 상황은 대부분 이런 거임. 채용을 잘못했거나(→ 채용 프로세스를 고쳐라), 프로세스가 엉망이거나(→ 프로세스를 고쳐라), 기술 부채가 쌓였거나(→ 꾸준히 리팩토링해라), 사람이 부족하거나(→ 더 뽑아라). "10x 한 명 뽑으면 해결됨"은 "로또 당첨되면 해결됨"과 같은 수준의 경영 전략임.

같은 브룩스의 또 다른 법칙도 기억해둘 만함. **"늦은 프로젝트에 인력을 추가하면 더 늦어진다."** 좋은 개발자 한 명이 망가진 시스템을 구원할 수는 없음. 하지만 망가진 시스템이 좋은 개발자를 0.1x로 만들 수는 있음.

<Callout type="warning" title="마지막 경고">
면접에서 "우리는 10x 엔지니어를 찾고 있습니다"라고 하는 회사는, 높은 확률로 인력은 부족한데 돈은 안 쓰고 싶은 회사임. 1명한테 10명분 일을 시키겠다는 완곡어법일 수 있음. 레드 플래그임.
</Callout>
