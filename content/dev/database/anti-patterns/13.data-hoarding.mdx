## 이게 뭔데

데이터 호더. 이름은 좀 무서운데, 증상은 다들 한 번씩 앓아본 거다. **"데이터는 자산이니까 전부, 영원히 보관해야 한다"는 믿음** 하나로, 보존 정책 없이 테이블에 행을 무한정 쌓기만 하는 습관이다.

TV에 가끔 나오는 저장강박 다큐 본 적 있을 거다. 10년 전 영수증, 다 떨어진 신발, 언젠가 쓸지도 모른다며 모은 페트병이 천장까지 쌓여서 방을 못 쓰는 그 집. 본인은 "이거 다 필요한 거예요"라고 하는데, 정작 마지막으로 그걸 꺼내 쓴 게 언제인지는 본인도 모른다. DB 호더도 똑같다. 다만 차이가 있다면, 우리 집 거실은 안 비싸지지만 **DB는 행이 쌓일수록 매달 청구서가 두꺼워지고 쿼리가 느려진다**는 거다.

핵심은 "데이터를 많이 가진 것"이 문제가 아니라는 점이다. 문제는 **버릴 것과 둘 것을 구분하는 설계, 즉 보존 정책이 아예 없다는 것**이다. 들어온 건 무조건 영원히 살아남는다. 청소 안 하는 방에 짐만 들이는 거랑 똑같다.

<Callout type="warning" title="한 줄 요약">
데이터를 안 지우는 건 "신중한 결정"처럼 보이지만, 사실 대부분은 "결정을 안 한 것"이다. 보존 정책 없이 쌓기만 하면, 테이블이 비대해지며 쿼리·백업·비용이 전부 같이 부푼다.
</Callout>

## 시나리오: 이런 적 있을 거임

로그 테이블, 이벤트 테이블, 알림 테이블. 행이 끝없이 쌓인다. 누군가 회의에서 "이거 오래된 거 좀 지울까요?" 하고 운을 띄우면, 회의실에 정적이 흐른다. 그리고 나오는 말은 항상 똑같다. "혹시 나중에 필요할지 모르잖아. 그냥 두자." 다들 고개를 끄덕인다. 아무도 책임지고 싶지 않으니까.

```sql
-- 처음엔 다들 이렇게 시작한다. 순수했다.
CREATE TABLE events (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT,
  type        TEXT,
  payload     JSONB,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 그리고 아무도 DELETE를 쓰지 않는다. INSERT만 영원히.
```

그렇게 2년이 지난다. 어느 날 `events` 테이블이 **5억 행**이 됐다. 인덱스도 같이 비대해져서 메모리(버퍼 캐시)에 다 안 올라간다. 예전엔 0.1초면 끝나던 단순 집계 쿼리가 디스크를 긁어가며 **분 단위**로 느려진다. 대시보드 한 장 띄우는 데 커피를 한 잔 내려야 한다.

```sql
-- 예전엔 순식간이었던 이 쿼리가, 이제 콜드 데이터를 통째로 헤집는다
SELECT type, count(*)
FROM events
WHERE created_at >= now() - interval '7 days'
GROUP BY type;
-- created_at에 인덱스가 있어도, 인덱스 자체가 너무 커져서 캐시 미스가 줄줄이 난다
```

웃긴 건, 정작 **자주 보는 데이터는 최근 7일치뿐**이라는 거다. 나머지 99%는 2년 동안 아무도 안 들여다본 콜드 데이터다. 그런데 이 99%의 콜드가 한 테이블에 같이 섞여 살면서, 핫 데이터의 발목을 잡는다. 같은 디스크, 같은 인덱스, 같은 캐시를 두고 경쟁하는 거다.

여기서 끝이 아니다. 후폭풍이 줄줄이 따라온다.

- **백업이 길어진다.** 5억 행을 매일 백업하니 백업 윈도우가 늘어지고, 복구 시간(RTO)도 같이 늘어난다.
- **마이그레이션이 공포가 된다.** 컬럼 하나 추가하거나 인덱스 하나 거는 DDL이 5억 행을 건드려야 해서, 다들 "그거 건드리면 락 걸려서 서비스 멈춘다"며 손대기를 무서워한다. 테이블이 성역이 된다.
- **스토리지 청구서가 매달 오른다.** 그리고 그 추세는 절대 꺾이지 않는다. 우상향 그래프다.

그러다 누군가 정곡을 찌르는 질문을 한다. **"그 2년 전 데이터, 마지막으로 누가 조회한 게 언제예요?"** ...아무도 모른다. 모니터링도 안 했다. 결국 깨닫는다. 그 데이터는 **필요해서 둔 게 아니라, 지우는 게 무서워서** 둔 거였다는 걸.

테이블을 비대하게 만든 건 데이터의 가치가 아니라, 지우지 못하는 습관이다. 그 습관의 정체를 까보면 대충 이렇다.

- "나중에 필요할지 모르니까 다 보관하자." — **막연한 미래**의 불확실한 이득을 위해, **확실한 현재 비용**을 매달 지불하고 있다.
- "지우는 건 위험하잖아. 잘못 지우면 큰일 나니까 그냥 두자." — 삭제라는 결정의 **책임을 아무도 안 지려는** 회피. 안 지우면 적어도 내 탓은 아니니까.
- "스토리지 싸잖아, 기가에 몇 원인데." — 스토리지 단가만 보고, **느려진 쿼리·길어진 백업·무서워진 마이그레이션이라는 진짜 비용**은 장부에 안 적었다.

이 셋 중 하나라도 입에서 나와본 적 있다면, 뭐 어쩌겠나. 우리 다 한 번쯤 데이터를 쟁여본 호더다.

## 무슨 일이 벌어지나 (비용)

"행 좀 많은 게 뭐 어때서?" 싶을 수 있다. 비용을 갈래별로 뜯어보면 생각보다 안 가볍다.

**1. 테이블·인덱스 비대 → 전부 느려진다.** 테이블이 커지면 같이 커지는 게 인덱스다. 인덱스 트리가 깊고 무거워지면, 메모리에 다 못 올라가서 캐시 미스가 늘고, 인덱스를 타도 디스크를 긁게 된다. 여기에 백업은 데이터양에 비례해 길어지고, DDL 마이그레이션은 테이블 전체를 건드려야 하니 락과 시간이 함께 폭증한다. **데이터양 하나가 쿼리·백업·마이그레이션 세 군데를 동시에 때린다.**

**2. 핫/콜드가 섞여 캐시 효율이 박살난다.** DB의 버퍼 캐시는 한정돼 있다. 핫 데이터(최근 7일)만 있으면 캐시에 쏙 들어가서 디스크를 거의 안 본다. 그런데 콜드 데이터 99%가 같은 테이블·인덱스에 섞여 있으면, 캐시가 콜드와 핫을 구분 못 하고 자리를 나눠 쓴다. 결국 정작 자주 쓰는 핫 데이터마저 캐시에서 밀려난다. **안 보는 데이터가 보는 데이터의 자리를 뺏는 구조.**

**3. 비용은 우상향으로만 간다.** 스토리지 비용은 행이 쌓이는 한 절대 안 줄어든다. 거기에 큰 데이터셋을 다루는 운영 부담(느린 백업, 무서운 마이그레이션, 길어진 복구 시간)까지 더하면, 단가 곱하기 용량으로 계산한 것보다 실제 청구서가 훨씬 두껍다.

**4. 규제 리스크 — 이건 성격이 다르다.** 위 셋은 "느려지고 비싸지는" 성능·비용 문제지만, 이건 **법적 리스크**다. 개인정보가 섞인 데이터(이메일, 결제 내역, 행동 로그)는 많은 규제에서 **목적을 다한 데이터를 보존기한 내에 파기하라**고 요구한다. GDPR, 국내 개인정보보호법 모두 "필요 없어진 개인정보는 지체 없이 파기"가 원칙이다. 즉 **무지성으로 다 쌓아두는 것 자체가 컴플라이언스 위반**이 될 수 있다. 호더 입장에선 자산이라 믿고 모은 게, 법적으로는 시한폭탄을 모은 셈이다.

<Callout type="error" title="뭐가 문제냐면">
- **느려지는데 원인이 분산돼 있다**: 특정 쿼리 하나가 느린 게 아니라, 테이블이 커서 캐시·백업·DDL이 다 같이 느려진다. "이 쿼리 튜닝하면 되겠지"로 안 풀린다.
- **콜드가 핫의 발목을 잡는다**: 안 보는 데이터 99%가 자주 보는 1%의 캐시·인덱스 효율을 갉아먹는다.
- **비용이 단조 증가한다**: 안 지우면 스토리지·운영 비용은 영원히 우상향. 꺾을 방법이 보존 정책밖에 없다.
- **법적 리스크가 쌓인다**: 보존기한 지난 개인정보를 쌓아두는 것 자체가 규제 위반 소지.
</Callout>

## 정석: 보존 정책 + 파티셔닝/아카이빙

해법의 출발점은 의외로 SQL이 아니라 **합의**다. "무엇을, 얼마나, 왜 보관하는가"를 정하는 일.

**1. 보존 정책(retention policy)을 먼저 정의한다.** 코드 짜기 전에 정해야 할 건 이거다. 데이터 종류별로 **얼마나 보관하고, 언제 버릴지**를 명문화한다. 예를 들면 이런 식이다.

| 데이터 종류 | 핫 보관 | 아카이브 | 파기 |
| --- | --- | --- | --- |
| 디버그 로그 | 7일 | 없음 | 7일 후 드롭 |
| 이벤트(분석용) | 90일 | 1년(콜드 스토리지) | 1년 후 파기 |
| 결제 내역 | 영구(운영 DB) | 법정 보존기한 후 이전 | 법정 보존기한 종료 시 파기 |

핵심은 **"언제 버릴지"를 빈칸으로 두지 않는 것**이다. 빈칸으로 두는 순간 그게 곧 "영원히 보관"이고, 호더의 시작이다.

**2. 시간 단위 파티셔닝으로 콜드를 분리한다.** 시간(보통 월·일 단위)으로 테이블을 파티션으로 쪼개면 두 가지가 공짜로 따라온다. 첫째, 최근 데이터만 조회하는 쿼리가 **해당 파티션만 스캔(partition pruning)**해서 빨라진다. 둘째, 오래된 파티션을 통째로 떼어내는 게 **`DROP`이나 `DETACH` 한 방**이라 엄청 싸다. 5억 행에서 `DELETE WHERE created_at < ...` 돌리는 것과 비교가 안 된다. (대량 DELETE는 락 걸고, WAL 잔뜩 쓰고, 죽은 튜플 남겨서 VACUUM까지 부른다. 파티션 드롭은 그냥 메타데이터 한 줄 지우고 끝.)

**3. 아카이빙으로 콜드를 싼 곳으로 옮긴다.** 당장 안 쓰지만 버리긴 애매한 데이터(분석용, 감사용)는 비싼 운영 DB에서 빼서, 저렴한 스토리지(객체 스토리지, 컬럼 포맷 데이터레이크, 별도 아카이브 테이블)로 이전한다. 운영 DB는 핫 데이터만 들고 날렵하게, 콜드는 싼 데서 자게 만든다.

**4. 규제·감사 요건과 보존기한을 맞춘다.** 보존 정책의 숫자는 엔지니어 감으로 정하면 안 된다. 법무·보안과 함께 **법정 보존기한**을 확인해서 맞춘다. "더 길게 보관하면 안전하겠지"는 틀렸다. 보존기한을 **넘겨서** 보관하는 게 오히려 규제 위반이다. 적당히가 아니라 정확히 맞춰야 한다.

<Callout type="info" title="파티셔닝이 만능은 아니다">
파티셔닝은 "시간으로 자르고 시간으로 조회·삭제하는" 패턴에 최적이다. 로그·이벤트·알림이 딱 그렇다. 반대로 파티션 키와 무관한 컬럼으로 자주 조회하면 모든 파티션을 다 뒤지게 돼서 오히려 손해다. 파티셔닝을 도입하기 전에 "내 쿼리가 정말 시간 범위로 거르는가?"를 먼저 확인하자. 안 그러면 파티션만 잔뜩 만들고 이득은 없는 상황이 된다.
</Callout>

## 이렇게 하면 됨

말로만 하면 안 와닿으니 코드로 보자. Before는 우리가 순수했을 때 만든 단일 거대 테이블이다.

```sql
-- Before: 모든 이벤트가 한 테이블에 영원히. INSERT만 있고 DELETE는 없다.
CREATE TABLE events (
  id          BIGSERIAL,
  user_id     BIGINT,
  type        TEXT,
  payload     JSONB,
  created_at  TIMESTAMPTZ DEFAULT now()
);
-- 2년 뒤: 5억 행, 비대한 인덱스, 분 단위 집계, 무서워서 못 건드리는 마이그레이션
```

After는 같은 테이블을 **월 단위로 파티셔닝**한 모습이다. 핫과 콜드가 물리적으로 분리되고, 오래된 건 드롭 한 방으로 정리된다.

```sql
-- After: 월 단위 레인지 파티셔닝
CREATE TABLE events (
  id          BIGSERIAL,
  user_id     BIGINT,
  type        TEXT,
  payload     JSONB,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- 월별 파티션을 미리 만들어 둔다 (보통 잡으로 자동 생성)
CREATE TABLE events_2026_04 PARTITION OF events
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
CREATE TABLE events_2026_05 PARTITION OF events
  FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
CREATE TABLE events_2026_06 PARTITION OF events
  FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');
```

이제 최근 데이터만 보는 쿼리는 해당 월 파티션만 스캔한다. 옵티마이저가 파티션 가지치기를 해주니, 5억 행이 아니라 그 달치만 뒤진다.

```sql
-- 이 쿼리는 events_2026_06 파티션만 본다. 나머지는 거들떠도 안 본다.
SELECT type, count(*)
FROM events
WHERE created_at >= '2026-06-01'
GROUP BY type;
```

그리고 보존 정책을 잡(cron, 스케줄러)으로 자동화한다. 매달 한 번, 보존기한 지난 파티션을 떼어내거나 드롭한다.

```sql
-- 보존 정책 잡: 90일 지난 파티션을 통째로 드롭 (DELETE가 아니라 DROP!)
DROP TABLE IF EXISTS events_2026_01;

-- 버리긴 아까우면 떼어내서(DETACH) 아카이브로 옮기고 운영 DB에서 분리
ALTER TABLE events DETACH PARTITION events_2026_01;
-- 이후 events_2026_01을 객체 스토리지로 export → 운영 DB에서 DROP
```

```typescript
// 매달 1일 도는 보존 정책 잡 (의사코드)
async function enforceRetention() {
  const cutoff = monthsAgo(3);           // 핫 보관 3개월
  const oldPartitions = await listPartitionsBefore("events", cutoff);

  for (const part of oldPartitions) {
    if (mustArchive(part)) {
      await detachPartition("events", part);   // 운영 DB에서 분리
      await exportToColdStorage(part);         // 싼 스토리지로 이전
    }
    await dropTable(part);                      // 그리고 드롭. 끝.
  }
}
```

요점은 5억 행을 한 줄씩 `DELETE`로 지우는 게 아니라, **파티션 단위로 통째로 떼고 드롭**한다는 거다. 락도 거의 안 걸리고, WAL도 안 불고, VACUUM도 안 부른다.

마지막으로, 코드보다 중요한 판단 기준 하나. 데이터를 둘지 말지 고민될 때 스스로에게 이렇게 물어보자.

> **"이거, 필요해서 두는 거야? 무서워서 두는 거야?"**

- **필요해서 두는 것** — "이 데이터는 분기 정산에 쓰고, 법정 보존기한이 5년이라 그때까지 둔다." 근거가 있고 기한이 있다. 이건 보관이 맞다.
- **무서워서 두는 것** — "혹시 모르니까. 언젠가 필요할지도. 지우면 누가 뭐라 할까 봐." 근거도 기한도 없다. 이건 보관이 아니라 **결정의 회피**다.

전자는 두고, 후자는 보존 정책 표에 기한을 적어 넣자. "혹시 모르니까"에 기한을 못 적겠으면, 그게 바로 안 필요하다는 신호다.

## 정리

데이터 호더는 악의로 생기지 않는다. 누구도 "DB를 느리게 만들어야지" 하고 데이터를 쟁이지 않는다. 그냥 "지우기 무서워서", "나중에 필요할까 봐", "스토리지 싸니까" 한 번씩 미뤘을 뿐이고, 그 미룸이 2년 쌓여 5억 행이 됐을 뿐이다. 하지만 원리는 단순하다.

> **보관에도 설계가 필요하다 — 안 지우는 건 결정이 아니라, 결정의 회피다.**

들어오는 데이터마다 "얼마나 두고 언제 버릴지"를 정하면, 테이블은 적정 크기를 유지하고, 핫 데이터는 캐시에 살고, 백업은 짧고, 마이그레이션은 안 무섭고, 청구서는 안 부푼다. 파티셔닝과 아카이빙은 그 정책을 싸게 실행하는 도구일 뿐이고, 진짜 핵심은 **"버릴 것을 버리기로 결정하는 일"**이다.

---

그리고, 이게 시리즈의 마지막 편이다. 잠깐 묵은 짐을 정리하는 김에, 13개를 쌓아온 이 시리즈도 회고 한 스푼 떠보자.

[JOIN 공포증](/docs/dev/database/anti-patterns/1.fear-of-joins)부터 여기까지 13편을 죽 늘어놓고 보면, 안티패턴마다 옷은 다른데 속살은 닮았다. JOIN이 무서워서 코드로 합치고, 데드락이 무서워서 트랜잭션을 피하고, 삭제가 무서워서 데이터를 쌓는다. 인덱스를 막연히 다 걸고, 정규화/비정규화를 감으로 정하고, NULL을 대충 다룬다. 관통하는 한 줄은 이거였다.

> **DB를 울리는 결정은 대개 "측정 없이, 두려움으로, 편의로" 내려진다.**

- **측정 없이** — `EXPLAIN`을 안 보고, 실행 통계를 안 재고, 마지막 조회 시점을 모니터링 안 한 채 감으로 결정한다.
- **두려움으로** — JOIN이, 데드락이, 삭제가 무서워서 엔진이 가장 잘하는 일을 일부러 피해 간다.
- **편의로** — 당장 짜기 편한 쪽으로 가고, 그 비용은 미래의 나(혹은 후임)에게 청구한다.

반대로 말하면, 이 셋만 뒤집으면 안티패턴 대부분이 안 생긴다. **재보고(측정), 엔진을 믿고(두려움 대신), 조금 귀찮아도 정석으로(편의 대신).** 거창한 비법이 아니라 그냥 이거였다.

13편을 끝까지 따라온 당신은, 이제 코드 리뷰에서 N+1을 알아보고, 인덱스를 의심하고, 보존 정책 없는 테이블에 위화감을 느낄 거다. 그거면 충분하다. 안티패턴은 외우는 게 아니라 **냄새를 맡는 것**이고, 냄새를 맡기 시작했다면 이미 반은 고친 거다.

긴 시리즈 함께 와줘서 고맙다. 이제 가서, 당신의 DB를 덜 울려주자.
