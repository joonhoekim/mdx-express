## 방어형 개발자의 탄생

모든 방어형 개발자는 처음부터 방어형이 아니었음. 첫 직장에서 눈을 반짝이며 "뭐든 해볼게요!"라고 했던 그 사람이, 3년 후 "그건 제 담당이 아닌데요"라고 말하게 되는 데는 다 이유가 있음.

<Callout type="info" title="방어형 개발자 생성 조건">
다음 조건이 충족되면 약 2~3년 내에 방어형 개발자가 생성됨:
- 범위가 불명확한 업무 할당 ("이것도 해주세요, 저것도 해주세요")
- 다른 사람의 실수를 떠안아야 하는 경험 반복
- 성과 평가에서 "내 일 + 떠안은 일"이 구분 안 됨
- 야근의 원인이 항상 "남의 일을 대신 해줘서"
- 보상 없는 헌신이 당연시되는 문화
</Callout>

방어형은 보통 4단계를 거쳐 완성됨. **1단계(0~1년차)는 순수한 신입.** "저 뭐든 할 수 있어요!" 하면서 다른 팀 버그도 고쳐주고 문서도 써주고 배포 스크립트도 만들어줌. **2단계(1~2년차)는 번아웃의 시작.** 다른 팀 버그를 고쳐줬는데 내 성과로 안 잡히고, 만들어준 배포 스크립트의 유지보수가 어느새 내 일이 되고, 열심히 쓴 문서는 아무도 안 읽음. "이건 원래 제 일이 아닌데... 근데 해야겠죠?"

**3단계(2~3년차)는 방어 본능의 발현.** "죄송한데 그건 제 담당이 아닙니다." JIRA 티켓 없으면 안 하고, 슬랙 DM으로 부탁하면 "티켓 만들어주세요"라고 응답함. **4단계(3년차+)는 완전체.** "그건 저 아니고요, 백엔드팀한테 물어보세요"가 모든 요청의 디폴트 응답이 됨. 본인 담당 범위를 극도로 좁게 정의하고, 회의에서는 침묵하다가 본인 이슈만 처리함.

## 방어형은 성격이 아니라 구조의 산물

여기서 중요한 포인트. 방어형 개발자는 **개인의 성격 문제가 아니라 조직의 구조적 문제**임. 세 가지 구조가 사람을 방어형으로 만듦.

**첫째, 책임과 권한의 불일치.** 방어형이 되기 전의 개발자는 책임(responsibility)이 산더미임 — 프론트 개발, API 연동, 코드 리뷰, 주니어 멘토링, 배포 관리, 모니터링, 레거시 마이그레이션, 새벽 2시 장애 대응, 다른 팀 API 변경 대응, 기획서에 없는 예외 처리까지. 그런데 권한(authority)은 "음... 코드 스타일 정도?"고, 책무(accountability)는 "위에 나열한 전부 + 다른 팀이 일으킨 장애의 연대 책임"이며, 보상(reward)은 "연봉 3% 인상 + 회식 때 칭찬 한마디"임. 이 불균형이 2년쯤 누적되면, 책임 범위를 "내 JIRA 티켓에 있는 것만"으로 줄이고 보상란에 "마음의 평화"를 적는 방어형이 완성됨.

**둘째, 유능함의 역설.** 일 잘하면 보상이 "고마워요! 이것도 부탁할게요"임. 더 잘하면 "역시! 이것도 좀..."이고, 3명분 일을 하면 "너 없으면 안 돼"(연봉 인상은 없음)다가, 결국 번아웃이 오면 "요즘 좀 기운이 없어 보이네요?"라는 피드백이 돌아옴. 반대로 처음부터 "그건 제 일이 아닌데요"를 시전한 동료는 아무도 추가 업무를 안 줘서 성과 평가 B+를 받음. 3배 일한 사람은 A-. 반 등급 차이. 이걸 보고도 방어형이 안 되면 그게 오히려 비정상임.

<Callout type="warning" title="유능함의 역설">
조직에서 '잘하는 사람'에게 일이 몰리는 건 자연스러운 현상임. 하지만 보상 없이 책임만 늘어나면, 유능한 사람이 무능한 척하거나 방어형이 됨. 이건 개인의 문제가 아니라 조직의 인센티브 구조 문제임.
</Callout>

**셋째, 모호한 경계와 '그레이 존'.** "이거 프론트가 해야 하나요 백엔드가 해야 하나요?"는 대한민국 IT 회사의 영원한 떡밥임. API 응답을 누가 정렬할지, 데이터 검증을 어느 쪽에서 할지, 에러 메시지를 누가 관리할지 — 이런 그레이 존에서 누가 이기든 **진 쪽이 방어형이 됨.** 둘 다 안 하면 보안 취약점이 생기고, 둘 다 하면 "그건 이중 작업인데..."가 됨. 그레이 존 해결에 실패하면 다 같이 방어형이 되어 "제 담당이 아닙니다"의 무한 루프에 빠짐.

## "내 코드만 돌아가면 됨"의 함정

방어형의 극단적 형태는 **내 모듈만 동작하면 끝**이라는 사고방식임.

방어형 프론트엔드 개발자는 API 응답을 `any`로 받음("스펙은 백엔드 담당이니까"). 에러가 나면 빈 배열을 조용히 리턴함("API 에러는 백엔드 담당이니까"). 결과는 타입 안전성 제로, 스펙 변경 시 런타임 에러, 에러가 나도 아무도 모르는 상태. 그리고 익숙한 대사. "내 코드에서는 에러 안 나는데요?" 기술적으로는 맞음. 에러를 삼켰으니까.

PR 전략도 방어적임. 변경을 최소화해서 책임 범위를 줄이고, 테스트는 내 컴포넌트만 하고("통합 테스트는 QA 담당"), PR 설명에 면책 조항을 달아둠("이 PR은 UserCard만 수정합니다. 다른 컴포넌트 이슈는 별도 티켓으로"). 리뷰어가 "이 변경이 X에도 영향 주지 않나요?"라고 하면 "좋은 지적이네요. 별도 티켓으로 만들게요"라고 답함. (그 별도 티켓은 영원히 만들어지지 않음.)

<Callout type="error" title="방어가 만든 장애">
실제 사례: A팀과 B팀 사이 API 연동에 타임존 이슈가 있었음. A팀은 "우리는 UTC로 보냅니다. B팀에서 변환하세요", B팀은 "우리는 KST 기준입니다. A팀에서 변환해서 보내세요". 아무도 변환을 안 해서 모든 예약이 9시간씩 어긋났음. 고객 CS 3,000건이 들어온 후에야 수습됐고, 양쪽 다 "그건 우리 담당이 아니었습니다"라고 함.
</Callout>

이런 일이 왜 반복될까. 멜빈 콘웨이가 1967년에 한 말이 있음. **"시스템을 설계하는 조직은, 그 조직의 의사소통 구조를 그대로 닮은 시스템을 만든다."** 콘웨이의 법칙임. 팀 사이에 대화가 없으면 시스템의 그 경계에서도 데이터가 안 흐름. UTC↔KST 변환이 아무 데서도 안 일어난 건 코드 문제가 아니라 **두 팀이 그 경계에서 대화하지 않았기 때문**임. 방어형 개발자가 만든 벽은 결국 시스템의 벽이 됨.

## 오너십과 방어의 경계

여기서 자연스러운 질문. **그럼 모든 일에 오너십을 가져야 하나?** 답은 "아니"임. 무분별한 오너십은 번아웃의 지름길임. 핵심은 **건강한 경계 설정**임.

방어적 경계와 건강한 경계의 차이는 딱 하나임 — **거기서 대화가 끝나느냐, 이어지느냐.**

> **방어적 경계:** "내 티켓 아니면 거절. 리스크 있으면 거절. 최소한만 함."
>
> **건강한 경계:** "내 담당이면 바로 처리. 내 담당은 아니지만 도울 수 있고 여유가 있으면, 도와주되 담당자를 명확히 지정. 도울 수 없으면 적절한 담당자를 안내."

겉으로는 둘 다 "이건 제 일이 아니에요"로 시작할 수 있음. 하지만 방어형은 거기서 문을 닫고, 건강한 경계는 "이건 OO팀 XX님 담당이에요. 연결해드릴까요?"로 문을 열어둠. 이 한 끗이 전부임.

<Callout type="success" title="건강한 경계 설정 5원칙">
**1. 명확한 소유권 정의**: 담당 범위를 문서화. 모호한 영역은 미리 합의. 그레이 존이 방어형을 만듦

**2. "No"라고 하되 대안 제시**: "제 담당이 아닙니다" 대신 "이건 XX팀 담당인데, 제가 연결해드릴게요"

**3. 도움은 주되 소유권은 넘기지 않기**: "방향만 잡아드릴게요. 구현은 담당 팀에서 해주세요"

**4. 기록 남기기**: 내가 한 추가 작업은 기록으로. 성과 평가 때 근거가 됨

**5. 반복 요청은 프로세스화**: 같은 "그건 누구 담당이죠?"가 반복되면 담당 매트릭스를 만들어 공유
</Callout>

## 조직이 풀어야 할 문제

방어형 개발자는 개인의 문제가 아니라 조직의 증상임. 담당 범위가 불명확하고, 추가 업무에 보상이 없고, 장애 시 개인에게 책임을 전가하고, 소유권이 문서화 안 돼 있고, "야근해서라도" 문화가 있고, 팀 간 협업이 고통스러운 조직 — 이 중 3개 이상이면 방어형을 양산하는 조직임.

처방은 정해져 있음. **RACI 매트릭스**로 누가 책임(R)·승인(A)·자문(C)·통보(I) 대상인지 명확히 하고, **추가 업무에 인센티브**를 줘서 돕는 게 커리어 플러스가 되게 하고, **온콜 로테이션**으로 장애 대응이 1~2명에게 몰리지 않게 하고, **비난 없는 포스트모템**으로 "누구 잘못"이 아니라 "어떤 시스템 문제"를 묻는 문화를 만드는 것.

그리고 가장 실용적인 건 **담당 매트릭스**를 한 장 그려두는 거임.

| 영역 | 담당 팀 | 백업 팀 | 결정권자 |
|------|--------|--------|---------|
| 사용자 인증 | 플랫폼팀 | 백엔드팀 | 플랫폼 TL |
| 결제 처리 | 결제팀 | 백엔드팀 | 결제 TL |
| UI 컴포넌트 | 프론트팀 | - | 프론트 TL |
| API 게이트웨이 | 인프라팀 | 백엔드팀 | CTO |
| 모니터링/알림 | SRE팀 | 인프라팀 | SRE TL |
| API 스펙 변경 | 변경 주체 | 영향 팀 | 합의 |

"그건 제 담당이 아닌데요"를 막는 유일한 방법은 "그건 XX의 담당이다"를 미리 명확히 정의해두는 거임.

## 방어에서 협업으로

개인 차원에서는 단어 하나만 바꾸면 됨. "그건 제 일이 아닌데요" → "그건 XX팀 담당인데, 같이 미팅 잡을까요?" "스펙에 없는 건데요" → "스펙엔 없지만 사용자 경험상 필요해 보여서 PM한테 확인해볼게요." **방어형은 벽을 세우고, 협업형은 다리를 놓음.** 방어형은 "내 일이 아닙니다"에서 끝나지만, 협업형은 "내 일은 아니지만, 이렇게 하면 해결됩니다"까지 감.

팀 차원에서는 **인터페이스 계약**을 도입하면 됨. 프론트와 백엔드 사이에 엔드포인트, 요청/응답 스키마, 에러 코드, 소유자, breaking change 정책을 명시한 계약서를 두는 것. 계약이 있으면 "이건 누구 담당이죠?"는 계약서를 보면 되고, "스펙 바뀐 거 왜 안 알려줬어요?"는 계약의 통지 정책으로 해결되고, "에러 코드가 문서에 없는데요"는 계약 위반이라 수정을 요청할 수 있음. **계약이 없으면 방어만 남고, 계약이 있으면 협업할 근거가 생김.**

<Callout type="note" title="방어형 개발자에게 보내는 메시지">
당신이 방어형이 된 건 당신 잘못이 아닐 수 있음. 시스템이 당신을 그렇게 만들었을 가능성이 높음.

하지만 방어형에 머무르면 당신의 성장도 멈춤. "그건 제 일이 아닙니다"를 100번 말하는 사이에 기술도, 인간관계도, 커리어도 좁아짐.

완벽한 오너십을 가질 필요는 없음. 다만 "다리를 놓는 사람"이 되면 조직에서의 가치가 기술 실력 이상으로 올라감. 그리고 그게 시니어의 진짜 의미이기도 함.
</Callout>

## 마무리: 경계는 전략이다

방어 자체가 나쁜 건 아님. 명확한 경계 없이는 모든 게 무너짐. 하지만 **벽을 세우는 방어**와 **다리를 놓는 경계**는 다름.

벽을 세우는 사람은 "그건 제 담당이 아닙니다"로 대화를 끝냄. 안전하지만 성장하지 못함. 다리를 놓는 사람은 담당자를 연결하고 맥락을 전달하고 필요하면 같이 논의함. 그 결과 더 많은 신뢰를 얻고, 더 많은 맥락을 이해하고, 더 좋은 평가를 받고, 더 빨리 시니어가 됨.

경계는 나를 보호하는 도구지, 나를 가두는 감옥이 되면 안 됨.

*"'그건 제 담당이 아닙니다'는 때로는 건강한 경계이고, 때로는 조직의 실패를 보여주는 증상이다. 그 차이를 구분하는 게 진짜 실력."*
